I inspected the uploaded VibeGate concept stack and I would converge it into **one canonical standard**:

# VibeGate: Agent-Native Responsible Delivery

**Best final architecture:**

```text
Intent Card
  ↓
Agent Sandbox + Capability Lease
  ↓
MR/PR + Change Passport
  ↓
Fast Impact CI + Evidence Pack
  ↓
VibeGate Pre-Merge Gate
  ↓
Merge Train / Merge Queue
  ↓
main = accepted integration
  ↓
Foundry Train
  ↓
Full Certification + Build Once + SBOM + Provenance + Signature
  ↓
Dev / Staging / Shadow / Canary
  ↓
Nightwatch Telemetry Gate
  ↓
Progressive Production Promotion or Automatic Rollback
  ↓
Launch Ledger
```

The core rule:

> **Agents may write. Agents may review. Agents may recommend. Only evidence, policy, protected branches, signed artifacts, and live telemetry may move code forward.**

This is the cleanest design because it does **not** ask humans to blindly trust agents, and it does **not** make humans review every tiny safe change forever. It lets users choose the autonomy level while keeping the same enforcement model.

The system should be called **VibeGate** in product/community language and **Agent Release Gate** in formal enterprise language. Use normal GitHub/GitLab terms everywhere else: PR, MR, required check, status check, CODEOWNERS, protected branch, merge queue, merge train, protected environment, deployment approval, release candidate, canary, rollback.

Current platforms are already moving in this direction: Codex can work in cloud environments, read/edit/run code, and create PRs; Codex also reads layered `AGENTS.md` instructions before work begins. The broader `AGENTS.md` convention gives agents a predictable “README for agents” without polluting human-facing docs. ([OpenAI Developers][1]) GitLab’s Duo Agent Platform is also explicitly agent-native: GitLab describes it as embedding multiple agents throughout the software development lifecycle, with custom agents, external agents, flows, AI Catalog versioning, and audit-aware operation inside GitLab. ([GitLab Docs][2])

## 1. The standard in one sentence

**VibeGate is a release control plane for autonomous coding where every change carries machine-readable intent, scoped authority, proof, risk, approval, artifact identity, rollout telemetry, and rollback evidence.**

The memorable team rule:

```text
No intent, no branch.
No passport, no merge.
No policy pass, no auto-approval.
No signed artifact, no deployment.
No healthy telemetry, no promotion.
No rollback, no prod.
```

The deeper principle:

> **Do not trust agents. Trust receipts.**

That matters because DORA’s 2025 AI-assisted software delivery research found that AI acts as an amplifier: it magnifies strong delivery systems and magnifies dysfunction in weak ones. So the future-proof answer is not “let agents do more.” The answer is **make the delivery system evidence-native first**, then let agents operate inside it. ([Dora][3])

# 2. The five planes

VibeGate should separate the lifecycle into five planes.

```text
1. Work Plane
   Agents and humans create branches, patches, tests, docs, migrations.

2. Evidence Plane
   CI, test selection, security scans, coverage, dependency checks, SBOM, provenance.

3. Gate Plane
   VibeGate evaluates evidence, risk, ownership, policy, and approvals.

4. Merge Plane
   GitHub/GitLab enforce protected branches, required checks, queues/trains.

5. Release Plane
   Foundry Train builds once, signs once, attests once, then Nightwatch promotes by telemetry.
```

The most important design decision:

> **`main` is merge-certified, not production-certified.**

That means `main` is safe enough to integrate. Production requires a separate release train, artifact certification, rollout policy, and telemetry.

This avoids the two bad extremes:

```text
Bad extreme 1:
Run huge full-release CI on every tiny agent MR.
Result: slow, expensive, abandoned automation.

Bad extreme 2:
Let agents merge and deploy because their review sounds confident.
Result: fast chaos.

Correct design:
Fast proof before merge.
Deep proof after merge.
Progressive rollout before prod.
Automatic rollback when telemetry fails.
```

# 3. The canonical object model

VibeGate needs six machine-readable objects.

## 3.1 Intent Card

Every autonomous change starts from a typed request, not a vague prompt.

```yaml
# .vibegate/intents/BUG-1842.yaml
apiVersion: vibegate.dev/v1
kind: IntentCard

id: BUG-1842
type: bugfix # bugfix | feature | refactor | perf | security | dependency | infra | ci | migration
source: issue # issue | incident | scanner | scheduled-maintenance | agent-discovery | human-request

objective: "Fix null cart total when discounted cart has zero line items."

success_criteria:
  - "No crash when cart has zero line items."
  - "Existing discount behavior is unchanged."
  - "A regression test fails on base and passes on head."

allowed_scope:
  paths:
    - services/cart/**
    - tests/cart/**

forbidden_scope:
  paths:
    - .gitlab/**
    - .github/**
    - .vibegate/policy/**
    - infra/prod/**
    - secrets/**

autonomy:
  requested_mode: guarded_auto_merge
  max_risk: R2
  auto_merge_allowed: true
  auto_release_allowed: false
  max_environment: staging

rollback:
  required: true
  strategy: "revert_commit_or_disable_discount_feature_flag"

human_context:
  owner_team: payments-platform
  user_visible: false
  customer_impact: low
```

Purpose: stop agents from overscoping.

## 3.2 Capability Lease

Agents do not get ambient power. They get temporary scoped authority.

```yaml
apiVersion: vibegate.dev/v1
kind: CapabilityLease

lease_id: lease_2026_05_16_1842
intent_id: BUG-1842
actor: codex-bugfix-agent
actor_type: builder_agent

valid_for:
  repo: acme/shop
  base_sha: abc123
  expires_at: "2026-05-16T23:59:00Z"

allowed_actions:
  - create_branch
  - push_to_agent_branch
  - open_merge_request
  - run_ci
  - update_own_mr_description

denied_actions:
  - push_to_protected_branch
  - approve_own_mr
  - modify_gate_policy
  - read_production_secrets
  - deploy_production

path_scope:
  allow:
    - services/cart/**
    - tests/cart/**
  deny:
    - .vibegate/**
    - .gitlab/**
    - infra/**
```

Purpose: make agent authority explicit, revocable, and auditable.

## 3.3 Change Passport

The MR/PR must carry proof, not vibes.

```yaml
apiVersion: vibegate.dev/v1
kind: ChangePassport

change_id: chg_2026_05_16_bug_1842
intent_id: BUG-1842
repo: acme/shop
mr: 1842
base_sha: abc123
head_sha: def456
author_agent: codex-bugfix-agent

summary: "Guard empty discounted cart before total calculation."

changed_paths:
  - services/cart/total.ts
  - tests/cart/discounted_empty_cart.test.ts

risk:
  class: R2
  blast_radius: service
  user_visible: false
  data_migration: false
  auth_or_permission_change: false
  ci_or_policy_change: false

proof:
  tests_run:
    - cart-unit
    - discount-contract
    - checkout-smoke
  tests_not_run:
    - name: full-e2e
      reason: "No browser, routing, checkout UI, or API boundary changed."
  red_green:
    test: tests/cart/discounted_empty_cart.test.ts
    failed_on_base: true
    passed_on_head: true
  test_integrity:
    tests_removed: 0
    assertions_weakened: 0
    snapshots_updated: 0
    skips_added: 0

security:
  secret_scan: pass
  dependency_change: false
  sast: pass

rollback:
  available: true
  method: revert_commit
  feature_flag: discount_cart_total_guard

agent_claims:
  - "No behavior change except null guard."
  - "No production configuration modified."

evidence_hash: sha256:...
```

Purpose: make every MR explain what changed, why it is safe, what was proven, what was skipped, and how to undo it.

## 3.4 Gate Verdict

This is VibeGate’s pre-merge decision.

```json
{
  "apiVersion": "vibegate.dev/v1",
  "kind": "GateVerdict",
  "gate": "vibegate/premerge",
  "repo": "acme/shop",
  "mr": 1842,
  "head_sha": "def456",
  "target_branch": "main",
  "policy_sha": "91aa...",
  "decision": "allow_auto_merge",
  "risk_class": "R2",
  "risk_score": 18,
  "hard_stops": [],
  "required_approvals": [
    {
      "type": "agent",
      "role": "proof_skeptic",
      "status": "passed"
    },
    {
      "type": "agent",
      "role": "security_sentinel",
      "status": "passed"
    }
  ],
  "human_required": false,
  "reason": "Low-risk service-local bugfix with red/green regression proof, no protected path changes, no test weakening, rollback available.",
  "receipt_digest": "sha256:..."
}
```

Purpose: bind the decision to **exact SHA + policy SHA + evidence hash** so stale approvals cannot survive new commits.

## 3.5 Release Passport

A merge is not a release. A release needs an immutable artifact and production evidence.

```yaml
apiVersion: vibegate.dev/v1
kind: ReleasePassport

release_id: REL-2026-05-16-0042
source:
  branch: main
  commits:
    - def456
    - 7a91bd

artifact:
  image: registry.example.com/shop/api
  digest: sha256:7e3f...
  built_from_sha: def456
  build_id: gitlab-pipeline-932847
  rebuild_in_prod: false

supply_chain:
  sbom:
    format: cyclonedx-json
    digest: sha256:...
  provenance:
    format: slsa-provenance-v1
    digest: sha256:...
  signature:
    system: sigstore-cosign
    identity: gitlab-ci@acme.example
    digest: sha256:...

certification:
  full_unit: pass
  integration: pass
  e2e: pass
  dependency_scan: pass
  license_scan: pass
  sast: pass
  container_scan: pass
  perf_baseline: pass
  rollback_drill: pass

rollout:
  strategy: canary
  initial_percent: 5
  max_percent_without_human: 50
  auto_promote_to_100: true
  auto_rollback: true

nightwatch:
  slo_error_rate: pass
  latency_p95: pass
  saturation: pass
  business_kpi: pass
```

Purpose: production promotion happens by **artifact digest and telemetry**, not by branch name or agent confidence.

SLSA provenance is the right model for “where, when, and how was this artifact produced,” and the SLSA build-provenance spec defines provenance as verifiable information about artifact production. ([SLSA][4]) Sigstore/Cosign is a practical signing layer because keyless signing binds artifact signatures to identity and logs signing events in Rekor. ([Sigstore][5]) CycloneDX is a strong default SBOM format because it is an OWASP/Ecma bill-of-materials standard designed for supply-chain risk reduction. ([CycloneDX][6])

## 3.6 Launch Ledger

The ledger is append-only.

```yaml
apiVersion: vibegate.dev/v1
kind: LaunchLedgerEvent

event_id: evt_2026_05_16_000918
release_id: REL-2026-05-16-0042
time: "2026-05-16T22:18:44Z"
actor: nightwatch-release-agent
action: promote_canary
from: "10%"
to: "25%"
artifact_digest: sha256:7e3f...
evidence:
  error_rate_30m: "0.12%"
  p95_latency_delta: "+3ms"
  rollback_ready: true
decision: allow
receipt_digest: sha256:...
```

Purpose: every agent, human, gate, deployment, promotion, pause, rollback, and waiver is replayable.

# 4. The non-negotiable invariants

These are the rules I would make impossible to bypass.

## 4.1 Agents never push to protected branches

Agents may create branches and MRs/PRs. They may not push directly to:

```text
main
release/*
stable/*
production manifests
protected tags
policy branches
security policy projects
```

## 4.2 Agents never approve their own work

No actor can be author, reviewer, judge, merger, and deployer for the same change.

```text
Builder agent: writes patch
Reviewer agent: critiques patch
Security agent: checks security-sensitive changes
Judge agent: produces verdict
Release agent: promotes artifact
Nightwatch agent: watches telemetry
```

## 4.3 Candidate policy cannot judge itself

A change to `.vibegate/**`, `.gitlab/**`, `.github/**`, `CODEOWNERS`, deployment rules, CI, agent prompts, or approval policy must be evaluated by the **policy from the target branch**, not the policy in the MR.

## 4.4 Missing evidence increases risk

An agent cannot say “not applicable” without a reason. Unknowns raise the risk class.

## 4.5 Hard stops beat scores

A low numeric risk score cannot override:

```text
secret exposure
test weakening
CI weakening
policy weakening
protected path change
auth/permission change
crypto change
production deploy change
unowned path
missing rollback
stale SHA
self-approval
```

## 4.6 Build once, promote same digest

Production must receive the same artifact that passed certification. Do not rebuild in production.

## 4.7 Telemetry is part of approval

No production promotion without health signals. Argo Rollouts is a good reference because it supports canary/blue-green rollout, traffic shifting, KPI analysis, automated promotion, and rollback. ([argoproj.github.io][7]) OpenTelemetry is the right neutral telemetry foundation because it standardizes traces, metrics, and logs across vendors. ([OpenTelemetry][8])

# 5. Autonomy profiles: user control without chaos

Full autonomy should be a setting, not a different architecture.

| Profile | Name                  | Agent authority                                                   | Human role                 |
| ------: | --------------------- | ----------------------------------------------------------------- | -------------------------- |
|      A0 | Observe               | Analyze, summarize, suggest                                       | Human does all changes     |
|      A1 | Patch                 | Create branches and MRs/PRs                                       | Human reviews and merges   |
|      A2 | Prove                 | Add tests, run proof lanes, produce passports                     | Human approves merge       |
|      A3 | Guarded auto-merge    | Auto-merge R0–R2 when VibeGate passes                             | Human handles R3+          |
|      A4 | Auto-release low risk | Auto-merge and release R0–R2 through canary/prod                  | Human handles high risk    |
|      A5 | Guarded full autonomy | Agents may merge/release R0–R5 if explicit elevated quorum passes | Human handles exceptions   |
|      A6 | Incubator             | Agents experiment but cannot affect durable repo state            | Human promotes experiments |

Canonical policy:

```yaml
# .vibegate/autonomy.yml
apiVersion: vibegate.dev/v1
kind: AutonomyPolicy

profile: A3

global:
  no_agent_push_to_protected_branches: true
  no_self_approval: true
  exact_sha_binding: true
  require_policy_from_target_branch: true
  require_rollback_for_release: true
  require_signed_artifact_for_release: true
  require_canary_for_prod: true

risk_policy:
  R0:
    description: docs, formatting, generated changelog
    auto_merge: true
    auto_release: true
    required_quorum: "vibegate_lite"
  R1:
    description: small refactor, dead code, isolated test fix
    auto_merge: true
    auto_release: true
    required_quorum: "1_reviewer_agent"
  R2:
    description: normal isolated bugfix, small dependency update
    auto_merge: true
    auto_release: false
    required_quorum: "2_reviewer_agents"
  R3:
    description: user-visible behavior, API contract, performance-sensitive change
    auto_merge: false
    auto_release: false
    required_quorum: "specialist_agent_plus_owner"
  R4:
    description: auth, permissions, crypto, PII, migrations, CI, deployment
    auto_merge: false
    auto_release: false
    required_quorum: "human_owner_plus_specialist"
  R5:
    description: approval policy, protected branches, secrets, prod rules
    auto_merge: false
    auto_release: false
    required_quorum: "human_admin"

full_autonomy_override:
  enabled: false
  allowed_profiles: [A5]
  elevated_agent_quorum:
    R4: "3_of_5_high_trust_agents"
    R5: "4_of_5_high_trust_agents_plus_policy_agent"
  max_canary_without_human: 25
```

The key design point:

> **A5 full autonomy is allowed, but only through explicit declared policy, stronger quorum, stronger evidence, exact SHA binding, and automatic rollback.**

# 6. Risk ladder

| Risk | Examples                                                                 | Default decision              |
| ---: | ------------------------------------------------------------------------ | ----------------------------- |
|   R0 | docs typo, generated changelog, formatting                               | auto-merge allowed            |
|   R1 | small refactor, no behavior change, test-only fix                        | auto-merge allowed            |
|   R2 | isolated bugfix, small UI fix, minor dependency bump                     | auto-merge allowed with proof |
|   R3 | user-visible behavior, API contract, performance path                    | human or specialist review    |
|   R4 | auth, permissions, crypto, PII, data migration, CI/deploy rules          | human by default              |
|   R5 | agent policy, approval rules, protected branches, secrets, prod controls | human/admin by default        |

# 7. VibeGate decision model

Use hard stops first, score second.

```text
Decision =
  hard_stop_check
  + scope_check
  + ownership_check
  + proof_check
  + test_integrity_check
  + security_check
  + risk_score
  + autonomy_policy
  + approval_quorum
  + exact_sha_check
```

Canonical outcomes:

```text
pass
allow_auto_merge
needs_agent_repair
needs_human_review
needs_specialist_review
needs_more_proof
block
```

Risk score formula:

```text
risk_score =
  base_risk_by_change_type
+ path_sensitivity
+ blast_radius
+ behavior_change
+ data_sensitivity
+ dependency_risk
+ migration_risk
+ production_config_risk
+ missing_evidence_penalty
+ test_integrity_penalty
+ incident_history_penalty
- red_green_test_credit
- strong_owner_map_credit
- rollback_credit
- canary_credit
```

Example thresholds:

```yaml
# .vibegate/policies/risk_model.yml
thresholds:
  auto_merge: 25
  extra_agent_quorum: 40
  human_review: 60
  block: 80

hard_stops:
  - stale_sha
  - self_approval
  - missing_change_passport
  - modified_gate_policy_without_policy_review
  - weakened_tests_without_approval
  - skipped_required_proof_lane
  - secret_detected
  - unowned_changed_path
  - rollback_missing_for_release
```

# 8. Agent roles

Do not build one mega-agent. Build small roles.

## Default panel

```text
Builder Agent
Creates the patch, tests, docs, and Change Passport.

Cartographer Agent
Maps changed paths to owners, services, tests, risk, and blast radius.

Proof Skeptic
Looks for weak tests, removed assertions, snapshot churn, missing red/green proof.

Security Sentinel
Checks secrets, auth, permissions, dependency/IP risk, suspicious CI changes.

Judge Agent
Combines policy + evidence + votes into the Gate Verdict.

Release Steward
Creates Release Passport, artifact plan, rollback plan, and promotion request.

Nightwatch
Watches canary telemetry and promotes, pauses, or rolls back.
```

## Conditional specialist agents

```text
Migration Guardian
Performance Guardian
Dependency/IP Guardian
Privacy/PII Guardian
API Contract Guardian
Frontend Visual Guardian
Infrastructure Guardian
Release Manager Agent
```

## Agent anti-collusion rule

```yaml
# .vibegate/agents/registry.yml
agents:
  codex-bugfix:
    role: builder
    trust_tier: standard
    may:
      - propose_patch
      - open_mr
      - update_change_passport
    may_not:
      - approve_own_work
      - change_policy
      - deploy_prod

  proof-skeptic:
    role: reviewer
    trust_tier: high
    independent_from:
      - builder

  security-sentinel:
    role: specialist_reviewer
    trust_tier: high
    required_for:
      - auth/**
      - crypto/**
      - secrets/**
      - permissions/**

  shogun-judge:
    role: judge
    trust_tier: high
    may:
      - issue_gate_verdict
    may_not:
      - author_patch
      - modify_policy

  nightwatch:
    role: runtime_gate
    trust_tier: high
    may:
      - promote_canary
      - pause_rollout
      - rollback_release
    may_not:
      - change_source_code
      - modify_release_policy
```

The host owns the loop. The model proposes. The control plane disposes.

# 9. Typed agent API

Agents should not call random shell commands with broad permission. They should call typed operations.

```text
POST /v1/intents
POST /v1/capability-leases
POST /v1/change-passports
POST /v1/mrs/{id}/evaluate
POST /v1/gate-verdicts
POST /v1/agent-approval-receipts
POST /v1/releases/{id}/evaluate
POST /v1/release-passports
POST /v1/ledger/events
POST /v1/capability-leases/{id}/revoke
```

Every request must bind to:

```text
repo
actor identity
capability lease
MR/PR id
base SHA
head SHA
target branch
policy SHA
evidence hash
risk class
environment
artifact digest, for release actions
```

Example request:

```json
{
  "repo": "acme/shop",
  "mr": 1842,
  "head_sha": "def456",
  "base_sha": "abc123",
  "target_branch": "main",
  "policy_sha": "91aa",
  "change_passport_digest": "sha256:...",
  "ci_evidence": {
    "fast_unit": "passed",
    "targeted_integration": "passed",
    "secret_scan": "passed",
    "test_integrity": "passed"
  }
}
```

Example response:

```json
{
  "decision": "allow_auto_merge",
  "required_platform_status": "pass",
  "status_name": "vibegate/premerge",
  "subject_sha": "def456",
  "risk_class": "R2",
  "risk_score": 18,
  "receipt": "sha256:..."
}
```

Never support:

```text
approve latest
merge branch
deploy current
trust this agent
skip checks
```

Only support exact, bound, replayable requests.

# 10. GitLab-first implementation

GitLab should be the primary target because its primitives map extremely cleanly to this system:

```text
GitLab protected branches
GitLab MR approvals
GitLab CODEOWNERS
GitLab external status checks
GitLab merge trains
GitLab auto-merge
GitLab protected environments
GitLab deployment approvals
GitLab resource groups
GitLab pipeline execution policies
GitLab CI/CD components and inputs
GitLab Duo Agent Platform / external agents
```

GitLab merge trains queue MRs and compare each MR with earlier queued MRs so changes are tested together before landing, which directly solves the “passed alone, failed together” problem. ([GitLab Docs][9]) GitLab external status checks can send MR data to an external service, receive pass/fail states, block merges when configured, and reject stale responses that do not match the current MR head SHA. ([GitLab Docs][10]) GitLab auto-merge only merges after required checks such as approvals, successful pipelines, resolved discussions, external status checks, and policies are satisfied. ([GitLab Docs][11])

Use a tiny root `.gitlab-ci.yml`. Do not make one giant CI file.

GitLab CI/CD components are reusable pipeline configuration units with input parameters, and GitLab CI/CD inputs provide typed parameters with validation at pipeline creation time. That makes them a good fit for agent-controlled proof lanes and reusable release jobs. ([GitLab Docs][12])

## Non-bypassable GitLab policy

Use GitLab Pipeline Execution Policies for the jobs that must run even if the repo CI file changes or someone tries to skip CI. GitLab documents that pipeline execution policies can enforce CI/CD jobs across projects, remain in effect even when a project lacks CI config, and can prevent `[skip ci]` or `[no_pipeline]` from bypassing enforced policy jobs. ([GitLab Docs][14]) Protect the security policy project itself with protected branches and MR approvals. GitLab’s policy docs explicitly recommend restricting who can modify pipeline execution policies, protecting security policy project branches, requiring MR approvals, and monitoring policy changes. ([GitLab Docs][15])

```yaml
# .gitlab/security-policies/policy.yml
pipeline_execution_policy:
  - name: "VibeGate non-bypassable release controls"
    description: "Enforce VibeGate policy jobs even if project CI is modified."
    enabled: true
    pipeline_config_strategy: inject_policy
    content:
      include:
        - project: platform/vibegate-policy
          file: /policy-ci/vibegate-required.yml
          ref: main
    skip_ci:
      allowed: false
    no_pipeline:
      allowed: false
```

# 11. GitHub implementation equivalent

GitHub should use the same model with GitHub-native surfaces:

```text
GitHub App: VibeGate
Required check: vibegate/premerge
Ruleset: main_protected
Required status checks
Expected source: VibeGate GitHub App
CODEOWNERS
Merge queue
Environments
Deployment protection rules
Artifact attestations/signing
```

GitHub merge queue ensures a PR’s changes pass required status checks against the latest target branch plus earlier queued PRs. ([GitHub Docs][16]) GitHub rulesets can require status checks before merging and can require that a status check comes from a specific GitHub App source, which is exactly what VibeGate needs. ([GitHub Docs][17]) GitHub environments can use deployment protection rules for manual approval, delays, branch restrictions, or custom GitHub App-powered deployment gates. ([GitHub Docs][18]) Branch protection rules can require approving reviews and passing status checks before PRs merge. ([GitHub Docs][19])

The GitHub version:

```text
PR opened
  ↓
VibeGate GitHub App reads Change Passport
  ↓
Actions runs fast proof
  ↓
VibeGate posts check run: vibegate/premerge
  ↓
Ruleset requires check from VibeGate App
  ↓
Merge Queue validates latest-base compatibility
  ↓
main
  ↓
Release workflow builds once, signs, attests
  ↓
Environment protection rule calls VibeGate release gate
  ↓
Canary + Nightwatch
  ↓
prod or rollback
```

For open source, the bot must be polite:

```text
One PR summary.
One evidence table.
One risk score.
One clear blocker list.
No comment spam.
No hidden permissions.
No auto-merge unless maintainers opted in.
No policy changes without maintainer review.
```

# 12. Human-in-the-loop UX

The user experience should be boring and trustworthy.

## MR/PR comment format

```md
## VibeGate Verdict: Needs human review

Risk: R4
Reason: This change modifies authentication and CI policy.

What changed:
- src/auth/session.ts
- .gitlab/ci/20_fast_proof.yml

Proof:
- Unit tests: passed
- Targeted integration: passed
- Secret scan: passed
- Test integrity: passed
- Policy self-change check: failed

Why I blocked:
- Candidate CI policy cannot approve itself.
- Auth path requires Security Sentinel + human owner.

Safe next actions:
1. Security owner reviews auth behavior.
2. Platform owner reviews CI change.
3. Re-run VibeGate after approval.
```

## Human behavior rules

Humans will reject the system if it creates noise, hides authority, or makes them feel responsible for decisions they did not understand.

So the system should:

```text
Summarize, do not spam.
Show receipts, not raw logs first.
Make the next action obvious.
Let humans override only with reason.
Require waivers to expire.
Teach from failures.
Never shame humans for stopping automation.
Never make “full autonomy” the default.
```

## Good default UX

```text
Green:
  “Safe to auto-merge. Evidence attached.”

Yellow:
  “Needs more proof. Agent can repair.”

Orange:
  “Needs specialist review.”

Red:
  “Blocked by hard rule.”

Black:
  “Break-glass only.”
```

# 13. Anti-vibe-coding protections

VibeGate should explicitly detect the patterns that make AI-generated code dangerous.

```yaml
# .vibegate/policies/anti_vibe.yml
anti_vibe_rules:
  test_weakening:
    block_if:
      - assertions_removed
      - tests_deleted_without_replacement
      - snapshots_mass_updated
      - skip_or_only_added
      - coverage_drop_without_waiver

  fake_confidence:
    block_if:
      - change_summary_claims_tests_but_no_test_artifacts
      - says_no_behavior_change_but_public_api_changed
      - says_no_security_impact_but_auth_paths_changed

  ci_weakening:
    escalate_if:
      - required_job_removed
      - status_check_name_changed
      - policy_job_moved_to_allow_failure
      - deploy_job_rules_relaxed

  dependency_risk:
    escalate_if:
      - new_runtime_dependency
      - license_unknown
      - dependency_score_below_threshold
      - transitive_vulnerability_high_or_critical

  generated_code:
    require:
      - generated_zone_declared
      - source_generator_identified
      - regeneration_command_recorded
      - generated_diff_not_manually_modified
```

OpenSSF Scorecard is useful as an input signal for dependency and project health because it checks risks across source code, build, dependencies, testing, and maintenance, but it should be one signal, not the sole gate. ([undefined][20])

# 14. Supply-chain and security baseline

VibeGate should align with NIST SSDF, SLSA, SBOM, signatures, and policy-as-code.

NIST SP 800-218 SSDF is a common secure software development framework intended to integrate secure practices into SDLC implementations. ([NIST Computer Security Resource Center][21]) OPA is a strong fit for policy-as-code because it decouples policy decision-making from enforcement and evaluates structured input like JSON against policies. ([Open Policy Agent][22])

SLSA provenance

Append-only ledger

# 15. Release control: Foundry Train + Nightwatch

## Foundry Train

Foundry Train is the post-merge certification system.

It does:

```text
Coalesce merged changes
Create release candidate
Run full/deep CI
Build once
Generate SBOM
Generate provenance
Sign artifact
Deploy to dev/staging/shadow
Create Release Passport
```

Why after merge?

Because full CI is expensive. Running it on every tiny MR kills autonomy. The better model is:

```text
MR/PR: fast, honest, impacted proof
Merge queue/train: latest-base compatibility
Release train: full certification
Canary: runtime proof
```

## Nightwatch

Nightwatch is the runtime release agent.

It watches:

```text
error rate
latency p50/p95/p99
saturation
crash loops
log anomaly rate
business KPI
support tickets
security alerts
rollback readiness
```

Canary policy:

```yaml
# deploy/rollout.yml
strategy: canary

steps:
  - percent: 5
    duration: 20m
    auto_promote_if_healthy: true
  - percent: 10
    duration: 30m
    auto_promote_if_healthy: true
  - percent: 25
    duration: 60m
    auto_promote_if_healthy: true
  - percent: 50
    duration: 60m
    require_human_unless_profile: A5
  - percent: 100
    require:
      - all_slos_pass
      - rollback_ready
      - no_open_sev1_or_sev2

rollback:
  automatic_if:
    - error_rate_delta_gt: "2%"
    - p95_latency_delta_gt: "20%"
    - crash_loop_detected: true
    - business_kpi_drop_gt: "5%"
  method_order:
    - traffic_weight_to_stable
    - feature_flag_disable
    - previous_artifact_digest
    - database_mitigation_plan
```

# 16. The canonical `AGENTS.md`

Put this at repo root.

```md
# Agent Delivery Standard

This repository uses VibeGate.

## Core rules

1. All durable repo changes go through PR/MR.
2. Agents may create branches and MRs; agents may not push to protected branches.
3. Agents may not approve their own work.
4. Every autonomous change must start from an Intent Card.
5. Every MR must include a Change Passport.
6. No proof, no merge.
7. No rollback, no production.
8. Policy, CI, auth, crypto, secrets, deploy, and migration changes require elevated review.
9. Every VibeGate verdict must be bound to the exact HEAD SHA.
10. Releases are built once and promoted by artifact digest.
11. Production promotion requires canary telemetry.
12. All decisions are logged and replayable.

## Required MR evidence

- Objective
- Changed paths
- Risk class
- Tests run
- Tests intentionally not run, with reason
- Red/green proof for bug fixes
- Test integrity summary
- Security summary
- Rollback plan

## Agent behavior

- Prefer small diffs.
- Do not broaden scope without updating the Intent Card.
- Do not weaken tests to pass CI.
- Do not modify VibeGate policy unless the task explicitly requests policy work.
- When blocked, explain the blocker and propose the smallest safe repair.
```
