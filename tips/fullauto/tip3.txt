After reading the uploaded VibeGate notes and cross-checking against current GitHub/GitLab and supply-chain practices, my final recommendation is:

# Build **VibeGate: Evidence-Carrying Autonomous Delivery**

The core idea:

> **Agents may write code. Agents may review code. Agents may release code. But only evidence, policy, platform enforcement, signed artifacts, telemetry, and rollback may move code forward.**

The simplest canonical flow is:

```text
Intent Card
  ↓
Scoped Capability Lease
  ↓
Ephemeral Agent Sandbox
  ↓
agent/* branch
  ↓
MR/PR + Change Passport
  ↓
Fast impacted CI
  ↓
VibeLock Airlock
  ↓
GitLab Merge Train / GitHub Merge Queue
  ↓
main = merge-certified
  ↓
Foundry Train
  ↓
certified-main / release candidate
  ↓
Build once, sign, attest, SBOM
  ↓
Dev / staging / shadow
  ↓
Nightwatch canary
  ↓
Progressive production
  ↓
Launch Ledger
```

The operating law:

```text
No intent, no work.
No lease, no privileged action.
No passport, no merge.
No receipt, no trust.
No signed artifact, no deploy.
No telemetry, no promotion.
No rollback, no production.
```

This fits where the ecosystem is going. OpenAI Codex already works as a cloud software-engineering agent that can edit repositories, run tasks in sandboxed environments, and propose pull requests; `AGENTS.md` is now a widely used open convention for repository-level agent instructions; GitLab Duo Agent Platform is explicitly moving toward multiple specialized agents across the SDLC. ([OpenAI][21]) The important lesson from DORA’s 2025 AI-assisted software delivery research is that AI amplifies the delivery system around it, including both strengths and dysfunctions; so the best future-proof design is not “trust better models,” it is **make the software delivery system evidence-native**. ([Google Research][22])

# 1. The final naming system

Use fun names internally, but map them to boring GitHub/GitLab terms externally.

| Layer            | Canonical product name |             Plain enterprise name | Purpose                                                                |
| ---------------- | ---------------------: | --------------------------------: | ---------------------------------------------------------------------- |
| Repo standard    |  **Jankurai Standard** |           Agent delivery standard | The anti-vibe-coding rules for a repo.                                 |
| Control plane    |           **VibeGate** | Autonomous delivery control plane | Owns policy, evidence, leases, verdicts, telemetry, release authority. |
| Pre-merge gate   |   **VibeLock Airlock** |   Required pre-merge status check | Blocks unproven or unsafe MRs/PRs.                                     |
| Proof object     |    **Change Passport** |                   Evidence bundle | Machine-readable proof attached to every change.                       |
| Merge proof      |     **Merge Passport** |              Gate verdict receipt | Exact-SHA-bound decision allowing merge queue/train entry.             |
| Release queue    |      **Foundry Train** |       Release certification queue | Batches full validation and artifact production.                       |
| Runtime observer |         **Nightwatch** |     Canary and rollout controller | Promotes, pauses, or rolls back based on telemetry.                    |
| Audit trail      |      **Launch Ledger** |          Append-only decision log | Stores every intent, lease, decision, artifact, deployment, rollback.  |

The most important naming decision is this:

> **VibeGate is the system. VibeLock is the gate. Foundry Train certifies releases. Nightwatch watches production.**

That avoids the confusion of calling every layer “the gate.”

# 2. The core philosophy

The wrong architecture is:

```text
agent writes code → agent says it is good → agent merges → agent deploys
```

The right architecture is:

```text
agent writes code
  → CI proves facts
  → independent agents critique evidence
  → deterministic policy decides
  → GitHub/GitLab enforce branch rules
  → release system builds one artifact
  → canary telemetry proves runtime health
  → rollback remains armed
```

The key doctrine:

> **Do not trust agents. Trust receipts.**

Agents are powerful, useful, and untrusted. The trusted control plane is VibeGate. The forge remains GitHub/GitLab. CI remains the proof engine. The release platform remains the deployment authority. Humans and agents are both approvers inside a policy model, but neither bypasses the model.

# 3. The five-plane architecture

## Plane 1: Work Plane

Builder agents create patches, tests, docs, migrations, refactors, or dependency updates.

They can:

```text
create branches
open MRs/PRs
run tests
propose evidence
repair failures
explain changes
```

They cannot:

```text
push to protected branches
approve their own work
weaken policy silently
access production secrets
deploy directly
edit release gates without escalation
```

## Plane 2: Proof Plane

CI, static analysis, test selection, coverage, security scanners, dependency scanners, benchmarks, contract tests, and runtime simulation create evidence.

The rule:

> **Agents may claim almost nothing unless a tool produced the proof.**

Good proof examples:

```text
regression test failed on base SHA and passed on head SHA
impacted test map selected these tests and skipped these with reasons
security scanner produced no new critical findings
dependency delta has license and vulnerability receipt
migration has forward and rollback plan
performance claim has before/after benchmark
```

## Plane 3: VibeLock Airlock

VibeLock is the pre-merge decision layer.

It asks:

```text
Did this change stay inside the Intent Card?
Did it use only granted capabilities?
Did it include a valid Change Passport?
Did impacted CI pass?
Were skipped tests skipped for defensible reasons?
Did it weaken tests, CI, policy, observability, or rollback?
Did it touch protected control-plane paths?
Is rollback clear?
Is canary required?
Does this autonomy profile allow auto-merge?
```

VibeLock should be a **required status check / external status check**, not just an AI review comment. GitLab external status checks can be configured so merge requests cannot merge unless all required status checks pass, and GitLab returns conflicts for out-of-date status responses that do not match the current HEAD SHA. ([GitLab Docs][23])

## Plane 4: Merge Plane

GitLab/GitHub remain the merge authority.

Use:

```text
protected branches
required checks
CODEOWNERS
approval rules
merge trains / merge queues
signed commits where needed
linear history where needed
no force-push to protected branches
```

GitLab merge trains queue merge requests and compare each request against earlier queued requests to ensure they work together before reaching the default branch. ([GitLab Docs][24]) GitHub merge queue similarly ensures queued pull request changes pass required status checks when applied to the latest target branch plus earlier queued pull requests. ([GitHub Docs][25])

## Plane 5: Release Plane

`main` is not production. `main` is merge-certified.

Release certification happens after merge:

```text
main
  ↓
Foundry Train
  ↓
full CI / deep security / integration / perf
  ↓
immutable artifact
  ↓
SBOM + provenance + signature
  ↓
dev/staging/shadow
  ↓
Nightwatch canary
  ↓
progressive rollout
```

Production promotion requires live health, not agent confidence. Argo Rollouts is a strong reference model because it supports canary, blue-green, canary analysis, experimentation, and progressive delivery features for Kubernetes. ([Argo Project][26])

# 4. The mandatory machine-readable objects

## 4.1 Intent Card

Every autonomous change starts with a typed request.

```yaml
# .autonomy/intents/BUG-1842.yaml
apiVersion: vibegate.dev/v1
kind: IntentCard

id: BUG-1842
type: bugfix # docs | test | bugfix | feature | dependency | security | refactor | infra | ci | release
source: issue
owner: payments-platform

objective: "Fix null cart total when discounted cart has no line items"

success_criteria:
  - "Regression test fails on base SHA and passes on head SHA"
  - "No behavior change for non-empty carts"
  - "Checkout smoke test remains green"

allowed_scope:
  paths:
    - services/cart/**
    - tests/cart/**

forbidden_scope:
  paths:
    - .gitlab/**
    - .github/**
    - .autonomy/policies/**
    - infra/prod/**
    - secrets/**

autonomy:
  requested_mode: guarded_auto_merge
  max_merge_risk: R2
  max_release_environment: staging
  production_allowed: false

rollback:
  required: true
  strategy: "revert_commit_or_disable_cart_discount_flag"
```

This prevents the most common agent failure: broad, confident work from an underspecified prompt.

## 4.2 Capability Lease

The lease is the agent’s temporary permission.

```yaml
apiVersion: vibegate.dev/v1
kind: CapabilityLease

lease_id: lease_2026_05_16_1842
intent_id: BUG-1842
agent_id: codex-bugfix-v4
actor_role: builder

valid_for:
  base_sha: "abc123"
  branch_pattern: "agent/bugfix/BUG-1842/*"
  expires_at: "2026-05-16T23:59:00Z"

allowed_actions:
  - repo.read
  - branch.create
  - branch.push.agent
  - ci.run.impacted
  - mr.create
  - evidence.write

denied_actions:
  - protected_branch.push
  - approval.self
  - policy.modify
  - production.deploy
  - secrets.read.production

scope:
  paths:
    - services/cart/**
    - tests/cart/**

signature:
  issuer: vibegate-control-plane
  alg: ed25519
  value: "..."
```

The lease must be scoped, short-lived, exact-SHA-aware, signed, revocable, and audited.

## 4.3 Change Passport

The passport is the evidence bundle for the MR/PR.

```yaml
apiVersion: vibegate.dev/v1
kind: ChangePassport

change_id: chg_2026_05_16_BUG_1842
intent_id: BUG-1842
mr: 812

author:
  agent_id: codex-bugfix-v4
  lease_id: lease_2026_05_16_1842

git:
  base_sha: abc123
  head_sha: def456
  target_branch: main
  source_branch: agent/bugfix/BUG-1842/fix-null-cart-total

summary:
  plain_english: "Guard empty discounted cart before total calculation."
  changed_paths:
    - services/cart/total.ts
    - tests/cart/discounted_empty_cart.test.ts

risk:
  declared_class: R2
  blast_radius: service
  protected_paths_touched: false
  user_visible: false
  data_model_change: false
  auth_change: false
  migration: false
  ci_policy_change: false

proof:
  tests:
    - name: cart-unit
      status: passed
      artifact: artifacts/cart-unit.xml
    - name: checkout-smoke
      status: passed
      artifact: artifacts/checkout-smoke.xml

  regression:
    base_failed: true
    head_passed: true
    test: tests/cart/discounted_empty_cart.test.ts

  impact:
    selected_tests:
      - tests/cart/**
      - tests/checkout/smoke/**
    skipped_tests:
      tests/billing/**: "No dependency edge, no shared contract, no historical co-failure"

  security:
    secret_scan: passed
    sast_delta: no_new_high_or_critical

rollback:
  strategy: revert_commit
  estimated_time: "<5m"

integrity:
  evidence_hash: sha256:...
  generated_at: "2026-05-16T05:44:00Z"
```

## 4.4 Gate Verdict / Merge Passport

The gate verdict is the exact-SHA-bound decision.

```json
{
  "apiVersion": "vibegate.dev/v1",
  "kind": "MergePassport",
  "change_id": "chg_2026_05_16_BUG_1842",
  "intent_id": "BUG-1842",
  "mr": 812,
  "base_sha": "abc123",
  "head_sha": "def456",
  "decision": "allow_merge_train",
  "risk_class": "R2",
  "risk_score": 18,
  "confidence": 0.92,
  "human_required": false,
  "hard_rule_hits": [],
  "required_status_checks": [
    "pr-fast-ci",
    "vibelock"
  ],
  "agent_reviews": [
    {
      "agent": "cartographer-v2",
      "role": "impact",
      "decision": "pass"
    },
    {
      "agent": "test-skeptic-v3",
      "role": "test_integrity",
      "decision": "pass"
    },
    {
      "agent": "judge-v2",
      "role": "judge",
      "decision": "pass"
    }
  ],
  "policy_ref": "main:.autonomy/policies@sha256:...",
  "evidence_hash": "sha256:...",
  "expires_at": "2026-05-17T00:00:00Z",
  "signature": "..."
}
```

The most important field is `policy_ref`.

> **Candidate policy cannot judge itself.**

A PR/MR changing `.autonomy/policies/**`, `.gitlab-ci.yml`, `.github/**`, deploy manifests, or agent prompts must be evaluated by the policy from the protected target branch, not by the policy being introduced in the change.

## 4.5 Release Passport

The release passport proves what entered production.

```yaml
apiVersion: vibegate.dev/v1
kind: ReleasePassport

release_id: REL-2026-05-16-0042
source:
  ref: certified-main
  sha: "9b1..."
  included_changes:
    - chg_2026_05_16_BUG_1842
    - chg_2026_05_16_UPG_4412

certification:
  full_ci: passed
  integration: passed
  security: passed
  performance: passed
  migration_dry_run: not_applicable

artifact:
  name: cart-service
  digest: sha256:...
  built_once: true
  sbom: artifacts/sbom.cdx.json
  provenance: artifacts/provenance.intoto.jsonl
  signature: artifacts/cosign.bundle

rollout:
  strategy: canary
  rings:
    - name: staging
      status: passed
    - name: production-1-percent
      status: passed
    - name: production-10-percent
      status: passed
    - name: production-100-percent
      status: pending

rollback:
  method: previous_digest
  previous_digest: sha256:...
  kill_switch: cart_discount_enabled
```

in-toto is a good fit for supply-chain evidence because it is designed to make transparent what steps were performed, by whom, and in what order; OPA is a good fit for policy-as-code because it can enforce policies across CI/CD and other systems using structured inputs. ([in-toto][27])

# 5. Autonomy profiles

Full autonomy must be easy to enable, but never ambiguous. Make autonomy a repo-level profile.

```yaml
# .autonomy/profile.yaml
apiVersion: vibegate.dev/v1
kind: AutonomyProfile

active_profile: A3

profiles:
  A0_observe:
    agents_can:
      - analyze
      - suggest
      - draft_intent
    auto_merge: false
    auto_release: false

  A1_patch:
    agents_can:
      - create_branch
      - open_mr
      - run_ci
    auto_merge: false
    auto_release: false

  A2_prove:
    agents_can:
      - create_change_passport
      - request_reviews
      - repair_failures
    auto_merge: false
    auto_release: false

  A3_guarded_auto_merge:
    auto_merge:
      max_risk: R2
      required_gate: vibelock
      require_merge_train: true
    auto_release: false
    humans_required_for:
      - R3
      - R4
      - R5
      - protected_paths
      - policy_changes
      - production

  A4_auto_release_low_risk:
    auto_merge:
      max_risk: R2
    auto_release:
      max_risk: R2
      max_environment: production
      require_canary: true
      require_rollback: true
    humans_required_for:
      - R4
      - R5
      - policy_changes
      - auth
      - secrets
      - migrations

  A5_sovereign_autopilot:
    auto_merge:
      max_risk: R5
      require_elevated_agent_quorum: true
    auto_release:
      max_environment: production
      require_canary: true
      require_slo_guard: true
      require_rollback: true
    humans_required_for: []
    policy_mutation:
      allowed_by_elevated_agent_quorum: true
      judged_by_previous_protected_policy: true
      require_public_receipt: true
```

This is the key to making human-in-the-loop and 100% autonomy compatible:

> **Full autonomy is not a different architecture. It is the same architecture with a different approval profile.**

A0 through A4 should be normal for most teams. A5 should exist because the system must allow 100% autonomous production if the owner explicitly configures it. But A5 still uses exact-SHA receipts, independent agent quorum, protected branch enforcement, canary, rollback, and immutable audit logs.

# 6. Risk ladder

Use a simple risk ladder developers can understand.

|   Risk | Examples                                                                             | Default behavior                                          |
| -----: | ------------------------------------------------------------------------------------ | --------------------------------------------------------- |
| **R0** | Docs typo, formatting, generated changelog                                           | Auto-merge eligible after minimal proof.                  |
| **R1** | Tests, small cleanup, dead-code removal                                              | Auto-merge eligible with fast CI and one reviewer agent.  |
| **R2** | Isolated bug fix, small UI fix, safe dependency patch                                | Auto-merge eligible under A3+ with VibeLock pass.         |
| **R3** | User-visible behavior, API contract, performance-sensitive code                      | Human or specialist agent approval depending on profile.  |
| **R4** | Auth, permissions, crypto, PII, billing, migrations, CI, release config              | Human required by default; elevated quorum in A5.         |
| **R5** | Production credentials, policy root of trust, emergency override, rollback disabling | Human by default; A5 only with explicit sovereign policy. |

Risk scoring should be max-plus, not averaging.

```text
final_risk =
  max(hard_rule_risk)
  + evidence_gap
  + blast_radius
  + rollback_difficulty
  + test_integrity_risk
  + security_data_dependency_risk
  + agent_trust_risk
  + uncertainty_penalty
```

Why max-plus?

Because ten green checks must not hide one severe failure. A single hard stop should dominate the score.

Hard stops:

```text
changed CI but no elevated approval
changed policy but evaluated by candidate policy
deleted tests without replacement proof
changed auth/permissions without security review
added dependency without license/security receipt
changed generated files outside generated-zone map
missing owner map for changed path
missing rollback for production-affecting change
attempted self-approval
status check not bound to current HEAD SHA
```

# 7. Agent roles

Do not use a giant fixed swarm for every change. Use a small default set, then specialist agents only when triggered.

## Default VibeLock pack

| Agent                  | Purpose                                                                      |
| ---------------------- | ---------------------------------------------------------------------------- |
| **Builder Agent**      | Writes the patch. Cannot approve it.                                         |
| **Cartographer Agent** | Maps changed paths to owners, modules, tests, contracts, risk.               |
| **Test Skeptic Agent** | Looks for missing tests, weakened assertions, snapshot abuse, skipped proof. |
| **Judge Agent**        | Combines policy, CI, evidence, and reviews into a verdict.                   |

## Conditional specialist agents

| Trigger                     | Specialist                    |
| --------------------------- | ----------------------------- |
| Auth / permissions / crypto | Security Sentinel             |
| Dependencies / licenses     | Dependency and IP Guardian    |
| Migrations / data model     | Data Migration Auditor        |
| CI / policy / release files | Control Plane Guardian        |
| Performance claim           | Benchmark Skeptic             |
| Frontend visual change      | UI Regression Agent           |
| Docs / public API           | Developer Experience Reviewer |
| Production rollout          | Release Shepherd              |
| Runtime telemetry           | Nightwatch                    |

## Anti-collusion rules

```text
author_agent != reviewer_agent
reviewer_agent did not write to the branch
judge_agent did not author the patch
release_agent cannot override VibeLock
same lease cannot both author and approve
same agent session cannot both create and validate evidence
agent approval must be exact-SHA-bound
agent approval expires after branch update
```

For high-risk changes, require diversity:

```yaml
elevated_agent_quorum:
  required_roles:
    - security
    - test_integrity
    - release
    - judge
  independence:
    min_distinct_agent_instances: 3
    min_distinct_leases: 3
    author_excluded: true
```

# 8. Approval routing

Approval should be a policy outcome, not a hardcoded “human required” branch.

```yaml
# .autonomy/policies/routing.yaml
approval_routing:
  R0:
    auto_merge_allowed_profiles: [A3, A4, A5]
    required:
      - fast_ci
      - vibelock_lite

  R1:
    auto_merge_allowed_profiles: [A3, A4, A5]
    required:
      - fast_ci
      - cartographer
      - test_skeptic

  R2:
    auto_merge_allowed_profiles: [A3, A4, A5]
    required:
      - fast_ci
      - vibelock
      - merge_train

  R3:
    auto_merge_allowed_profiles: [A5]
    required:
      - full_relevant_ci
      - specialist_agent
      - owner_or_elevated_quorum

  R4:
    auto_merge_allowed_profiles: [A5]
    default_human_required: true
    required:
      - full_relevant_ci
      - specialist_agent
      - security_or_platform_owner
      - explicit_rollout_plan

  R5:
    auto_merge_allowed_profiles: [A5]
    default_human_required: true
    required:
      - emergency_or_sovereign_policy
      - elevated_quorum
      - break_glass_receipt
```

The best UX is:

```text
VibeLock decision: Needs human
Reason: CI policy changed + policy file changed.
Required approver: Platform Owner or A5 elevated quorum.
One-click actions:
  - Approve once
  - Send back to agent
  - Require more proof
  - Escalate to sovereign quorum
  - Reject
```

Humans should see one decision, one risk score, one reason, and one set of actions. Not ten bot comments.

# 10. GitHub implementation mapping

Use the same model with GitHub-native primitives:

| VibeGate concept        | GitHub equivalent                                            |
| ----------------------- | ------------------------------------------------------------ |
| VibeLock                | Required status check from GitHub App                        |
| Protected branch        | Branch protection / rulesets                                 |
| Merge train             | Merge queue                                                  |
| Human/elevated approval | Required reviewers / CODEOWNERS / app-enforced check         |
| Protected production    | Environments + deployment protection rules                   |
| Release gate            | GitHub App custom deployment protection rule                 |
| Evidence                | Checks artifacts, workflow artifacts, PR summary, app ledger |
| Launch Ledger           | External append-only store + Checks API annotations          |

GitHub branch protection/rulesets can require status checks before merging, and merge queue validates required checks against the target branch plus queued changes. ([GitHub Docs][32]) GitHub environments can require reviewers, prevent self-review, use wait timers, restrict branches, and support custom deployment protection rules through GitHub Apps. ([GitHub Docs][33]) If using GitHub merge queue with Actions, workflows for required checks need the `merge_group` trigger, or the queue may fail because required checks are not reported. ([GitHub Docs][34])

Minimal GitHub workflow shape:

```yaml
name: VibeGate

on:
  pull_request:
  merge_group:
  push:
    branches: [main]

jobs:
  pr-fast:
    if: github.event_name == 'pull_request' || github.event_name == 'merge_group'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: vibegate impact plan
      - run: vibegate proof run --lane fast
      - run: vibegate passport create

  vibelock:
    if: github.event_name == 'pull_request' || github.event_name == 'merge_group'
    runs-on: ubuntu-latest
    needs: [pr-fast]
    steps:
      - uses: actions/checkout@v4
      - run: vibegate gate evaluate --policy-source protected-target-branch

  foundry:
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: vibegate proof run --lane full-certification
      - run: vibegate artifact build
      - run: vibegate attest
```

# 11. CI strategy: fastest version that remains safe

The winning CI strategy is:

```text
PR/MR: fast, impact-based, evidence-focused
Merge queue/train: latest-base integration proof
Release train: full/deep/batched certification
Canary: production health proof
```

## Tier 1: Always-on sentinels

Run on every MR/PR:

```text
schema validation
intent validation
lease validation
changed-path classification
secret scan
lint of policy/CI syntax
basic unit tests for touched modules
test weakening detection
passport generation
```

## Tier 2: Impacted proof

Run only the tests mapped to the blast radius:

```text
owner map
service map
dependency graph
coverage map
contract map
historical co-failure map
runtime trace map
```

The rule:

> **Run no test outside the provable blast radius, but never pretend the blast radius is smaller than the evidence proves.**

If the impact map is weak, outdated, or missing, risk increases and the selected proof expands.

## Tier 3: Merge queue/train proof

Validate against the latest target branch plus queued changes. This prevents many individually-green MRs from breaking `main` together.

## Tier 4: Foundry Train certification

Batch expensive checks:

```text
full integration
slow e2e
deep SAST/DAST where applicable
dependency diff
license review
migration dry-run
performance benchmarks
SBOM/provenance/signing
release candidate creation
```

## Tier 5: Nightwatch runtime proof

Production is certified by live telemetry:

```text
error rate
latency
saturation
conversion or product metric
security signal
support ticket spike
logs/anomaly signal
rollback readiness
```

OpenTelemetry already defines CI/CD semantic conventions for spans, metrics, and logs, which makes it a good telemetry foundation for agent runs, CI, gate decisions, deployments, and rollbacks. ([OpenTelemetry][35])

# 12. Anti-vibe-coding protections

VibeGate should explicitly detect the patterns maintainers hate.

## 12.1 Test gaming

Block or escalate when the agent:

```text
deletes assertions
widens expected values
updates snapshots without explanation
skips failing tests
moves tests out of path
changes CI to avoid tests
claims "tested" without artifact
```

## 12.2 Scope creep

Block or escalate when:

```text
changed paths exceed Intent Card
unrelated files modified
large refactor hidden inside bugfix
generated files changed without source change
dependency lockfile changes unexplained
```

## 12.3 False proof

Block or escalate when:

```text
test passed on head but was never shown failing on base for a bugfix
benchmark lacks baseline
security fix lacks threat statement
migration lacks rollback
canary lacks SLO
evidence artifact missing or hash mismatch
```

Always high risk:

```text

CODEOWNERS

k8s/**

release/**
secrets/**
auth/**
crypto/**
```

Default: human required.

A5: elevated autonomous quorum allowed, judged by previous protected policy.

# 15. Policy-as-code example

Use Rego/OPA for deterministic decisions and agent reviewers for evidence critique.

```rego
package vibegate.premerge

default allow_merge := false
default needs_human := false
default block := false

protected_path_touched if {
  some p in input.changed_paths
  some pattern in data.protected_paths.control_plane.paths
  glob.match(pattern, ["/"], p)
}

self_approval if {
  input.author.agent_id == input.review.approver_agent_id
}

missing_passport if {
  not input.passport.valid
}

policy_judged_itself if {
  input.changed_paths[_] == ".autonomy/policies/hard_stops.rego"
  input.policy.source != "protected_target_branch"
}

block if missing_passport
block if self_approval
block if policy_judged_itself

needs_human if {
  protected_path_touched
  input.profile != "A5_sovereign_autopilot"
}

needs_human if {
  input.risk.class == "R4"
  input.profile != "A5_sovereign_autopilot"
}

allow_merge if {
  not block
  not needs_human
  input.passport.valid
  input.ci.fast_passed
  input.rollback.exists
  input.risk.score <= data.thresholds[input.profile].auto_merge
}
```

The policy engine decides. Agents supply context and critique.

# 19. Security baseline

The minimum responsible baseline:

```text
agents use ephemeral identities
all leases are signed and TTL-bound
all privileged actions require a lease
agents never receive broad long-lived tokens
production secrets only available inside protected deployment jobs
no agent can approve its own work
no agent can push to protected branches
policy evaluated from protected target branch
artifact is built once and promoted by digest
SBOM/provenance/signature required before production
rollback is mandatory for production
all decisions are append-only logged
```

For sandboxing:

```text
ephemeral worktrees or containers for low-risk code
network egress limits for unknown dependencies
microVM or hardened container for untrusted code execution
secretless test environments by default
tainted cache when dependency/security inputs change
deterministic replay for failure capsules where feasible
```

# 21. Why this is the best design

It is fast because:

```text
agents work in parallel
PR/MR CI is impact-based
review agents are selected dynamically
merge train/queue handles integration
expensive proof is batched in Foundry Train
production proof happens through canary telemetry
```

It is safe because:

```text
agents cannot self-approve
agents cannot push to protected branches
policy cannot judge itself
evidence is exact-SHA-bound
merge is platform-enforced
release uses signed immutable artifacts
production promotion requires telemetry
rollback is mandatory
```

It is simple because every stage has one job:

```text
Intent Card: what should happen
Capability Lease: what the agent may do
Change Passport: what changed and how it was proven
VibeLock: may this enter main?
Foundry Train: is this release candidate certified?
Nightwatch: is production healthy?
Launch Ledger: what happened and why?
```

It is future-proof because it does not depend on one model, one vendor, one forge, or one CI system. It depends on stable software-delivery primitives: PR/MR, protected branches, required checks, policy-as-code, signed artifacts, progressive delivery, telemetry, and rollback.
