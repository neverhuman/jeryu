# VibeGate RADS v1.0 — Responsible Autonomous Delivery Standard

**Final recommendation:** build the system around one simple contract:

> **Agents author. Evidence proves. Policy decides. GitLab/GitHub enforces. Telemetry promotes.**

The best design is not “let agents code and hope CI catches it.” It is a **release-control standard** where every autonomous action is bounded by a lease, every merge is backed by proof, every production release is tied to a signed artifact, and every automated approval is traceable to a policy that the user deliberately configured.

The canonical name I would use is:

> **VibeGate RADS v1.0**
> **Responsible Autonomous Delivery Standard**

The key product object names should be simple and memorable:

| Object                     | Purpose                                                                                   |
| -------------------------- | ----------------------------------------------------------------------------------------- |
| **Intent Card**            | What work is being requested, by whom, under what autonomy level.                         |
| **Capability Lease**       | What the agent may do, for how long, on which repo/branch/files/tools.                    |
| **Change Passport**        | The merge-ready evidence bundle for an MR/PR.                                             |
| **VibeLock**               | The merge gate that says “proof is complete” or “repair required.”                        |
| **Release Passport**       | The production-release evidence bundle for one immutable artifact digest.                 |
| **Nightwatch**             | The release agent that watches canary/prod telemetry and promotes, pauses, or rolls back. |
| **Launch Ledger**          | The append-only audit trail of decisions, evidence, approvals, artifacts, and rollbacks.  |
| **Agent Approval Receipt** | A signed, scoped approval by an independent release/review agent.                         |

The design is intentionally small at the top level:

```text
Intent Card
  → Capability Lease
    → Agent Branch / MR
      → Change Passport
        → VibeLock Merge Gate
          → Foundry Build
            → Release Passport
              → Nightwatch Release Gate
                → Canary / Production / Rollback
                  → Launch Ledger
```

That is the “genius in simplicity.” Everything else is implementation detail.

## 1. Definition of “100% responsible autonomous coding”

“100% autonomous” must **not** mean “the model can do whatever it wants.” It means:

> The human pre-delegates authority through policy, and the system can execute all the way to production only when the delegated policy, evidence, telemetry, rollback plan, and independent approval rules are satisfied.

So **100% responsible** means:

1. **Explicit authorization:** the user or org chooses the autonomy profile.
2. **Least privilege:** the agent receives a temporary Capability Lease.
3. **No self-approval:** an agent cannot approve its own code, release, policy change, or evidence.
4. **Evidence-gated merge:** no proof, no merge.
5. **Build once, promote same artifact:** production deploys the exact signed digest that passed checks.
6. **Independent release decision:** Nightwatch or a human decides production promotion.
7. **Observable rollout:** canary/progressive release with SLO-based pause/rollback.
8. **Rollback armed:** no rollback plan, no production.
9. **Tamper-evident audit:** every decision produces a Launch Ledger event.
10. **Human override:** pause, reject, edit, rollback, or revoke autonomy at any time.

No system can eliminate all software risk. This standard makes the risk **explicit, bounded, observable, reversible, and auditable**.

# 2. The three-gate mental model

Every maintainer, founder, engineer, and future AI developer should be able to understand the system in one minute:

| Gate             | Question                                    | Artifact                       | Enforcer                           |
| ---------------- | ------------------------------------------- | ------------------------------ | ---------------------------------- |
| **Work Gate**    | May the agent work?                         | Intent Card + Capability Lease | Agent host                         |
| **Merge Gate**   | May this change enter the protected branch? | Change Passport                | VibeLock + GitLab/GitHub           |
| **Release Gate** | May this artifact reach production?         | Release Passport               | Nightwatch + protected environment |

The system should be sold and documented as:

> **Work requires a lease. Merge requires proof. Production requires a passport.**

# 3. Autonomy profiles

The user should not have to understand 40 policy flags before getting started. They should choose an autonomy profile.

| Profile | Name           | What it allows                                                                   | Best for                          |
| ------: | -------------- | -------------------------------------------------------------------------------- | --------------------------------- |
|  **A0** | Observe        | Agent can inspect, summarize, and recommend only.                                | Regulated or new repo onboarding. |
|  **A1** | Suggest        | Agent can draft patches but not write to repo.                                   | Conservative teams.               |
|  **A2** | Branch         | Agent can create branch/MR, run tests, attach evidence. Human merges.            | Default for open source.          |
|  **A3** | Repair Loop    | Agent can iterate on MR until green. Human merges/releases.                      | CI-fix bots, routine work.        |
|  **A4** | Auto-Merge     | Low-risk MRs can merge automatically after VibeLock passes. Human controls prod. | Mature internal repos.            |
|  **A5** | Auto-Release   | Low-risk releases can deploy to staging/canary with Nightwatch approval.         | SaaS teams with strong telemetry. |
|  **A6** | Full Autopilot | Agent can go branch → merge → canary → production under explicit policy.         | Highly mature systems only.       |

**A6 is not “no humans.”** A6 means humans have pre-designed the release policy, risk limits, evidence requirements, rollback behavior, and emergency stop conditions.

The default should be **A2** for open source and **A3/A4** for internal mature repos. A6 must be opt-in.

# 4. Risk levels

Autonomy is meaningless without risk. Every Intent Card and MR should receive a risk classification.

|   Risk | Examples                                                                     | Default autonomy                       |
| -----: | ---------------------------------------------------------------------------- | -------------------------------------- |
| **R0** | Docs, comments, formatting, non-behavioral tests                             | A4/A5 allowed                          |
| **R1** | Small internal bug fix, narrow dependency bump, isolated refactor            | A4 allowed                             |
| **R2** | Feature behind flag, UI change, non-critical service behavior                | A5 allowed with canary                 |
| **R3** | Database migration, infra, performance-sensitive code, public API            | Human or senior release agent required |
| **R4** | Auth, payments, secrets, PII, permissions, destructive changes               | Human code owner + security required   |
| **R5** | Incident, regulated, irreversible, production hotfix with broad blast radius | Incident commander required            |

The important rule:

> **Autonomy is capped by risk.**

Even if the repo is configured for A6, an R4/R5 change must not silently auto-release.

---

# 5. The core laws of the standard

These should be written in the project README, docs, and policy files.

## The ten laws

1. **Agents never push to protected branches.**
2. **Agents never approve their own changes.**
3. **Policy is evaluated from the protected target branch or protected policy project, not the agent branch.**
4. **No proof, no merge.**
5. **No rollback, no production.**
6. **No exact SHA/digest binding, no trust.**
7. **Build once, promote the same artifact.**
8. **Long-lived secrets are forbidden for agents; use short-lived OIDC/federated credentials.**
9. **Human emergency stop always wins.**
10. **Every autonomous decision creates a signed receipt.**

These are the lines that prevent “responsible autonomy” from degrading into “a bot with admin rights.”

# 6. User control over release agents and approvals

The user needs strong control without drowning in configuration.

## Approval modes

| Mode                       | Meaning                                                                |
| -------------------------- | ---------------------------------------------------------------------- |
| **manual**                 | A human must approve.                                                  |
| **agent_recommend**        | Agent recommends, human approves.                                      |
| **agent_approval_receipt** | Independent agent may approve within a narrow scope.                   |
| **auto_when_policy_green** | System auto-approves only when all evidence and risk constraints pass. |
| **sovereign_autopilot**    | Full delegated autonomy under explicit A6 production policy.           |

## Release-agent design knobs

Users should configure:

| Knob                   | Example                                                       |
| ---------------------- | ------------------------------------------------------------- |
| Release agent identity | `nightwatch-primary`, `nightwatch-security`, `nightwatch-sre` |
| Scope                  | staging only, canary only, production ring 1, full production |
| Max risk               | R1, R2, R3                                                    |
| Required evidence      | tests, SBOM, SLSA provenance, rollback plan, canary health    |
| Required independence  | release agent must not be author, planner, or builder         |
| Rollout style          | manual, canary, progressive, blue/green                       |
| Watch window           | 10m, 30m, 2h                                                  |
| SLO thresholds         | error rate, latency, crash loops, security signal             |
| Escalation             | maintainer → SRE → incident commander                         |
| Emergency stop         | user, maintainer, security, incident commander                |

## `.vibegate/autonomy.yml`

```yaml
version: vibegate.rads/v1

default_profile: A2

production_autonomy:
  allowed: true
  max_profile: A6
  max_risk: R2
  require_independent_release_agent: true
  require_build_once_promote_same_digest: true
  require_rollback_armed: true
  require_canary: true
  emergency_stop_roles:
    - maintainer
    - security
    - sre
    - incident_commander

risk:
  human_required_for:
    - R4
    - R5
    - auth
    - secrets
    - payments
    - pii
    - destructive_db
    - public_api_breaking
    - policy_change

approvals:
  modes:
    low_risk_change:
      mode: auto_when_policy_green
      max_risk: R1
      require:
        - vibelock.green
        - no_policy_regression
        - no_test_weakening
        - no_secret_findings

    production_canary:
      mode: agent_approval_receipt
      max_risk: R2
      reviewer_agent: nightwatch-primary
      require:
        - release_passport.signed
        - artifact.provenance_verified
        - artifact.sbom_present
        - rollback.armed
        - canary.enabled

    critical_change:
      mode: manual
      roles:
        - code_owner
        - security

release:
  rings:
    - name: staging
      strategy: full

    - name: prod-canary
      strategy: canary
      percent: 1
      watch: 30m

    - name: prod-main
      strategy: progressive
      steps: [5, 25, 50, 100]
      watch_each: 30m

  slo_budgets:
    error_rate_max_delta: 0.003
    p95_latency_max_delta: 0.10
    rollback_on:
      - slo_burn
      - crash_loop
      - security_signal
      - manual_stop
```

# 8. Change Passport

A Change Passport is the one object VibeLock judges.

```json
{
  "schema": "vibegate.change_passport.v1",
  "repo": "group/project",
  "provider": "gitlab",
  "mr": 123,
  "source_branch": "agent/fix-checkout-timeout",
  "target_branch": "main",
  "commit_sha": "abc123",
  "autonomy_profile": "A4",
  "risk": "R1",
  "intent_card": "sha256:...",
  "capability_lease": "sha256:...",
  "changed_paths": [
    "apps/web/src/checkout/Checkout.tsx",
    "apps/web/src/checkout/Checkout.test.tsx"
  ],
  "owner_route": ["web", "checkout"],
  "proof_lanes_required": ["unit", "typecheck", "security", "anti_vibe"],
  "proof_lanes_present": ["unit", "typecheck", "security", "anti_vibe"],
  "anti_vibe": {
    "tests_deleted": 0,
    "assertions_weakened": 0,
    "ts_ignore_added": 0,
    "silent_catches_added": 0
  },
  "policy_bundle_sha": "sha256:...",
  "verdict": "merge_allowed",
  "signature": "..."
}
```

VibeLock should return one of five decisions:

| Decision           | Meaning                          |
| ------------------ | -------------------------------- |
| `merge_allowed`    | All required proof exists.       |
| `repair_required`  | Agent may fix and retry.         |
| `human_required`   | Autonomy cap exceeded.           |
| `blocked_policy`   | Policy violation.                |
| `blocked_security` | Security/supply-chain violation. |

    "sha256:release-passport",

# 11. GitLab CI organization

GitLab is the cleanest first-class implementation target because it has strong CI composition, protected branches, protected environments, deployment approvals, merge trains, external status checks, pipeline execution policies, and OIDC support. GitLab supports including local/external YAML in pipelines, typed CI/CD inputs, and reusable CI/CD components, so the CI should be modular rather than one giant file. ([GitLab Docs][20])

Use this structure:

# 12. GitLab enforcement settings

The CI file is not enough. The repository settings must enforce the contract.

GitLab should be configured so:

1. **Protected branches** prevent agents from directly updating `main`.
2. **Code Owner approval** is required for protected/high-risk areas.
3. **External status check** `vibegate/merge-passport` is required before merge.
4. **Merge trains** are used so MRs are tested in the correct queued order.
5. **Protected environments** require release approval for production.
6. **Pipeline execution policies** enforce security and VibeGate jobs even if someone tries to skip CI.
7. **OIDC ID tokens** provide short-lived credentials for artifact signing and cloud deploys.

GitLab documents protected branch and Code Owner approval controls, external status checks for third-party pass/fail gating, merge trains for validating queued merge order, deployment approvals for protected environments, pipeline execution policies, and CI/CD ID tokens/OIDC for external services. ([GitLab Docs][21])

The policy principle:

> The agent may create an MR, but GitLab must be the thing that enforces whether it can merge or deploy.

# 13. GitHub open-source mapping

For GitHub, map the same objects to familiar community terms.

| VibeGate concept      | GitHub equivalent                                         |
| --------------------- | --------------------------------------------------------- |
| Change Passport       | PR evidence summary + required status check               |
| VibeLock              | Required status check / GitHub App check                  |
| Capability Lease      | GitHub App installation token + policy scope              |
| Release Passport      | Deployment evidence artifact                              |
| Nightwatch            | GitHub App + environment protection rule                  |
| Launch Ledger         | Checks, deployments, artifacts, attestations, signed logs |
| Merge train           | Merge queue                                               |
| Protected environment | GitHub Environment protection                             |

For GitHub, use stable required check names:

```text
vibegate/merge-passport
vibegate/release-passport
vibegate/supply-chain
```

Do **not** create dozens of dynamic required check names. Maintainers hate noisy bots and brittle checks. Keep the required check surface small and make the evidence expandable inside the passport.

GitHub supports protected branch required status checks, deployments/environments, custom deployment protection rules, merge queue behavior, and OIDC federation for short-lived cloud credentials. ([GitHub Docs][22])

Open-source social rule:

> The bot should never say “approved by AI.”
> It should say: **“VibeGate evidence complete. Maintainer policy allows merge.”**

That language respects maintainers. It avoids implying that the model is the source of trust.

# 14. Policy-as-code

Use OPA/Rego for the central decision layer.

OPA is a general-purpose policy engine for policy-as-code and can be used as an automated guardrail in CI/CD before production. ([Open Policy Agent][23])

## Example merge policy

```rego
package vibegate.merge

default allow := false

allow if {
  input.passport.schema == "vibegate.change_passport.v1"
  input.passport.verdict == "merge_allowed"
  input.passport.risk in ["R0", "R1"]
  input.autonomy.profile in ["A4", "A5", "A6"]
  input.proofs.unit.status == "pass"
  input.proofs.typecheck.status == "pass"
  input.proofs.security.critical_findings == 0
  input.proofs.anti_vibe.tests_deleted == 0
  input.proofs.anti_vibe.assertions_weakened == 0
  input.proofs.anti_vibe.ts_ignore_added == 0
  not touches_human_required_area
}

touches_human_required_area if {
  some path in input.passport.changed_paths
  path_matches(path, "auth/**")
}

touches_human_required_area if {
  some path in input.passport.changed_paths
  path_matches(path, "payments/**")
}

touches_human_required_area if {
  some path in input.passport.changed_paths
  path_matches(path, ".vibegate/**")
}
```

## Example release policy

```rego
package vibegate.release

default deploy_canary := false
default deploy_full := false

deploy_canary if {
  input.release_passport.schema == "vibegate.release_passport.v1"
  input.release_passport.rollback_plan.armed == true
  input.release_passport.ring == "prod-canary"
  input.release_passport.approval.mode in ["agent_approval_receipt", "manual"]
  input.release_passport.artifact_digest == input.provenance.subject.digest
  input.sbom.present == true
  input.security.critical_findings == 0
  input.autonomy.production_allowed == true
  input.release_passport.risk in ["R0", "R1", "R2"]
}

deploy_full if {
  deploy_canary
  input.nightwatch.canary_status == "healthy"
  input.nightwatch.error_rate_delta <= input.policy.slo_budgets.error_rate_max_delta
  input.nightwatch.p95_latency_delta <= input.policy.slo_budgets.p95_latency_max_delta
}
```

# 15. Supply-chain integrity

For autonomous production, supply-chain proof is not optional.

Minimum required production evidence:

| Evidence              | Purpose                                  |
| --------------------- | ---------------------------------------- |
| **Artifact digest**   | Exact thing being deployed.              |
| **Build provenance**  | Where, when, and how it was built.       |
| **SBOM**              | What dependencies/components are inside. |
| **Signature**         | Who/what signed the artifact.            |
| **Policy bundle SHA** | Which policy decided.                    |
| **Change Passport**   | Why this change was allowed to merge.    |
| **Release Passport**  | Why this artifact was allowed to deploy. |

SLSA defines tracks/levels and recommends attestation formats including provenance, while Sigstore provides signing and verification for artifacts and SBOMs with a transparency-log model. OpenSSF Scorecard provides automated checks for open-source security risks, and NIST’s SSDF provides secure software development practices. ([SLSA][24])

Production rule:

> The release agent does not approve “code.”
> It approves **one signed artifact digest** with a verified Release Passport.

# 16. Progressive delivery and Nightwatch

Nightwatch should not be a vague “AI reviewer.” It should be a release controller with clear responsibilities.

## Nightwatch responsibilities

| Responsibility          | Description                                                      |
| ----------------------- | ---------------------------------------------------------------- |
| Verify Release Passport | Confirm artifact, provenance, SBOM, rollback, risk, policy.      |
| Approve canary          | Allow limited production exposure.                               |
| Watch telemetry         | Observe error rate, latency, crashes, saturation, business KPIs. |
| Promote                 | Move from 1% → 5% → 25% → 50% → 100%.                            |
| Pause                   | Stop rollout when uncertain.                                     |
| Rollback                | Revert to previous digest when SLO/security thresholds fail.     |
| Escalate                | Request human decision when risk or ambiguity exceeds policy.    |

Canary/progressive rollout should be first-class. Argo Rollouts, for example, supports canary release strategies and analysis steps that can abort and send traffic back when analysis fails. ([Argo Rollouts][25])

# 19. Human behavior and open-source etiquette

## Good bot behavior

| Bad behavior                          | Good behavior                                            |
| ------------------------------------- | -------------------------------------------------------- |
| “AI says this is safe.”               | “Required evidence is complete under repository policy.” |

# 20. Anti-vibe rules

Autonomous coding dies when agents learn to “make CI green” by weakening the product. VibeGate should fail closed on anti-vibe behavior.

Block or require human approval for:

```text
deleted tests
weakened assertions
new @ts-ignore / type ignores
silent catch blocks
fake-data fallbacks
TODO/stub shipping paths
snapshot replacement without rendered proof
broad dependency upgrades
public API contract drift
generated file edits without source artifact change
secret-scan findings
permission expansion
policy file changes
```

Bug-fix rule:

> A bug fix should include or reference a failing test, reproduction, trace, or incident evidence.

# 21. Dynamic proof lanes

Do not run everything on every change. That is slow and expensive.

Use changed paths to select proof lanes:

| Changed path    | Required lanes                                |
| --------------- | --------------------------------------------- |
| `docs/**`       | lint docs                                     |
| `src/**`        | unit, typecheck, anti-vibe                    |
| `api/**`        | contract, integration, generated-client check |
| `db/**`         | migration test, rollback test, backup check   |
| `auth/**`       | security, code owner, human approval          |
| `infra/**`      | plan, policy, drift, human/SRE approval       |
| `.vibegate/**`  | policy validation, human maintainer approval  |
| generated files | generated-zone proof from source artifact     |

Dynamic child pipelines are a good fit for this, because the system can generate proof-lane CI based on changed paths and risk. GitLab supports child pipelines and dynamic pipeline configuration generated by a job. ([GitLab Docs][28])

# 25. The final GitLab operating mode

The best GitLab mode is:

```text
Agent opens MR
  ↓
GitLab MR pipeline runs proof lanes
  ↓
VibeLock posts one required external status check
  ↓
Merge train validates final merge result
  ↓
Protected branch accepts merge only when required checks pass
  ↓
Foundry builds and signs immutable artifact
  ↓
Release Passport created
  ↓
Protected environment blocks prod unless policy/human/agent receipt allows it
  ↓
Nightwatch deploys canary
  ↓
Telemetry decides promote / pause / rollback
  ↓
Launch Ledger records everything
```

This is fast because most changes stop at one stable check. It is safe because all evidence is hidden behind that check. It is understandable because humans only need to learn three gates.

# 27. Final canonical standard

The final standard should be stated like this:

> **VibeGate RADS v1.0 is a responsible autonomous delivery standard for agent-authored software. It allows autonomous coding all the way to production only through explicit autonomy profiles, temporary capability leases, evidence-gated merge, signed release passports, independent agent or human approval, progressive rollout, rollback readiness, and append-only audit.**

The shortest possible version:

```text
Lease the work.
Prove the change.
Sign the artifact.
Gate the release.
Watch production.
Record everything.
```

That is the best design: simple enough to understand, strict enough to trust, flexible enough for GitLab/GitHub, and powerful enough to support full autonomous production when the user deliberately enables it.
