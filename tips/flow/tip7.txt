Build the new TUI around a default view called **Flow Mission Control**. The current dashboard answers “is the system alive?” The new default should answer, at a glance:

**What version are we shipping? Where is it in the Git → CI → release flow? What is running now? What is blocked? What is pending? What is the ETA, and why?**

## 1. Product concept: Flow Mission Control

The core visual should be a **left-to-right execution graph**, not a job list. Treat GitLab, runner pools, tests, canary, gates, secrets, and production promotion as one observable delivery DAG.

The CEO-level mental model:

```text
Git change  →  Admission  →  Pipeline  →  Build  →  Parallel tests  →  Evidence gates  →  Canary  →  Prod
```

Parallel work should stack vertically inside each stage. The current critical path should be visually dominant. Everything pending should be categorized as either **queued**, **blocked by dependency**, **waiting for runner**, **manual gate**, **retrying**, **failed**, **superseded**, or **release-gated**.

A good first frame should look closer to this:

## 6. ETA: compute the critical path, not a naive average

The ETA should be based on **remaining critical path duration**. Never sum all parallel jobs. For a stage with parallel jobs, the stage ETA is the longest remaining job in that stage. For a DAG with dependencies, compute the longest remaining path.

### ETA inputs

Use these sources:

| Data                     | Source                                            |
| ------------------------ | ------------------------------------------------- |
| Active job status        | `GitlabClient::list_pipeline_jobs()`              |
| Job queue duration       | `job_events.queued_duration`, GitLab job metadata |
| Job state timestamps     | `job_events.received_at`, GitLab job timestamps   |
| Historical test duration | `test_executions`, past pipeline jobs             |
| Failure classification   | `evidence_capsules`, `retry_decisions`            |
| Runner pressure          | `pools`, `managers`, pending job count            |
| Release gate duration    | `release_attempts` history                        |
| Cache pressure           | `cache_requests`, `cache_metrics`                 |

### ETA model

Implement:

```rust
#[derive(Clone, Debug)]
pub struct EtaEstimate {
    pub remaining: std::time::Duration,
    pub expected_finish_at: chrono::DateTime<chrono::Utc>,
    pub confidence: EtaConfidence,
    pub basis: EtaBasis,
}

#[derive(Clone, Debug)]
pub enum EtaConfidence {
    High,
    Medium,
    Low,
    Unknown,
}

#[derive(Clone, Debug)]
pub enum EtaBasis {
    HistoricalSamples { p50_ms: i64, p80_ms: i64, p90_ms: i64, samples: usize },
    RunningElapsed { elapsed_ms: i64 },
    QueueOnly { queue_p80_ms: i64 },
    DefaultByStage { stage: String },
    Unknown,
}
```

Per-job estimate:

```text
running remaining =
  max(predicted_runtime_p80 - elapsed_runtime, minimum_visible_remaining)

pending/queued remaining =
  predicted_queue_p80(pool, tags) + predicted_runtime_p80(job_name, stage)

blocked remaining =
  max(predecessor remaining) + predicted_runtime_p80(job)

manual remaining =
  unknown unless configured; display "manual gate"
```

Pipeline ETA:

```text
ETA = longest remaining path from current incomplete nodes to terminal release node
```

Pseudo-code:

```rust
pub fn compute_critical_path(graph: &FlowGraph) -> CriticalPathResult {
    let order = topo_sort(&graph.nodes, &graph.edges);

    let mut finish: HashMap<FlowNodeId, Duration> = HashMap::new();
    let mut previous: HashMap<FlowNodeId, FlowNodeId> = HashMap::new();

    for node in order {
        let own = remaining_duration(node);
        let best_pred = predecessors(node)
            .map(|p| finish.get(&p.id).copied().unwrap_or_default())
            .max()
            .unwrap_or_default();

        finish.insert(node.id.clone(), best_pred + own);

        if let Some(pred) = predecessor_with_max_finish(node) {
            previous.insert(node.id.clone(), pred.id.clone());
        }
    }

    let terminal = finish.iter().max_by_key(|(_, d)| *d).map(|(id, _)| id.clone());
    backtrack_path(terminal, previous)
}
```

Display ETA with honesty:

```text
ETA 06m20s p80          enough samples
ETA ~08m low confidence few samples
ETA manual gate         human action required
ETA unknown             missing timestamps or no historical data
```

### Current blocker

A node is the current blocker when:

1. It is incomplete.
2. It lies on the critical path.
3. All predecessors are complete, or it is the first incomplete dependency.
4. It controls the earliest next flow transition.

Examples:

```text
build-enclave-server running 99%
unit-public queued: waiting for default pool
remote-gate blocked: canary not launched
prod-promotion manual: canary not fully passed
```

## 7. Rust TUI technology choice

Keep the stack as **Ratatui + Crossterm + Tokio**, and do not introduce a large UI framework. Ratatui is a strong fit because it is immediate-mode: each frame rebuilds the widgets from app state, and the application owns input handling and event/state updates. ([Ratatui][1]) Ratatui already provides the core widgets needed for this design, including tables, gauges, charts, lists, progress-style visualizations, and constraint-based dynamic layout. ([Ratatui][2])

Use **Crossterm** for keyboard, mouse, resize, raw mode, and alternate-screen terminal control. Its event module supports keyboard, mouse, and resize events, and its docs explicitly warn against reading events from multiple threads or mixing `read`/`poll` with `EventStream`, so the TUI should have exactly one input reader. ([Docs.rs][3])

Use **Tokio** for the async event loop and data refresh tasks. `tokio::select!` is appropriate for waiting on multiple concurrent branches, such as input events, ticks, snapshot updates, action responses, and shutdown signals; blocking work should stay out of those branches because the futures run concurrently on the current task, not in parallel. ([Docs.rs][4])

Use Ratatui’s **Canvas** only for optional mini-maps or high-level sparkline-style flow previews. The primary graph should be a custom `StatefulWidget` drawing directly into the `Buffer`, because crisp node boxes, labels, edge elbows, selection borders, and keyboard navigation are easier to control cell-by-cell. Ratatui Canvas can draw lines, rectangles, points, labels, and custom shapes, but its Braille/default marker behavior is better suited to shape drawing than dense job cards. ([Docs.rs][5])

### Event loop

Use one input task and one app loop:

```rust
pub enum TuiEvent {
    Tick,
    RenderTick,
    Input(crossterm::event::Event),
    Snapshot(TuiStateSnapshot),
    ActionResult(ActionResult),
    LogTail(LogTailUpdate),
    Shutdown,
}
```

The loop shape:

```rust
loop {
    tokio::select! {
        _ = render_tick.tick() => {
            app.on_event(TuiEvent::RenderTick)?;
            terminal.draw(|f| ui::render(f, &mut app))?;
        }

        _ = data_tick.tick() => {
            snapshot_tx.send(fetch_snapshot_request()).await?;
        }

        Some(snapshot) = snapshot_rx.recv() => {
            app.on_event(TuiEvent::Snapshot(snapshot))?;
        }

        Some(input) = input_rx.recv() => {
            app.on_event(TuiEvent::Input(input))?;
        }

        Some(result) = action_rx.recv() => {
            app.on_event(TuiEvent::ActionResult(result))?;
        }

        _ = shutdown.cancelled() => break,
    }
}
```

Render at a capped cadence, such as 10 FPS, while data refreshes at slower intervals. The running spinner can tick every 100 ms without forcing expensive GitLab calls.


### Grouping rule

When a column has too many nodes to fit:

```text
visible_count > available_rows / node_height
```

collapse into a group node:

```rust
FlowNodeKind::StageGroup {
    children: Vec<FlowNodeId>,
}
```

Auto-expand rules:

1. Always expand the critical-path stage.
2. Always expand a failed stage.
3. Always expand the selected stage.
4. Collapse non-blocking stages in Executive zoom.

### Edge routing

Draw edges before nodes. Use elbow routes:

```text

```

Route critical-path edges last or with heavier glyphs so they dominate.

## 12. Data builder

Create `FlowGraphBuilder`.

```rust
pub struct FlowGraphBuilder<'a> {
    pub db: &'a state::Db,
    pub gitlab: &'a GitlabClient,
    pub project_id: i64,
    pub ref_name: String,
    pub selected_pipeline_id: Option<i64>,
}

impl<'a> FlowGraphBuilder<'a> {
    pub async fn build(&self) -> anyhow::Result<FlowGraph> {
        // 1. Load selected/latest pipeline
        // 2. Load jobs from GitLab
        // 3. Load job events from RedlineDB
        // 4. Load release attempt
        // 5. Load evidence/retry records
        // 6. Load impact/supersedence/admission events
        // 7. Build canonical nodes/edges
        // 8. Compute ETA and critical path
        // 9. Return immutable FlowGraph
    }
}
```

### Data refresh cadence

| Data                         |                                    Cadence |
| ---------------------------- | -----------------------------------------: |
| DB pools/managers/job events |                                         1s |
| Active pipelines             |                                         2s |
| Selected pipeline jobs       |              2s while active, 5s when idle |
| GitLab traces                | only selected job, 2s while log panel open |
| Docker manager logs          |              only selected manager/job, 3s |
| Cache metrics                |                                         5s |
| Release status               |                                         5s |
| Historical ETA model         |                  30s or on pipeline finish |

Do not fetch every trace for every job. Only fetch traces on selection or failed jobs.

## 13. How to infer lanes

Implement a classifier:

```rust
pub fn classify_job_lane(job: &JobView) -> FlowLaneKind {
    let name = job.name.to_lowercase();
    let stage = job.stage.to_lowercase();
    let tags = job.tags.join(",").to_lowercase();

    if stage.contains("release") || name.contains("canary") || name.contains("promote") {
        return FlowLaneKind::ReleaseExecution;
    }

    if name.contains("research") || tags.contains("research") {
        return FlowLaneKind::Research;
    }

    if name.contains("extended") || tags.contains("extended") {
        return FlowLaneKind::Extended;
    }

    if name.contains("integration") || name.contains("e2e") || name.contains("api") {
        return FlowLaneKind::Integration;
    }

    if name.contains("unit") || name.contains("test") {
        return FlowLaneKind::Unit;
    }

    if name.contains("build") || name.contains("compile") || name.contains("docker") || name.contains("enclave") {
        return FlowLaneKind::ReleaseCritical;
    }

    FlowLaneKind::ReleaseCritical
}
```

Use the release schema when available. Classifier fallback should only be used when schema metadata is absent.

## 16. Actions mapped to existing API

The TUI should call internal Rust functions, not shell out to the CLI.

| UI action         | Internal API                                             |
| ----------------- | -------------------------------------------------------- |
| Retry job         | `GitlabClient::retry_job(project_id, job_id)`            |
| Cancel job        | `GitlabClient::cancel_job(project_id, job_id)`           |
| Play manual job   | `GitlabClient::play_job(project_id, job_id)`             |
| Cancel pipeline   | `GitlabClient::cancel_pipeline(project_id, pipeline_id)` |
| Job logs          | `GitlabClient::job_trace(project_id, job_id)`            |
| Failure capsule   | `Db::latest_evidence_for_job(project_id, job_id)`        |
| Pipeline explain  | existing `explain.rs` logic                              |
| Pipeline doctor   | existing pipeline doctor logic                           |
| Release status    | `release::status` logic                                  |
| Promote prod      | `release::promote_prod` logic, with confirmation         |
| Pool pause/resume | pool manager functions                                   |
| Runner logs       | `DockerClient::manager_logs(manager_id, lines)`          |

Destructive actions should require confirmation:

```text

```

Production promotion should require a stronger confirmation:

```text
Promote ci-54c658aac9c3 to production? type promote to confirm.
```

## 20. Performance constraints

The renderer must not degrade under large pipelines.

Rules:

```text
Only render visible nodes.
Only route visible edges.
Never fetch all logs.
Never re-layout if graph topology did not change.
Use stable node IDs so selection survives refresh.
Debounce GitLab refreshes.
Cap rendering at 10 FPS.
Cap data snapshots at 1–2 FPS.
```

Store layout cache:

```rust
pub struct FlowLayoutCache {
    pub graph_fingerprint: String,
    pub zoom: FlowZoom,
    pub terminal_size: (u16, u16),
    pub layout: FlowLayout,
}
```

Graph fingerprint:

```text
pipeline_id + sha + job_id/status/version list hash
```

## 22. Acceptance criteria

The implementation is done when these are true:

1. Opening `jeryu tui` defaults to the Flow view.
2. The latest active pipeline is selected automatically.
3. The graph shows Git → admission → pipeline → stage/test → release gates left-to-right.
4. Parallel jobs in the same stage are stacked vertically.
5. The active critical path is highlighted.
6. Header shows release version, phase, ETA, confidence, and current blocker.
7. Pending jobs are grouped by reason.
8. Selecting a node updates the inspector.
9. `Enter` drills into stages/jobs/release gates.
10. `l`, `e`, `r`, `c`, `p`, and `x` perform the expected job/pipeline actions.
11. `--once` renders a deterministic frame for CI smoke tests.
12. Small terminals fall back to critical-path list mode.
13. No background task fetches traces for unselected successful jobs.
14. Pipeline supersedence is visible.
15. Release canary and production gates are visible in the same flow.
