Below is the build spec I would hand to an implementation agent. The core idea is: **replace the current pane-heavy dashboard default with a left-to-right “Flow Board” that reads like a live Git/CI/release railway map.** The old dashboard can stay as Tab 2, but Tab 1 should become the executive flow view.

---

# jeryu TUI “Flow Board” Specification

## 1. Product goal

When the TUI opens, the CEO-level question should be answered immediately:

> “What changed, where is it in the delivery flow, what is blocking it, what is running now, what is pending, and when do we expect it to finish?”

The view should not feel like a list of jobs. It should feel like a **living delivery graph**:

```text
commit / branch → admission → impact analysis → pipeline → build → parallel tests → gates → canary → production
```

Parallel jobs should be stacked vertically under the same stage. Sequential flow should move left to right. The current blocker should be impossible to miss.

---

Use the current top bar, but make the main screen a full-width flow chart.

## 3. Main view modes

The TUI should have four top-level modes:

| Mode                | Key | Purpose                                        |
| ------------------- | --: | ---------------------------------------------- |
| **Flow Board**      | `1` | Default left-to-right delivery graph           |
| **Pipeline Detail** | `2` | Dense job table, sortable/filterable           |
| **Infra / Pools**   | `3` | Current runner pools, Docker managers, cache   |
| **Logs / Capsules** | `4` | Trace tails, failure capsules, retry decisions |

The Flow Board remains the command center. The other modes are drill-down surfaces.

## 5. Standard left-to-right flow

The graph should always use a stable delivery vocabulary, even when the actual GitLab stages vary.

### 5.1 Canonical columns

| Column           | Meaning                                     | Source                                      |
| ---------------- | ------------------------------------------- | ------------------------------------------- |
| **Source**       | Push, branch, MR, agent branch, jeryu sync | push hook, tracked pipeline, agent metadata |
| **Admission**    | Server hook / policy / trust tier           | admission events, event ledger              |
| **Impact**       | Full, Unit, Integration, DocsOnly           | impact analysis event                       |
| **Pipeline**     | GitLab pipeline status                      | `tracked_pipelines`, GitLab API             |
| **Prepare**      | cache, sandbox, dependency setup            | jobs with prepare/setup stage               |
| **Build**        | compile, docker build, package build        | job stage/name/tags                         |
| **Tests**        | unit, integration, e2e, smoke, local RC     | job stage/name/tags                         |
| **Supply Chain** | honeypot, witness, taint, cache verdict     | capsule/cache/witness tables                |
| **Package**      | artifacts/images                            | jobs/artifacts                              |
| **Canary**       | release canary launch                       | `release_attempts`                          |
| **Gates**        | remote, telemetry, e2e gates                | release manager diagnostics                 |
| **Production**   | promote-prod state                          | release manager                             |
| **Ledger**       | evidence, replay, final audit               | events, capsules, secret audit              |

Not every run will have every column. Empty columns should be hidden unless the user toggles `A` for “all columns.”

### 5.2 Job grouping

Jobs are grouped by:

1. Pipeline stage order.
2. Lane classification.
3. Blocking status.
4. Running before pending.
5. Longest ETA before shorter ETA.

The UI must avoid a flat job list. Instead:

```text
Build column
┌────────────────────┐
│ ● build-enclave    │
│ 99% ETA :20        │
└────────────────────┘
┌────────────────────┐
│ ✓ lint             │
│ 1m12s              │
└────────────────────┘

Tests column
┌────────────────────┐
│ ○ test-local-rc    │
│ starts ~:20        │
├────────────────────┤
│ ○ publish-dry-run  │
│ starts ~:20        │
├────────────────────┤
│ ○ test-live-public │
│ starts ~:20        │
└────────────────────┘
```

Parallel tests should stack vertically inside the same column, not spread across the whole width.

## 6. Lane model

Use visible swimlanes inside columns, but keep them compact.

| Lane                  | Purpose                                  |
| --------------------- | ---------------------------------------- |
| **Mainline**          | push, admission, pipeline creation       |
| **Release-critical**  | blocking build/test jobs                 |
| **Extended**          | non-blocking but useful validation       |
| **Research**          | experimental, allow-failure, exploratory |
| **Release-execution** | canary, gates, prod                      |
| **Infra**             | runner, cache, sandbox, queue state      |
| **Audit**             | capsules, retry, witness, secret audit   |

A CEO mode should default to showing **Mainline + Release-critical + Release-execution**. Engineer mode shows all lanes.

Key toggle:

```text

b  Blocking-only
a  All lanes / canonical lanes
```

## 7. “Where are we?” logic

The Flow Board should calculate a single `current_node_id`.

Priority order:

1. First failed blocking node.
2. First running blocking node on the critical path.
3. First blocked gate.
4. First pending blocking node whose dependencies are satisfied.
5. Latest completed node if the run is green.
6. Release gate if CI is green but release is waiting.

```rust
pub fn choose_current_node(graph: &FlowGraph) -> Option<FlowNodeId> {
    graph.nodes.iter()
        .filter(|n| n.is_blocking)
        .find(|n| matches!(n.status, FlowStatus::Failed { .. }))
        .or_else(|| graph.nodes.iter()
            .filter(|n| n.is_blocking && n.is_critical_path)
            .find(|n| matches!(n.status, FlowStatus::Running)))
        .or_else(|| graph.nodes.iter()
            .filter(|n| n.is_blocking)
            .find(|n| matches!(n.status, FlowStatus::Blocked { .. })))
        .or_else(|| graph.nodes.iter()
            .filter(|n| n.is_blocking)
            .find(|n| matches!(n.status, FlowStatus::Pending | FlowStatus::Manual)))
        .map(|n| n.id.clone())
}
```

The selected/current node should be rendered with the strongest visual treatment. Downstream pending nodes should be dimmed and annotated with the reason:

```text
blocked by build-enclave-server
waiting for manual gate
waiting for canary evidence
waiting for runner capacity
superseded by #408
```

## 8. ETA engine

The ETA system should be conservative, explainable, and visible.

### 8.1 User-facing ETA fields

Header should show:

```text
ETA green: 08m12s
ETA canary: 14m30s
ETA prod: waiting on approval
Confidence: B
```

Node cards should show:

```text
● build-enclave
99% ETA :20
```

Pending cards should show:

```text
○ test-local-rc
starts ~:20
ETA 3m40s
```

Stage headers should show:

```text
Tests: 0/4 passed, ETA 04m10s
```

### 8.2 ETA calculation

Use a critical-path model:

```rust
pipeline_remaining =
    sum_over_sequential_columns(
        max_remaining_of_parallel_blocking_nodes_in_column
    )
```

For each job:

```rust
running_remaining =
    max(p80_historical_duration - elapsed, minimum_display_remaining)

pending_remaining =
    queue_wait_estimate + p80_historical_duration

done_remaining = 0
failed_remaining = unknown until action
manual_remaining = unknown until played
```

### 8.3 Historical samples

The existing `job_events` table can derive duration if it has `running` and terminal status events for the same `job_id`. Use:

```text
job_id + status + received_at
```

Derive:

```text
queue time = pending/created → running
run time   = running → success/failed/canceled
```

Use the sample key:

```rust
pub struct JobDurationKey {
    pub project_id: i64,
    pub job_name: String,
    pub stage: String,
    pub lane: FlowLaneKind,
    pub pool_name: Option<String>,
}
```

Use last 30 successful samples when possible.

| Samples | Confidence             |
| ------: | ---------------------- |
|     20+ | A                      |
|    5–19 | B                      |
|     1–4 | C                      |
|       0 | Unknown / elapsed-only |

### 8.4 Recommended DB addition

For best ETA quality, add a compact table populated by webhook processing:

```sql
CREATE TABLE IF NOT EXISTS job_duration_samples (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    pipeline_id INTEGER,
    job_id INTEGER NOT NULL,
    job_name TEXT NOT NULL,
    stage TEXT NOT NULL,
    lane TEXT NOT NULL,
    pool_name TEXT,
    runner_system_id TEXT,
    status TEXT NOT NULL,
    queued_ms INTEGER,
    run_ms INTEGER,
    created_at TEXT NOT NULL,
    finished_at TEXT
);

CREATE INDEX IF NOT EXISTS idx_job_duration_samples_key
ON job_duration_samples(project_id, job_name, stage, lane, pool_name, finished_at);
```

This table is not required for MVP, but it makes ETA much better than estimating from current elapsed time only.

### 8.5 Queue ETA

Approximate pool capacity:

```rust
effective_pool_slots =
    active_managers * min(pool.concurrent, pool.request_concurrency)
```

For pending jobs assigned to a pool:

```rust
queue_wait =
    jobs_ahead_in_same_pool / effective_pool_slots * median_job_duration_for_pool
```

If the pool has zero active managers, show:

```text
blocked: no active runner in pool build
```

If the pool is paused:

```text
blocked: pool build paused
```

## 9. Data collection pipeline

Do **not** call GitLab or Docker from the render function. Build a background snapshot actor.

### 9.1 Collector loop

```rust
pub async fn run_flow_collector(
    db: Db,
    gitlab: GitlabClient,
    docker: DockerClient,
    selected: watch::Receiver<RunSelection>,
    tx: watch::Sender<Arc<FlowSnapshot>>,
    action_rx: mpsc::Receiver<TuiAction>,
) -> anyhow::Result<()> {
    let mut interval = tokio::time::interval(Duration::from_millis(1000));

    loop {
        tokio::select! {
            _ = interval.tick() => {
                let raw = collect_raw_snapshot(&db, &gitlab, &docker, &selected).await?;
                let graph = FlowBuilder::new().build(raw)?;
                let graph = EtaEstimator::new(&db).annotate(graph).await?;
                let snapshot = Arc::new(FlowSnapshot::from_graph(graph));
                let _ = tx.send(snapshot);
            }
            Some(action) = action_rx.recv() => {
                execute_tui_action(action, &db, &gitlab).await?;
            }
        }
    }
}
```

Use a `watch` channel for the latest snapshot because the UI only needs the newest state. Use `mpsc` for user actions. Tokio documents `watch` as keeping only the most recent value for receivers, while `mpsc` supports many values from multiple producers to one consumer. ([tokio.rs][1])

### 9.2 Raw sources

Collect:

| Source          | Calls                                                                                                                                     |
| --------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| SQLite          | `list_pools`, `list_managers`, `recent_job_events`, `recent_tracked_pipelines`, `recent_release_attempts`, `recent_events`, cache metrics |
| GitLab          | `get_pipeline`, `list_pipelines`, `list_pipeline_jobs`, selected `job_trace` only                                                         |
| Docker          | managed containers, runner liveness                                                                                                       |
| Release manager | status/reconcile data                                                                                                                     |
| Cache           | `/cache/summary` or `Db::get_cache_metrics()`                                                                                             |
| Evidence        | `latest_evidence_for_job` for selected/failed jobs                                                                                        |

### 9.3 Refresh rates

| Data                         |                           Refresh |
| ---------------------------- | --------------------------------: |
| Active pipeline jobs         |         1s while running, 5s idle |
| Selected job trace           |                                2s |
| Runner pool/container status |                                2s |
| Release status               | 2s while release active, 10s idle |
| Cache summary                |                                5s |
| Historical ETA samples       |     30s or on pipeline completion |

## 10. Rust TUI technology choice

Keep the existing stack: **Ratatui + Crossterm + Tokio**.

Ratatui is suitable here because it is immediate-mode: every frame renders the complete UI, and `Terminal::draw` performs a diff so only changes are written to the terminal. It also has built-in layout primitives for splitting the terminal into responsive regions. ([Docs.rs][2])

Use Crossterm for keyboard, mouse, focus, and resize events. Its `poll`/`read` model allows non-blocking input loops, but those calls should stay on the same thread and raw mode must be enabled for keyboard behavior. ([Docs.rs][3])

Ratatui 0.30 also keeps the main `ratatui` crate as the recommended application crate and defaults to the Crossterm backend family, which matches your current architecture. ([Docs.rs][2])

## 13. Custom FlowGraph widget

Do not try to build the flow graph from only `Table` and `List`. Implement a custom widget that writes directly to the Ratatui buffer.

```rust
pub struct FlowGraphWidget<'a> {
    pub graph: &'a FlowGraph,
    pub theme: &'a FlowTheme,
}

impl<'a> ratatui::widgets::StatefulWidget for FlowGraphWidget<'a> {
    type State = FlowViewState;

    fn render(
        self,
        area: ratatui::layout::Rect,
        buf: &mut ratatui::buffer::Buffer,
        state: &mut Self::State,
    ) {
        let layout = compute_flow_layout(area, self.graph, state);

        draw_edges(buf, &layout, self.graph, self.theme, state);
        draw_nodes(buf, &layout, self.graph, self.theme, state);
        draw_column_headers(buf, &layout, self.graph, self.theme);
        draw_minimap_or_scrollbars(buf, area, &layout, state);

        state.hit_map = layout.hit_boxes;
    }
}
```

### 13.1 Why custom buffer rendering?

A flow chart needs:

* Orthogonal arrows.
* Node boxes with variable sizes.
* Stacked parallel jobs.
* Horizontal and vertical scrolling.
* Hit detection for mouse and keyboard.
* Critical-path highlighting.
* Collapsed stack cards.
* Terminal-size responsive layout.

This is much easier with a custom widget than with nested lists.

### 13.2 Glyph set

Use Unicode by default, ASCII fallback with `--ascii`.

## 14. Layout algorithm

The graph is not an arbitrary DAG. It is a **layered delivery graph**, so layout can be deterministic and fast.

### 14.1 Column layout

```rust
column_width = match zoom {
    Executive => 18,
    Normal => 22,
    Dense => 14,
};

column_gap = 2;
x(column_n) = left + n * (column_width + column_gap) - scroll_x;
```

### 14.2 Node layout

Within each column:

1. Group by lane.
2. For each lane, sort nodes.
3. If too many nodes fit, collapse into a stack card.
4. Preserve the selected/current/failed node even when collapsing.

```rust
pub struct FlowLayout {
    pub node_rects: HashMap<FlowNodeId, Rect>,
    pub edge_routes: HashMap<FlowEdgeId, EdgeRoute>,
    pub hit_boxes: Vec<NodeHitBox>,
    pub virtual_width: u16,
    pub virtual_height: u16,
}
```

### 14.3 Stack collapse

If a column has more nodes than visible height:

```text
┌─ Tests ───────────┐
│ ○ test-local-rc   │
├───────────────────┤
│ ○ publish-dry-run │
├───────────────────┤
│ +7 pending jobs   │
│ ETA max 06m20s    │
└───────────────────┘
```

Press `x` or `Enter` on the stack to expand.

### 14.4 Edge routing

Draw edges before nodes.

Use orthogonal routing:

```text
node A right midpoint → horizontal → vertical if needed → horizontal → node B left midpoint
```

Critical-path edges should use bold/bright styling. Non-critical edges should be dim.

## 17. Action palette

Write actions should go through an explicit action palette. The TUI should be safe by default.

```rust
pub enum TuiAction {
    RetryJob { project_id: i64, job_id: i64 },
    CancelJob { project_id: i64, job_id: i64 },
    PlayJob { project_id: i64, job_id: i64 },
    ReconcileRelease { project_id: i64, ref_name: String },
    PromoteProd { project_id: i64, ref_name: String, version: Option<String> },
    PausePool { name: String },
    ResumePool { name: String },
    DrainPool { name: String },
}
```

Recommended safety rule:

```text
Read-only actions: always enabled.
Write actions: require `jeryu tui --actions` or `JERYU_TUI_ACTIONS=1`.
Destructive actions: require confirmation modal.
```

Confirmation modal should show the exact equivalent CLI command:

```text
Confirm cancel?

jeryu job cancel 2 12345

Press y to confirm, Esc to abort.
```

All write actions should append to the event ledger.

### 22.1 Job lane classification

```rust
pub fn classify_job_lane(job: &GitlabJob) -> FlowLaneKind {
    let name = job.name.to_ascii_lowercase();
    let stage = job.stage.to_ascii_lowercase();

    if stage.contains("release") || name.contains("canary") || name.contains("promote") {
        return FlowLaneKind::ReleaseExecution;
    }

    if name.contains("research") || job.allow_failure.unwrap_or(false) {
        return FlowLaneKind::Research;
    }

    if name.contains("extended") || name.contains("nightly") {
        return FlowLaneKind::Extended;
    }

    if name.contains("docs") || stage == "docs" {
        return FlowLaneKind::Extended;
    }

    FlowLaneKind::ReleaseCritical
}
```

### 22.2 Job column classification

Prefer real GitLab stage order. If unavailable, use name/stage heuristics:

```rust
pub fn classify_job_column(job: &GitlabJob) -> FlowColumnKind {
    let name = job.name.to_ascii_lowercase();
    let stage = job.stage.to_ascii_lowercase();

    if stage.contains("prepare") || name.contains("prepare") || name.contains("cache") {
        FlowColumnKind::Prepare
    } else if stage.contains("build") || name.contains("build") || name.contains("compile") {
        FlowColumnKind::Build
    } else if stage.contains("test") || name.contains("test") || name.contains("e2e") {
        FlowColumnKind::Tests
    } else if stage.contains("package") || name.contains("artifact") || name.contains("image") {
        FlowColumnKind::Package
    } else if stage.contains("release") || name.contains("canary") {
        FlowColumnKind::Release
    } else {
        FlowColumnKind::Stage(job.stage.clone())
    }
}
```

## 29. Event loop architecture

Keep input and rendering responsive.

```rust
pub async fn run_tui(mut app: App) -> anyhow::Result<()> {
    let mut terminal = ratatui::init();
    let result = run_tui_loop(&mut terminal, &mut app).await;
    ratatui::restore();
    result
}

async fn run_tui_loop(
    terminal: &mut ratatui::DefaultTerminal,
    app: &mut App,
) -> anyhow::Result<()> {
    let mut render_tick = tokio::time::interval(Duration::from_millis(100));
    let mut snapshot_rx = app.snapshot_rx.clone();

    loop {
        tokio::select! {
            _ = render_tick.tick() => {
                drain_input_events(app)?;
                if snapshot_rx.has_changed().unwrap_or(false) {
                    app.latest_snapshot = Some(snapshot_rx.borrow_and_update().clone());
                }
                terminal.draw(|f| draw(f, app))?;
            }
            _ = snapshot_rx.changed() => {
                app.latest_snapshot = Some(snapshot_rx.borrow_and_update().clone());
                terminal.draw(|f| draw(f, app))?;
            }
        }

        if app.should_quit {
            break;
        }
    }

    Ok(())
}
```

Input should use Crossterm on one thread. Do not mix `read`/`poll` from multiple threads.

## 30. Acceptance criteria

The implementation is successful when:

1. On launch, Tab 1 shows a left-to-right flow graph for the latest active `main` pipeline.
2. Parallel jobs are stacked vertically inside their stage.
3. The current blocker/current running job is highlighted and named in the footer.
4. The header shows ETA to green and ETA to canary when applicable.
5. Pressing `Enter` on any node opens details.
6. Pressing `t` on a job shows trace tail.
7. Failed jobs show capsule classification and retry recommendation.
8. Superseded pipelines are visible but dim.
9. Pending jobs show why they are pending: dependency, runner capacity, manual gate, or blocked release evidence.
10. The UI remains responsive with 100+ jobs.
11. No GitLab/Docker call happens in the render function.
12. The TUI still supports `--once` for smoke testing.
