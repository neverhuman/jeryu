## Canonical rule

**Every change enters through a PR. Every PR is tested in the merge queue. Only `main` produces release artifacts. Artifacts are built once, signed/attested once, and promoted by digest through environments. No rebuilds per environment. No human bypass.**

That is the whole system.

## The nonnegotiable workflow

```text
branch -> PR -> fast CI -> review/agent-review -> merge queue -> full CI -> main
main -> build once -> sign/attest/SBOM -> local smoke -> dev -> prod-canary -> prod-full
rollback -> redeploy previous signed digest, never rebuild
```

GitHub branch protection/rulesets support the core enforcement knobs: require PRs, status checks, signed commits, linear history, merge queue, successful deployments, no bypass, and restricted pushes. Merge queue is especially important for noisy repos because it retests PRs against the latest target branch and queued changes before merge. ([GitHub Docs][1])

# 1. Repository shape: one trunk, no options

**Allowed branches**

```text
main                      protected trunk
agent/<agent>/<ticket>    agent branches
human/<user>/<ticket>     human branches
release/*                 forbidden unless generated by release automation
```

**Allowed merge strategy**

```text
squash merge only
linear history required
merge commits forbidden
rebase-to-main allowed only before PR/queue
```

Linear history keeps reverts clean and prevents branch archaeology; GitHub rulesets explicitly support requiring linear history. ([GitHub Docs][2])

**Allowed releases**

```text
Only main can release.
Only CI can tag.
Only signed/attested artifacts can deploy.
Only immutable digests deploy.
```

No feature branch deploys to prod. No manual releases from laptops. No “I rebuilt it locally and pushed it.”

# 2. Main branch ruleset

For `main`, set this and do not allow bypass:

```yaml
main_rules:
  require_pull_request: true
  required_approvals: 0_or_1
  require_code_owner_review_for_sensitive_paths: true
  dismiss_stale_reviews_on_new_commits: true
  require_status_checks: true
  require_merge_queue: true
  require_linear_history: true
  require_signed_commits: true
  require_conversation_resolution: true
  require_deployments_to_succeed: optional_for_pre_merge_staging_only
  restrict_pushes: ci_release_bot_only_if_needed
  allow_force_pushes: false
  allow_deletions: false
  allow_admin_bypass: false
```

GitHub’s protected branch settings include required PR reviews, required status checks, signed commits, linear history, merge queue, required deployments, no bypass, push restrictions, and force-push/delete controls. Required status checks must pass before merge, and “do not allow bypassing” applies restrictions even to admins when enabled. ([GitHub Docs][1])

**Approval rule**

```text
0 approvals:
  allowed only for low-risk PRs where:
    CI passes
    merge queue passes
    agent review passes
    no sensitive files changed
    no dependency boundary changed
    no security/deploy/workflow files changed

1 approval:
  required for everything else

2+ approvals:
  forbidden as a default; use only for exceptional regulated repos
```

Sensitive files require human/code-owner review:

```text
.github/**
.github/workflows/**
.github/actions/**
Dockerfile*
docker/**
k8s/**
helm/**
terraform/**
infra/**
scripts/deploy*
package manager manifests
lockfiles, if dependency changes matter
security policy files
authn/authz code
crypto/secrets/config paths
```

GitHub specifically recommends using `CODEOWNERS` to control workflow-file changes, because workflow files are security-critical. ([GitHub Docs][3])

# 3. Secrets policy: secrets never enter Git

Use four layers. All four are mandatory.

```text
Layer 1: local pre-commit secret scan
Layer 2: local pre-push secret scan
Layer 3: server-side push protection
Layer 4: CI full repo/history secret scan
```

GitHub push protection is designed to block hardcoded credentials before they reach the repository, including command-line pushes, GitHub UI commits, file uploads, REST API writes, and GitHub MCP interactions for public repositories. ([GitHub Docs][4])

**Secret rules**

```text
No .env in git.
No private keys in git.
No cloud keys in GitHub secrets if OIDC is possible.
No deploy tokens in workflow YAML.
No secrets printed in logs.
No secret exceptions without security owner approval.
Any leaked secret is considered compromised and rotated immediately.
```

For deployments, use OIDC instead of long-lived cloud credentials. GitHub Actions OIDC lets workflows request short-lived cloud tokens directly from the cloud provider, avoiding duplicated long-lived cloud secrets in GitHub. ([GitHub Docs][5])

# 4. Hooks: useful, but never trusted

Client-side Git hooks are guardrails, not enforcement. Git’s own docs note that pre-commit can be bypassed with `--no-verify`; server-side rules and CI must enforce the same checks. ([Git][6])

Use exactly these hooks:

```text
pre-commit:
  format
  lint staged files
  secret scan staged files
  block private keys
  block huge files
  block generated/vendor junk unless allowlisted

commit-msg:
  require ticket/id
  require conventional shape or simple standard:
    <type>: <short description>

pre-push:
  block push to main
  secret scan pushed range
  run fast unit/smoke checks
  verify branch name
```

**Canonical local setup**

```bash
git config --global init.defaultBranch main
git config --global pull.rebase true
git config --global rebase.autoStash true
git config --global commit.gpgsign true
git config --global tag.gpgSign true
```

**Repo bootstrap**

```bash
make bootstrap
```

`make bootstrap` installs hooks, validates tool versions, installs formatters, and runs the fast local suite. Nobody hand-installs pieces.

# 5. CI gates

Use three CI levels.

## PR CI: fast gate

Runs on every PR.

```text
format
lint
typecheck
unit tests
fast integration tests
secret scan
dependency audit
license/SBOM check
changed-file policy
agent review
workflow-safety check
```

Goal: fail bad PRs quickly.

## Merge queue CI: correctness gate

Runs on the synthetic queued merge commit.

```text
full test suite
full integration suite
migration tests
backward compatibility tests
security/code scanning
dependency audit
build dry-run
```

Goal: prove this exact PR will not break `main` when merged with queued neighbors.

## Main CI: release gate

Runs after merge to `main`.

```text
full CI again
build release artifact once
generate SBOM
sign artifact
generate provenance attestation
push artifact by immutable digest
create release/tag only from CI
```

GitHub Actions artifact attestations can establish build provenance for binaries and container images, and GitHub documents verifying attestations with the GitHub CLI. ([GitHub Docs][7])

# 6. Artifact policy

This is nonnegotiable:

```text
Build once.
Sign once.
Attest once.
Promote the same digest everywhere.
Never rebuild for dev/staging/prod.
Never deploy mutable tags.
Never deploy unsigned artifacts.
Never roll back by rebuilding old code.
```

Canonical artifact identity:

```text
image: ghcr.io/org/app@sha256:<digest>
source: main@<commit_sha>
build: github_run_id=<id>
sbom: attached
signature: attached
provenance: attached
```

Sigstore is a common signing/provenance foundation for artifacts such as binaries, container images, release files, and SBOMs; its model uses ephemeral signing keys and records signing events in a tamper-resistant transparency log for public-good signing. ([Sigstore][8])

---

# 7. Deployment policy

Use only this progression:

```text
local smoke:
  run artifact locally
  verify boot, healthcheck, basic behavior

dev:
  automatic deploy from signed main artifact
  automatic smoke/integration tests

prod-limited:
  canary or limited release
  automatic metrics/SLO guard
  automatic rollback on failure

prod-full:
  progressive rollout
  automatic halt/rollback on failure
```

GitHub environments can require deployment protection rules before a job runs or accesses environment secrets, and can require reviewers, wait timers, no admin bypass, custom protection rules, and branch/tag deployment restrictions. ([GitHub Docs][9])

**Rollback rule**

```text
rollback = redeploy previous known-good signed digest
rollback != revert code and rebuild under panic
```

A revert PR can follow later. Production recovery uses the previous artifact.

# 8. Agent rules

Agents are PR producers, not release authorities.

```text
Agents may:
  create branches
  open PRs
  update their PRs
  run local tests
  propose fixes
  comment/review

Agents may not:
  push to main
  bypass branch rules
  approve their own PRs
  change workflow/deploy/security files without code-owner approval
  access production secrets
  deploy to production
  create release tags
  weaken CI
```

Set default `GITHUB_TOKEN` permissions to read-only, then elevate per job only when needed. GitHub’s Actions security guidance recommends least privilege for workflow credentials and setting the default `GITHUB_TOKEN` permission to read-only for repository contents. ([GitHub Docs][3])

Also pin all third-party Actions by full commit SHA. GitHub says pinning an Action to a full-length SHA is the only way to use an Action as an immutable release, and it supports org/repo policies requiring SHA pinning. ([GitHub Docs][3])

# 9. Minimal repo files

Every repo gets this shape:

```text
.github/
  workflows/
    pr.yml
    merge-queue.yml
    main-release.yml
    deploy.yml
  CODEOWNERS
  ruleset.yml              # desired-state repo rules
.husky/ or .githooks/      # local hooks, installed by make bootstrap
.pre-commit-config.yaml
Makefile
SECURITY.md
README.md
```

`CODEOWNERS` minimum:

```text
.github/**        @org/platform-security
.github/workflows/** @org/platform-security
infra/**          @org/platform-infra
terraform/**      @org/platform-infra
k8s/**            @org/platform-infra
helm/**           @org/platform-infra
scripts/deploy*   @org/platform-release
SECURITY.md       @org/platform-security
```

# 10. The canonical Makefile commands

No bespoke human commands. Every repo exposes the same interface.

```makefile
bootstrap:
	./scripts/bootstrap.sh

check:
	./scripts/check.sh

test:
	./scripts/test.sh

test-full:
	./scripts/test-full.sh

build:
	./scripts/build.sh

smoke:
	./scripts/smoke.sh

release:
	@echo "Release is CI-only"; exit 1

deploy:
	@echo "Deploy is CI-only"; exit 1
```

Humans and agents run:

```bash
make bootstrap
make check
make test
```

CI runs:

```bash
make check
make test-full
make build
make smoke
```

Release/deploy from laptops is forbidden.

---

# 11. The shortest possible doctrine

```text
1. Nothing goes to main except through PR.
2. Nothing merges unless required checks pass in the merge queue.
3. Nothing deploys unless built from main.
4. Nothing deploys unless signed, attested, and addressed by digest.
5. Nothing secret is stored in git, workflows, logs, or long-lived deploy keys.
6. Nothing bypasses rules, including admins and agents.
7. Nothing changes CI, deploy, or security policy without code-owner approval.
8. Nothing rolls back by rebuilding; rollback redeploys the previous signed digest.
```

This is the minimal golden path: **PR-only trunk, merge queue, signed immutable artifacts, progressive deploy, automatic rollback, zero bypass.**
