Absolutely ŌĆö this is a great fit for JeRyu. I would implement this as a **single multi-project control plane** with **project-scoped runner bindings, project-scoped filesystem roots, and a weighted roulette scheduler**. The key is: do **not** treat ŌĆ£projectŌĆØ as just a GitLab `project_id` argument anymore. Make it a first-class object in JeRyu.

JeRyu is already close to this shape: it is a single-binary Rust control plane with CI/CD tooling, smart test selection, runner orchestration, remote server management, and a proof-rich TUI. The current README still shows test execution through a numeric `--project-id`, and the TUI is already organized around operational tabs like Mission, Release, Jobs, Agents, Tests, Pools, Cache, Evidence, Secrets, and LLMs. ([GitHub][1]) The current TUI code has explicit `ActiveTab` variants and a shared `TuiStateSnapshot`, while the header/tab renderer is already centralized in `draw_header_tabs`, which makes the project strip a clean additive change rather than a rewrite. ([GitHub][2]) ([GitHub][2]) ([GitHub][3])

## Recommendation

Build this around three new concepts:

1. **`LocalProject` / `ProjectScope`**: the canonical JeRyu-managed local repo identity.
2. **`ProjectScheduler`**: a weighted roulette scheduler that decides which project receives runner capacity.
3. **`ProjectIsolationRoot`**: per-project filesystem, cache, artifact, log, and workspace roots.

The big rule should be:

> A runner may share the same JeRyu server, GitLab instance, Vault instance, and global control database, but it must never share mutable caches, workspaces, artifacts, logs, temp files, or secrets between local projects.

The only acceptable shared resources should be read-only or content-addressed immutable resources, such as pinned container images or globally installed JeRyu binaries.

---

## Current shape to build from

Today, JeRyuŌĆÖs data model already records `project_id` in many places, such as job events, CI job runs, tracked pipelines, release attempts, evidence capsules, retries, test plans, and build signatures. That is a good foundation, but it is currently a **GitLab project identifier**, not a full local-project identity with repo root, cache root, scheduler weight, and isolation metadata.

The current path model is also pool-centered: JeRyu uses a global `~/.jeryu` data root, a global cache root, per-manager cache directories, local Cargo cache roots, and pool-scoped cache roots such as `cache/pools/<pool>`. That is exactly the part I would change, because pool-scoped caches are too broad once many unrelated local repos share one server.

The runner config currently mounts a supplied `cache_dir` into `/cache` for Docker runners or `/pool-cache` for custom runners, and it sets `JERYU_CARGO_CACHE_ROOT` to that mounted cache path. That means the right isolation fix is to make that `cache_dir` project-scoped, not just pool-scoped.

# Engineering spec

## 1. Project registry

Add a first-class project registry in the state layer.

### New table: `local_projects`

```sql
CREATE TABLE IF NOT EXISTS local_projects (
    id                    INTEGER PRIMARY KEY AUTOINCREMENT,
    slug                  TEXT NOT NULL UNIQUE,
    display_name          TEXT NOT NULL,

    -- Local repository identity
    repo_root             TEXT NOT NULL,
    repo_fingerprint      TEXT NOT NULL UNIQUE,
    default_ref           TEXT NOT NULL DEFAULT 'main',

    -- GitLab mirror/link
    gitlab_project_id     INTEGER,
    gitlab_path           TEXT,

    -- Scheduling
    scheduler_weight      INTEGER NOT NULL DEFAULT 100,
    scheduler_enabled     INTEGER NOT NULL DEFAULT 1,
    concurrency_limit     INTEGER NOT NULL DEFAULT 1,
    trust_tier            TEXT NOT NULL DEFAULT 'local',

    -- Isolation roots
    state_root            TEXT NOT NULL,
    work_root             TEXT NOT NULL,
    cache_root            TEXT NOT NULL,
    artifact_root         TEXT NOT NULL,
    log_root              TEXT NOT NULL,
    tmp_root              TEXT NOT NULL,

    -- TUI ordering
    last_run_at           TEXT,
    last_selected_at      TEXT,

    created_at            TEXT NOT NULL,
    updated_at            TEXT NOT NULL
);
```

Important naming choice: keep current GitLab `project_id` semantics where needed, but introduce **`local_project_id`** for JeRyuŌĆÖs own registry. This avoids confusion between:

```rust
gitlab_project_id: i64
local_project_id: i64
```

### New Rust model

```rust
pub struct LocalProject {
    pub id: i64,
    pub slug: String,
    pub display_name: String,
    pub repo_root: PathBuf,
    pub repo_fingerprint: String,
    pub gitlab_project_id: Option<i64>,
    pub default_ref: String,
    pub scheduler_weight: u32,
    pub scheduler_enabled: bool,
    pub concurrency_limit: u32,
    pub trust_tier: String,
    pub roots: ProjectRoots,
    pub last_run_at: Option<DateTime<Utc>>,
}

pub struct ProjectRoots {
    pub state_root: PathBuf,
    pub work_root: PathBuf,
    pub cache_root: PathBuf,
    pub artifact_root: PathBuf,
    pub log_root: PathBuf,
    pub tmp_root: PathBuf,
}

pub struct ProjectScope {
    pub local_project_id: i64,
    pub slug: String,
    pub gitlab_project_id: i64,
    pub repo_root: PathBuf,
    pub cache_namespace: String,
    pub trust_tier: String,
    pub roots: ProjectRoots,
}
```

`ProjectScope` should become the object passed into `test_runner`, `release`, `cache`, `pool`, `exec`, and TUI collectors. Avoid passing raw `i64 project_id` through new code.

## 3. Runner scheduling model

Use **weighted roulette scheduling with starvation protection**.

Each project gets a weight:

```text
docs-site      priority 50
api-server     priority 200
compiler-core  priority 400
```

Expected access ratio:

```text
docs-site : api-server : compiler-core
1         : 4          : 8
```

But the scheduler should also enforce:

```text
project concurrency limit
pool capacity
trust tier
runner tags
current queue depth
aging / anti-starvation bonus
manual pause state
```

### New table: `project_scheduler_state`

```sql
CREATE TABLE IF NOT EXISTS project_scheduler_state (
    local_project_id     INTEGER PRIMARY KEY REFERENCES local_projects(id),
    running_count        INTEGER NOT NULL DEFAULT 0,
    queued_count         INTEGER NOT NULL DEFAULT 0,
    deficit              REAL NOT NULL DEFAULT 0,
    last_selected_at     TEXT,
    last_missed_at       TEXT,
    updated_at           TEXT NOT NULL
);
```

### New table: `scheduler_decisions`

```sql
CREATE TABLE IF NOT EXISTS scheduler_decisions (
    id                   INTEGER PRIMARY KEY AUTOINCREMENT,
    pool_name            TEXT NOT NULL,
    selected_project_id  INTEGER REFERENCES local_projects(id),
    free_slots           INTEGER NOT NULL,
    eligible_json        TEXT NOT NULL,
    reason               TEXT NOT NULL,
    decided_at           TEXT NOT NULL
);
```

This is important for the TUI: the scheduler needs to be explainable. A user should be able to ask: ŌĆ£Why did `api-server` get the next runner?ŌĆØ

### Roulette algorithm

Use weighted random selection, but with a bounded aging bonus:

```rust
tickets = scheduler_weight
        * runnable_job_count.min(4)
        * aging_multiplier
        * trust_pool_multiplier;
```

Where:

```rust
aging_multiplier = 1.0 + min(waiting_secs / 60.0, 3.0);
```

Then:

```rust
let total = candidates.iter().map(|c| c.tickets).sum();
let r = rng.uniform(0, total);

for candidate in candidates {
    cumulative += candidate.tickets;
    if r < cumulative {
        return candidate;
    }
}
```

For deterministic testing, seed the RNG with:

```text
scheduler_epoch + pool_name + sorted_candidate_project_ids + queue_version
```

That makes tests stable while still giving roulette behavior in production.

### Starvation guard

Even with weighted roulette, low-priority projects must not starve forever.

Add:

```text
max_starvation_secs = 300
```

If a project has runnable work and has missed selection longer than this, boost it once or force-pick it within its concurrency cap.

## 4. Runner pool architecture

Keep `Pool` as the capacity class:

```text
default
build
untrusted
gpu
release
```

But add project bindings.

### New table: `project_pool_bindings`

```sql
CREATE TABLE IF NOT EXISTS project_pool_bindings (
    local_project_id     INTEGER NOT NULL REFERENCES local_projects(id),
    pool_name            TEXT NOT NULL REFERENCES pools(name),

    runner_tag           TEXT NOT NULL,
    gitlab_runner_id     INTEGER,
    auth_token_ref       TEXT,

    min_warm             INTEGER NOT NULL DEFAULT 0,
    max_slots            INTEGER NOT NULL DEFAULT 1,
    paused               INTEGER NOT NULL DEFAULT 0,

    created_at           TEXT NOT NULL,
    updated_at           TEXT NOT NULL,

    PRIMARY KEY(local_project_id, pool_name)
);
```

The secure default should be:

> Runner managers are bound to one project and one pool.

So instead of:

```text
pool: default
cache: ~/.jeryu/cache/pools/default
```

use:

```text
project: api-server
pool: default
cache: ~/.jeryu/projects/api-server/cache/pools/default
```

This keeps the same server and same GitLab instance, but prevents project A from sharing mutable runner state with project B.

The current `pool.rs` path creates manager cache, pool cache, pool Cargo target, and pool sccache directories before starting a runner manager. That should change to project-aware paths, not pool-only paths.

### Runner tags

Generated CI jobs should include project tags:

```yaml
test:unit:
  tags:
    - jeryu:project:api-server
    - jeryu:pool:default
```

The binding should create runner managers that only accept those tags.

This is the safest way to ensure GitLab does not bypass JeRyuŌĆÖs roulette scheduler. If all projects share one generic runner tag, GitLab can schedule jobs independently of JeRyuŌĆÖs desired project priority.

## 5. Filesystem isolation

Use this layout:

```text
~/.jeryu/
  state.redline                 # global control DB only
  projects/
    api-server/
      project.toml
      state/
      work/
        runs/
          run-000001/
            repo/
            tmp/
      cache/
        cargo-home/
        cargo-target/
        sccache/
        npm/
        pnpm/
        yarn/
        docker/
        action/
      artifacts/
        run-000001/
      logs/
        run-000001/
      secrets/
        refs/
```

### Strict rules

Every job gets:

```text
JERYU_PROJECT_ID=<local_project_id>
JERYU_PROJECT_SLUG=<slug>
JERYU_PROJECT_ROOT=<project root>
JERYU_RUN_ROOT=<project work run root>
JERYU_CACHE_ROOT=<project cache root>
```

For Rust jobs:

```text
CARGO_HOME=$JERYU_CACHE_ROOT/cargo-home
CARGO_TARGET_DIR=$JERYU_CACHE_ROOT/cargo-target/<toolchain-hash>
SCCACHE_DIR=$JERYU_CACHE_ROOT/sccache
```

For Node jobs:

```text
npm_config_cache=$JERYU_CACHE_ROOT/npm
PNPM_STORE_PATH=$JERYU_CACHE_ROOT/pnpm
YARN_CACHE_FOLDER=$JERYU_CACHE_ROOT/yarn
```

For temp files:

```text
TMPDIR=$JERYU_RUN_ROOT/tmp
```

Do not let jobs fall back to:

```text
~/.cargo
~/.npm
/tmp shared across projects
~/.cache
global /cache
pool-only /pool-cache
```

### Workspace isolation

Do **not** run CI directly in the userŌĆÖs working tree.

For each run:

```text
~/.jeryu/projects/<slug>/work/runs/<run_id>/repo
```

Populate it by one of:

```text
git worktree add --detach
git clone --reference-if-able
local GitLab mirror checkout
```

Then verify:

```rust
canonical(run_root).starts_with(project.work_root)
canonical(cache_root).starts_with(project.cache_root)
canonical(artifact_root).starts_with(project.artifact_root)
```

Reject symlink escapes.

### Build cache keys

Every cache key must include the project namespace:

```text
project:<local_project_id>:<repo_fingerprint>:trust:<trust_tier>:epoch:<epoch>:<cache-key>
```

That applies to:

```text
action cache
Cargo cache metadata
sccache metadata
cache proxy objects
build signatures
image signatures
cache leases
cache taints
cache verdicts
cache epochs
```

The current schema already has some useful cache namespace ideas, such as `action_cache.namespace`, `cache_promotions.source_namespace`, `cache_promotions.target_namespace`, and `cache_epochs.scope`; the change is to make project namespace mandatory everywhere cache state is read or written.

## 11. Cache and file isolation enforcement

Add a small but strict API layer:

```rust
impl ProjectRoots {
    pub fn assert_contains(&self, path: &Path) -> Result<()> {
        let canonical = path.canonicalize()?;
        ensure!(
            canonical.starts_with(&self.project_root_canonical),
            "path escapes project root"
        );
        Ok(())
    }
}
```

Every module that creates paths should take `&ProjectScope`.

Bad new API:

```rust
fn pool_cache_root(pool_name: &str) -> PathBuf
```

Good new API:

```rust
fn project_pool_cache_root(project: &ProjectScope, pool_name: &str) -> PathBuf
```

Bad:

```rust
cache::put(key, value)
```

Good:

```rust
cache::put(&project_scope, key, value)
```

Bad:

```rust
cargo_cache::render_runner_cargo_pre_build_script("/cache", executor)
```

Good:

```rust
cargo_cache::render_project_runner_cargo_pre_build_script(
    &project_scope,
    "/cache",
    executor,
)
```

This is the fastest path to security: make it hard to compile code that touches caches without a project scope.

# Best implementation direction

The strongest design is:

```text
One JeRyu server
One global control DB
Many LocalProject records
Project-bound runner managers
Project-scoped cache/work/artifact/log/secret roots
Weighted roulette scheduler
TUI project strip sorted by last_run_at
```

I would **not** rely on one shared runner tag and hope GitLab gives the right project more capacity. That would make the roulette scheduler advisory only. To make priority real, JeRyu needs either project-bound runner tags or custom-executor reservations. The safest first version is project-bound runner tags/managers, because it is easy to audit, easy to display in the TUI, and naturally prevents mutable filesystem sharing.

The result is exactly the UX you described: many local repos managed by the same JeRyu server, visible in one thin top bar, navigable with Enter plus arrows, with higher-priority projects receiving more runner access while every project remains isolated.
