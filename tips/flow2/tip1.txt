# Engineering Spec: JeRyu Workflow Mission Control TUI

## 1. Product Vision

Build the **developer’s dream CI/CD visualization TUI**: a terminal-native “mission control” where a developer can see the entire journey from **PR → local checks → merge readiness → test matrix → build → deploy → canary → production**, with every node, dependency, failure, cache hit, skip reason, and live status visible in one navigable DAG.

The Workflow tab should answer four questions instantly:

1. **Where are we?**
   The active/running node is visually obvious, centered when follow mode is enabled, and reflected in the header timeline.

2. **What is blocking us?**
   Failed, blocked, skipped, and waiting-on-dependency nodes show the exact upstream cause.

3. **What ran in parallel?**
   Parallel test shards appear as fixed-width cards side by side inside phase bands, not stretched rows.

4. **How far are we from production?**
   The graph runs from layer 0 to production gate, with critical path, ETA, phase progress, and deploy confidence shown continuously.

The feel should be: **NASA mission control + GitLab/GitHub Actions + production deploy war room**, but rendered entirely in Rust terminal UI.

---

## 2. Core UX Philosophy

The core model is:

> **Glance → Navigate → Inspect → Act**

At a glance, the user should see phase health, active work, failures, and pipeline progress. With keyboard or mouse, they should move quickly through a large DAG. Pressing Enter or clicking a node opens a focused inspector with full details. Later, action hooks can support retry, skip, cancel, expand logs, or open in GitLab.

The UI should be **dense but calm**. It should never feel like a log wall. It should feel like a live system map.

---

## 3. Name the Experience

Internally, call this view:

## **Workflow Mission Control**

Key concepts:

| Concept                  | Meaning                                                                         |
| ------------------------ | ------------------------------------------------------------------------------- |
| **Mission Strip**        | Top summary banner showing PR-to-prod progress, mode, confidence, failures, ETA |
| **Phase Bands**          | Horizontal layers: lint, unit, integration, deploy, canary, prod                |
| **Test Swarms**          | Parallel fixed-width node cards within a phase                                  |
| **Edge Highway**         | Dependency connectors routed through gutters between phases                     |
| **Critical Beam**        | Highlighted critical path from PR to production                                 |
| **Stall Beacon**         | The exact node or dependency causing the pipeline to stop                       |
| **Flight Recorder**      | Inspect overlay with status history, duration, logs, deps, events               |
| **Minimap Rail**         | Compact viewport indicator showing where the user is within the full DAG        |
| **Follow Active**        | Auto-center on running or failing work                                          |
| **Failure Blast Radius** | Downstream nodes affected by a failed upstream dependency                       |

## 4. Rust TUI Feasibility

This design is very realistic for the current stack.

Ratatui is an immediate-mode rendering library: applications draw into a `Frame`, widgets render into an intermediate buffer, and Ratatui flushes that buffer to the terminal at the end of the draw call. That matches the current JeRyu architecture of pure render functions and explicit app state. ([Ratatui][1])

The Ratatui buffer is a grid of terminal cells, where each cell contains a grapheme plus foreground/background style. That makes it suitable for precise edge drawing, custom card borders, port connectors, minimaps, and overlay composition. ([Docs.rs][2])

Ratatui supports normal widgets, custom widgets, and stateful widgets, which fits this design well: the workflow renderer can remain pure, while navigation, hover, selection, scrollbars, and mouse drag state live in `WorkflowNav` or adjacent state. ([Ratatui][3])

Ratatui also includes `Scrollbar` support, including vertical and horizontal orientations; the docs note that `ScrollbarState::content_length` must be set or the scrollbar renders blank. That should replace the currently unreliable viewport indicator with first-class scrollbars. ([Docs.rs][4])

For graph drawing, Ratatui’s `Canvas` can draw lines, rectangles, points, and labels, but for this DAG I would use direct buffer rendering rather than Canvas for the main graph because card layout, clipping, Unicode box drawing, hit testing, and exact terminal-cell routing matter more than geometric plotting. Canvas can still be useful for tiny inspector visualizations such as duration history or dependency micrographs. ([Docs.rs][5])

For mouse support, Crossterm exposes `MouseEventKind::ScrollUp`, `ScrollDown`, `ScrollLeft`, `ScrollRight`, and `Drag`, and mouse events are available after enabling mouse capture. That supports wheel scrolling, horizontal trackpad panning, click selection, and drag-to-pan. ([Docs.rs][6])

# 6. Visual Language

## 6.1 Theme

Do not hardcode colors. Extend `Theme` if needed.

Desired style:

| UI Element              | Visual Intent                                           |
| ----------------------- | ------------------------------------------------------- |
| Background              | Deep charcoal / black terminal calm                     |
| Phase headers           | Muted steel / blue-gray                                 |
| Selected node           | Electric cyan border + `[SEL]` badge                    |
| Running node            | Pulsing cyan/blue border, spinner glyph                 |
| Failed node             | Red border, red failure beacon, downstream blast radius |
| Blocked node            | Amber/yellow border, dependency icon                    |
| Cached node             | Cool green/teal, cache badge                            |
| Skipped node            | Muted gray with reason visible                          |
| Critical path           | Bright, continuous highlighted connector                |
| Non-critical edges      | Dim but readable                                        |
| VTI accelerated/skipped | Purple/fire accent via existing `vti_fire` style        |
| Production gate         | Distinct “launch” visual, not just another node         |

Add semantic theme fields only if needed:

```rust
pub struct Theme {
    // existing fields...
    pub workflow_selected: Color,
    pub workflow_edge: Color,
    pub workflow_edge_dim: Color,
    pub workflow_critical: Color,
    pub workflow_phase: Color,
    pub workflow_mouse_hover: Color,
    pub workflow_stall: Color,
}
```

But prefer existing theme methods first: `status_color`, `status_glyph`, `bold`, `muted`, `primary`, `secondary`, `border_style_for`.

## 6.3 Intelligent Label Abbreviation

The 34-character card width means labels must be compressed intelligently.

Rules:

1. Preserve high-signal suffixes: `payments`, `auth`, `deploy`, `prod`, `canary`, shard number.
2. Collapse common prefixes:

   * `test_` → remove
   * `integration/` → `int/`
   * `end_to_end/` → `e2e/`
   * `production` → `prod`
   * `staging` → `stage`
   * `javascript` → `js`
   * `typescript` → `ts`
3. Preserve shard identity:

   * `unit-api-shard-07-of-24` → `unit/api 07/24`
4. Use middle ellipsis only when necessary:

   * `integration/payments/refund-flow` → `int/payments/…refund`
5. Never truncate status badges, `[SEL]`, shard count, or failure glyph.

Implement as:

```rust
fn compact_node_label(label: &str, max: usize) -> String
fn compact_test_metadata(node: &WorkflowNode, max: usize) -> String
```

Keep this in `workflow/widget.rs` or a small `workflow/format.rs`.

# 7. DAG Layout Specification

## 7.1 Virtual Canvas

The DAG should live on a virtual canvas with its own coordinate system.

```rust
struct WorkflowLayout {
    canvas_w: i32,
    canvas_h: i32,
    phase_rects: Vec<PhaseRect>,
    node_rects: HashMap<String, Rect>,
    edge_routes: Vec<EdgeRoute>,
    hit_regions: Vec<HitRegion>,
}
```

The renderer should compute this layout every frame from immutable inputs:

```rust
let layout = compute_workflow_layout(snapshot, area, nav, theme);
render_workflow_canvas(frame, area, snapshot, nav, &layout, theme);
```

The actual render clips by:

```rust
screen_x = canvas_x - nav.viewport_x + dag_area.x
screen_y = canvas_y - nav.viewport_y + dag_area.y
```

Anything outside the visible `dag_area` is skipped.

## 7.2 Phase Bands

Each `WorkflowPhase` becomes a horizontal band.

```text
L2  Unit Test Swarm ─────────────────────  8/12 done · 2 running · 1 failed
```

Phase header should include:

* Layer index
* Phase name
* Status rollup
* Running count
* Failed count
* Cached/skipped count
* Duration or ETA
* Optional mode badge: `selected`, `full`, `vti`, `deploy`

Phase bands should be visually stable. Nodes should not jump around every frame. Running progress can update, but layout should remain deterministic.

## 7.3 Parallel Test Layout

Parallel nodes in the same phase are arranged left-to-right in lanes:

```text
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ unit/core    │ │ unit/api     │ │ unit/db      │
└──────────────┘ └──────────────┘ └──────────────┘
```

For many parallel nodes, wrap into rows within the same phase band:

```text
Phase L3 Integration
  card card card card
  card card card card
  card card card card
```

Layout formula:

```rust
let lane_gap_x = 3;
let row_gap_y = 1;
let card_w = NODE_CARD_W as i32;
let card_h = NODE_CARD_H as i32;

cards_per_row = max(1, available_virtual_width / (card_w + lane_gap_x));
row = node_idx / cards_per_row;
col = node_idx % cards_per_row;
x = PHASE_LABEL_W + col * (card_w + lane_gap_x);
y = phase_y + PHASE_HEADER_H + row * (card_h + row_gap_y);
```

Important: `available_virtual_width` should not be the terminal width. It should be a layout target derived from the largest phase, capped by a sane maximum, so horizontal panning works.

## 7.4 Stable Ordering

Within each phase, order nodes by:

1. Critical path first
2. Running/failing nodes next
3. Dependency order
4. Natural label order
5. Original input order as final tiebreaker

This prevents visual noise while keeping important nodes near the left.

# 8. Edge Routing Specification

## 8.1 Edge Philosophy

Edges must be readable, not mathematically perfect. The goal is to show dependency flow and blockage.

Use “ports” on cards:

```text
       upstream card
            │ output port
            ▼
──────── edge gutter ────────
            ▲ input port
       downstream card
```

Each node has:

```rust
struct NodePorts {
    top: Position,
    bottom: Position,
    left: Position,
    right: Position,
}
```

## 8.2 Edge Types

Use existing `EdgeKind` glyph/badge concepts:

| Edge Kind       | Visual                    |
| --------------- | ------------------------- |
| `GitlabNeeds`   | Solid line                |
| `ArtifactDep`   | Solid with artifact glyph |
| `StageOrder`    | Dim line                  |
| `VtiSkipped`    | Dashed/dim purple         |
| `Blocked`       | Amber/red blocking line   |
| `ChildPipeline` | Branch connector          |

## 8.3 Edge Rendering Order

Render order must be:

1. Clear DAG canvas background
2. Phase backgrounds and headers
3. Edge gutters
4. Edge labels/badges
5. Node cards
6. Selection/hover overlays
7. Scrollbars/minimap
8. Inspector overlay, if open

Edges should be under cards, not over them.

## 8.4 Connector Characters

Use Unicode box drawing:

```text
│ ─ ┌ ┐ └ ┘ ├ ┤ ┬ ┴ ┼ ═ ║ ╔ ╗ ╚ ╝
```

Critical path:

```text
━━━┃━━━
```

Blocked/failure propagation:

```text
╍╍╍ or ┄┄┄
```

For crossing edges, use `┼`. For selected-node edges, upgrade to stronger characters.

## 8.5 Edge Deconfliction

Assign each edge a gutter lane:

```rust
struct EdgeRoute {
    from_id: String,
    to_id: String,
    points: Vec<Position>,
    lane: usize,
    kind: EdgeKind,
    critical: bool,
    blocked: bool,
}
```

Route algorithm:

1. Start at upstream bottom port.
2. Move vertically into edge gutter.
3. Move horizontally to downstream x.
4. Move vertically into downstream top port.
5. If multiple edges overlap, merge shared vertical trunks.
6. If selected node is active, brighten only its inbound/outbound edges.

## 8.6 Dependency Highlight Modes

When a node is selected:

* Brighten inbound dependencies.
* Brighten outbound dependents.
* Dim unrelated edges.
* Show “upstream chain” and “downstream blast radius.”
* In footer: `3 deps · 8 downstream · critical path +4m`.

This is essential for “why am I stuck?”

| `/`           | Search node/test               |
| `n / N`       | Next/previous search hit       |

| `e`           | Toggle edge detail             |

## 9.2 Mouse

Mouse support should feel first-class, not bolted on.

| Mouse Action                    | Behavior                                  |
| ------------------------------- | ----------------------------------------- |
| Wheel up/down                   | Pan vertical                              |
| Shift+wheel or horizontal wheel | Pan horizontal                            |
| Click card                      | Select node                               |
| Double-click card               | Open inspector                            |
| Drag empty canvas               | Pan x/y                                   |
| Click phase header              | Collapse/expand phase                     |
| Click minimap                   | Jump viewport                             |
| Click scrollbar thumb           | Jump/pan                                  |
| Right click node                | Context mini-menu: inspect, logs, copy id |

Implementation requires extending `runtime/input.rs` / `general.rs` to handle `Event::Mouse`. Crossterm supports scroll and drag event kinds, and mouse events are enabled through mouse capture. ([Docs.rs][6])

Important UX rule: mouse capture should be **toggleable** with `m`. Developers still need native terminal text selection sometimes. Default can be enabled in demo mode and configurable in live mode.

## 9.3 Follow Active

Follow mode should center on the most important live node:

Priority:

1. Failed node on critical path
2. Running node on critical path
3. Any failed node
4. Any running node
5. First blocked node
6. Selected node
7. Start node

When follow is active:

```text
FOLLOW ACTIVE: e2e/payments-03 · 63% · ETA 48s
```

If the user manually pans or drags, temporarily suspend follow:

```text
FOLLOW PAUSED — press f to resume
```

# 13. Rendering Architecture

## 13.1 New Internal Modules

Recommended structure:

```text
src/tui/workflow/
  mod.rs
  model.rs
  nav.rs
  widget.rs
  builder.rs
  collector.rs

  layout.rs       // virtual canvas layout, node rects, phase rects
  render.rs       // card, phase, edge, minimap render helpers
  format.rs       // compact labels, durations, ETA, counts
  hit.rs          // mouse hit testing
  inspect.rs      // inspector rendering
```

Keep `widget.rs` as the public renderer entry point:

```rust
pub fn draw_workflow_tab(
    f: &mut Frame,
    area: Rect,
    app: &App,
    theme: &Theme,
)
```

Then delegate:

```rust
let layout = WorkflowLayout::compute(snapshot, nav, area);
render_mission_strip(...);
render_canvas(...);
render_scrollbars(...);
render_footer_hints(...);
```

## 13.2 Pure Render Rule

Rendering functions may compute temporary layout, but must not mutate app state except through explicitly passed mutable local widget states where Ratatui requires it. Existing architecture requires widgets to remain pure.

For actual state updates, use App actions:

```rust
app.workflow_mouse_scroll(dx, dy)
app.workflow_mouse_drag(from, to)
app.workflow_select_node(id)
app.workflow_open_inspector(id)
app.workflow_toggle_mouse()
```

## 13.3 Layout Cache

For performance, cache layout metadata when snapshot and terminal size are unchanged:

```rust
pub struct WorkflowLayoutCache {
    snapshot_fingerprint: u64,
    area: Rect,
    layout: WorkflowLayout,
}
```

But do not over-optimize early. The first priority is correctness and visual quality.

# 14. Live Data and Refresh Strategy

Current issue: `refresh_workflow_snapshot()` runs every draw frame.

Better model:

```text
Input/event tick ──> App state update ──> mark workflow dirty
Timer tick ────────> refresh if stale
Draw ──────────────> render latest cached snapshot only
```

## 14.1 Refresh Rules

Refresh the workflow snapshot when:

* New job event arrives
* Pipeline state changes
* VTI plan changes
* User manually presses refresh
* Demo tick changes simulated status
* Stale interval expires, e.g. 500ms or 1s in live mode

Do not rebuild from scratch every render frame.

## 14.2 Snapshot Fingerprint

Use a cheap fingerprint:

```rust
struct WorkflowFingerprint {
    pipeline_id: Option<String>,
    job_count: usize,
    edge_count: usize,
    status_counts: StatusCounts,
    latest_event_seq: u64,
}
```

If unchanged, skip rebuild.

# 15. Demo Mode Animation

Demo mode should feel alive.

## 15.1 Animation Loop

Every tick:

1. Advance one or more running nodes.
2. Move nodes through `Waiting → Running → Ran`.
3. Occasionally mark cache hit/skipped.
4. Optionally inject a recoverable failure scenario.

# 16. PR-to-Production Workflow Model

The ideal DAG should include the whole lifecycle.

## 16.1 Canonical Phases

```text
L0  Local / Developer Preflight
L1  PR Created / Source Gate
L2  VTI Selection / Change Impact
L3  Static Checks
L4  Unit Test Matrix
L5  Integration Test Matrix
L6  E2E / Browser / Contract Tests
L7  Security / Compliance / SBOM
L8  Build / Package / Image
L9  Merge Queue / Merge Train
L10 Staging Deploy
L11 Smoke Tests
L12 Canary Deploy
L13 Canary Health
L14 Production Gate
L15 Production Rollout
L16 Post-Deploy Verification
```

Not every pipeline needs every phase. Empty phases should be hidden unless expanded.

## 16.2 Production Gate Card

Production gate should feel special:

```text
╔══════════════════════════════╗
║ 🚀 prod-gate                 ║
║ waiting: canary health       ║
║ policy: 2 approvals + SLO    ║
╚══════════════════════════════╝
```

For terminal compatibility, avoid relying on emoji by default; make emoji optional. ASCII/Unicode fallback:

```text
║ ^ prod-gate                  ║
```

# 17. Failure Experience

Failures should be impossible to miss but not chaotic.

## 17.1 Failure Beacon

When a failure exists:

* Header shows red failure count.
* First failure gets a beacon.
* `b` jumps to root blocker.
* Downstream affected nodes are dimmed/amber.
* Edges from failed dependency are highlighted.
* Inspector explains cause and downstream impact.

## 17.2 Failure Root Cause

Compute:

```rust
fn root_blocker(snapshot: &WorkflowSnapshot) -> Option<&WorkflowNode>
```

Priority:

1. Failed critical-path node
2. Failed node with most downstream blocked nodes
3. Long-running critical-path node beyond expected duration
4. Manual gate blocking production
5. Waiting-for-runner node with longest queue time

Display:

```text
STALL BEACON: integration/auth failed · blocks 8 nodes · prod delayed ~14m
```

# 18. Performance Targets

The view should remain responsive on large pipelines.

## 18.1 Target Scale

| Metric             | Target                                         |
| ------------------ | ---------------------------------------------- |
| Nodes              | 500 visible DAG nodes                          |
| Edges              | 2,000 dependency edges                         |
| Render             | Smooth enough at 10–20 FPS during live updates |
| Input latency      | Under 50ms perceived                           |
| Snapshot rebuild   | Avoid every-frame rebuild                      |
| Horizontal panning | No layout jump                                 |
| Vertical panning   | No layout jump                                 |

## 18.2 Rendering Optimizations

* Skip offscreen cards.
* Skip offscreen edge segments.
* Precompute node rects and edge routes.
* Use compact strings allocated once per layout pass.
* Avoid cloning whole snapshots in render.
* Render only to visible `Rect`.
* Keep inspector data lazy.

# 22. Non-Negotiable Acceptance Criteria

The Workflow TUI is not done until all of these are true:

1. **Fixed-width cards**: every node card is 34 chars wide.
2. **Parallel visibility**: parallel tests appear side by side.
3. **Full DAG visibility**: user can pan from layer 0 to production.
4. **Live status**: running, waiting, blocked, cached, skipped, failed, and done are visually distinct.
5. **Selected node obvious**: selected card has accent border and `[SEL]`.
6. **Edges visible**: dependency edges render between phases.
7. **Critical path visible**: user can trace the longest/most important path.
8. **Stall obvious**: blocker/failure root cause is shown in header and jumpable.
9. **Mouse panning**: wheel and drag work for navigation.
10. **Keyboard remains fast**: no mouse dependency.
11. **Inspector is useful**: Enter explains node status, deps, command, reason, and downstream impact.
12. **No hardcoded widget colors**: theme only.
13. **No render-time mutation of app state**: preserve architecture.
14. **No every-frame snapshot rebuild in live mode**.
15. **Demo mode animates**.

# 24. Final Design Principle

The Workflow tab should not merely display CI jobs.

It should become the **operational truth surface** for software delivery.

A developer should be able to open Tab 0 and immediately know:

> “What changed, what ran, what is still running, what failed, what is blocked, what is safe to merge, and how close we are to production.”

That is the bar.
