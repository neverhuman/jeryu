## Verdict

There is **no single global GitHub/GitLab bug workflow standard**. The closest formal standards are: **IEEE 1044** for classifying software anomalies, **ISO/IEC 23531** for issue-management tool capabilities, and, for security defects, **NIST SSDF + CVSS**. IEEE 1044 gives a uniform anomaly-classification approach across a product/system lifecycle, while ISO/IEC 23531 is explicitly about issue-management tool capabilities and is independent of Waterfall, Agile, or any other lifecycle method. ([CTest Labs][1])

The canonical engineering answer is therefore: **do not copy GitHub’s or GitLab’s default workflow. Define a provider-neutral bug schema and lifecycle, then map it onto GitHub Issues/Projects or GitLab Issues/Work Items/Boards.**

Call it **BugOps v1**:

> Every bug is a structured issue with enough evidence to reproduce, classify, assign, fix, verify, release, and learn from it.
> The issue is the source of truth. Pull requests or merge requests are execution records, not the bug record.

GitHub supports this with Issues, YAML issue forms, labels, assignees, issue types, milestones, Projects, custom fields, and automation; GitHub issue-form responses are converted into Markdown issue bodies, and forms can set labels, assignees, projects, and issue type. ([GitHub Docs][2]) GitLab supports the same canonical process through issues/work items, Markdown description templates, group/project/instance templates, quick actions, labels, scoped labels, milestones, iterations, and issue boards. ([GitLab Documentation][3])

# BugOps v1: canonical bug tracking engineering spec

## 1. Non-negotiable rules

1. **One bug, one issue.**
   A bug issue represents one observable failure mode. If one report contains three independent failures, split it into three issues and link them.

2. **The issue is the source of truth.**
   All reproduction info, triage decisions, severity, priority, owner, fix evidence, release status, and closure reason live on the issue.

3. **Users describe impact and evidence; maintainers assign severity and priority.**
   Users should not be asked to pick engineering priority. They should report what happened, how often, who is affected, and whether there is a workaround.

4. **Severity and priority are different.**
   Severity means objective impact. Priority means scheduling order.

5. **No secrets, credentials, tokens, private customer data, or exploit details in public issues.**
   Security-sensitive reports go to a private security process, not a public issue.

6. **A bug is not accepted until it is triaged.**
   New issues are reports, not commitments.

7. **A bug is not complete when code is written.**
   It is complete when the fix is merged, tested, verified, and released or otherwise made available to affected users.

8. **Every fixed bug gets regression protection.**
   The default expectation is a failing test, reproduction fixture, monitor, assertion, contract test, or documented reason why a test cannot be added.

9. **Every terminal closure must have a reason.**
   Valid closure reasons: `fixed`, `duplicate`, `not-a-bug`, `out-of-scope`, `cannot-reproduce`, `wont-fix`, `stale-needs-info`, `superseded`.

10. **Agents may act only on actionable issues.**
    A minimal-context agent should never have to infer product, version, environment, failure, expected behavior, or success criteria from memory.

# 2. Canonical bug issue data model

Use this schema regardless of provider.

## 2.1 User-supplied fields

These are the fields the reporter should provide.

| Field                               |                Required | Purpose                                                             |
| ----------------------------------- | ----------------------: | ------------------------------------------------------------------- |
| Summary                             |                     Yes | One-sentence observable failure.                                    |
| Product / repo / component          |                     Yes | Routes the bug to the right team.                                   |
| Affected version / build / commit   |                     Yes | Prevents chasing already-fixed or wrong-branch bugs.                |
| Environment                         |                     Yes | OS, browser, device, runtime, config, deployment, package versions. |
| Current behavior                    |                     Yes | What actually happened.                                             |
| Expected behavior                   |                     Yes | What should have happened.                                          |
| Steps to reproduce                  |       Yes when possible | Deterministic reproduction path.                                    |
| Reproduction artifact               |      Strongly preferred | Minimal repo, script, input file, URL, test case, payload, fixture. |
| Frequency                           |                     Yes | Always, intermittent, once, unknown.                                |
| User impact                         |                     Yes | Who is affected and how badly.                                      |
| Workaround                          |                Optional | Helps priority and support response.                                |
| Regression info                     |                Optional | Last known good version/date.                                       |
| Logs / traces / screenshots / video | Optional but encouraged | Evidence. Must be scrubbed.                                         |
| Contact / follow-up channel         |                Optional | Needed when more info is required.                                  |
| Security/privacy flag               |                     Yes | Determines public vs private handling.                              |
| “No secrets included” confirmation  |                     Yes | Prevents accidental exposure.                                       |

GitHub’s own bug issue form examples include the same core ideas: what happened, expected behavior, version, browser/environment, logs, screenshots, and required validations. ## 3.3 Severity definitions

Severity is **impact**, not scheduling.

| Severity | Meaning                       | Examples                                                                        |
| -------- | ----------------------------- | ------------------------------------------------------------------------------- |
| `S0`     | Emergency / active incident   | Total outage, active exploit, irreversible data loss, safety issue.             |
| `S1`     | Critical user-blocking defect | Major production functionality broken for many users, no reasonable workaround. |
| `S2`     | High-impact defect            | Important workflow broken, partial workaround exists, limited blast radius.     |
| `S3`     | Normal defect                 | Incorrect behavior with workaround or limited user impact.                      |
| `S4`     | Low-impact defect             | Cosmetic, copy, minor UX, edge-case annoyance.                                  |

## 3.4 Priority definitions

Priority is **when we work on it**.

| Priority | Meaning                               |
| -------- | ------------------------------------- |
| `P0`     | Drop everything; incident response.   |
| `P1`     | Current release / current sprint.     |
| `P2`     | Next planned cycle.                   |
| `P3`     | Backlog, accepted but not scheduled.  |
| `P4`     | Parking lot; valid but unlikely soon. |

Security bugs should additionally use **CVSS** when they are vulnerabilities. CVSS v4.0 is an open framework for communicating software, hardware, and firmware vulnerability severity, using Base, Threat, Environmental, and Supplemental metric groups, with scores and vectors intended to make severity derivation transparent. ([FIRST][6])

# 4. Canonical lifecycle

## 4.1 State machine

```text
needs-triage
  -> needs-info
  -> triaged
  -> accepted
  -> scheduled
  -> in-progress
  -> fix-proposed
  -> ready-for-verification
  -> verified
  -> released
  -> closed
```

Allowed side exits:

```text
needs-triage -> closed:duplicate
needs-triage -> closed:not-a-bug
needs-triage -> closed:out-of-scope
needs-info   -> closed:stale-needs-info
triaged      -> closed:cannot-reproduce
triaged      -> closed:wont-fix
accepted     -> closed:superseded
```

## 4.2 Status definitions

| Status                   | Definition                                                                |
| ------------------------ | ------------------------------------------------------------------------- |
| `needs-triage`           | New report. No commitment yet.                                            |
| `needs-info`             | Missing information prevents action.                                      |
| `triaged`                | Type, component, severity, priority, and reproducibility status are set.  |
| `accepted`               | The team agrees this is an in-scope defect worth fixing or investigating. |
| `scheduled`              | Milestone, iteration, release, or owner commitment exists.                |
| `in-progress`            | An owner is actively working.                                             |
| `blocked`                | Work cannot proceed until a named dependency is resolved.                 |
| `fix-proposed`           | PR/MR exists.                                                             |
| `ready-for-verification` | Fix is merged or deployed to a testable environment.                      |
| `verified`               | Acceptance criteria passed.                                               |
| `released`               | Affected users can obtain the fix.                                        |
| `closed`                 | Terminal state with closure reason.                                       |

# 5. Acceptance criteria: when a bug becomes “accepted”

A bug is **accepted** only when all of these are true:

1. The report is in scope for the repo/team.
2. The observed behavior is plausibly wrong.
3. Expected behavior is clear.
4. Affected version/build/environment is known.
5. Impact is understood enough to assign severity.
6. Priority is assigned by the owning team.
7. Primary component is assigned.
8. Reproducibility is classified.
9. Duplicate search has been performed.
10. A next action exists: investigate, fix, ask for info, defer, or close with reason.
11. If public handling would expose sensitive information, the issue is moved to the private security/support path.

A bug may be accepted even if it is not yet reproduced, but only when the impact justifies investigation. In that case use:

```text
status::accepted
repro::not-reproduced
```

and add an explicit investigation task.

---

# 6. Completion criteria: when a bug is actually done

A bug is **complete** only when all required gates pass:

1. The root cause is identified or the fix rationale is documented.
2. The fix PR/MR is linked.
3. A regression test, fixture, monitor, or explicit “no test possible” rationale exists.
4. CI passes.
5. The fix is reviewed.
6. The fix is merged.
7. QA, maintainer, or agent verifies the original reproduction path.
8. User-facing change is released or deployed.
9. Reporter is notified when appropriate.
10. Issue is closed with reason `fixed`.

Important rule: **do not use `Fixes #123` if merge does not mean user-visible completion.** Use `Refs #123` until release/deployment is complete. Close only after the fix is actually available to affected users.

For severe bugs, add a follow-up root-cause review. NIST SSDF explicitly recommends recording vulnerabilities in issue tracking software, making risk-based remediation decisions, and analyzing root causes for vulnerabilities; the same discipline is useful for severe non-security bugs. ([NIST Publications][7])

## 7.2 GitHub automation requirements

Add workflows or app automation for:

```text
on issue opened:
  - add to canonical Bug Triage project
  - validate required headings/fields
  - parse component/version/environment/repro fields
  - scan for obvious secrets
  - if security/privacy=yes, comment with private-report instructions and restrict/move where possible
  - set status::needs-triage
  - assign triage rotation

on issue edited:
  - re-run validation
  - if missing info resolved, move from needs-info to needs-triage

on triage complete:
  - enforce exactly one severity, priority, component, status, repro value
  - perform duplicate search
  - assign owner or team

on PR opened:
  - require linked issue for bugfix PRs
  - require regression test or documented exception
  - copy issue severity/priority into PR context

on PR merged:
  - if merge means fix is user-visible, move to ready-for-verification or verified
  - if deployment/release is still required, do not close; move to ready-for-verification

on release/deploy:
  - move verified fixed bugs to released
  - notify reporter if appropriate
  - close with reason fixed
```

# 9. Minimal-context agent contract

An agent should be able to pick up an accepted issue and work from the issue alone.

## 9.1 Agent readiness rule

An issue is agent-ready only when:

```text
type::bug exists
status is accepted, scheduled, or in-progress
component is known
affected version is known
expected behavior is known
current behavior is known
reproduction path or investigation path exists
severity is known
priority is known
security/private flag is not public-sensitive
```

## 9.2 Agent tasks

The agent’s job is not “fix vaguely.” The agent must follow this sequence:

1. Read the issue body and linked evidence.
2. Confirm scope and component.
3. Reproduce the bug or document why reproduction failed.
4. Add or propose a failing test.
5. Identify likely root cause.
6. Make the smallest safe fix.
7. Run relevant tests.
8. Open PR/MR linked to the issue.
9. Update the issue with:

   * reproduction result,
   * root cause,
   * files changed,
   * tests added,
   * tests run,
   * residual risk,
   * verification instructions.

## 9.3 Agent stop conditions

The agent must stop and ask for triage/human help when:

```text
security/private concern is detected
reproduction requires secrets or private data
requirements are ambiguous
the expected behavior conflicts with existing docs/specs
the fix requires product decision
the issue affects public API compatibility
the bug appears to be data corruption
the fix requires migration or irreversible change
```

# 10. Triage process

## 10.1 New issue triage checklist

For every new bug:

```text
[ ] Is this actually a bug, not a feature request or support question?
[ ] Is it in scope for this repo/team?
[ ] Is it a duplicate?
[ ] Is sensitive/security information present?
[ ] Are required fields present?
[ ] Is the component known?
[ ] Is the affected version known?
[ ] Is expected behavior clear?
[ ] Are reproduction steps present or is impact high enough to investigate anyway?
[ ] Is severity assigned?
[ ] Is priority assigned?
[ ] Is owner/team assigned?
[ ] Is the next action clear?
```

## 10.2 Missing information policy

When information is missing:

```text
status::needs-info
```

The triage response should ask for **one concise set of missing facts**, not a vague “please provide more details.”

Example:

```md
Thanks for the report. We need three things before this is actionable:

1. The affected version or commit.
2. The exact command or click path that triggers the failure.
3. The expected output versus the actual output.

Please also confirm that no secrets or private customer data are included.
```

Recommended closure policy:

```text
needs-info after 14 days with no reporter response -> closed::stale-needs-info
```

The issue may be reopened any time the missing information arrives.

# 11. Fix PR/MR requirements

Every bugfix PR/MR must include:

```text
[ ] Linked bug issue
[ ] Root cause summary
[ ] Reproduction confirmed or investigation documented
[ ] Regression test / fixture / monitor added
[ ] Manual verification steps
[ ] Risk assessment
[ ] Rollback notes if production-facing
[ ] Release note if user-visible
```

PR/MR title format:

```text
fix(<component>): <concise behavior corrected>
```

Branch format:

```text
fix/<issue-number>-<short-slug>
```

Use:

```text
Refs #123
```

when merge does not equal completion.

Use:

```text
Fixes #123
```

only when merging the PR/MR satisfies the issue’s definition of done.

# 12. Security bugs

Security bugs are not normal public bugs.

Required behavior:

1. Public issue form asks whether the report has security/privacy impact.
2. If yes or unsure, redirect to private reporting.
3. Security issue gets:

   * private visibility,
   * security owner,
   * CVSS vector when applicable,
   * embargo/disclosure status,
   * affected versions,
   * fixed versions,
   * advisory/CVE status if relevant,
   * coordinated release plan.
4. Public issue should not include exploit details, secrets, or sensitive logs.

For vulnerability work, NIST SSDF’s “Respond to Vulnerabilities” practices include identifying and confirming vulnerabilities, having policy and response roles, using issue tracking to record vulnerabilities, planning risk responses, and analyzing root causes. ([NIST Publications][7])

| PR/MR integration           | Linked PRs, Actions, APIs          | Linked MRs, quick actions, APIs               | Require linked fix record.        |
| Organization-wide templates | Org/community health and templates | Group/instance templates                      | Centralize templates.             |

GitHub labels categorize issues, pull requests, and discussions, and GitHub provides default labels such as `bug`, `duplicate`, and `enhancement`; the canonical spec should replace vague defaults with scoped labels that encode workflow state and classification. ([GitHub Docs][9]) GitLab has project, group, and instance labels, so for multi-repo organizations, group labels are the natural place for the canonical taxonomy. ([GitLab Documentation][10])

What version are you using?

## Environment

- Browser/device:

What happened?

What should have happened?

Always / often / intermittent / once / unknown.

Who is affected? How badly? Is there a workaround?

# 20. Final canonical standard

Use this as the policy:

> A bug is a structured anomaly report.
> It enters as `needs-triage`, becomes `accepted` only after classification and ownership, becomes `in-progress` only with an assigned owner, becomes `fix-proposed` only with a linked PR/MR, becomes `verified` only after the original failure is proven fixed, becomes `released` only when users can obtain the fix, and becomes `closed` only with a terminal reason.

That is the cleanest cross-provider standard: **IEEE-style anomaly discipline, GitHub/GitLab-native implementation, strict state machine, clear labels, evidence-driven acceptance, test-backed completion, and agent-readable structure.**
