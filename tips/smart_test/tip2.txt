Yes. The best path is **not** “teach GitLab path filters to be clever.” The best path is to make jeryu itself a **Test Intelligence Plane**: a deterministic Rust subsystem that owns impact analysis, builds a source→test dependency graph, generates the smallest safe CI pipeline, records what it skipped, learns from nightly/full runs, and escalates to broader testing only when confidence is low.

The current `impact.rs` design is a good seed, but it is too coarse: `src/**/*.rs → Unit`, `tests/ → Integration`, `Cargo.toml/.gitlab-ci → Full`. That is useful for a first-pass lane selector, but it will not get you to 100×. To get there, jeryu needs to reason at the level of **files, crates, modules, public APIs, test binaries, individual nextest tests, CI jobs, runtime resources, and historical failure edges**.

## The core recommendation

Build a new subsystem: **VTI — jeryu Test Intelligence**.

It should become the authority behind:

```bash
jeryu test plan --base <sha> --head <sha> --json
jeryu test run --affected
jeryu test explain --base <sha> --head <sha>
jeryu ci generate --affected --out .gitlab/jeryu-generated.yml
jeryu test learn --from-pipeline <id>
jeryu test audit-misses
```

The default PR/push path should run only:

1. the affected compile/check/lint surface;
2. the tests directly or transitively affected by changed files;
3. tests that recently failed or are flaky/quarantined;
4. newly added tests;
5. a tiny random “sentinel” sample of skipped tests.

Full testing should run on nightly, release gates, risky files, low-confidence graph cases, and periodic audit runs.

That is exactly the pattern used by serious systems: Nx computes affected projects from Git diff + project graph + dependents, and recommends using the latest successful main commit as the base in CI; it also says affected selection should be paired with remote caching and distributed execution. ([Nx][1]) Pants exposes project targets and dependents so a file can be mapped to tests through graph introspection. ([Pants][2]) Bazel’s model is explicit action inputs/outputs plus cacheable action results; it reruns tests when the test or dependencies changed, when the test failed, or under other explicit invalidation conditions. ([Bazel][3]) Microsoft’s Test Impact Analysis pattern also includes impacted tests, previously failing tests, newly added tests, safe fallback to all tests when the system cannot reason about a change, and configurable periodic full runs. ([Microsoft Learn][4])

## Why GitLab `rules:changes` alone is not enough

Use GitLab path rules as a coarse outer optimization, but do not make them the brain. GitLab’s `rules:changes` checks whether listed files changed, but it is path/glob based, OR-based, and has documented special cases: for new branches or pipelines without a Git push event, it can evaluate true unless `compare_to` is used; merge request pipelines compare to the target branch, while branch pipelines compare to the previous branch commit. ([GitLab Docs][5]) It also has hard limits: max 50 patterns per `rules:changes` section, and after 50,000 path checks patterned globs assume a match. ([GitLab Docs][5])

For jeryu, GitLab should execute the plan, not decide the plan.

The better GitLab architecture is:

```yaml
plan:
  stage: plan
  script:
    - jeryu test plan \
        --base "${CI_MERGE_REQUEST_DIFF_BASE_SHA:-$CI_COMMIT_BEFORE_SHA}" \
        --head "$CI_COMMIT_SHA" \
        --emit-child target/jeryu-selected-ci.yml \
        --emit-json target/jeryu-test-plan.json
  artifacts:
    paths:
      - target/jeryu-selected-ci.yml
      - target/jeryu-test-plan.json

selected-tests:
  stage: test
  needs: ["plan"]
  trigger:
    include:
      - artifact: target/jeryu-selected-ci.yml
        job: plan
    strategy: depend
```

GitLab supports dynamic child pipelines generated by a job artifact, and its own docs call this “very powerful” for pipelines targeting changed content or generated target matrices. ([GitLab Docs][6]) Use `strategy: depend` or `strategy: mirror` so child-pipeline reports are visible to the parent/MR where supported. ([GitLab Docs][6]) GitLab scheduled pipelines are the right primitive for nightly full validation because they run independently of code changes. ([GitLab Docs][7])

Cargo itself can already avoid building/running everything if target-selection flags are used. ([Rust Documentation][13]) Nextest gives stronger machine integration: it can list tests without running them, emit JUnit, filter via expressions, partition work, rerun failures from recorded runs, and run tests with retries. ([Nexte.st][14])

One important caveat: because jeryu is currently a **single-binary Rust project**, selecting one Rust test can still require compiling a large test binary. Runtime can drop by 100×, but compile time may not unless you also split the codebase into smaller workspace crates and lean hard on sccache/SmartCache. The long-term structure should still produce one `jeryu` binary, but the internals should be crates aligned to the four planes:

## The selection algorithm

The planner should produce a signed, explainable `TestPlan`.

### Step 1: Normalize the diff

Inputs:

```text
base_sha
head_sha
changed_paths
change_kind: added | modified | deleted | renamed
line ranges
file hashes
```

Base resolution order:

1. MR diff base SHA, if present.
2. Push hook `before` SHA.
3. Latest successful `main` pipeline SHA.
4. Fallback: full test plan.

Nx’s affected-CI guidance uses exactly this concept: affected calculation needs both base and head, and in CI the base should usually be the latest successful commit on main so all changes since the last known-green point are accounted for. ([Nx][1])

### Step 2: Classify changed files

Example hard-coded first-pass:

| Change                                                                                                    | Default lane                                                                     |
| --------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------- |
| `README.md`, docs-only files                                                                              | docs lint only, no Rust tests unless docs include commands/templates.            |
| `src/decision.rs`                                                                                         | decision unit tests + impact planner tests.                                      |
| `src/impact.rs` / `src/test_intel/**`                                                                     | planner tests + targeted integration of CI generation.                           |
| `src/state.rs`                                                                                            | RedlineDB memory CRUD tests + affected schema tests.                                |
| `src/config.rs`                                                                                           | config template tests + generated compose/runner config snapshot tests.          |
| `src/gitlab_client.rs`                                                                                    | API client unit/mock tests + live smoke only if endpoint semantics changed.      |
| `src/docker.rs`, `src/pool.rs`                                                                            | pool tests + Docker controller tests; no agent/release/e2e unless graph says so. |
| `src/engine.rs`                                                                                           | webhook parser, supersedence, retry, release-trigger tests.                      |
| `src/agent.rs`, `src/capability.rs`                                                                       | agent/capability tests + risk-gated merge tests.                                 |
| `src/exec.rs`, `src/sandbox.rs`, `src/honeypot.rs`, `src/capsule.rs`                                      | custom executor, sandbox, honeypot, capsule replay tests.                        |
| `src/cache*.rs`, `src/gateway/**`, `src/epoch.rs`, `src/taint.rs`, `src/witness.rs`, `src/sccache_mgr.rs` | cache integration tests only.                                                    |
| `src/release.rs`                                                                                          | release state-machine tests + pipeline explain tests.                            |
| `src/secrets.rs`                                                                                          | Vault mock tests; live Vault tests only if Vault lifecycle paths changed.        |
| `src/tui/**`                                                                                              | TUI `--once` smoke + widget rendering tests.                                     |
| `Cargo.toml`, `Cargo.lock`, `.gitlab-ci.yml`, workflow files, build images                                | broad/full, because dependencies and CI semantics changed.                       |
| `tests/e2e.rs`, `tests/common/**`                                                                         | e2e lane, but only those e2e cases affected.                                     |

### Step 3: Build the affected graph

Graph nodes:

```text
SourceFile
RustModule
CargoPackage
CargoTarget
PublicSymbol
TestCase
TestBinary
TestSuite
CiJob
RuntimeResource      # docker, gitlab, vault, network, apfs, cache
SecretSurface
PolicySurface
```

Graph edges:

```text
SourceFile -> RustModule
RustModule -> CargoPackage
CargoPackage -> CargoTarget
CargoTarget -> TestBinary
TestBinary -> TestCase
SourceFile -> TestCase          # dynamic coverage edge
GlobRule -> TestSuite           # declared rule
RuntimeResource -> TestSuite
FailedTest -> TestCase          # previous failure inclusion
FlakyTest -> TestCase           # retry/quarantine inclusion
```

A test is selected if any of these are true:

```text
test file changed
test covers changed file/function
test imports affected package/module
test belongs to affected declared suite
test depends on changed runtime resource
test failed recently
test is flaky/quarantined and relevant
test is new
test is sentinel-sampled
policy requires lane expansion
```

### Step 4: Confidence scoring

Each candidate test gets a reason and confidence:

| Source of edge                             |                                 Confidence |
| ------------------------------------------ | -----------------------------------------: |
| Test file changed directly                 |                                       1.00 |
| Dynamic coverage edge from recent full run |                                       0.95 |
| Static Cargo package reverse dependency    |                                       0.80 |
| Explicit `jeryu.test.toml` rule             |           0.75–0.95 depending on freshness |
| Name/module heuristic                      |                                       0.50 |
| LLM suggestion only                        | 0.20, never sufficient to skip other tests |
| Unknown file type                          |                            0.00 → escalate |

Then choose the narrowest safe plan:

```text
Exact tests        confidence >= 0.95
Test binary/file   confidence >= 0.85
Package/lane       confidence >= 0.70
Full               confidence < 0.70 or force_full rule
```

The key principle: **AI may expand the plan, never shrink it below deterministic evidence.**

## Self-management loop

This is the most important part. A smart selector becomes dangerous if it never audits itself. jeryu is already built around ledgers, failure capsules, witness attestation, cache verdicts, and policy gates, so use those strengths.

### 1. Every selected run records evidence

For every pipeline, store:

```json
{
  "base": "abc",
  "head": "def",
  "changed_paths": ["src/pool.rs"],
  "selected_tests": ["pool_tests::scale_up", "pool_tests::drain"],
  "skipped_tests": ["agent_tests::*", "cache_integration_test::*"],
  "selection_reasons": {
    "pool_tests::scale_up": ["changed:src/pool.rs", "declared-suite:pool-docker"]
  },
  "confidence": 0.91,
  "fallbacks": [],
  "policy": "trusted-pr-fast"
}
```

Then append that to the jeryu event ledger and create a witness attestation. That makes “we skipped these tests” auditable.

### 2. Nightly full run audits skipped tests

Nightly does:

```bash
jeryu test run --full --record-coverage --record-junit
jeryu test audit-misses
jeryu test learn
```

If a skipped test fails on nightly and the failure traces back to a prior diff that skipped it, jeryu records a **selector miss**:

```text
selector_miss:
  changed_path: src/pool.rs
  skipped_test: e2e::runner_picks_up_job
  failure_kind: regression
  repair: add edge src/pool.rs -> e2e-full-lifecycle
```

Then jeryu can either:

1. open an agent MR updating `jeryu.test.toml`;
2. lower confidence for similar future changes;
3. temporarily broaden that module’s testing lane;
4. block auto-merge if misses exceed budget.

### 3. AI proposes, policy disposes

Use AI where it is strongest:

* read diffs and propose likely impacted suites;
* explain why a plan is safe or risky;
* propose missing `jeryu.test.toml` mappings;
* generate new tests for uncovered changed code;
* summarize selector misses.

Do **not** let the LLM be the final authority for skipping. The final skip decision should be deterministic and stored in RedlineDB. The LLM can add tests or request escalation; it should not remove deterministic tests from the plan.

### 4. Random sentinel testing

On each PR, run a tiny sample of skipped tests:

```text
sentinel_sample = min(5 tests, 3% of skipped tests)
bias toward:
  historically flaky tests
  recently failed tests
  expensive-but-critical e2e tests at low frequency
  tests with stale coverage edges
```

This catches silent graph drift without turning every PR into a full run.

### 5. Confidence decay

Dynamic source→test mappings should expire. For example:

```text
coverage edge age < 7 days      full confidence
7–30 days                       decayed confidence
>30 days                        stale; broaden test lane
test changed since edge learned stale; relearn
source file heavily refactored  stale; broaden
```

## Dynamic source→test learning

Use two learning modes.

### Fast mode: every PR

Ingest:

* nextest JUnit status/duration;
* job logs and failure capsules;
* test names from nextest metadata;
* changed files;
* selected/skipped reasons.

Nextest can emit JUnit where each test binary is a `<testsuite>` and each test is a `<testcase>`. ([Nexte.st][15]) It also exposes runtime environment variables such as `NEXTEST_BINARY_ID`, `NEXTEST_TEST_NAME`, attempt IDs, test groups, and workspace root, which are useful for correlating test execution and jeryu telemetry. ([Nexte.st][16])

### Slow mode: nightly/full

Run coverage-instrumented tests to learn file/function coverage. Rust’s LLVM coverage support instruments functions/branches and writes profiling data that can be turned into coverage reports. ([Rust Documentation][12])

Do not put per-test coverage learning in every PR. It is too expensive. Use nightly and selective audits. For very high-value tests, jeryu can run individual tests under coverage after enumerating them with nextest metadata.

```yaml
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
      - name: Install nextest
        run: cargo install cargo-nextest --locked
      - name: Run affected tests
        run: cargo run -- test run --plan target/jeryu-test-plan.json

  nightly-full:
    if: github.event_name == 'schedule'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
      - name: Full test learning run
        run: cargo run -- test run --full --record-coverage
```

Keep `fmt` and `clippy`, but narrow them:

```bash
# changed Rust files only for formatting
git diff --name-only "$BASE..$HEAD" -- '*.rs' | xargs -r rustfmt --check

# affected packages only once you split into workspace crates
cargo clippy -p jeryu-decision --tests -- -D warnings
```

## Test suite restructuring for jeryu

To make this work well, restructure tests around resource cost and dependency ownership:

```text
tests/
├── unit/
│   ├── decision_tests.rs
│   ├── impact_tests.rs
│   ├── config_template_tests.rs
│   └── state_tests.rs
├── integration/
│   ├── pool_tests.rs
│   ├── agent_tests.rs
│   ├── cache_integration_test.rs
│   ├── secrets_vault_tests.rs
│   └── engine_webhook_tests.rs
├── e2e/
│   └── full_lifecycle.rs
└── common/
```

Use Cargo features or nextest groups for heavy tests:

```toml
# .config/nextest.toml

[profile.default]
retries = 0
fail-fast = false

[profile.ci]
retries = 1
fail-fast = false

[profile.ci.junit]
path = "target/nextest/ci/junit.xml"

[test-groups.docker]
max-threads = 1

[test-groups.gitlab]
max-threads = 1

[[profile.ci.overrides]]
filter = 'test(/e2e|full_lifecycle/)'
test-group = 'gitlab'
slow-timeout = { period = "120s", terminate-after = 2 }

[[profile.ci.overrides]]
filter = 'test(/pool|docker/)'
test-group = 'docker'
retries = 1
```

Nextest supports per-test settings selected by filtersets, including retries, test groups, thread requirements, priority, and slow timeouts. ([Nexte.st][17])

## Safety policy

Selected testing is safe only under a strict policy.

### Always include

```text
recently failed tests
new tests
tests touching changed files
tests with high-confidence coverage edges
tests mapped by explicit rules
small sentinel sample
```

### Always broaden or full-run

```text
unknown file type
Cargo.lock changed
Cargo.toml dependency/features changed
CI YAML changed
security/policy/admission changed under untrusted tier
coverage graph stale
too many files changed
base SHA unavailable
selector misses above budget
release promotion
nightly schedule
```

### Trust tier behavior

| Trust tier   | Test selection policy                                                                     |
| ------------ | ----------------------------------------------------------------------------------------- |
| `Trusted`    | Narrow selected plan allowed if confidence high and no recent selector misses.            |
| `Privileged` | Selected plan allowed, but include more sentinels and release-critical smoke.             |
| `Untrusted`  | Selected plan for feedback, but no auto-merge without broader or human-approved evidence. |

This plugs directly into your existing `TrustTier`, `RiskGate`, and `RequiredEvidencePolicy`.

## What jeryu should invent

You do **not** need to invent nextest, coverage, Git diffing, or Cargo graph parsing.

You **do** need to invent the jeryu-native part:

1. **The ledgered test plan** — exact record of why each test ran or skipped.
2. **The source→test graph** — combining declared rules, Cargo metadata, historical failures, and coverage.
3. **The self-healing loop** — nightly miss detection and automatic rule repair.
4. **The policy gate** — deterministic admission rule that says whether selected testing is enough.
5. **The dynamic pipeline generator** — per-diff child CI instead of static `.gitlab-ci.yml`.
6. **The AI-assisted maintainer** — proposes mappings/tests, but cannot silently reduce coverage.

That is where jeryu can be uniquely strong.

## Implementation sequence

### Phase 1 — Immediate speedup

1. Add `cargo-nextest`.
2. Emit JUnit in CI.
3. Move live GitLab/Docker/Vault tests into explicit nextest groups.
4. Add `jeryu test plan --base --head` with path rules from `jeryu.test.toml`.
5. Replace `cargo test --tests` with selected nextest commands.
6. Add nightly full schedule.

Expected result: large runtime drop for docs/unit/module-only changes.

### Phase 2 — Graph-backed selection

1. Add `cargo_metadata` + `guppy`.
2. Add `nextest_metadata` test discovery.
3. Store `test_cases`, `test_edges`, `test_plans`, `test_runs`.
4. Generate GitLab dynamic child pipelines.
5. Add `jeryu test explain`.

Expected result: reliable package/test-binary selection and explainable skips.

### Phase 3 — Self-learning

1. Ingest JUnit and failure capsules.
2. Track recent failures and flaky scores.
3. Nightly full run records coverage.
4. Detect selector misses.
5. Auto-open agent MR to update `jeryu.test.toml`.

Expected result: the system improves rather than rots.

### Phase 4 — True 100× potential

1. Split jeryu internals into workspace crates aligned with architecture planes.
2. Cache build artifacts through SmartCache/sccache.
3. Add symbol-level Rust analysis.
4. Add per-test coverage learning for high-value modules.
5. Add sentinel sampling and selector-miss budgets to admission control.

Expected result: narrow code changes compile and test only a small crate/test subset.

## The main principle

Do not ask, “Which tests can we skip?”

Ask:

> “What is the smallest test/evidence set that proves this diff is safe under the current trust tier, with an auditable fallback if the selector is uncertain?”

That framing fits jeryu perfectly. Git remains the ledger, RedlineDB stores the evolving test graph, GitLab Runner executes only the generated plan, SmartCache avoids repeated work, Failure Capsules teach the selector, and the Risk Gate decides whether the evidence is enough.
