Yes. The right answer is **not** “add more `rules:changes` to CI.” The right answer is to make test selection a **first-class jeryu subsystem**: a deterministic, auditable **Test Impact Kernel** that decides what evidence is required for each change, generates the minimal pipeline, records why every test ran or skipped, and audits itself with nightly/full runs.

The design principle should be:

> **AI may propose test scope. Rust must prove, enforce, execute, and audit it.**

## 1. What best-in-class systems do

The mature pattern is consistent across modern CI systems:

**Affected graph first.** Nx’s affected mode uses Git changed files plus a project graph to find the minimum affected projects, then runs only tasks for that subset. It also recommends using the latest successful main commit as the base SHA, not just `origin/main`, so missed failures from earlier broken runs are not ignored. ([Nx][1])

**Transitive dependents matter.** Pants explicitly recommends running tests for changed files plus transitive dependents, because a change to a dependency can break tests elsewhere even if the test file itself did not change. ([Pants][2])

**Safe fallback is mandatory.** Microsoft’s Test Impact Analysis model includes impacted tests, previously failing tests, newly added tests, safe fallback to all tests when it cannot reason about a change, and configurable periodic full test runs. That is exactly the safety posture jeryu should copy. ([Microsoft Learn][3])

**Coverage and tracked invalidators beat pure prediction.** Datadog’s Test Impact Analysis uses code coverage to skip irrelevant tests, recommends excluding critical/default branches from skipping, supports tracked files that force full runs, and requires a baseline run on the default branch. ([Datadog Monitoring][4])

**GitLab YAML should execute the decision, not make the decision.** GitLab `rules` are evaluated before jobs run, so they cannot depend on a selector job’s generated dotenv output. jeryu should therefore generate a dynamic child pipeline from a YAML artifact after it has computed the test plan. GitLab officially supports dynamic child pipelines generated by a job artifact, and `needs` can start independent jobs immediately without waiting for unrelated stages. ([GitLab Docs][5])

**Rust already has the primitives.** Cargo can emit workspace and dependency metadata as JSON, `guppy` can query Cargo dependency graphs, and `cargo-nextest` can list tests as JSON and run precise filtersets by test name, package, binary, reverse dependency, and kind. ([Rust Documentation][6])

So the best jeryu system should combine: **changed files + Rust dependency graph + declarative test contracts + coverage history + failure history + cache/witness attestation + nightly audit.**

# 2. The system to build: jeryu Test Impact Kernel

Add a new subsystem, conceptually:

```text
src/test_index.rs      # discovers tests, crates, binaries, files, symbols
src/test_impact.rs     # computes affected tests from diff + graph + coverage + policy
src/test_plan.rs       # turns impact into executable lanes and child-pipeline YAML
src/test_cache.rs      # witnessed test-result reuse and skip accounting
src/test_audit.rs      # nightly/full-run comparison and selector self-healing
src/test_explain.rs    # human/machine explanation of why tests ran/skipped
```

This extends your existing `impact.rs` and `test_runner.rs`, rather than replacing them. Today, `impact.rs` has coarse lanes like `Unit`, `Integration`, `DocsOnly`, and `Full`. The new kernel makes those lanes **evidence-backed and per-test/per-binary**, while still preserving conservative escalation to full testing.

The key architecture fit is excellent:

| jeryu plane          | New testing responsibility                                                                                                    |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| **Admission Plane** | Reject merges when the test plan is below policy, selector changed without full audit, or critical invalidators were skipped. |
| **Control Plane**   | Store every selection, skip, cache hit, and audit result in SQLite/event ledger.                                              |
| **Execution Plane** | Run selected tests in sandboxes; run E2E/detonation lanes only when required or scheduled.                                    |
| **Cognition Plane** | Let AI propose mappings and explain risks, but never be the sole authority for skipping.                                      |

The selector should behave like a mini build system inside jeryu: **Git is the ledger, Rust is the policy kernel, GitLab Runner is the executor.**

# 3. The core algorithm

For every push, MR, or agent patch:

```rust
fn select_tests(diff: Diff, ctx: PipelineContext) -> TestPlan {
    let changed = normalize_changed_files(diff);

    if force_full(ctx) {
        return full_plan("forced by label/env/schedule");
    }

    if global_invalidator_changed(&changed) {
        return full_plan("global invalidator changed");
    }

    let graph_nodes = index.map_files_to_nodes(&changed);
    let static_impacted = graph.reverse_dependents(graph_nodes);
    let declared_impacted = testmap.match_rules(&changed);
    let coverage_impacted = coverage.tests_covering(&changed);
    let historical_impacted = history.tests_that_failed_for_similar_changes(&changed);
    let mandatory = policy.mandatory_tests(ctx, &changed);

    let candidates = union(
        static_impacted,
        declared_impacted,
        coverage_impacted,
        historical_impacted,
        mandatory,
        newly_added_tests(&changed),
        previously_failing_tests(ctx),
    );

    let confidence = score_evidence(&candidates, &changed, ctx);

    if confidence < policy.min_confidence {
        return escalate(candidates, confidence);
    }

    attach_witnessed_cache_verdicts(candidates)
}
```

The important rule:

> jeryu may skip a test only when it can produce a machine-readable explanation: changed input set, selected graph path, coverage relationship, declarative rule, or witnessed cache key.

No “AI intuition” skip decisions.

# 4. Evidence sources

## 4.1 Git diff source of truth

Use `git2`, which jeryu already uses in `impact.rs`, to compute changed paths against:

1. MR target branch merge base for MRs.
2. Latest successful main SHA for branch pipelines.
3. Previous green release-critical pipeline for release validation.
4. Explicit `--base/--head` for local CLI.

Do **not** default to “previous commit” in CI. That misses earlier unvalidated commits. Nx explicitly recommends using the latest successful main commit as the base in CI. ([Nx][1])

## 4.2 Rust package graph

Use `cargo_metadata` or `guppy`:

```rust
cargo_metadata::MetadataCommand::new()
    .manifest_path("Cargo.toml")
    .features(CargoOpt::AllFeatures)
    .exec()
```

`cargo metadata` exposes workspace members and dependency resolution as JSON, and Cargo recommends specifying `--format-version` for compatibility. ([Rust Documentation][6])

`guppy` then gives jeryu an in-Rust queryable package graph. ([Docs.rs][7])

For jeryu specifically, this is useful but not sufficient, because your current system is mostly a **single binary crate**. To unlock true 100x CI speedups, split the codebase into workspace crates while preserving the single `jeryu` binary:

```text
crates/
  jeryu-core/          # state, config, policy, decision
  jeryu-gitlab/        # GitLab client + endpoint tests
  jeryu-docker/        # Docker control plane
  jeryu-pool/          # runner pool manager
  jeryu-agent/         # agent/capability logic
  jeryu-exec/          # custom executor, sandbox, honeypot, capsules
  jeryu-cache/         # SmartCache, gateway, taint, witness
  jeryu-release/       # release + secrets
  jeryu-tui/           # Ratatui app
  jeryu-cli/           # clap command wiring
src/main.rs           # thin binary shell
```

This keeps the product a single binary but makes the dependency graph meaningful. Without this, file-level test selection still helps, but Cargo may rebuild too much.

## 4.3 Test inventory

Use `cargo nextest list --message-format json` to index every test and binary. Nextest documents machine-readable test and binary listings and recommends the `nextest-metadata` crate for Rust parsers. ([Nexte.st][8])

Store:

```sql
CREATE TABLE test_catalog (
    test_id            TEXT PRIMARY KEY,
    package_name       TEXT NOT NULL,
    binary_name        TEXT NOT NULL,
    binary_kind        TEXT NOT NULL, -- lib, test, bin, proc-macro
    test_name          TEXT NOT NULL,
    lane               TEXT NOT NULL,
    requires_docker    BOOLEAN NOT NULL DEFAULT FALSE,
    requires_gitlab    BOOLEAN NOT NULL DEFAULT FALSE,
    requires_vault     BOOLEAN NOT NULL DEFAULT FALSE,
    is_e2e             BOOLEAN NOT NULL DEFAULT FALSE,
    is_flaky           BOOLEAN NOT NULL DEFAULT FALSE,
    p50_duration_ms    INTEGER,
    p95_duration_ms    INTEGER,
    last_seen_sha      TEXT NOT NULL
);
```

Nextest filtersets can select by package, reverse dependencies, binary, test name, and binary kind, which maps cleanly to your lanes and integration binaries. ([Nexte.st][9])

## 4.4 Declarative test contracts

Create a repository-owned file:

```text
.jeryu/testmap.toml
```

This is the human-readable contract between source areas and required tests. jeryu can generate, lint, and update it, but it should live in Git so changes are reviewable.

Example:

```toml
[settings]
base_strategy = "latest-green-main"
min_confidence = 0.86
full_on_selector_change = true
full_on_unknown_file = true
nightly_full_required = true

[global_invalidators]
paths = [
  "Cargo.lock",
  "Cargo.toml",
  "rust-toolchain.toml",
  ".cargo/**",
  ".gitlab-ci.yml",
  ".github/workflows/**",
  ".jeryu/testmap.toml",
  "src/impact.rs",
  "src/test_impact.rs",
  "src/test_plan.rs",
  "src/policy.rs"
]

[[lanes]]
name = "unit"
runner_tags = ["default", "rust", "test"]
command = "cargo nextest run --workspace -E '${filter}'"

[[lanes]]
name = "integration"
runner_tags = ["default", "rust", "test"]
command = "cargo nextest run --workspace -E '${filter}' --profile ci"

[[lanes]]
name = "e2e"
runner_tags = ["build", "docker-build"]
command = "cargo nextest run -E '${filter}' --profile e2e"

[[rules]]
id = "decision-core"
changes = ["src/decision.rs", "src/policy.rs"]
select = [
  "binary(jeryu)",
  "test(/decision/)",
  "test(/risk_gate/)",
  "test(/retry/)"
]
lane = "unit"
confidence = 0.95

[[rules]]
id = "pool-manager"
changes = ["src/pool.rs", "src/docker.rs", "src/config.rs"]
select = [
  "binary(pool_tests)",
  "test(/pool/)",
  "test(/runner_config/)"
]
lane = "integration"
requires = ["docker"]

[[rules]]
id = "engine-webhooks"
changes = ["src/engine.rs", "src/gitlab_client.rs", "src/state.rs"]
select = [
  "binary(job_tests)",
  "binary(pool_tests)",
  "test(/webhook/)",
  "test(/tracked_pipeline/)"
]
lane = "integration"

[[rules]]
id = "docs-only"
changes = ["**/*.md"]
select = ["docs-lint"]
lane = "docs"
confidence = 0.90
```

This makes “which tests depend on which files” explicit, but jeryu should also learn from coverage and history so the config does not become a giant hand-maintained mess.

## 4.5 Coverage-derived dependencies

Nightly full runs should generate coverage maps. For Rust, use `cargo llvm-cov nextest` as the baseline collection mechanism; nextest documents that coverage tools usually integrate by setting compiler/runtime environment variables, and cargo-llvm-cov supports nextest. ([Nexte.st][10])

Start with **file-level and module-level coverage**, then move toward **per-test coverage** using nextest’s process-per-test model plus a jeryu wrapper/profiler once validated. Treat coverage as strong evidence, but not the only evidence.

Store:

```sql
CREATE TABLE test_dependencies (
    test_id          TEXT NOT NULL,
    dep_kind         TEXT NOT NULL, -- file, module, symbol, crate, service, env
    dep_value        TEXT NOT NULL,
    source           TEXT NOT NULL, -- declared, static, coverage, history, ai_candidate
    confidence       REAL NOT NULL,
    last_verified_sha TEXT,
    PRIMARY KEY (test_id, dep_kind, dep_value, source)
);
```

## 4.6 Historical failure learning

Use existing `evidence_capsules`, `retry_decisions`, `test_executions`, and the event ledger to learn:

* Which tests fail after changes to which files.
* Which tests are flaky.
* Which tests are slow.
* Which tests often catch regressions.
* Which modules produce infrastructure vs. regression failures.

A previously failing test should always be included until it passes on the current head, matching the Microsoft TIA safety pattern. ([Microsoft Learn][3])

# 6. Pipeline shape

Use jeryu to generate the pipeline. Do not try to encode all intelligence in YAML.

Parent `.gitlab-ci.yml`:

```yaml
stages:
  - plan
  - selected

select-tests:
  stage: plan
  needs: []
  image: rust:1.92.0
  tags: ["default", "rust", "test"]
  script:
    - cargo run -- test select
        --base "$JERYU_BASE_SHA"
        --head "$CI_COMMIT_SHA"
        --emit-plan target/jeryu-test-plan.json
        --emit-child target/jeryu-selected-tests.yml
        --explain
  artifacts:
    when: always
    paths:
      - target/jeryu-test-plan.json
      - target/jeryu-selected-tests.yml
    reports:
      dotenv: target/jeryu-test-plan.env

run-selected-tests:
  stage: selected
  needs: ["select-tests"]
  trigger:
    include:
      - artifact: target/jeryu-selected-tests.yml
        job: select-tests
    strategy: mirror
```

GitLab supports dynamic child pipelines generated by a job and included via `include: artifact`, and `strategy: mirror` makes the trigger job reflect the downstream status. ([GitLab Docs][11])

A generated child pipeline might be:

```yaml
unit-decision:
  stage: test
  image: rust:1.92.0
  tags: ["default", "rust", "test"]
  script:
    - cargo nextest run --workspace
        -E 'test(/decision/) + test(/risk_gate/) + test(/retry/)'
  artifacts:
    when: always
    reports:
      junit: target/nextest/default/junit.xml

integration-pool:
  stage: test
  image: rust:1.92.0
  tags: ["build", "docker-build"]
  script:
    - cargo nextest run --workspace -E 'binary(pool_tests)'
  artifacts:
    when: always
    reports:
      junit: target/nextest/ci/junit.xml
```

Use GitLab `rules:changes` only for very coarse outer gates, such as skipping the selector on docs-only scheduled contexts. The selector itself should live in Rust because GitLab rules are evaluated before job scripts, so they cannot consume the selector’s computed output. ([GitLab Docs][5])

# 7. Witnessed test-result caching

Skipping is only one optimization. The bigger win is **reusing known-good results** when inputs are unchanged.

jeryu already has the right primitives:

* `cache_verdicts`
* `cache_taints`
* `cache_epochs`
* `action_cache`
* `material_objects`
* `witness.rs`
* SmartCache
* append-only event ledger

Extend these into a test-result cache:

```sql
CREATE TABLE test_result_verdicts (
    id                INTEGER PRIMARY KEY AUTOINCREMENT,
    test_id           TEXT NOT NULL,
    action_key        TEXT NOT NULL,
    inputs_hash       TEXT NOT NULL,
    result            TEXT NOT NULL, -- pass, fail, flaky, skipped
    witness_digest    TEXT NOT NULL,
    produced_by_job_id INTEGER,
    produced_at       TEXT NOT NULL,
    selector_version  TEXT NOT NULL,
    trust_tier        TEXT NOT NULL,
    expires_at        TEXT
);
```

The `inputs_hash` should include:

* Test source files.
* Covered production files.
* `Cargo.toml`, `Cargo.lock`, features, target triple.
* Rust toolchain.
* Docker image digest.
* Relevant env vars.
* jeryu binary version.
* `.jeryu/testmap.toml` hash.
* Coverage map version.
* Cache epoch.
* Taint state.
* Trust namespace.

A skipped test should not simply disappear. It should become:

```json
{
  "test_id": "decision::risk_gate_denies_failed_jobs",
  "decision": "reused_by_witness",
  "reason": "inputs_hash matched prior passing verdict",
  "witness_digest": "sha256:...",
  "last_verified_sha": "abc123",
  "selector_version": "tik-1"
}
```

That gives you an auditable CI transcript: **this test did not run because its exact inputs already had a passing witnessed verdict.**

# 8. Safety policy

The selector should be aggressive only when the evidence is strong.

## Always run full or expanded tests when:

* `.jeryu/testmap.toml` changes.
* Selector code changes: `impact.rs`, `test_impact.rs`, `test_plan.rs`, `policy.rs`.
* `Cargo.lock`, `Cargo.toml`, `rust-toolchain`, `.cargo/**`, Dockerfiles, CI YAML, or runner templates change.
* Coverage map is missing or stale.
* A path is unknown.
* A file was renamed/deleted and the mapping is ambiguous.
* A dependency manifest changed.
* A test was newly added.
* A previously failing test exists.
* The change is on a protected/release branch.
* The author/agent is untrusted.
* The selected set has confidence below threshold.

## Always run scheduled full suites:

* Nightly: full unit + integration + E2E.
* Weekly: full plus coverage, mutation testing for critical modules, supply-chain detonation, and longer soak tests.
* Before production promotion: release-critical full evidence, not just selected tests.

This copies the proven TIA pattern: automatic selection for speed, safe fallback, and periodic full runs. ([Microsoft Learn][3])

# 9. Self-management loop

This is where jeryu becomes much smarter than normal CI.

Every nightly full run compares:

```text
tests selected during daytime PRs
vs.
tests that would have failed if full had run
```

If a skipped test fails in the nightly full run, jeryu should:

1. Insert `selector_false_negative` into the event ledger.
2. Mark the related dependency mapping as tainted.
3. Lower confidence for the affected rule.
4. Force expanded/full testing for that area until repaired.
5. Open an agent issue/MR proposing a `.jeryu/testmap.toml` update.
6. Attach the failure capsule and replay script.
7. Require human or trusted-agent approval before restoring aggressive skipping.

This is the “self-managing” part. The system learns, but it learns through **audited evidence**, not silent model drift.

# 10. AI’s role

AI should not be the judge. AI should be the **maintainer of the test graph**.

Good AI tasks:

* Explain why a change selected certain tests.
* Suggest missing tests for a changed module.
* Propose `.jeryu/testmap.toml` updates after a false negative.
* Classify new tests by lane/resource requirement.
* Detect when a test name no longer matches its actual coverage.
* Recommend splitting an integration test into smaller contract tests.
* Generate mocks/fixtures to replace heavy E2E runs.
* Summarize nightly audit failures.

Bad AI tasks:

* “Trust me, this test is unrelated.”
* Skipping without deterministic evidence.
* Lowering safety thresholds.
* Marking a critical test skippable without audit history.

So: **AI can propose. jeryu policy must dispose.**

# 13. E2E strategy

Full E2E should be rare, but E2E protection should not disappear.

Split E2E into rings:

| Ring               | When it runs                                                                                                                          |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------- |
| **E2E smoke**      | Changes to bootstrap, pool, Docker, GitLab client, engine webhook handling, runner config, custom executor, or random audit sampling. |
| **E2E targeted**   | Specific modules: agent lifecycle, release lifecycle, Vault, cache gateway, jeryu sync.                                              |
| **E2E full**       | Nightly, release candidate, selector changes, global invalidators, or manual force.                                                   |
| **E2E detonation** | Dependency/lockfile changes, untrusted agent changes, custom executor/sandbox changes, supply-chain risk.                             |

For example:

* `src/tui/**` should never bootstrap GitLab.
* `src/decision.rs` should never run Docker.
* `src/pool.rs` should not run agent MR tests unless GitLab API contracts changed.
* `Cargo.lock` should run supply-chain/detonation and broader integration.
* `src/exec.rs` should run sandbox/honeypot/capsule tests, not release promotion.

This is where most of the 100x win comes from.

# 14. Immediate CI rewrite

Your current GitHub Actions workflow runs:

```bash
cargo test --lib
cargo test --tests
```

That is intentionally blunt. Replace the integration job with:

```yaml
test-selected:
  name: Selected Tests
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
      with:
        fetch-depth: 0

    - name: Set up Rust
      uses: dtolnay/rust-toolchain@stable

    - name: Rust cache
      uses: Swatinem/rust-cache@v2

    - name: Install nextest
      uses: taiki-e/install-action@v2
      with:
        tool: nextest

    - name: Build selector
      run: cargo build --bin jeryu

    - name: Select tests
      run: |
        ./target/debug/jeryu test select \
          --base "$JERYU_BASE_SHA" \
          --head "$GITHUB_SHA" \
          --emit-plan target/jeryu-test-plan.json \
          --emit-github-matrix target/jeryu-test-matrix.json \
          --explain

    - name: Run selected tests
      run: |
        ./target/debug/jeryu test run-plan target/jeryu-test-plan.json
```

But because jeryu’s core product is GitLab-native, the better long-term path is to let the local GitLab instance run the dynamic child pipeline. GitHub Actions can remain a public mirror/smoke, while jeryu/GitLab becomes the smart authoritative test runner.

# 15. Rollout plan

## Phase 0 — fast win

* Install and standardize on `cargo-nextest`.
* Add `cargo nextest list --message-format json` ingestion.
* Add `.jeryu/testmap.toml`.
* Add `jeryu test select --base --head --explain`.
* Generate selected `cargo nextest -E ...` commands.
* Make unknown/global changes fall back to full.
* Keep nightly full unchanged.

Expected win: **2x–10x** on many changes, mostly from avoiding E2E/integration.

## Phase 1 — dynamic child pipelines

* Add `jeryu test select --emit-child`.
* Parent GitLab pipeline becomes selector + dynamic child pipeline.
* Record `test_selection_runs` and `test_selection_items`.
* TUI shows selected/skipped/reused tests.
* `pipeline explain` includes “why this test plan exists.”

Expected win: **5x–30x** on isolated code changes.

## Phase 2 — coverage and history

* Nightly full generates coverage baseline.
* Import file/module coverage into `test_dependencies`.
* Always include previous failures and new tests.
* Use `test_executions` duration data to order quick/high-signal tests first.
* Add false-negative audit events.

Expected win: **10x–50x** with strong confidence.

## Phase 3 — witnessed result reuse

* Add test-result cache keys and witness attestation.
* Reuse passing verdicts when exact inputs match.
* Invalidate through cache epochs and taints.
* Namespace by trust tier.

Expected win: **20x–100x** for isolated or repetitive changes.

## Phase 4 — workspace crate split

* Split jeryu internals into crates aligned with modules.
* Use `guppy` reverse dependency selection.
* Use nextest package/binary filters.
* Improve compile-time isolation.
* Consider `cargo-hakari`/workspace-hack after the workspace split if feature unification starts dominating build time.

Expected win: **structural 100x potential** for small changes, because both compilation and tests become more granular.

# 16. The final operating model

For normal PRs and agent branches:

```text
diff → jeryu selector → exact tests/cache/E2E lane → dynamic child pipeline → ledgered evidence
```

For nightly:

```text
full suite → coverage refresh → false-negative audit → selector repair → baseline update
```

For release:

```text
selected tests allowed for development feedback
full release-critical evidence required for promotion
```

For untrusted agents:

```text
broader test expansion + no unsafe cache reuse + risk gate escalation
```

This gets you the speed target without lying to yourself. **Most changes should run only a handful of tests. Some should run one test. Docs should run none of the runtime stack. Dependency/toolchain/CI/executor changes should still go broad. Full exhaustive testing becomes a scheduled oracle, not the default tax on every commit.**

The core implementation should be Rust inside jeryu, with these external helpers:

```toml
cargo_metadata = "..."
guppy = "..."
nextest-metadata = "..."
globset = "..."
ignore = "..."
petgraph = "..."
syn = "..."
toml = "..."
serde = "..."
```

And the strategic product name I’d use internally:

> **jeryu Test Impact Kernel** — a self-auditing, policy-driven test selector with witnessed skips and nightly oracle validation.
