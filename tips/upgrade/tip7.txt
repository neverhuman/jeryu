This is a seriously ambitious control plane. The big thing that jumps out: **jeryu already has most of the ingredients for an agent-native delivery system**, but the current surfaces are still split between “CLI commands humans can call,” “engine automation,” “partial capability intents,” and “TUI observability.” The dream version would unify those into a **real-time command console where every action, gate, blocker, agent task, runner, cache decision, release state, and log stream is visible, explainable, and actionable from one place.**

Below is my best design-review feedback.

---

# 1. Highest-level assessment

`jeryu` has an unusually strong foundation:

* It owns the local GitLab lifecycle, runners, webhooks, release automation, Vault-backed secrets, SmartCache, failure evidence, VTI test selection, agent MRs, and a TUI.
* It has a durable SQLite state plane, which is exactly what an agent-operable system needs.
* It already thinks in terms of evidence, risk gates, trust tiers, pipeline products, cache taint, selector misses, release attempts, and audit events.
* It intentionally avoids “just shell into prod,” which is excellent. Production promotion is mediated through GitLab pipelines and release gates.

The main gap is not conceptual. The main gap is **surface unification**.

Right now, many powerful things exist, but they are spread across:

* CLI commands
* engine background behavior
* TUI display
* hidden hook commands
* partial capability socket intents
* raw GitLab events
* SQLite ledgers
* test-intel subcommands
* release/secrets/cache commands

A developer or agent should not have to mentally stitch those together. The dream system would expose one coherent model:

> “Here is the current delivery graph. Here is what is blocking it. Here is the exact evidence. Here are the safe actions available now. Here is what will happen if you take one.”

That should be the heart of the redesign.

---

# 2. Clear current gaps

## 2.1 Capability API is too small relative to the system’s ambition

The capability API is supposed to be the agent-facing least-privilege control plane, but only two intents have meaningful active behavior:

* `FetchCapsule`
* `RunTests`

Meanwhile these deserialize and return success while being deferred:

* `ProposePatch`
* `RacePatches`
* `RequestMerge`

That is dangerous from an agent UX perspective. An agent may interpret “success” as “the requested operation was performed,” when actually the operation was accepted but not routed.

**Fix:** make deferred intents return an explicit structured status, such as:

```json
{
  "success": false,
  "message": "Intent is recognized but not implemented",
  "data": {
    "status": "deferred",
    "intent": "RequestMerge",
    "recommended_surface": "jeryu agent merge"
  }
}
```

Even better, add a capability-discovery intent:

```json
{
  "intent": "DescribeCapabilities"
}
```

Response:

```json
{
  "capabilities": [
    {
      "name": "RunTests",
      "implemented": true,
      "risk": "low",
      "requires": ["project_id", "target_ref", "test_scope"]
    },
    {
      "name": "RequestMerge",
      "implemented": false,
      "recommended_cli": "jeryu agent merge"
    }
  ]
}
```

That would make agents much safer and less confused.

---

## 2.2 The TUI has hidden state/actions that are not visibly represented

The active TUI still has pane concepts for:

* Pools
* Pipelines
* Jobs

But only the Jobs list is visibly represented in the active Flow layout. The old pool and pipeline renderers exist but are inactive.

That creates a mismatch:

* `p` can pause/resume a pool, but the pool pane is not visibly active in the current layout.
* `d` can delete a selected pipeline/job record, but the visible pipeline list is absent.
* `Tab` cycles through panes that may not have visible focus targets.

**Fix:** either restore visible Pool and Pipeline panes, or change the active pane model to match what the user can actually see. Hidden focus is frustrating in a power console.

A better approach would be a **left navigation rail** with visible selectable entities:

```text
▸ Flow
  Pipelines
  Jobs
  Pools
  Agents
  Releases
  Cache
  Secrets
  Audit
```

The current active selection should always correspond to something visible.

---

## 2.3 The Audit tab is a placeholder despite the system having a rich event ledger

This is one of the biggest missed opportunities. The architecture already has an append-only `events` table, retry decisions, release attempts, selector misses, cache verdicts, secret audit events, job events, pipeline tracking, and release secret sets.

The Audit tab should be one of the crown jewels.

It should answer:

* What happened?
* Who or what did it?
* Why did it happen?
* What evidence was used?
* What changed afterward?
* Was this automatic, agent-initiated, or human-triggered?

**Fix:** turn Audit into a chronological, filterable event stream:

```text
12:44:01  PIPELINE  #531 success on main fa51a52
12:44:03  RELEASE   Canary claim accepted for ci-fa51a52
12:44:04  GITLAB    Triggered release-execution pipeline #533
12:46:11  JOB       e2e-canary failed
12:46:12  CAPSULE   Failure classified as Regression
12:46:12  RETRY     DoNotRetry
12:46:13  RELEASE   Production blocked: canary E2E failed
```

With filters:

```text
[All] [Release] [Jobs] [Agents] [Secrets] [Cache] [Security] [Errors]
```

This would immediately make the system feel trustworthy.

---

## 2.4 Flow Board only shows the first active pipeline

The current Flow Board renders only the first active pipeline in `active_pipelines`.

That is fine for a first version, but a power user needs to see:

* all active pipelines
* superseded pipelines
* release-execution pipeline
* production-promotion pipeline
* downstream child pipelines
* scratch `jeryu-test-*` pipelines
* agent branch pipelines
* blocked/manual jobs
* stale pipelines with no progress
* critical-path pipeline vs background pipelines

**Fix:** the Flow Board should have a pipeline selector or stacked overview.

Example:

```text
Pipelines
● #533 release-execution main ci-fa51a52  running   61% ETA 4m
● #532 main              fa51a52          success   upstream
○ #534 prod-promotion    fa51a52          blocked   waiting gates
○ #535 jeryu-test-abc     agent/fix-x      pending   unit
```

Then the center graph updates based on the selected pipeline.

---

## 2.5 Graph edges exist but are not computed

The `FlowGraph.edges` field exists, but the docs say edges are currently empty. This limits the Flow Board from being a true DAG.

A dream console needs dependency visibility:

* Which job unblocks which lane?
* Which downstream pipeline is waiting?
* Which release gate depends on which artifact?
* Which skipped test was omitted because of VTI?
* Which manual job is blocking production?

**Fix:** compute edges from GitLab stages, job needs, bridges/downstream pipelines, release attempt linkage, and jeryu semantic events.

Even approximate edges would be powerful:

```text
Impact → Build → Unit Tests → Security → Package → Canary → E2E → Prod
```

The graph should make the critical path visually obvious.

---

## 2.6 ETA and progress are useful but too heuristic

The current ETA model uses lane defaults:

* Unit: 60s
* Integration: 300s
* Security: 120s
* Build: 180s
* ReleaseExecution: 120s
* Other: 90s

That is a good placeholder, but this system already stores historical timing in `ci_job_runs` and test execution data.

**Fix:** ETAs should be based on:

* job name
* stage
* runner pool
* image
* branch/ref
* historical p50/p90 runtime
* queue duration by pool/tag
* current runner capacity
* cache hit/miss likelihood
* previous runs of same job on same ref
* whether this is a release-critical lane

A developer dream console should show not just:

```text
ETA 4m
```

but:

```text
ETA 4m p50 / 8m p90
Reason: integration-tests historically 5.8m, queue p90 52s, cache cold
```

---

## 2.7 Release state is too stringly typed

The docs show release/canary states like:

* `green`
* `released`
* `in-flight`
* `canary-authorized`
* `waiting`
* `ready-for-canary`
* `blocked`
* `blocked-by-upstream`
* `failed`
* `e2e-passed`

These are powerful states, but they appear to be represented as strings in several places.

**Fix:** use typed enums internally and at the API boundary:

```rust
enum CanaryState {
    Waiting,
    ReadyForCanary,
    CanaryAuthorized,
    InFlight,
    E2ePassed,
    BlockedByUpstream,
    BlockedByCanary,
    Failed,
    Released,
}
```

Then render strings only at the edge.

This matters because production promotion depends on exact state checks like `canary_state == "e2e-passed"`. String mismatch bugs here could be painful.

---

## 2.8 Production gate should validate artifact provenance, not just existence

Production promotion currently requires:

* release attempt exists
* `canary_state == "e2e-passed"`
* C handoff artifact exists
* C validation artifact exists
* no existing production-promotion pipeline for the same SHA/ref

That is good, but for a production-grade system, “file exists” is not enough.

**Fix:** validate artifact provenance:

* artifact SHA matches release SHA
* artifact version matches release version
* artifact was produced by the attached release-execution pipeline
* artifact timestamp is after the upstream pipeline
* artifact contains expected schema version
* artifact is signed or checksum-recorded
* validation result references the same handoff bundle
* no stale artifact from previous release can satisfy the gate

In the TUI, the gate should show:

```text
Canary E2E     PASS  gate-canary-e2e.json     sha fa51a52 verified
Handoff        PASS  c-handoff.json           produced by pipeline #533
Validation     PASS  c-validation.json        schema v3, version ci-fa51a52
Prod Pipeline  READY no existing prod pipeline for sha
```

That would make the release system feel rock-solid.

---

## 2.9 Risk gate should verify required evidence, not just counts

The risk gate currently considers:

* successful jobs
* pending jobs
* failed jobs
* trust tier
* policy

That is a good start, but an MR merge gate should care about **which jobs** passed, not just how many.

For example:

* one successful docs job should not satisfy evidence for a risky executor change
* success from an older SHA should not count
* success from a superseded pipeline should not count
* untrusted code should require sandboxed/custom executor evidence
* release-affecting changes should require release-critical jobs

**Fix:** evolve from count-based evidence to required evidence sets:

```text
Required:
✓ unit
✓ integration or justified skip
✓ security lane
✓ admission policy
✓ VTI selector result
✓ no failed release-critical jobs
✓ evidence SHA == MR head SHA
```

The TUI should render this as a merge-readiness matrix.

---

## 2.10 Webhook server binds to `0.0.0.0`

The engine binds to:

```text
0.0.0.0:9777
```

`POST /hooks` is token-protected, but `/cache/summary` is unauthenticated. `/health` is fine unauthenticated, but cache summary may reveal operational information.

**Fix options:**

* bind to localhost by default
* require token for `/cache/summary`
* add explicit `--public-webhook-bind` or env opt-in
* split health from operational routes
* add request size limits, replay logging, and source audit events

This is not necessarily urgent if the environment is isolated, but it is worth tightening.

---

## 2.11 Custom executor and hidden commands need stronger self-protection

The docs correctly warn:

* custom executor commands are GitLab Runner protocol hooks
* server hook reads pre-receive lines from stdin
* agents should not call these directly

But they are still process-invocable commands.

**Fix:** hidden protocol commands should verify expected environment where possible.

For example:

* `jeryu exec prepare` should require GitLab Runner custom executor env vars.
* `jeryu exec run` should reject suspicious manual contexts unless `JERYU_ALLOW_PROTOCOL_DRIVER_TEST=1`.
* `server-hook pre-receive` should validate hook-like stdin format and maybe log direct manual use.
* Protocol commands should emit clear “driver protocol only” errors when invoked incorrectly.

This reduces accidental misuse.

---

## 2.12 Jeryu sync has a placeholder command

`jeryu jeryu sync-now --source <dir>` currently tells the user to ensure `jeryu serve` is running.

For an operator console, that is unsatisfying. Manual sync is a natural power-user action.

**Fix:** implement actual sync-now semantics:

* load the jeryu config
* run one sync cycle
* return pushed SHA / no-op / conflict / error
* append audit event
* expose it in the TUI as an action

---

## 2.13 Secrets support only `dougx`

That is okay as a current limitation, but the system should make it visually explicit.

In the TUI and CLI, show:

```text
Secrets repo: dougx only
Release repo root: /home/ubuntu/dougx
```

If an agent or user tries another repo, return a typed unsupported-repo error, not a vague failure.

---

# 3. What an agent would dream to be able to do

An agent’s dream is not “give me a shell.” A shell is too broad and too dangerous.

An agent’s dream is:

> “Give me typed, least-privilege abilities to observe, explain, patch, validate, compare hypotheses, and request gated actions — with every action recorded and reversible when possible.”

Here is the agent capability surface I would want.

---

## 3.1 Capability discovery

Agents should be able to ask:

```json
{
  "intent": "DescribeCapabilities"
}
```

And receive:

* available intents
* required parameters
* risk level
* permission tier
* whether action is read-only or mutating
* whether action requires human confirmation
* related CLI fallback
* current availability

This is foundational.

---

## 3.2 Unified observe/read capabilities

Agents need rich read access before write access.

Dream intents:

```text
GetSystemStatus
GetReleaseStatus
GetProgressReport
ExplainPipeline
DoctorPipeline
ListActivePipelines
ListPipelineJobs
GetJobTrace
FetchCapsule
ListRecentEvents
GetCacheStatus
GetPoolStatus
GetShadowStatus
GetSecretsStatusRedacted
GetTestPlan
ExplainTestPlan
GetAgentWork
```

The agent should be able to answer:

* What is broken?
* What is blocking release?
* What evidence exists?
* What changed?
* What should be tested?
* What is safe to retry?
* Which runner pool is overloaded?
* Which cache rule caused a cold execution?

---

## 3.3 Patch and MR workflow

The deferred intents are exactly what agents want, but they need to become real.

Dream intents:

```text
ProposePatch
ApplyPatchToBranch
CreateBranch
CommitFiles
OpenMergeRequest
CommentOnIssue
RequestReview
RequestMerge
ExplainMergeGate
```

Each should be scoped:

* allowed project
* allowed branch prefix
* allowed file paths
* maximum diff size
* allowed trust tier
* required validation before merge

A great agent flow:

```text
Observe failure
→ Fetch capsule
→ Classify likely cause
→ Propose patch
→ Run VTI-selected tests
→ Open MR
→ Explain risk gate
→ Request merge
```

---

## 3.4 Hypothesis racing

`RacePatches` is a fantastic idea. It should become one of the signature features.

An agent should be able to submit several repair hypotheses:

```json
{
  "intent": "RacePatches",
  "payload": {
    "base_branch": "main",
    "commit_message": "fix flaky release gate",
    "hypotheses": [
      {
        "name": "increase timeout",
        "modifications": [...]
      },
      {
        "name": "fix missing artifact path",
        "modifications": [...]
      },
      {
        "name": "repair validation schema",
        "modifications": [...]
      }
    ]
  }
}
```

The system would:

1. Create branches.
2. Apply patches.
3. Run selected tests.
4. Compare outcomes.
5. Return a ranked evidence report.
6. Optionally open an MR for the best hypothesis.

This would be incredible for CI repair, flaky tests, release gate failures, and compiler errors.

---

## 3.5 Typed validation oracle

Agents should not guess which tests to run. They should ask jeryu:

```text
PlanValidationForChange
RunValidationPlan
ExplainValidationResult
```

Inputs:

* changed files
* branch
* risk tier
* target goal: merge, release, docs-only, security-sensitive, executor-sensitive

Outputs:

* selected tests
* skipped tests
* required lanes
* cache keys
* runner tags
* expected duration
* risk explanation
* whether merge evidence is enough

This would unify VTI, cargo-vrc, cargo-witness, release explain, and risk gate.

---

## 3.6 Failure repair loop

Given a failed job, an agent should be able to do:

```text
DiagnoseFailure(job_id)
SuggestRepair(job_id)
CreateRepairBranch(job_id)
RunRepairTests(branch)
OpenRepairMR(branch)
```

The diagnosis should combine:

* failure capsule
* trace tail
* job metadata
* recent similar failures
* changed files
* owner/module routing
* cache verdict
* runner pool
* whether failure is infrastructure/transient/regression
* whether retry has already been attempted

---

## 3.7 Streaming observations

The current capability API is one request / one response. That is useful, but agents also need event streams.

Dream:

```text
SubscribeEvents
SubscribePipeline
SubscribeJobLog
SubscribeRelease
```

Over a Unix socket, this could use newline-delimited JSON.

Events:

```json
{
  "type": "job_status_changed",
  "job_id": 14445,
  "from": "pending",
  "to": "running",
  "pipeline_id": 533,
  "timestamp": "..."
}
```

This would let agents react without polling.

---

## 3.8 Action budgets and permission envelopes

Agents should operate under explicit budgets:

```json
{
  "max_pipelines": 3,
  "max_runtime_minutes": 30,
  "allowed_actions": ["RunTests", "ProposePatch", "OpenMergeRequest"],
  "forbidden_actions": ["PromoteProd", "RotateSecrets"],
  "branch_prefix": "agent/",
  "path_allowlist": ["src/test_intel/**", "docs/**"]
}
```

This creates safe autonomy without handing over broad credentials.

The answer must be unambiguous.

# 11. Architectural redesign for the TUI

The current TUI has several background workers that merge snapshots. That is okay, but the dream version should evolve toward a **unified event/state model**.

## 11.1 Introduce a TUI state aggregator

Instead of each screen pulling from DB/GitLab/Docker ad hoc, create a central aggregator:

```text
GitLab events
DB events
Docker events
Cache metrics
Release reports
Agent events
Log streams
        ↓
TuiStateAggregator
        ↓
Unified TuiModel
        ↓
Widgets
```

Every field should carry freshness metadata:

```rust
struct Fresh<T> {
    value: T,
    generated_at: DateTime<Utc>,
    source: DataSource,
    stale_after: Duration,
    error: Option<String>,
}
```

That lets every widget show whether data is live, stale, partial, or failed.

---

## 11.2 Add an action registry

Define actions once:

```rust
struct ActionSpec {
    id: &'static str,
    label: &'static str,
    risk: ActionRisk,
    scope: ActionScope,
    input_schema: ActionInputSchema,
    confirmation: ConfirmationPolicy,
    command_equivalent: Option<String>,
}
```

Then use this in:

* TUI command palette
* capability API
* CLI docs
* agent permissioning
* audit logs

This prevents drift between surfaces.

---

## 11.3 Treat every action as a transaction with a receipt

When a user or agent does something, the system should return an action receipt:

```json
{
  "action_id": "retry_job",
  "requested_by": "tui",
  "target": "job:14445",
  "accepted": true,
  "gitlab_result": {
    "new_job_id": 14450
  },
  "events": [
    "job_retry_requested"
  ],
  "timestamp": "..."
}
```

Then the bottom console can show:

```text
✓ retry_job accepted, new job #14450
```

This builds confidence.

---

## 11.4 Make release gates first-class model objects

Instead of rendering release status from a collection of strings/files, model gates explicitly:

```rust
struct ReleaseGate {
    id: GateId,
    name: String,
    state: GateState,
    blocking: bool,
    evidence: Vec<EvidenceRef>,
    required_for: Vec<ReleasePhase>,
    failure_reason: Option<String>,
    next_action: Option<ActionSpecId>,
}
```

Then the TUI, CLI, API, and agents can all consume the same gate model.

# 13. Issues around agent safety

## 13.1 Agents need precise permissions

Do not give agents blanket access to:

* secrets
* production promotion
* pool token rotation
* host reclaim apply
* hidden executor commands
* server hook commands
* arbitrary GitLab PAT operations

Instead, give scoped permissions:

```text
Read status: yes
Run tests: yes
Open MR: yes
Retry transient job: maybe
Cancel pipeline: maybe
Merge: gated
Promote prod: no or human-confirmed
Rotate secrets: no
Host reclaim apply: no or admin-confirmed
```

## 13.2 Every agent action should be attributed

The event ledger should distinguish:

```text
actor_type: human | agent | engine | gitlab | system
actor_id: ...
surface: cli | tui | capability | webhook | reconcile_loop
```

This is crucial for trust.

## 13.3 Agents need dry-run/preflight

Before doing anything mutating, agents should be able to ask:

```text
Can I do this?
What would happen?
What gates block it?
What evidence is required?
```

This is especially important for merge, release, secrets, and storage reclaim.

# 15. What I would not do

A few things I would avoid:

## Do not make the TUI a thin CLI launcher only

It should not merely run shell commands. It should call typed domain actions and show structured receipts.

## Do not overload the main Flow Board

The graph should stay readable. Details belong in the inspector.

## Do not hide dangerous actions behind single-key shortcuts

For example:

* production promotion
* secrets rotation
* pool deletion
* host reclaim apply
* runner token rotation

These should require command palette confirmation and show blockers/evidence.

## Do not let agents interpret “success” ambiguously

For agent APIs, “recognized but deferred” must not look like “done.”

## Do not rely only on polling forever

Polling is fine now, but the long-term dream is an event stream from the engine/state aggregator.
