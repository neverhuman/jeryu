Based on the three documents, my strongest read is: **jeryu is already much more than a CLI/TUI wrapper; it is a local delivery-control plane.** The architecture is exciting because it has the raw ingredients for something rare: agent-operable CI/CD with typed gates, failure evidence, release automation, cache intelligence, and a supervisory terminal interface.

The biggest opportunity is to make the system’s **intent, safety gates, blockers, evidence, and available actions visible in one place**. Right now, a lot of power exists, but much of it is scattered across CLI commands, hidden executor surfaces, RedlineDB state, release logic, and partially wired TUI panes.

## My high-level feedback

The system is strongest where it does three things:

1. **Turns dangerous actions into gated workflows**
   Production promotion, risk-gated merge, Vault secret rotation, runner trust tiers, and custom executor tripwires are all good signs. This is much safer than giving agents raw shell or broad GitLab tokens.

2. **Keeps durable evidence**
   The append-only event ledger, failure capsules, retry decisions, release attempts, cache verdicts, test plans, and secret audit events are exactly the kind of substrate an agent or human operator needs.

3. **Treats CI/CD as an observable state machine**
   The release attempt model, tracked pipelines, VTI plans, pipeline explain/doctor commands, SmartCache state, and TUI flow board point in the right direction.

But the visible interface has not yet caught up to the internal power. The current TUI is useful, especially for jobs/logs/release visibility, but it is not yet the “command center” that the architecture deserves.

# 1. Clear gaps and issues I see

## A. Capability API is the biggest agent-facing gap

The Capability API is conceptually excellent, but currently underpowered.

The most concerning issue is that these intents deserialize and return success even though routing is deferred:

* `ProposePatch`
* `RacePatches`
* `RequestMerge`

That is dangerous for agents. An agent may interpret `success: true` as “the patch was proposed” or “the merge was requested,” when in reality nothing meaningful happened. I would change these to return something like:

```json
{
  "success": false,
  "message": "Intent accepted by schema but not implemented",
  "data": {
    "state": "deferred",
    "implemented": false,
    "recommended_surface": "jeryu agent spawn"
  }
}
```

Better yet, add a formal response state:

```rust
enum CapabilityOutcome {
    Completed,
    Deferred,
    Rejected,
    RequiresApproval,
    Failed,
}
```

Right now, the API has the shape of a least-privilege agent control surface, but not yet the breadth agents would dream of.

## B. `FetchCapsule` should not scan only the latest 500 event rows

The API says `FetchCapsule` scans the latest 500 event log rows looking for a `failure_capsule` event. But the state layer already has:

* `evidence_capsules`
* `latest_evidence_for_job`
* `insert_evidence_capsule`

I would strongly prefer `FetchCapsule` query the dedicated evidence table first, then optionally fall back to the event ledger. A latest-500 scan can miss older failures, especially in a busy system where Docker events, cache events, release events, and job events are all flowing.

This matters because failure capsules are not just logs. They are agent evidence.

## C. Capability `RunTests` duplicates the safer `jeryu test run` path

The CLI `jeryu test run` is richer than the capability `RunTests` path. It infers tags/risk/timeout, checks cache, creates scratch branches, records test execution, waits for completion, fetches trace tails, and cleans up branches when possible.

The capability `RunTests` flow appears simpler:

1. Create branch.
2. Generate `.gitlab-ci.yml`.
3. Trigger pipeline.

That creates divergence. Agents should probably call into the same planning/execution machinery as `jeryu test run`, or at least return a full structured test handle:

```json
{
  "pipeline_id": 123,
  "branch": "main-ci-...",
  "jobs": [...],
  "cleanup_policy": "ttl",
  "risk": "trusted",
  "tags": ["rust", "test"],
  "cache_status": "miss"
}
```

Also: the capability version should have explicit cleanup or TTL semantics for ephemeral branches.

## D. The TUI has invisible active panes

The active pane model still includes:

* Pools
* Pipelines
* Jobs

But the active Flow layout visibly renders only the Live Jobs list. Pipeline focus affects the Flow Board border, and pool actions still exist, but the Pools and Pipelines lists are not visibly present.

That creates a real UX hazard:

* `Tab` can focus a pane the user cannot see.
* `p` can pause/resume a selected pool without an obvious visible pool list.
* `d` can delete a selected pipeline/job local record depending on invisible focus state.

This is one of the first things I would fix.

Either restore visible Pools/Pipelines panes, or remove those focus modes from the active Flow layout and move their actions into a command palette or inspector panel.

## E. The Audit tab is far below the rest of the system’s maturity

The system has an event ledger, retry decisions, release attempts, secret audit events, selector misses, cache verdicts, and production pipeline tracking.

But the Audit tab currently says:

```text
Audit CI Pipeline running...
Check Dashboard/Pipelines to see live progress.
```

That undersells the system dramatically. This tab should be one of the most powerful parts of the product.

It should show:

* recent engine decisions
* release gate decisions
* auto-retry decisions
* supersedence cancellations
* production promotion decisions
* secret rotation/finalization events
* admission denials
* cache taints/quarantines
* agent actions
* selector misses
* disk pressure GC events

The audit ledger is the system’s memory. It deserves a real UI.

## F. Flow Board currently hides too much

The Flow Board renders only the first active pipeline. The collector prioritizes the release pipeline if available, which is useful, but it can hide the wider story:

* upstream pipeline
* release-execution pipeline
* production-promotion pipeline
* downstream bridges
* superseded pipelines
* blocked pipelines
* agent scratch test pipelines

For a release control plane, the dream view is not “one active pipeline.” It is the **release chain**:

```text
Push → Upstream CI → Release Execution → Canary/E2E → Production Promotion
```

The TUI should show that as a linked chain, not as whichever pipeline happens to be first.

## G. Flow graph edges are missing

The docs say `FlowGraph.edges` exists but is currently empty. That means the graph visually groups jobs, but does not yet explain dependencies.

Edges are what let the UI answer:

* “What is blocking what?”
* “What can still run in parallel?”
* “What is on the critical path?”
* “Why is production not unlocked?”
* “Which job failure invalidates which gate?”

Without edges, the board is a status wall. With edges, it becomes a control graph.

## H. ETA is currently too heuristic for a “dream” power tool

The lane-based ETA defaults are a good placeholder:

* Unit: 60s
* Integration: 300s
* Security: 120s
* Build: 180s
* ReleaseExecution: 120s

But the state layer already has historical timing tables:

* `ci_job_runs`
* `test_executions`
* `get_test_bottlenecks`
* `ci_job_bottlenecks`

The dream TUI should use historical p50/p90 duration by job name, stage, lane, runner pool, ref, and maybe image/toolchain. A power user needs to know:

```text
ETA: 4m 20s
Confidence: medium
Historical p50: 3m 10s
Historical p90: 7m 45s
Currently slower than usual by 42%
Likely cause: queued on untrusted pool
```

## I. Release-state vocabulary should be normalized

The release color logic references states like:

* `green`
* `released`
* `in-flight`
* `canary-authorized`
* `waiting`
* `ready-for-canary`
* `blocked`
* `blocked-by-upstream`
* `failed`

Elsewhere, production promotion requires:

* `canary_state == "e2e-passed"`

That may be implementation-compatible, but as a user-facing model it wants a single canonical state machine.

For example:

```text
NoAttempt
UpstreamRunning
UpstreamFailed
EligibleForCanary
CanaryClaimed
CanaryRunning
CanaryE2EPassed
CanaryFailed
ReadyForProduction
ProductionRunning
ProductionSucceeded
ProductionFailed
Blocked
Superseded
```

The TUI, CLI, release engine, and event ledger should all speak the same release-state language.

## J. Destructive TUI actions need stronger affordances

`d` deletes local DB records for selected jobs/pipelines. That is not the same as canceling a GitLab job or pipeline, but in a fast terminal UI, “delete” can feel destructive and ambiguous.

I would rename this behavior in the UI as:

```text
Forget local record
```

And make the command palette show:

```text
Forget local job record
Does not cancel the GitLab job.
Evidence may no longer appear in recent local views.
```

For important audit data, I would prefer soft-delete or hidden/archived state rather than hard deletion.

## K. Webhook/cache endpoints should be reviewed for exposure

The engine binds to:

```text
0.0.0.0:9777
```

`POST /hooks` is token-protected, which is good. But `GET /cache/summary` has no auth. It is only summary data, but it still exposes operational state.

Depending on deployment assumptions, I would consider one of:

* bind engine to localhost by default
* require token for `/cache/summary`
* expose unauthenticated `/health` only
* make unauthenticated summary opt-in

## L. Agent auditability needs first-class correlation IDs

The system has many ledgers, but I would want every agent-triggered action to carry an `intent_id` or `correlation_id`.

For example:

```text
agent_intent_id: agt-20260425-abc123
actor: local-agent:doug-assistant
source: capability-api
target: project=2 ref=main
action: RunTests
result: pipeline_triggered
pipeline_id: 9001
```

Then the TUI can show a beautiful story:

```text
Agent proposed patch → VTI selected tests → pipeline ran → failure capsule captured → retry denied → MR blocked
```

## M. Secrets and release safety are good, but visibility is thin

The secrets model is promising. But a dream operations UI should show secret lifecycle status without exposing secret values:

```text
Vault: healthy
Authority: jeryu-managed
Latest secret set: v1.2.3 / canary
Rotated: 12m ago
Finalized: no
Deploy env handoff: present
Runtime env handoff: present
Recovery bundle: present
Audit events: 4
```

This should appear in Release and Audit views.

## N. `jeryu sync-now` being a placeholder is likely to confuse agents

The docs clearly say it is a placeholder, but the command name implies action. For agents, placeholder commands are risky. I would either implement it or rename the output very explicitly:

```text
jeryu sync-now is not implemented as a direct trigger.
No sync was performed.
Run `jeryu serve` to allow the background sync loop to process this source.
```

## O. Project/ref are too hardcoded in the TUI

The TUI currently centers around:

```text
project_id = 2
ref_name = main
```

That is fine for the release control plane, but a dream TUI needs a project/ref selector. Even if `2/main` remains the default, the UI should show that assumption loudly:

```text
Project: dougx / #2
Ref: main
Release root: /home/ubuntu/dougx
```

Power users need to switch context without relaunching.

# 2. What an agent would dream to be able to do

The dream is **not** to give the agent a shell. The dream is to give it typed, inspectable, least-privilege capabilities with evidence, budgets, approval gates, and audit trails.

I would evolve the Capability API into the main “agent nervous system.”

## Agent dream capability set

### 1. Full system snapshot

```json
{
  "intent": "GetSystemSnapshot",
  "payload": {
    "project_id": 2,
    "ref_name": "main"
  }
}
```

Returns:

* GitLab readiness
* pools and manager health
* active pipelines
* current release attempt
* blockers
* recent failures
* cache health
* disk pressure
* Vault status
* jeryu sync status
* pending approvals
* agent task state

This would let agents orient themselves before acting.

### 2. Subscribe to events

Current API is request/response. Agents would love an event stream:

```json
{
  "intent": "SubscribeEvents",
  "payload": {
    "filters": ["release", "job", "failure", "agent", "cache", "secret"]
  }
}
```

For a Unix socket, this could be a separate streaming mode or line-delimited JSON feed.

Agents should not have to poll RedlineDB or scrape CLI output.

### 3. Explain current blocker

```json
{
  "intent": "ExplainBlocker",
  "payload": {
    "project_id": 2,
    "ref_name": "main"
  }
}
```

Returns:

```json
{
  "blocker": "integration-test failed",
  "scope": "release-critical",
  "job_id": 123,
  "classification": "Regression",
  "retry_advice": "DoNotRetry",
  "next_safe_actions": [
    "FetchCapsule",
    "RunFocusedTests",
    "SpawnAgentTask"
  ]
}
```

This is the kind of function agents dream of.

### 4. Plan a patch before making it

```json
{
  "intent": "PlanPatch",
  "payload": {
    "changed_paths": ["src/release.rs", "src/state.rs"],
    "summary": "Add production gate state"
  }
}
```

Returns:

* owning modules
* risk tier
* required proof lane
* VTI selected tests
* witness-diff implications
* high-risk security modules touched
* release impact
* docs that must be updated

This should combine:

* `cargo-vrc`
* `cargo-witness`
* `cargo-aer`
* VTI
* API surface audit
* AGENTS.md routing

### 5. Propose patch for real

The deferred `ProposePatch` should become active, but with constraints:

```json
{
  "intent": "ProposePatch",
  "payload": {
    "base_branch": "main",
    "branch_name": "agent/release-gate-fix",
    "commit_message": "Fix production gate display",
    "patch": "...",
    "declared_risk": "release-ui"
  }
}
```

Returns:

* branch
* commit SHA
* MR URL
* selected validation plan
* required approvals
* risk gate status

Large patches should be chunked or referenced by file handles, not squeezed through a 64 KiB payload.

### 6. Race hypotheses

`RacePatches` is a fantastic idea. It should become one of the signature features.

An agent should be able to submit several candidate fixes:

```json
{
  "intent": "RacePatches",
  "payload": {
    "base_branch": "main",
    "hypotheses": [
      {
        "name": "fix-state-query",
        "patch": "..."
      },
      {
        "name": "fix-renderer-state",
        "patch": "..."
      }
    ],
    "validation": "minimal-safe"
  }
}
```

The system would:

* create branches
* run VTI-selected tests
* compare outcomes
* summarize the winning hypothesis
* archive losing branches
* preserve evidence

This would make jeryu feel genuinely agent-native.

### 7. Request merge with transparent gate output

```json
{
  "intent": "RequestMerge",
  "payload": {
    "project_id": 2,
    "mr_iid": 17,
    "trust_tier": "trusted"
  }
}
```

Returns:

```json
{
  "decision": "Escalate",
  "reasons": [
    "1 pending job",
    "touched high-risk module: release.rs"
  ],
  "required_evidence": [
    "release::tests",
    "repo audit-agent-surface",
    "pipeline explain eligible"
  ]
}
```

This should be the agent version of `jeryu agent merge`.

### 8. Run validation plan, not just tests

Agents do not merely want to run `cargo test`. They want the system to say:

```text
For this change, the minimal safe validation is:
- cargo check -p jeryu
- release::tests
- state::tests::test_production_pipeline_tracking
- repo audit-agent-surface
```

So expose:

```json
{
  "intent": "RunValidationPlan",
  "payload": {
    "plan_id": "vrc-plan-abc123"
  }
}
```

### 9. Ask for release status and gated actions

Agents should be able to query release state:

```json
{
  "intent": "GetReleaseStatus",
  "payload": {
    "project_id": 2,
    "ref_name": "main"
  }
}
```

And request gated actions:

```json
{
  "intent": "RequestProductionPromotion",
  "payload": {
    "project_id": 2,
    "ref_name": "main",
    "version": "ci-fa51a52a7882"
  }
}
```

The response should be very explicit:

```json
{
  "decision": "Rejected",
  "reason": "canary_state is running; required e2e-passed",
  "missing_gates": [
    "gate-canary-e2e.json"
  ]
}
```

### 10. Action budgets and permissions

Agents need constraints.

Every agent session should have a capability profile:

```json
{
  "max_pipelines": 3,
  "allowed_projects": [2],
  "allowed_actions": [
    "GetSystemSnapshot",
    "FetchCapsule",
    "RunTests",
    "ProposePatch"
  ],
  "forbidden_actions": [
    "RotateSecrets",
    "PromoteProduction",
    "HostReclaimApply"
  ],
  "approval_required": [
    "RequestMerge",
    "PromoteProduction"
  ]
}
```

That would let the system grant agents real autonomy without giving them unsafe power.

# 3. The dream TUI: what I would redesign

I would redesign the TUI around one idea:

> **Every entity should answer: What is happening, why is it happening, what is blocked, what evidence exists, and what actions are safe right now?**

The current TUI is a dashboard. The dream TUI should be a **command console**.

## The dream layout

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ JERYU COMMAND CENTER  Project dougx/#2  Ref main  GitLab ONLINE  Vault OK    │
│ Release ci-fa51a52a  Canary RUNNING  Prod LOCKED  Disk 68%  Cache 82% hit   │
├───────────────┬──────────────────────────────────────────────┬───────────────┤
│ NAVIGATOR     │ FLOW THEATER                                 │ INSPECTOR     │
│               │                                              │               │
│ ▸ Release     │ Push → Admission → Impact → Build → Tests     │ Selected:     │
│ ▸ Pipelines   │              ↓        ↓        ↓              │ integration   │
│ ▸ Jobs        │          Security → Package → Canary → Prod   │ Status: FAIL  │
│ ▸ Agents      │                                              │ Why: panic    │
│ ▸ Tests       │ [Upstream CI]     success        100%         │ Retry: deny   │
│ ▸ Cache       │ [Release Exec]    running         62% ETA 4m  │ Evidence: yes │
│ ▸ Storage     │ [Production]      locked           gate wait  │ Actions:      │
│ ▸ Audit       │                                              │  Explain      │
│               │ Critical path: build-image → e2e → prod-gate │  Fetch logs   │
├───────────────┴──────────────────────────────────────────────┴───────────────┤
│ EVENT + COMMAND CONSOLE                                                       │
│ 12:04:12 release gate waiting: gate-canary-e2e.json missing                  │

│ > Ctrl-K command palette | / search | ? help | Enter inspect | a actions     │
└──────────────────────────────────────────────────────────────────────────────┘
```

The important change is that the TUI becomes **selection-driven**. Anything selected on the left or in the graph populates the right inspector and bottom actions.

# 5. The critical information that matters most

For a developer power user, these are the highest-value facts.

At all times, the UI should answer:

```text
Are we green?
If not, what is the blocker?
Is production safe/unlocked?
What changed?
What is running?
What should I do next?
```

Critical release information:

* current version

# 6. Color and visual design

I would use a dark, high-contrast, color-rich terminal style, but never rely on color alone. Every important state should also have a label or icon.

## Suggested visual language

| Meaning              | Color                 | Shape/text  |

# 7. The dream command palette

The TUI should have a `Ctrl-K` command palette. This is where the “all actions” dream comes alive.

Examples:

```text
Ctrl-K
  retry selected failed job
  explain selected job
  fetch failure capsule
  cancel selected pipeline
  doctor selected pipeline
  run VTI test selection
  run validation plan
  spawn agent task from failure
  request MR merge
  reconcile release
  promote production
  rotate release secrets
  run cache GC dry-run
  apply cache GC
  scale pool
  pause pool
  resume pool
  host doctor
  audit agent surface
```

Each action should display:

```text
Action: Promote production
Status: unavailable
Reason: canary_state is running; requires e2e-passed
Missing:
  - gate-canary-e2e.json
  - C validation artifact
```

Or:

```text
Action: Retry job #14445
Status: available
Safety:
  classification: Infrastructure
  previous retries: 0
  decision: RetryOnce
```

This would be spectacular for power users.

# 9. Improvements to the current TUI implementation

If I were improving the existing TUI incrementally, I would do it in this order.

## First wave: fix confusing or misleading behavior

1. **Remove or visibly restore invisible Pools/Pipelines panes**
   Do not allow focus/actions against panes the user cannot see.

2. **Replace Audit placeholder with real event ledger view**
   Even a simple recent-events list would be a major improvement.

3. **Rename delete actions to “forget local record”**
   Make it clear this does not cancel remote GitLab work.

4. **Expose project/ref context in the header**
   Show `project_id=2`, `ref=main`, and release repo root.

5. **Normalize release-state labels**
   Make release engine, CLI, and TUI use the same state vocabulary.

6. **Show exact missing gates for production promotion**
   The Release Watch pane should always explain why prod is locked.

7. **Make unsupported capability intents return explicit deferred/unsupported status**
   Do not return success for no-op agent intents.

## Second wave: make it operationally powerful

8. **Wire the job inspector into the active layout**
   The inspector already exists but is inactive. Bring it back as a right-side pane or modal.

9. **Add command palette**
   This is probably the single biggest UX unlock.

10. **Add pipeline doctor/explain actions from the TUI**
    The CLI has excellent commands. The TUI should expose them.

11. **Add release gate checklist**
    Make canary/prod gates visible and actionable.

12. **Show historical ETA and bottleneck context**
    Use `ci_job_runs` and `test_executions`.

13. **Show pipeline chain, not just first active pipeline**
    Upstream → release-execution → production-promotion.

14. **Use event correlation IDs**
    Tie agent actions, pipelines, retries, and release decisions together.

## Third wave: make it magical

15. **Implement graph edges and critical path**
    The Flow Board becomes a real dependency graph.

16. **Add event streaming instead of polling where possible**
    Especially for logs and engine events.

17. **Integrate proof-scoped tools into the TUI**
    Run/view:

* `cargo-witness`
* `cargo-vrc`
* `cargo-aer`
* `repo audit-agent-surface`

18. **Add Agent screen with MR/risk/validation state**
    This would make the product feel agent-native.

19. **Add “spawn fix agent from failure”**
    Select failed job → fetch capsule → create task → MR → validation.

20. **Add replayable action manifests**
    Every action produces a structured record:

```text
actor, intent, target, preconditions, result, evidence
```

# 10. What I would not do

I would not give agents unrestricted access to:

* shell commands
* raw GitLab PATs
* arbitrary Docker control
* direct production deploy commands
* direct Vault secret reads
* unconstrained host reclaim
* custom executor commands as general-purpose shell wrappers

The current architecture is right to make production promotion pipeline-driven and gate-mediated. Preserve that. The dream is not “more power by bypassing controls.” The dream is **more power through typed, visible, auditable controls**.

---

# 11. My strongest product recommendation

The north star should be:

> **jeryu should become a real-time, evidence-driven CI/CD command center where every action is gated, explainable, and agent-operable.**

The TUI should not merely show state. It should show:

```text
State → Cause → Evidence → Gate → Available actions → Expected outcome
```

For example:

```text
Production is locked
because canary E2E evidence is missing
from release attempt ci-fa51a52a7882
after release pipeline #812
next safe action is watch/reconcile, not promote
```

Or:

```text
Job integration-db failed
classified as Regression
retry denied
failure capsule available
next safe action is spawn fix agent or inspect logs
```

That is the difference between a dashboard and a command console.

# 12. The top 10 highest-leverage changes

If you only did ten things, I would pick these:

1. **Make Capability API unsupported intents return explicit non-success.**
2. **Use `evidence_capsules` for `FetchCapsule`, not latest-500 event scanning.**
3. **Fix invisible TUI panes/actions.**
4. **Add a real Audit/Event Ledger tab.**
5. **Add a command palette with safe/disabled action explanations.**
6. **Add a release gate checklist showing exact missing production gates.**
7. **Show upstream → release-execution → production-promotion as one linked release chain.**
8. **Wire the job inspector into the active Flow layout.**
9. **Use historical job/test timing for ETA and bottleneck explanations.**
10. **Add agent task/MR/risk/validation visibility as a first-class TUI screen.**

That combination would make the system feel dramatically more powerful without weakening the safety model.
