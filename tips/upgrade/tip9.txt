Absolutely — this system is *already* unusually ambitious. My strongest read is that `jeryu` is not merely a GitLab helper; it is becoming a **local delivery operating system for agents and humans**. The architecture has real depth: typed control surfaces, runner pools, release gates, test intelligence, failure evidence, SmartCache, Vault-backed secrets, and a TUI that is trying to become an operational nerve center.

The biggest opportunity is this: the backend is much more powerful than the human/agent interface currently exposes. The system has the ingredients for a world-class command center, but the visible UI and capability API are still behind the orchestration engine.

Below is my best critique and redesign proposal.

---

# 1. High-level verdict

The strongest parts of the system are:

1. **Production is pipeline-mediated, not shell-mediated.**
   This is excellent. Production promotion through `CI_PIPELINE_PRODUCT=production-promotion` with explicit gate variables is the right safety posture.

2. **State is durable and evidence-oriented.**
   SQLite as the coordination/audit layer gives the system memory. Tables like `release_attempts`, `evidence_capsules`, `retry_decisions`, `test_plans`, `cache_verdicts`, and `selector_misses` are exactly the kind of substrate agents need.

3. **Agents are intended to operate through typed capabilities.**
   That is much better than handing agents broad PATs and shell access.

4. **The TUI already solves one hard UX problem: partial data stability.**
   The anti-blanking flow behavior is smart. Keeping the last meaningful board visible and marking it stale is much better than flickering between “real state” and empty state.

5. **The release architecture is unusually disciplined.**
   Canary, release-execution, and production-promotion are separate phases. This gives you a credible path toward agent-assisted deployment without reckless autonomy.

The main weaknesses are:

1. **The system has more power than its interfaces reveal.**
   Many controls exist, but the TUI does not yet make them discoverable, explainable, or safely actionable.

2. **The Capability API is too small and partly misleading.**
   Several intents deserialize and return success even though routing is deferred. For an agent-facing API, that is dangerous.

3. **The TUI active-pane model and visible layout are out of sync.**
   Pools and Pipelines can be focused by keybindings, but the active Flow layout visibly shows only Live Jobs and logs. That creates hidden state and possible accidental actions.

4. **The release view does not yet answer the most important operator question: “Why are we not in production yet?”**
   It shows state, but it needs a gate matrix with exact blockers.

5. **The Audit tab is a missed opportunity.**
   Given how rich the event/state model is, the Audit tab should be one of the most powerful parts of the product, not a placeholder.

---

# 2. The most important strategic gap

The biggest architectural gap is the absence of a unified **Action + Evidence Graph**.

Right now, the system has:

* CLI actions
* TUI actions
* webhook reactions
* release automation
* DB state
* event ledgers
* capability intents
* GitLab pipelines
* runner pools
* cache decisions
* test plans
* secrets lifecycle events

But from an operator or agent perspective, there is no single object that says:

> Here is what is happening.
> Here is why it is happening.
> Here is what is blocked.
> Here is what you can safely do next.
> Here is what evidence supports each action.
> Here is what risk tier each action carries.
> Here is what will change if you execute it.

That should become the center of the system.

I would introduce a first-class concept like:

```text
ActionAffordance
```

with fields such as:

```text
id
label
target_type
target_id
risk_level
trust_required
preconditions
blocked_by
would_change
undo_hint
cli_equivalent
capability_equivalent
requires_confirmation
requires_approval
evidence_links
```

Then the TUI, CLI, and Capability API could all expose the same action model.

For example, on a failed job, the action list might be:

```text
Retry job                    allowed       low risk
Fetch failure capsule         allowed       read-only
Open trace                    allowed       read-only
Explain failure               allowed       read-only
Quarantine cache key          allowed       medium risk
Cancel pipeline               allowed       medium risk
Escalate to human             allowed       no-op/audit
Merge MR                      blocked       failed job present
Promote production            blocked       not release context
```

That one concept would make the whole system feel dramatically more coherent.

# 3. Clear issues and gaps I would address

## 3.1 Capability API issues

The Capability API is the most important agent-facing surface, but it currently feels like an early stub compared with the rest of the system.

### Issue: deferred intents return success

The docs say these deserialize and return success, but active routing is deferred:

* `ProposePatch`
* `RacePatches`
* `RequestMerge`

That is risky. An agent may interpret `success: true` as “the work happened.”

Better response shape:

```json
{
  "success": false,
  "status": "unsupported",
  "message": "RequestMerge is recognized but not currently implemented.",
  "data": {
    "intent": "RequestMerge",
    "recommended_surface": "jeryu agent merge"
  }
}
```

Or:

```json
{
  "success": true,
  "status": "deferred",
  "performed": false,
  "message": "Intent accepted but no mutation was performed."
}
```

But `success: true` alone is too ambiguous.

### Issue: `FetchCapsule` scans recent event rows

The docs say `FetchCapsule` scans the latest 500 event log rows looking for `failure_capsule`.

That should probably use the dedicated state method:

```text
latest_evidence_for_job
```

or a direct indexed query against `evidence_capsules`.

A 500-row event scan can fail exactly when the system is busiest, which is when agents most need the evidence. Failure capsules are important enough to be retrieved through the state API directly.

### Issue: no capability introspection

Agents need to ask:

```text
What intents are supported?
What parameters do they accept?
Which ones mutate state?
Which ones require approval?
Which ones are currently blocked?
```

I would add:

```text
DescribeCapabilities
ListAffordances
PreflightAction
ExecuteAction
SubscribeEvents
ExplainBlocker
```

This would let agents operate safely without reverse engineering CLI docs.

### Issue: no action preflight

Every mutating capability should support dry-run/preflight.

For example:

```json
{
  "intent": "PreflightAction",
  "payload": {
    "action": "RetryJob",
    "project_id": 2,
    "job_id": 14445
  }
}
```

Response:

```json
{
  "success": true,
  "message": "Retry is allowed.",
  "data": {
    "risk": "low",
    "would_call": "GitlabClient::retry_job",
    "preconditions": [
      "job status is failed",
      "project is reachable"
    ],
    "blocked_by": []
  }
}
```

### Issue: socket security is underspecified

The docs mention a Unix socket and 64 KiB payload cap. I would also want explicit guarantees around:

* socket file permissions
* peer credential checks, such as Unix `SO_PEERCRED`
* allowed user/group
* request IDs
* audit logging
* rate limits
* per-intent authorization
* payload schema versioning

The capability server is the ideal place to enforce least privilege.

## 3.2 CLI and control-surface gaps

The CLI is broad and powerful, but its power creates discoverability and safety concerns.

### Issue: no generated action manifest

The docs are strong, but the system should generate a machine-readable action manifest from clap plus domain metadata.

Something like:

```bash
jeryu surface list --json
```

Output:

```json
[
  {
    "command": "jeryu job retry <project_id> <job_id>",
    "category": "job",
    "mutates": true,
    "risk": "low",
    "requires_gitlab": true,
    "requires_db": false,
    "safe_for_agent": true,
    "preflight_available": true
  },
  {
    "command": "jeryu release promote-prod",
    "category": "release",
    "mutates": true,
    "risk": "high",
    "requires_gitlab": true,
    "requires_release_gates": true,
    "safe_for_agent": "only_with_gate"
  }
]
```

This would be incredibly useful for agents, TUI command palettes, docs validation, and tests.

### Issue: placeholder commands should be visibly marked

Examples:

* `jeryu jeryu sync-now` is currently a placeholder.
* MR Hook is logged but has no active behavior.
* Audit TUI tab is placeholder.
* Capability `ProposePatch`, `RacePatches`, `RequestMerge` are deferred.

Those should all be exposed as “known but inactive” surfaces. Placeholders are fine, but they should not look operational.

### Issue: hidden command misuse risk

The docs correctly warn that custom executor commands and server hook commands are not general-purpose APIs. But I would go further:

```bash
jeryu exec run ...
```

should probably detect direct human/agent invocation unless required runner environment variables exist, and print something like:

```text
This command is a GitLab Runner custom executor protocol hook.
It is not intended for direct use.
```

The same applies to `server-hook pre-receive`.

## 3.3 TUI issues

The TUI has a strong foundation, but it is currently not yet the “power console” the backend deserves.

### Issue: active pane model does not match visible UI

The TUI still cycles:

```text
Pools -> Pipelines -> Jobs
```

But the active Flow layout visibly exposes only:

* Flow Board
* Live Jobs
* Log Preview

Pools and standalone Pipelines are not visible in the active Flow layout.

This creates a dangerous UX problem:

* The user may focus `Pipelines` without seeing a pipeline list.
* Pressing `d` can delete a selected local pipeline record.
* The visible object and the acted-on object may not be obvious.

This is one of the first things I would fix.

Either:

1. Bring visible Pools and Pipelines panes back into the layout, or
2. Remove Pools/Pipelines from the active pane cycle in the current Flow tab, or
3. Make the right-side Inspector always show the currently focused target and pending action.

The third option is best.

### Issue: `d` deletes local state, not GitLab state

That is useful, but the UI should say very clearly:

```text
Delete local record only
```

not just delete.

A user could easily think they are deleting/canceling a GitLab pipeline or job.

I would rename the action in the UI to:

```text
Hide local job record
Forget local pipeline record
```

and reserve `Cancel` for actual GitLab cancellation.

### Issue: Flow Board renders only the first active pipeline

This is understandable for an initial implementation, but operationally limiting.

The system can have:

* upstream main pipeline
* release-execution pipeline
* production-promotion pipeline
* scratch test pipelines
* downstream child pipelines
* superseded pipelines
* agent MR pipelines

A dream TUI needs to show multiple concurrent flows, or at least provide a pipeline selector strip.

### Issue: graph edges exist but are empty

The Flow Board will become much more powerful once edges are real.

Edges should show:

* job dependency
* bridge/downstream relationship
* release trigger relationship
* supersedence relationship
* “this failed job blocked that release gate”
* “this test was selected because of these changed paths”

Without edges, the board is a status wall. With edges, it becomes a causal map.

### Issue: release state color mapping may miss critical states

The production gate requires:

```text
canary_state == "e2e-passed"
```

But the TUI color mapping described includes:

```text
green
released
in-flight
canary-authorized
waiting
ready-for-canary
blocked
blocked-by-upstream
failed
```

I would make sure states like these are explicitly styled:

```text
e2e-passed
ready-for-prod
prod-triggered
prod-running
prod-succeeded
prod-failed
handoff-missing
validation-missing
```

`e2e-passed` should not fall into a generic gray/default state. It is one of the most important release states.

### Issue: Release Watch needs blocker explanations

The release pane currently shows useful fields, but it needs to answer:

```text
Why has production not triggered?
```

For example:

```text
Production: blocked

Missing:
  ✕ C handoff artifact
  ✓ C validation artifact
  ✓ canary E2E passed
  ✓ no existing prod pipeline for same SHA

Next safe action:
  Open release-execution job artifacts
```

That single pane would radically improve operator confidence.

### Issue: Audit tab should not be a placeholder

Given the system’s state model, the Audit tab could be incredible.

It should show:

* recent event ledger entries
* webhook events
* auto-retry decisions
* release gate decisions
* risk gate decisions
* agent actions
* cache taint/quarantine events
* secret lifecycle events
* disk pressure GC actions
* supersedence cancellations

Right now, a major part of the system’s value is invisible.

## 3.4 Engine and webhook issues

### Issue: unauthenticated `/cache/summary` on `0.0.0.0`

The webhook server binds to:

```text
0.0.0.0:9777
```

`POST /hooks` requires the GitLab token, which is good.

But:

```text
GET /health
GET /cache/summary
```

are unauthenticated.

That may be fine in a trusted local environment, but `/cache/summary` leaks operational information. I would either:

* bind non-webhook routes to localhost only,
* require a local token,
* or make unauthenticated cache summary intentionally minimal.

At minimum, docs should say whether port `9777` is expected to be exposed beyond localhost.

### Issue: webhook idempotency/replay details are not visible

GitLab webhooks can be retried. The engine likely handles some upserts, but from the docs I would want a more explicit idempotency story:

* event UUID if available
* duplicate suppression
* replay tolerance
* append-only event correlation
* safe retries for release triggers

Release/prod triggers especially need clear idempotency, and the TUI should show when something was ignored because it was duplicate or superseded.

### Issue: supersedence needs a visible safety story

Canceling superseded pipelines is excellent for capacity, but users need visibility into:

* which pipelines were canceled
* why
* what newer SHA replaced them
* whether release/prod pipelines are protected from cancellation

This belongs in the Audit tab and Flow Board.

## 3.5 State and observability gaps

### Issue: event ledger is not prominent enough

The `events` table is one of the most important assets in the architecture. It should power:

* Audit tab
* event tape in the Flow tab
* release timeline
* agent activity timeline
* “why did jeryu do that?” explanations

Every major operation should have a correlation ID:

```text
push_event_id
pipeline_id
job_id
release_attempt_id
agent_task_id
action_id
request_id
```

Then the UI can reconstruct stories:

```text
Push received
Impact decision: release-impacting
VTI plan recorded
Pipeline started
Pipeline succeeded
Canary claimed
Release pipeline triggered
Release E2E passed
Production promotion triggered
```

That is the kind of thing power users love.

### Issue: no obvious retention/pruning policy

The DB stores a lot:

* job events
* pipeline timings
* events
* cache requests
* cache verdicts
* test executions
* selector misses
* secret audit events

That is good, but it needs visible retention/compaction policy:

```text
Keep event ledger for N days
Keep job trace snippets for N days
Keep timing history for N runs
Keep cache request history for N days
```

The TUI Storage tab should show DB growth and retention status.

### Issue: TUI snapshot refresh needs deadlines and source freshness

The TUI pulls from DB, GitLab, Docker, filesystem, cache ports, and release reports. Each data source should have:

```text
last_success_at
last_error
latency
freshness_age
```

The UI should show source freshness per pane, not just global online/offline.

## 4.2 What agents especially want: hypothesis racing

The docs already include `RacePatches`, but it is deferred. This could become one of the most powerful parts of the system.

An agent could say:

```text
Hypothesis A: failure caused by missing feature flag
Hypothesis B: failure caused by cache taint
Hypothesis C: failure caused by changed API signature
```

`jeryu` could then:

1. Create three scratch branches.
2. Apply three candidate patches.
3. Run targeted VTI tests.
4. Compare results.
5. Keep the winning branch.
6. Delete losing branches.
7. Produce an evidence summary.

The TUI could show this as an “Agent Race” board:

```text
Hypothesis        Branch              Status       Tests       Confidence
A feature flag    agent/a-flag         failed       2/5         low
B cache taint     agent/b-cache        passed       5/5         medium
C API signature   agent/c-api          running      3/5         pending
```

That would be spectacular.

## 4.3 What agents need most: permissioned action leases

Instead of giving agents permanent broad power, give them scoped leases:

```text
Agent may:
  - read job logs
  - fetch artifacts
  - create branches matching agent/*
  - run up to 5 test pipelines
  - open MRs
  - request merge, but not accept
  - request production promotion, but not approve
```

Lease example:

```json
{
  "lease_id": "lease_123",
  "actor": "agent-ci-repair",
  "expires_at": "2026-04-25T23:30:00Z",
  "allowed_actions": [
    "FetchCapsule",
    "FetchTrace",
    "PlanTests",
    "RunTests",
    "ProposePatch",
    "OpenMergeRequest"
  ],
  "max_pipelines": 5,
  "branch_prefix": "agent/"
}
```

This would make the agent model much safer and more powerful.

# 5. Critical issues I would prioritize immediately

If I had to pick the most important fixes, I would start here.

## Priority 1: Make deferred capability intents explicitly unsupported

Do not return plain success for no-op/deferred agent actions.

This matters because agents will build plans around response semantics.

## Priority 2: Fix TUI focus/action mismatch

Visible focus and actionable target must always match.

If a key can mutate something, the selected object should be visible and clearly identified.

## Priority 3: Add a Release Gate Matrix

The Release Watch pane should show every gate:

```text
Canary started               ✓

C handoff artifact           ✓
C validation artifact        ✕ missing
Production already exists    ✓ no
Production promotion         blocked
```

This is probably the single highest-value TUI enhancement for deployment confidence.

## Priority 4: Make the Audit tab real

Turn the Audit tab into an event ledger viewer.

At minimum:

```text
time
event type
target
actor/source
summary
correlation id
```

Filters:

```text
release
pipeline
job
agent
cache
secret
security
disk
webhook
```

## Priority 5: Add a contextual Inspector + Action pane

Every selected object should have:

```text
Status
Why
Evidence
Risk
Next safe actions
Blocked actions
CLI equivalent
```

That one pane would make the TUI feel like a command console rather than a dashboard.

## Priority 6: Add action preflight

Before any mutating action:

```text

Pause pool

```

show:


## Priority 9: Add event correlation IDs

This will pay off everywhere: TUI, audit, release explanations, agent workflows, and debugging.

## Priority 10: Add source freshness indicators

Every major pane should show whether its data is fresh, stale, degraded, or unavailable.

# 7. Color-rich high-contrast visual language

I would use a dark background with aggressive semantic colors, but never rely on color alone.

## Recommended semantic colors

```text
Green       success, safe, complete
Blue        running, live work
Cyan        selected, inspectable, planned
Yellow      pending, waiting, warning
Orange      degraded, resource pressure, slow
Red         failed, blocker, destructive
Magenta     security, secrets, trust boundary
Purple      release/deployment identity
Gray        stale, skipped, canceled, unknown
White       primary text
Dim gray    metadata
```

## Icons / glyphs

Use icons plus text labels:

```text

● running
◌ pending
✕ failed
⏸ paused
↻ retrying
⚠ warning
⛔ blocked
🔒 gated/security
◇ unknown
· not started
```

For terminals without good glyph support, provide ASCII fallback:

```text
[OK]
[RUN]
[PEND]
[FAIL]
[WARN]
[BLOCK]
[LOCK]
```

## Visual rules

* **Critical path gets bright borders.**
* **Blocked release gates get red/magenta cards.**
* **Security events are magenta, not just red.**
* **Stale data is dimmed and explicitly labeled.**
* **Destructive actions are red and require confirmation.**
* **Disabled actions are visible but gray with reasons.**
* **Selected object is cyan with a clear inspector.**

The dream TUI should feel colorful, but not decorative. Every color should mean something.

# 9. Realtime architecture for the dream TUI

The current polling model is acceptable, but the dream version should be event-driven.

I would add an internal event stream from the engine:

```text
GET /events/stream
```

or a Unix socket stream for local TUI use.

Events:

```text
PipelineUpdated
JobUpdated
JobTraceAppended
ReleaseAttemptUpdated
GateChanged
PoolChanged
ManagerChanged
CacheMetricUpdated
DiskPressureChanged
AgentTaskUpdated
SecretSetUpdated
AuditEventAppended
```

The TUI should consume deltas, not rebuild everything every 1500ms.

Keep polling as fallback, but make the main path event-driven.

## Logs

Instead of polling full traces every 650ms, use one of:

* incremental trace fetching with byte offsets
* ETag/If-None-Match
* streaming proxy from engine
* local event relay

This would reduce load and make logs feel more live.

## State model

Internally, the TUI should normalize state by entity ID:

```text
jobs_by_id
pipelines_by_id
release_attempts_by_id
pools_by_name
agents_by_id
events_by_id
```

Then render derived views.

This avoids accidental state loss and makes selection preservation easier.

# 17. A better agent-control model

I would formalize agent control into four modes.

## Mode 1: Observe

Agent can read:

```text
status
events
logs
capsules
test plans
release state
```

No mutations.

## Mode 2: Validate

Agent can:

```text
run tests
retry safe jobs
create scratch branches
fetch artifacts
```

Limited budget.

## Mode 3: Propose

Agent can:

```text
create branches
commit patches
open MRs
comment with evidence
```

Cannot merge.

## Mode 4: Operate

Agent can:

```text
pause/resume pools
scale pools
request merge
request release reconciliation
request production promotion
rotate secrets
```

Only with gates and explicit policy.

This should be reflected in the capability server, TUI, and audit trail.

# 18. One subtle but important UX principle

The TUI should always distinguish between these four categories:

```text
Observed state
Inferred diagnosis
Recommended action
Executed action
```

For example:

```text
Observed:
  job #14445 failed

Inferred:
  likely regression

Recommended:
  do not retry; inspect failure capsule

Executed:
  no retry performed
```

This is important for trust. Agents and humans both need to know when the system is reporting facts versus making judgments.
