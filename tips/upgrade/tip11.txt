Based on the API, architecture, and TUI docs you shared, my overall read is: **jeryu is already much more than a CI helper. It is becoming a local delivery operating system.** The strongest parts are the typed control surfaces, release gates, durable RedlineDB state, failure evidence, VTI test selection, trust-tiered runners, Vault-backed release secrets, and the anti-blanking TUI behavior. The biggest opportunity is to turn this from a powerful but somewhat fragmented control plane into a **real-time, explainable, safety-gated command console** where a human or agent can always answer:

**What is happening? Why? What is blocked? What changed? What can I safely do next? What will happen if I do it?**

Below is my best feedback.

---

# 1. The biggest architectural strengths

The system has several unusually strong design choices.

**The control plane is stateful and evidence-driven.** RedlineDB is not just cache storage; it acts as a coordination and audit layer. That is the right move for agents because agents need memory, evidence, and explainability.

**Release and production are mediated by pipelines, not shell commands.** This is excellent. The fact that production promotion is represented as a GitLab pipeline with explicit variables and gates is a major safety feature.

**The system already separates trust tiers.** Trusted, privileged, and untrusted pools are a good foundation for agent work. An agent should not be treated as a magical admin shell. It should operate through scoped, typed, auditable capabilities.

**The TUI is designed for partial data.** The anti-blanking Flow Board behavior is very good. A power-user console must never flicker into “nothing is happening” merely because GitLab briefly returns empty data.

**Failure capsules are the right primitive.** A failed job should not just be “red.” It should be an evidence object: classification, snippet, retry advice, ref, commit, stage, replay context, and proposed next actions.

**VTI and proof-scoped tools are a major differentiator.** `cargo-witness`, `cargo-vrc`, `cargo-aer`, VTI plans, selector audits, and test cache keys give agents a way to reason about validation instead of brute-forcing everything.

# 2. The clearest gaps

## Gap 1: The Capability API is too partial for serious agents

The capability socket currently supports active `FetchCapsule` and `RunTests`, while `ProposePatch`, `RacePatches`, and `RequestMerge` deserialize but return deferred success.

That is risky semantically. A capability that returns “success” while not actually performing the named action can confuse agents and supervising systems.

I would change deferred intents to return something like:

```json
{
  "success": false,
  "message": "Intent RequestMerge is recognized but not implemented",
  "data": {
    "status": "not_implemented",
    "recommended_surface": "jeryu agent merge"
  }
}
```

An agent should be able to distinguish:

* implemented
* rejected by policy
* blocked by missing evidence
* not implemented
* accepted and queued
* completed
* completed with warnings

Right now, the capability API is promising a future agent control plane, but the CLI is still the real one.

## Gap 2: There is no obvious unified action registry

The API surface is rich, but it is mostly documented as commands. For a dream TUI and agent system, you want a machine-readable action registry.

Something like:

```bash
jeryu actions list --json
```

Each action should declare:

```json
{
  "id": "job.retry",
  "label": "Retry failed job",
  "entity_types": ["job"],
  "required_state": ["status == failed"],
  "risk": "low",
  "requires_confirmation": false,
  "required_scope": "ci:write",
  "dry_run_supported": true,
  "implemented_by": "GitlabClient::retry_job"
}
```

The TUI could then render **all possible actions** for the currently selected job, pipeline, pool, release, secret set, cache object, or agent task. Agents could also query the registry before attempting actions.

## Gap 3: The TUI has focus state for panes that are not visibly present

The docs say the active Flow layout visibly shows Live Jobs and Log Preview, while older Pools and Pipelines panes still exist but are not active renderers. Yet keybindings still cycle among Pools, Pipelines, and Jobs.

That is a UX mismatch. A user can focus “Pools” or “Pipelines” without seeing a clear corresponding pane. That will feel broken even if the internal state is correct.

This should be fixed one of two ways:

1. Reintroduce visible Pools and Pipelines panes into the Flow layout, perhaps as collapsible side panels.
2. Remove invisible pane focus from the active layout and move pool/pipeline actions to a command palette or dedicated tabs.

For a power-user TUI, hidden focus states are dangerous because actions like `p`, `d`, or `r` can become surprising.

## Gap 4: The Flow Board shows only the first active pipeline

The Flow Board is the centerpiece, but it currently renders only the first active pipeline. That leaves a lot of operational reality invisible.

A power user needs to see:

* current upstream `main` pipeline
* release-execution pipeline
* production-promotion pipeline
* scratch `jeryu-test-*` pipelines
* agent MR pipelines
* downstream child pipelines
* superseded/canceling pipelines
* blocked pipelines
* pipelines waiting for runners

The current release pipeline prioritization is good, but the TUI needs a multi-pipeline mode. Otherwise the operator sees one important thing while missing the rest of the system.

## Gap 5: Graph edges are not computed

The Flow Graph has `edges`, but they are currently empty. That limits the board’s ability to explain causality.

A dream Flow Board should show:

```text
Push -> Admission -> Impact -> Build -> Unit -> Integration -> Security -> Package -> Canary -> Prod
```

And it should visually distinguish:

* required edge
* optional edge
* skipped-by-VTI edge
* blocked edge
* failed edge
* superseded edge
* downstream bridge edge
* release gate edge

Without edges, the board is more of a lane-grouped job display than a true operational graph.

## Gap 6: Release visibility needs “why not?” explanations

The release system has strong gates, but the TUI should aggressively explain blocked states.

For any release attempt, the operator should immediately see:

```text
Production promotion: BLOCKED

Reason:
  Canary E2E gate: passed
  C handoff artifact: missing
  C validation artifact: present
  Existing prod pipeline: none
  Required release state: e2e-passed
  Current release state: waiting

Next safe action:
  Regenerate release handoff report
```

The current Release Watch pane shows key fields, but it should show the exact gate matrix. Power users and agents both need that.

## Gap 7: Status strings appear stringly typed

There are many status values across jobs, releases, canaries, production, cache, taint, tests, and pipeline products. The docs show strings like `green`, `released`, `in-flight`, `canary-authorized`, `waiting`, `ready-for-canary`, `blocked`, `blocked-by-upstream`, `failed`, `e2e-passed`, and others.

I would strongly prefer typed enums for all release and gate states, with a single canonical mapping to display labels, colors, and allowed actions.

Otherwise, small string drift can break rendering, gate logic, or automation.

## Gap 8: Capability socket authorization is underspecified

A Unix socket is a good local transport, but agent capability access still needs identity and scopes.

At minimum, the capability server should understand:

* actor identity
* session id
* allowed intents
* allowed project ids
* allowed refs
* allowed trust tier
* whether release actions are permitted
* whether destructive actions are permitted
* whether secret-related actions are permitted
* expiration time / lease duration

Without this, access to the socket can become equivalent to broad operational power.

## Gap 9: `RunTests` capability needs lifecycle controls

Capability `RunTests` creates a branch, writes dynamic CI, and triggers a pipeline. That is useful, but an agent also needs:

* returned pipeline id
* returned branch name
* cleanup policy
* cancellation endpoint
* wait/watch endpoint
* trace retrieval
* result summary
* failure capsule link
* TTL for scratch branches
* deduplication/cache behavior

The CLI `jeryu test run` sounds more complete than the capability intent. I would either make capability `RunTests` call the same logic as `jeryu test run`, or clearly expose two modes: `trigger_only` and `run_and_wait`.

## Gap 10: Webhook/cache endpoint exposure should be tightened

The engine binds to `0.0.0.0:9777`. `/hooks` is token-protected, but `/cache/summary` is unauthenticated.

That may be fine in a fully local network, but it is still worth tightening. Cache summaries can reveal operational state. I would either bind to localhost by default, require auth for non-health endpoints, or support explicit `--listen` / `--public` configuration.

## Gap 11: Runner scaling should be tag-aware

The reconciliation loop computes capacity from pending jobs and pool min/max. The docs make it sound like pending job count may be global rather than tag/pool-specific.

For best behavior, scaling should be based on:

* pending jobs by tag set
* trust tier
* executor type
* runner eligibility
* queue wait time
* historical job duration
* max concurrent jobs per manager
* warm pool deficit
* stuck runner detection

Otherwise, the system may scale the wrong pool or fail to scale the pool that actually has demand.

## Gap 12: TUI log polling is useful but eventually expensive

Polling job traces every 650ms is fine as a starting point, but it can become expensive with many users, large logs, or slow GitLab responses.

A better design would support:

* incremental trace ranges
* backoff when logs are unchanged
* event-driven refresh when job status changes
* local log cache
* stream compression
* “follow selected job only” by default
* multiple pinned logs only on demand

A websocket/SSE transport is not required immediately, but a richer log streaming abstraction would be valuable.

## Gap 13: Failure evidence lookup has two competing homes

The State API has `evidence_capsules` and methods like `latest_evidence_for_job`, but the Capability API `FetchCapsule` scans the latest 500 event ledger rows for `event_type == "failure_capsule"`.

That seems inconsistent. Capability `FetchCapsule` should probably use `Db::latest_evidence_for_job` or a dedicated evidence query. Scanning the event log is brittle and may miss older failures.

## Gap 14: Direct SQL in the TUI weakens state encapsulation

The architecture says callers should use `Db` methods rather than raw SQL except in migrations and narrowly scoped TUI metrics. The TUI currently does some direct count queries for taints and cache verdicts.

That is acceptable as a transitional implementation, but for long-term maintainability I would move those into state-owned metric methods, for example:

```rust
db.cache_health_summary()
db.security_taint_summary()
db.test_bottleneck_summary()
db.release_gate_summary()
```

## Gap 15: The Audit tab is a placeholder despite the system being audit-heavy

This is one of the biggest UX opportunities. The backend already has an append-only event ledger, retry decisions, secret audit events, release attempts, cache verdicts, taints, test plans, selector misses, and evidence capsules.

The Audit tab should become a first-class “flight recorder.”

It should answer:

* What did jeryu do?
* Who or what requested it?
* Which pipeline/job/release did it affect?
* What evidence justified it?
* What was the result?
* What changed after the action?
* Can I replay the decision?

Right now, that capability exists conceptually but is not surfaced.

# 3. What an agent would dream of being able to do

I would not give an agent raw shell, a broad GitLab PAT, Docker socket access, or Vault tokens. The dream agent experience should be **typed, scoped, auditable, reversible where possible, and always evidence-backed**.

The agent would dream of this loop:

```text
Observe -> Explain -> Plan -> Act safely -> Validate -> Report -> Escalate if needed
```

## The agent should be able to observe everything relevant

It should be able to ask:

* What pipelines are active?
* What jobs are blocked?
* What changed in this ref?
* Which tests were selected and skipped?
* Why was a pipeline superseded?
* Which runner pool is saturated?
* Which release gate is blocking production?
* Which artifact is missing?
* Which cache object was tainted?
* Which secret set is active?
* Which agent MRs are pending?
* What is the current risk gate decision?

This should be exposed through machine-readable APIs, not by scraping TUI text.

## The agent should be able to perform safe CI actions

Good agent-accessible actions:

```text
test.plan
test.run
test.batch
test.select
test.explain_plan
job.explain
job.trace
job.retry_failed
pipeline.explain
pipeline.doctor
pipeline.ingest
pipeline.cancel_superseded
```

`job.retry_failed` should be policy-gated. For example, allow retry only if the job failed, is not release-prod critical unless explicitly allowed, and retry count is below policy.

## The agent should be able to propose and validate patches

The deferred `ProposePatch` intent should become real.

A dream version:

```json
{
  "intent": "ProposePatch",
  "payload": {
    "project_id": 2,
    "base_branch": "main",
    "branch_name": "agent/fix-cache-taint-display",
    "commit_message": "Fix cache taint summary display",
    "patch": "...",
    "validation_plan": {
      "commands": [
        "rtk cargo check -p jeryu --message-format=json",
        "rtk cargo test -p jeryu -- tui -- --nocapture"
      ]
    }
  }
}
```

Response:

```json
{
  "success": true,
  "data": {
    "branch": "agent/fix-cache-taint-display",
    "commit": "...",
    "pipeline_id": 123,
    "mr_iid": 45,
    "risk": "trusted",
    "validation_status": "running"
  }
}
```

## The agent should be able to race hypotheses

`RacePatches` is potentially very powerful. It could let agents test multiple fixes in parallel through separate ephemeral branches.

For example:

```text
Hypothesis A: fix parser
Hypothesis B: adjust state migration
Hypothesis C: change renderer fallback
```

The system should run them as isolated branches, compare validation, and report:

```text
Winner: Hypothesis B
Reason:
  cargo check passed
  tui tests passed
  no witness API regression
  smallest diff
  no security-surface changes
```

This would be an incredible agent capability if gated correctly.

## The agent should be able to request merge, not force merge

`RequestMerge` should not blindly merge. It should produce a risk-gate decision.

Possible responses:

```text
ALLOW:
  All required jobs passed
  No pending jobs
  Trusted tier
  Evidence satisfied

ESCALATE:
  Work came from untrusted tier
  All jobs passed
  Human approval required

DENY:
  Failed jobs present
  Missing required release evidence
```

The agent should be able to prepare the MR and evidence packet, but the system should own the final merge policy.

## The agent should be able to ask “what should I do next?”

This is the killer capability.

A command like this would be huge:

```bash
jeryu next --project-id 2 --ref-name main --json
```

It would return prioritized actionable items:

```json
[
  {
    "priority": "critical",
    "entity": "release",
    "title": "Production promotion blocked",
    "reason": "C handoff artifact missing",
    "safe_actions": ["secrets.report", "release.reconcile"]
  },
  {
    "priority": "high",
    "entity": "runner_pool:build",
    "title": "Build queue waiting 14m",
    "reason": "No warm privileged managers available",
    "safe_actions": ["pool.scale", "pool.resume"]
  }
]
```

Agents thrive when the system can explain the next safe move.

Always visible:

Show:

* active pipelines
* active jobs
* queued jobs
* running release attempts
* active agent tasks
* active runner managers
* active cache operations
* ongoing host reclaim
* secret rotation/finalization status

## 3. What is blocked

This should be a first-class top banner.

Examples:

```text
BLOCKER: production promotion blocked by missing C handoff artifact
BLOCKER: build pool saturated, 6 jobs pending for 11m
BLOCKER: release pipeline failed at gate-canary-e2e
```

The TUI should identify the current bottleneck:

```text
Critical path:
  build-image -> integration-live -> gate-canary-e2e -> production-promotion

Current blocker:
  integration-live running for 18m, p95 is 11m
```

Not just:

```text
ETA 4m
```

```text
ETA 4m ± 2m
```

Or:

```text
ETA unknown
Reason: first run of this job signature
```

For each pipeline/ref:

```text
Changed paths:
```

# 8. Visual design: rich, high-contrast, and information dense

I would use a dark, high-contrast theme with a consistent semantic color system.

## Suggested color semantics

```text
Green      success, safe, complete
Blue       running, active work
Cyan       selected, live, agent-assisted
Yellow     waiting, pending, needs attention
Orange     slow, degraded, approaching threshold
Red        failed, blocked, destructive
Magenta    security, secrets, quarantine, policy hold
Purple     release/canary/prod lane
Gray       stale, unknown, disabled
White      primary text
Dim gray   metadata
```

Important: never rely only on color. Always pair color with symbols or text.

Examples:

```text
✓ success
● running
○ pending
✕ failed
⚠ warning
⏸ blocked
🔒 gated
↺ retrying
◇ skipped
```

## High-impact visual treatments

**Blocked items** should have red borders and an explicit reason.

```text
╔═ PRODUCTION BLOCKED ═════════════════════════╗
║ Missing C handoff artifact                   ║
║ Safe action: regenerate release handoff       ║
╚═══════════════════════════════════════════════╝
```

**Stale data** should be visibly dimmed with a timestamp.

```text
FLOW BOARD [stale 42s]
```

**Release gates** should look like a checklist, not prose.

```text
✓ upstream success
✓ release eligible
✓ canary e2e passed
✕ handoff artifact missing
⏸ production promotion blocked
```

**Risky actions** should be visually separated.

```text
Safe actions:
  retry job
  explain failure
  tail logs

Destructive/gated actions:
  cancel pipeline
  drain pool
  promote production
```

**Critical path** should be visually highlighted across the graph.

```text
build-image ━━━ integration-live ━━━ gate-canary-e2e ━━━ prod-promotion
```

# 15. Final recommendation

The system is already impressive. The main thing I would do is **unify observability, action, evidence, and policy**.

Right now, jeryu has many powerful surfaces:

* CLI
* TUI
* webhook server
* capability socket
* custom executor
* release engine
* state DB
* proof tools
* test intelligence
* cache intelligence
* Vault/secrets lifecycle

The next step is to make them feel like one system.

The most important upgrades are:

1. **Action registry**: every command/action exposed as machine-readable metadata.
2. **Entity inspector**: every job, pipeline, release, pool, cache object, and agent task has a “why / evidence / actions” panel.
3. **Real Audit tab**: event flight recorder with correlation IDs.
4. **Release gate matrix**: always show exactly why canary/prod is allowed or blocked.
5. **Agent capability completion**: real `ProposePatch`, `RacePatches`, and `RequestMerge`, with scoped permissions.
6. **Multi-pipeline Flow Board**: show upstream, release, prod, scratch, MR, and child pipelines.
7. **Graph edges and critical path**: make causality visible.
8. **Command palette**: fuzzy, keyboard-driven, previewing safe and dangerous actions.
9. **Event stream**: make the TUI real-time without relying only on polling.
10. **Typed states and policies**: reduce string drift and make display/action logic reliable.

The dream version is not just “a nicer TUI.” It is a **terminal-native DevOps cockpit** where the developer can see the entire delivery system, understand every decision, safely operate every action, supervise agents, and never wonder what is happening or why.
