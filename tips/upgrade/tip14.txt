## My overall read

You have built something much more interesting than a CLI wrapper. `jeryu` is already a local delivery-control OS: GitLab, runners, custom execution, cache policy, release automation, Vault, evidence, retries, and a supervisory TUI are all tied together through state and typed decisions.

The biggest gap is not “more features.” The biggest gap is **making the existing power legible, explainable, and safely operable by agents and humans in real time**.

Right now the control plane is stronger than the operator surface. The CLI/API has a lot of capability, but the TUI only exposes a fraction of the decision context. The agent surface has the right direction, but several important intents are deferred or not deeply auditable. The release system has strong gates, but the UI should make every gate, artifact, blocker, and “why not promoted?” reason impossible to miss.

If I were advising the project, I would push toward this north star:

> **Every entity should be inspectable. Every action should have a preview, risk rating, required evidence, audit trail, and next-best action. Every automation decision should be explainable.**

---

# 1. The strongest parts of the current design

The architecture already has several excellent instincts.

First, the system is properly **state-centered**. RedlineDB is not just a cache; it is the durable coordination layer for pools, jobs, releases, evidence, secrets, test plans, cache verdicts, and audit events. That gives you the foundation for a serious control console.

Second, the design separates **deployment from shell access**. Production promotion happens through GitLab pipelines with explicit variables and gates, not through arbitrary commands. That is exactly the right safety model.

Third, the runner pool architecture is promising. Trust tiers, custom executor isolation, taint, honeypots, cache witnesses, and BuildKit scoping are the right primitives for agent-driven development.

Fourth, the release model already knows that “green main” is not “production.” It separates upstream, release-execution/canary, and production-promotion pipelines. That is mature.

Fifth, the TUI already has two extremely important UX principles: stable selected jobs across refreshes, and anti-blanking for flow snapshots. Those are small details, but they matter enormously in a live operations UI.

# 2. Clear gaps and issues I would address

## P0: Capability API should not return “success” for deferred powers

The biggest agent-facing issue is this:

`ProposePatch`, `RacePatches`, and `RequestMerge` deserialize and return success, but active routing is deferred.

That is dangerous because an agent may believe it has actually proposed, raced, or requested something. A “successful no-op” is one of the worst states for autonomous systems.

I would change this immediately to one of these:

```json
{
  "success": false,
  "message": "Intent accepted by schema but not implemented",
  "data": {
    "status": "deferred",
    "implemented": false
  }
}
```

Or better, add a capability discovery endpoint:

```json
{
  "intent": "DescribeCapabilities"
}
```

Returning:

```json
{
  "RunTests": "active",
  "FetchCapsule": "active",
  "ProposePatch": "deferred",
  "RacePatches": "deferred",
  "RequestMerge": "deferred"
}
```

Agents need to know what powers are real.

---

## P0: `FetchCapsule` should query evidence directly, not scan the latest 500 events

The docs say `FetchCapsule` scans the latest 500 event ledger rows looking for a `failure_capsule` event with a matching job id.

But the state API already has:

* `evidence_capsules`
* `insert_evidence_capsule`
* `latest_evidence_for_job`
* `list_evidence_for_ref`

The capability API should use the evidence table directly. A noisy event stream could easily push the relevant capsule outside the latest 500 rows. An agent trying to debug an older but important failure would get a false negative.

Recommended behavior:

1. Query `latest_evidence_for_job(job_id)`.
2. Fallback to event ledger only if needed for compatibility.
3. Include capsule freshness and source in the response.

---

## P0: The TUI can perform invisible or confusing actions

The active pane model still includes `Pools`, `Pipelines`, and `Jobs`, but the active Flow layout visibly renders only the Live Jobs list. Pipeline focus affects the Flow Board border, but there is no visible standalone pipeline list. Pool renderers exist but are inactive.

That creates a serious UX mismatch:

* `Tab` can cycle to panes the user cannot clearly see.
* `p` can pause/resume a selected pool even though the active layout may not visibly show pool selection.
* `d` can delete pipeline/job records depending on active pane, but the visible screen may not make the target obvious.

This should be fixed before making the TUI a serious command console.

Options:

1. Restore visible Pools and Pipelines side panes.
2. Remove inactive panes from the active focus cycle.
3. Disable actions on hidden panes.
4. Move all destructive or operational actions behind a command palette with a target preview.

For example:

```text
Action: Pause pool
Target: pool=untrusted
Effect: GitLab runner paused + local DB paused
Risk: Medium
Confirm? y/N
```

---

## P0: Local deletion from the TUI should become soft-delete/archive with audit

The `d` key currently deletes selected local pipeline/job records from RedlineDB.

That is risky because local state is evidence. Even if it does not delete the GitLab job itself, it can hide important operational context.

I would change this to:

* `hide` or `archive` instead of hard delete
* require confirmation for evidence-bearing records
* append an audit event
* provide undo for the current session
* distinguish “remove from local view” from “cancel remote pipeline”

Suggested UI language:

```text
Archive local job record?
This does not cancel the GitLab job.
The record will remain in the audit ledger.
```

---

## P0: The webhook server binds broadly and exposes unauthenticated data

The engine binds to:

```text
0.0.0.0:9777
```

Routes:

* `GET /health` unauthenticated
* `GET /cache/summary` unauthenticated
* `POST /hooks` protected by `X-Gitlab-Token`

`/health` being open is fine. `/cache/summary` may or may not be fine depending on network exposure, but binding to `0.0.0.0` means this deserves scrutiny.

I would consider:

* binding to localhost by default unless explicitly configured
* requiring a token for `/cache/summary`
* adding allowlist checks if GitLab is local
* logging rejected webhook attempts
* adding rate limiting or at least basic request-size limits for webhooks
* optionally supporting HMAC-style webhook verification if future GitLab integration allows it

This is especially important because the system is an orchestration control plane, not just a metrics service.

---

## P0/P1: Production promotion needs a highly visible hold/kill switch

The production path is intentionally gated, which is good. But I would want an explicit release hold mechanism surfaced everywhere:

```bash
jeryu release hold --reason "investigating canary telemetry"
jeryu release unhold
```

And in the TUI:

```text
PRODUCTION HOLD: active
Reason: investigating canary telemetry
Set by: operator/agent
Since: 2026-04-25 14:03
```

Even if `maybe_trigger_production_promotion` is correct, a human operator needs a clear global brake.

I would also want every “did not promote” outcome recorded with a reason, not just returning `None`.

Examples:

```text
prod_not_triggered: canary_state != e2e-passed
prod_not_triggered: missing C handoff artifact
prod_not_triggered: production pipeline already exists
prod_not_triggered: release hold active
```

Negative decisions are just as important as positive actions.

## P1: The Audit tab being a placeholder is a major missed opportunity

For this system, Audit should be one of the most important tabs.

It should show:

* webhook events
* engine decisions
* agent actions
* retries
* cancellations
* pool changes
* release gate decisions
* secret rotations/finalizations
* cache taint/quarantine events
* admission hook rejections
* disk pressure GC events
* production promotion attempts

The audit view should answer:

> “What happened, who/what caused it, what evidence was used, and what changed?”

Right now the Audit tab says the audit CI pipeline is running. That undersells one of the strongest parts of the architecture: the event ledger.

## P1: Flow graph edges are missing

The Flow Board is visually promising, but `FlowGraph.edges` currently exists without computed edges.

That limits the graph from being a true dependency model. Without edges, the UI can show stages and nodes, but it cannot clearly answer:

* What is blocked by this job?
* What is on the critical path?
* Which downstream release gate is waiting?
* Which child pipeline contributes to prod readiness?
* Why is ETA what it is?

Edges would make the board dramatically more valuable.

Even approximate edges would help:

* stage order edges
* GitLab dependency/needs edges where available
* bridge/downstream edges
* release semantic edges: upstream → release-execution → canary evidence → production-promotion
* blocker edges from failed/pending required jobs to release eligibility

---

## P1: The Flow Board only shows the first active pipeline

The collector prioritizes the release pipeline when available, which makes sense. But in a power console, showing only the first active pipeline is too limiting.

A developer/operator may need to see:

* latest main upstream pipeline
* release-execution pipeline
* production-promotion pipeline
* agent MR pipelines
* scratch `jeryu-test-*` pipelines
* superseded/canceled pipelines
* downstream/child pipelines

I would add a pipeline strip above the graph:

```text
main#530 success | release#531 running | prod not-triggered | agent!17 failed | test#abc pending
```

Then the central graph changes based on selection.

## P1: Release gating should be displayed as a matrix, not just text

The Release Watch pane currently shows useful fields, but a release system this complex needs a gate matrix.

Example:

```text
Release ci-fa51a52a7882
SHA fa51a52a7882 | Ref main | Upstream #530 success

Gate                         Status     Evidence
────────────────────────────────────────────────────────────
Upstream pipeline             PASS       #530
Release eligibility           PASS       pipeline explain
Extended lane                 PASS       no blockers
Release-impacting change      PASS       veox-testctl ci-impact
Canary pipeline               RUNNING    #531
Canary E2E                    WAITING    gate-canary-e2e.json
C handoff artifact            PRESENT    /path/...
C validation artifact         MISSING    /path/...
Production hold               CLEAR      —
Production promotion          BLOCKED    missing validation artifact
```

That would immediately answer the most important release question:

> “What exactly is stopping production?”

## P1: ETA should learn from actual history

Current ETA is lane-based with default durations:

* Unit: 60s
* Integration: 300s
* Security: 120s
* Build: 180s
* ReleaseExecution: 120s
* Other: 90s

That is okay as a placeholder, but the system already stores `ci_job_runs` and test bottlenecks. ETA should eventually use:

* historical duration by job name
* historical duration by runner pool/tag
* queue time by pool
* cache hit/miss state
* current runner capacity
* previous failures/retries
* lane-specific confidence intervals

The UI should distinguish:

```text
ETA 4m20s, confidence medium, based on 17 prior runs
```

from:

```text
ETA 2m00s, heuristic fallback
```

## P1: Runner scheduling needs to be more visible

The system manages runner pools, tags, trust tiers, and scaling. The TUI should make queue pressure obvious.

A dream view should show:

```text
Pool        Trust        Warm  Active  Max  Queue  Oldest wait  Health
default     trusted      2     3       4    1      42s          OK
build       privileged   2     4       4    6      5m12s        SATURATED
untrusted   untrusted    1     1       2    3      2m01s        TAINT-CLEAN
```

And for a selected pending job:

```text
Waiting because:
- requires tags: build,docker-build,dind
- eligible pool: build
- active managers: 4/4
- queue position: 5
- oldest compatible runner heartbeat: 8s ago
Suggested action: scale build pool to 5? blocked by max_managers=4
```

That would be extremely valuable.

## P1: `jeryu sync-now` being a placeholder should be fixed or renamed

A command named `sync-now` that tells the user to ensure `jeryu serve` is running is likely to confuse both humans and agents.

I would either:

* implement an actual one-shot sync, or
* rename it to something like `jeryu nudge`, or
* mark it explicitly as unsupported/deferred with a non-zero status for automation

Agents especially need commands to mean what they say.

---

## P1: Secrets state is underrepresented in the TUI

The CLI can show latest release secret set, Vault status, and secret lifecycle information. The TUI should expose a safe, redacted secrets/release-handoff view.

It should never show raw secrets, but it should show:

* Vault health
* active authority
* latest secret set version
* target
* finalized/not finalized
* handoff files present/missing
* audit events
* recovery bundle status
* whether production promotion is blocked on secret artifacts

For a release-oriented power console, secret readiness is part of deployment readiness.

---

## P1: Agents need operation IDs and durable action tracking

When an agent runs tests, retries jobs, requests merge, or proposes a patch, the system should return an operation ID.

Example:

```json
{
  "success": true,
  "operation_id": "op_20260425_143012_7f2a",
  "kind": "RunTests",
  "project_id": 2,
  "pipeline_id": 531,
  "status_url": "...",
  "audit_event_id": 9912
}
```

Then later:

```json
{
  "intent": "GetOperation",
  "payload": {
    "operation_id": "op_20260425_143012_7f2a"
  }
}
```

Without operation IDs, agents have to infer state by searching GitLab/DB. That works, but it is not ideal for supervised autonomy.

# 3. What an agent would dream of being able to do

An agent does not need unrestricted shell access. It needs **typed, auditable, least-privilege powers**.

I would organize agent capabilities into five classes.

---

## A. Observe

The agent wants to ask:

```text
What is happening?
What is blocked?
What changed?
What failed?
What is safe to do next?
```

Useful capabilities:

```rust
GetSystemSnapshot
GetPipelineGraph
GetReleaseGateMatrix
GetRunnerQueueState
GetFailureCapsule
GetJobTraceTail
GetArtifactSummary
GetCacheVerdict
GetTaintStatus
GetTestPlan
GetAuditEvents
ExplainBlocker
ExplainWhyNotPromoted
ExplainWhyNotCanary
ExplainWhyJobPending
```

The most important one is probably:

```rust
ExplainCurrentBlockers
```

Returning a ranked list like:

```json
[
  {
    "kind": "release_gate",
    "severity": "blocking",
    "summary": "Production promotion blocked by missing C validation artifact",
    "evidence": ["release_attempts.id=42", "/path/gate-c-validation.json"],
    "suggested_actions": ["rerun release validation", "inspect release pipeline #531"]
  },
  {
    "kind": "runner_capacity",
    "severity": "warning",
    "summary": "Build pool saturated; 6 jobs waiting",
    "suggested_actions": ["scale build pool", "cancel superseded pipelines"]
  }
]
```

---

## B. Diagnose

The agent wants to turn evidence into hypotheses.

Capabilities:

```rust
DiagnoseFailedJob
CompareFailureToHistory
ClusterFailures
FindLikelyOwnerModule
SuggestRetryDecision
ExplainRiskGate
InspectSelectorMisses
AnalyzeSlowJob
```

A dream failure diagnosis response would include:

```text
Failure kind: Regression
Likely owner: src/release.rs
First bad commit candidate: fa51a52
Similar failures: 3 in last 14 days
Retry recommendation: DoNotRetry
Suggested validation: release::tests + state::tests::test_production_pipeline_tracking
```

This aligns perfectly with your existing `cargo-witness`, `cargo-vrc`, VTI, and capsule architecture.

---

## C. Act safely

The agent wants to do work without being handed broad credentials.

Capabilities:

```rust
ProposePatch
ApplyPatchToBranch
CreateMergeRequest
RunValidationPlan
RunFocusedTest
RetryFailedJob
CancelSupersededPipeline
RequestMerge
RequestCanaryReconcile
RequestProductionPromotion
RequestPoolScalePlan
ApplyPoolScalePlan
RequestHostReclaimPlan
ApplyHostReclaimPlan
```

The important pattern is **plan first, apply second**.

Example:

```json
{
  "intent": "PlanPoolScale",
  "payload": {
    "pool": "build",
    "target": 5
  }
}
```

Response:

```json
{
  "risk": "medium",
  "allowed": false,
  "reason": "target exceeds max_managers=4",
  "alternatives": [
    "raise max_managers with privileged approval",
    "cancel 3 superseded pipelines",
    "pause noncritical agent pipelines"
  ]
}
```

---

## D. Race hypotheses

This is where the system could become extraordinary.

The docs mention `RacePatches`, but it is deferred. A real implementation would be incredibly powerful.

The agent could submit several hypotheses:

```text
Hypothesis A: fix release artifact path
Hypothesis B: fix state upsert idempotency
Hypothesis C: fix pipeline product classification
```

`jeryu` would:

1. Create isolated branches.
2. Apply each patch.
3. Run VTI-selected validation.
4. Compare outcomes.
5. Keep the winning branch.
6. Close or archive losers.
7. Produce a decision report.

This would turn the system from “agent can try a fix” into “agent can run controlled experiments.”

---

## E. Learn

The agent should be able to improve the system’s own intelligence.

Capabilities:

```rust
RecordSelectorMiss
SuggestTestMapUpdate
SuggestAgentRoutingUpdate
UpdateAgentIndex
GenerateWitnessGraph
DiffWitnessGraph
RecordFailurePattern
SuggestCacheTaintRule
```

For example:

```text
The selector skipped integration-auth, but integration-auth failed.
Suggested .jeryu/testmap.toml update:
- paths: crates/auth/**
- tests: integration-auth
```

The agent should be able to propose this as an MR, not silently mutate production routing.

# 4. Agent access model I would use

I would not give all agents the same powers. I would define explicit capability profiles.

```text
read-only-observer
  Can inspect state, logs, capsules, release gates, audit events.

test-runner
  Can run focused tests and validation plans on ephemeral branches.

patch-author
  Can create branches, commit patches, open MRs.

ci-operator
  Can retry jobs, cancel superseded pipelines, ingest pipeline data.

fleet-operator
  Can propose pool scale/reclaim actions; apply only with approval.

release-operator
  Can reconcile release and request promotion, but gates still enforce.

secret-operator
  Can rotate/finalize release secrets but never read raw secret values.

admin
  Can change policies, trust tiers, pool limits, and security settings.
```

Every capability should be:

* auditable
* rate-limited
* scoped by project/ref
* tied to an agent identity
* denied by default
* explainable when denied

# 7. Critical information that matters most

For a power user, the TUI should prioritize these questions.

## “Are we shipping?”

Show:

* current ref
* current SHA
* release version
* upstream pipeline status
* release-execution pipeline status
* canary state
* E2E gate
* production status
* production blockers
* release hold state
* estimated time to next gate

This should be visible at all times.

---

## “What is blocking us?”

Show a ranked blocker list:

```text
1. production blocked: missing C validation artifact
2. build pool saturated: 6 jobs waiting
3. agent MR !17 failed required lint
4. cache taint active for rust-build namespace
```

Each blocker should have:

* severity
* owner/domain
* evidence
* age
* suggested next action

---

## “What is the critical path?”

Highlight the jobs/gates that determine time-to-green.

Not all running jobs matter equally. The UI should distinguish:

* critical path jobs
* parallel non-blocking jobs
* extended/research jobs
* release-critical jobs
* blocked jobs
* superseded jobs

The TUI should make it impossible to confuse an optional research job with a production-blocking gate.

---

## “What can I safely do?”

For the selected object, show actions with risk labels:

```text
Safe:
  view trace
  explain failure
  ingest pipeline jobs

Medium:
  retry failed job
  cancel superseded pipeline
  scale default pool

High:
  pause privileged pool
  request production promotion
  rotate release secrets
```

Actions should be visible, but gated.

---

## “Why did automation act or not act?”

Every engine decision should be inspectable.

Examples:

```text
Canary not launched because pipeline was not latest successful main.
Production not triggered because canary_state=running, required=e2e-passed.
Job not retried because failure classified as regression.
Pool not scaled because max_managers reached.
```

This is the difference between “automation happened” and “automation is trustworthy.”

# 11. Data architecture for the dream TUI

The current TUI polls several things. That is acceptable for now, but the dream version should have a unified event stream.

I would add an engine-local stream:

```text
GET /events/stream
```

or a Unix socket event stream.

Events:

```json
{
  "type": "job.status_changed",
  "time": "...",
  "project_id": 2,
  "pipeline_id": 530,
  "job_id": 14445,
  "old": "pending",
  "new": "running"
}
```

Other event types:

```text
pipeline.status_changed
release.gate_changed
release.prod_blocked
pool.scaled
runner.oom
cache.taint_added
cache.verdict_recorded
secret.rotated
agent.task_created
risk_gate.evaluated
disk_pressure.changed
```

Then the TUI subscribes once and updates in real time. Polling can remain as fallback.

# 14. The most important “dream agent” workflow

The killer workflow would be:

```text
A pipeline fails.
The agent sees the failure.
It fetches the capsule.
It compares historical failures.
It identifies likely owner modules.
It proposes two or three patch hypotheses.
It races them on isolated branches.
It runs VTI-selected validation.
It opens an MR for the winning patch.
It explains why the other patches lost.
It requests merge.
The risk gate evaluates evidence.
The TUI shows the whole story.
```

That is the dream.

Not “agent runs random shell commands.”

A typed, auditable, evidence-driven repair loop.
