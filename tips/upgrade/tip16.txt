## Overall read

This is already an unusually strong agent-operable delivery system. The best parts are the **typed control surfaces**, the **release/prod separation**, the **SQLite evidence layer**, the **trust-tiered runner pools**, the **failure capsule/retry model**, and the TUI’s current anti-blanking/log-follow behavior. The system has the right philosophy: agents should operate through **audited, typed, policy-mediated controls**, not by grabbing a PAT and shelling around.

The biggest opportunity is to turn `jeryu` from a powerful control plane with a useful dashboard into a **real-time command center**: one place where a developer can see every active pipeline, release gate, agent task, test selector decision, runner bottleneck, cache verdict, security tripwire, and allowed action, with “why” explanations and safe execution previews.

The most important missing abstraction is an **object-action graph**:

> Every object in the system — job, pipeline, MR, pool, runner, release attempt, secret set, cache object, agent task, jeryu sync, gate artifact — should have:
>
> 1. current state,
> 2. freshness/source,
> 3. blockers,
> 4. predicted next state / ETA,
> 5. available actions,
> 6. action risk level,
> 7. required permission / approval,
> 8. audit trail.

That abstraction would make both the TUI and agent API dramatically more powerful.

# 1. Clear strengths

## The architecture is agent-friendly in the right way

The system already avoids the worst pattern: giving agents raw shell or raw GitLab credentials and hoping policy happens elsewhere. `jeryu` centralizes runner state, GitLab access, release promotion, secrets, VTI, failure capsules, and merge gating. That is exactly the right direction.

The strongest pieces are:

| Area                           | Why it is strong                                                                                                                |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------- |
| Release automation             | Green `main` does not mean production. Canary/release-execution and production-promotion are separate GitLab pipeline products. |
| Risk gates                     | Merge is policy-mediated through trust tier, job state, and evidence.                                                           |
| Failure capsules               | Failed jobs become structured evidence rather than just text logs.                                                              |
| VTI/test intelligence          | Agents can run focused validation instead of brute-force everything.                                                            |
| SmartCache/taint/witness model | Cache is treated as a policy system, not just a directory.                                                                      |
| TUI anti-blanking              | Retaining the last meaningful flow snapshot and marking it stale is a good operational UX choice.                               |
| SQLite audit/state layer       | This gives the system memory, replayability, and a basis for explanations.                                                      |

## The release safety model is especially good

The production path being:

```text
main pipeline success
  -> release eligibility
  -> release-execution/canary pipeline
  -> e2e/handoff/validation gates
  -> production-promotion pipeline
```

is the right safety boundary. Production promotion should remain a pipeline product, not an operator shell action.

## The TUI already has the right foundations

The current TUI has three excellent design decisions:

1. **Stable selection across refreshes**
   The selected job survives reorderings.

2. **Log target decoupling**
   The selected job log is tracked independently from general snapshots.

3. **Anti-empty-flow behavior**
   Keeping the last non-empty flow board and marking it stale is much better than flickering to empty state.

Those three behaviors should be preserved in any redesign.

# 2. Important gaps and issues

## A. Capability API is too limited for serious agents

The capability server is conceptually excellent, but currently only two behaviors are active:

| Intent         | Current state |
| -------------- | ------------- |
| `FetchCapsule` | Active        |
| `RunTests`     | Active        |
| `ProposePatch` | Deferred      |
| `RacePatches`  | Deferred      |
| `RequestMerge` | Deferred      |

That means the agent-facing API does not yet match the architecture’s ambition. An agent can run tests and fetch some evidence, but cannot safely complete the full loop:

```text
inspect -> propose patch -> create branch/MR -> run validation -> explain failure -> repair -> request merge
```

The dream version should expose this as a first-class typed workflow.

### Suggested capability additions

| Capability            | Purpose                                                                   |
| --------------------- | ------------------------------------------------------------------------- |
| `ListAllowedActions`  | Given an object, return allowed actions and why/why not.                  |
| `DryRunAction`        | Preview exact side effects, risk, required gates, and audit record.       |
| `ExecuteAction`       | Execute a previously dry-run action with an idempotency token.            |
| `SubscribeEvents`     | Stream job/pipeline/release/cache/security events with cursors.           |
| `ExplainObject`       | Explain why a job/pipeline/release/MR is in its current state.            |
| `FetchArtifacts`      | Retrieve typed artifacts: gate reports, VTI plans, handoff files, traces. |
| `ProposePatch`        | Actually create branch commits through a constrained patch API.           |
| `RequestValidation`   | Run VTI-selected or explicit validation against a branch/MR.              |
| `RequestMerge`        | Route through the risk gate and return allow/deny/escalate evidence.      |
| `QuarantineObject`    | Quarantine runner/cache/artifact/job evidence under controlled policy.    |
| `ControlPool`         | Scale/pause/drain pools with dry-run and policy.                          |
| `ExplainCacheVerdict` | Explain exact cache hit/miss/taint/epoch decisions.                       |

The key principle: **do not give agents broader credentials; give them richer typed capabilities.**

---

## B. `FetchCapsule` should not scan only the latest 500 event rows

The API says `FetchCapsule` scans the latest 500 event log rows looking for a matching `failure_capsule`. But the state API already has:

```text
latest_evidence_for_job
insert_evidence_capsule
evidence_capsules
```

That is a likely correctness gap. If a job is older, or the event ledger is busy, the capsule may exist but be missed.

Better behavior:

```text
FetchCapsule(job_id)
  -> db.latest_evidence_for_job(job_id)
  -> fallback: event ledger scan only if needed
```

This would be more direct, faster, and more reliable.

---

## C. The TUI pane model and visible layout are out of sync

The app still has `Pools`, `Pipelines`, and `Jobs` active panes, and the keybindings still cycle among them. But the active Flow layout visibly shows only the Live Jobs list. Pool and pipeline renderers exist but are inactive.

That creates a mismatch:

```text
User presses Tab -> active pane changes
But visible pane may not clearly exist
Actions like p / d depend on hidden pane state
```

This is probably the single most obvious current TUI UX issue.

Fix options:

1. Restore visible Pools and Pipelines panes.
2. Remove inactive pane states from the active layout.
3. Replace pane focus with an object-centric selection model.
4. Add a command palette that only shows valid actions for the currently visible object.

The dream redesign should move to option 3 or 4.

---

## D. Flow Board only renders the first active pipeline

The Flow Board is the heart of the TUI, but it currently renders only the first active pipeline. The collector prioritizes release pipelines, which is useful, but it may hide concurrent upstream work, blocked branches, agent MRs, or superseded pipelines.

A power-user console should show:

```text
main upstream pipeline
release-execution pipeline
production-promotion pipeline
agent/MR pipelines
scratch jeryu-test-* pipelines
superseded/canceled pipelines
downstream bridges
```

Not all of them need equal visual weight, but they should exist in a navigable pipeline stack.

A strong design would be:

```text
Left: pipeline/ref stack
Center: selected pipeline DAG
Right: selected object inspector/actions
Bottom: event/log console
```

---

## E. Graph edges are not computed

`FlowGraph.edges` exists, but the current docs say graph edges are empty. That means the Flow Board is more of a lane board than a true dependency graph.

For a “developer dream” console, edges matter because they answer:

```text
What is blocking this?
What unblocks next?
What is on the critical path?
Which failure invalidates the release?
Which downstream pipeline belongs to which upstream job?
```

At minimum, edges should model:

| Edge type                    | Example                                       |
| ---------------------------- | --------------------------------------------- |
| stage dependency             | build -> unit tests                           |
| artifact dependency          | build image -> canary                         |
| release gate dependency      | e2e gate -> prod promotion                    |
| bridge/downstream dependency | parent pipeline -> child pipeline             |
| supersedence relationship    | old SHA pipeline -> canceled by new SHA       |
| agent dependency             | agent patch -> validation pipeline -> MR gate |

Edges should have types, not just source/target ids.

---

## F. ETA and progress are too heuristic

The current ETA model is useful as a placeholder, but it is lane-default based:

```text
Unit: 60s
Integration: 300s
Security: 120s
Build: 180s
ReleaseExecution: 120s
Other: 90s
```

A power-user system should use historical data from:

```text
ci_job_runs
test_executions
test_bottlenecks
runner queue durations
job name/stage/ref/image/tags
pool capacity
cache verdicts
```

The TUI should display ETA with confidence:

```text
ETA 4m 20s ± 1m 10s
Confidence: medium
Reason: runner queue is 2.3x normal, cache miss on build image
```

That “why” is more valuable than the raw ETA.

## G. Audit tab is currently a missed opportunity

The Audit tab is a placeholder, but this system has rich audit data:

```text
events
retry_decisions
evidence_capsules
selector_misses
secret_audit_events
release_secret_sets
cache_taints
cache_verdicts
pipeline_superseded
pipeline_cancel_requested
disk_pressure_gc
```

This should become one of the most powerful tabs.

A great Audit tab would answer:

```text
Who/what changed state?
What actions did jeryu take automatically?
Which actions did an agent request?
Which were denied/escalated?
Which release gates passed?
Which caches were tainted?
Which secrets were rotated/finalized?
Which jobs were retried and why?
```

It should be filterable by object:

```text
release
job
pipeline
MR
agent
pool
cache
secret
security
disk
```

## H. Webhook server exposes unauthenticated cache summary

The engine binds to:

```text
0.0.0.0:9777
```

`POST /hooks` is token-authenticated, but `GET /cache/summary` has no auth. That may be acceptable on a private host, but from a security review perspective, unauthenticated operational status on `0.0.0.0` is risky.

Suggested options:

1. Bind non-webhook diagnostics to localhost only.
2. Require a local admin token.
3. Split webhook listener from local diagnostic API.
4. Put `/cache/summary` behind the same auth policy as other operational endpoints.

Even if the data is not secret, it is operational intelligence.

---

## I. Custom executor commands should have invocation guards

The docs correctly say agents should not call:

```text
jeryu exec config
jeryu exec prepare
jeryu exec run
jeryu exec cleanup
```

as generic shell wrappers. But if these commands are callable by any local process, documentation alone is not a hard guardrail.

The custom executor path is security-sensitive because it touches sandboxing, honeypots, cache tainting, witness building, and potentially Docker socket access.

I would add runtime guard checks such as:

```text
required GitLab Runner env vars
expected manager identity
job id/project id consistency
executor invocation nonce
restricted filesystem context
audit event for every prepare/run/cleanup
```

The goal is not to make local root impossible; it is to prevent accidental or low-effort misuse by tools and agents.

---

## J. Release gate artifacts need stronger provenance

The production gate currently requires:

```text
canary_state == e2e-passed
C handoff artifact exists
C validation artifact exists
no existing prod pipeline
```

File existence is a useful minimum, but production safety would be stronger with artifact verification:

| Gate check      | Stronger version                                  |
| --------------- | ------------------------------------------------- |
| File exists     | File exists and parses as expected schema         |
| SHA present     | SHA matches release attempt SHA                   |
| Version present | Version matches release version                   |
| Artifact source | Artifact came from expected pipeline/job          |
| Integrity       | Artifact digest recorded in SQLite                |
| Freshness       | Artifact generated after release attempt creation |
| Signature       | Optional signature/attestation where appropriate  |

The TUI should show not merely “Evidence: path exists,” but:

```text
E2E gate: passed
SHA: matches
Version: matches
Artifact: validated
Generated by: job #12345
Age: 2m ago
Digest: verified
```

---

## K. Hardcoded defaults limit multi-project use

The system is currently strongly centered on:

```text
project_id = 2
ref = main
repo = dougx
release repo root = /home/ubuntu/dougx
```

That is fine for the current deployment, but the architecture is powerful enough to become multi-project. The more `jeryu` grows, the more these should become explicit profiles:

```toml
[profiles.dougx]
project_id = 2
release_ref = "main"
repo_root = "/home/ubuntu/dougx"
vault_prefix = "veox"
default_release = true

[profiles.jeryu]
project_id = 1
release_ref = "main"
repo_root = "/home/ubuntu/JeRyu"
```

Then the TUI can switch profiles, and agents can request capabilities scoped to a profile.

---

## L. Live log polling may become expensive

The TUI polls GitLab trace every 650ms. The local tail is capped at 160 KB, which protects UI memory, but depending on implementation, the GitLab request may still fetch the whole trace repeatedly.

The better version is:

```text
incremental trace fetch
range requests or offset cursor if available
trace digest/version check
adaptive polling
backoff on stale/completed jobs
streaming local event bus when possible
```

A dream TUI should make logs feel instant without hammering GitLab.

---

## M. Stringly typed statuses will eventually hurt

Many states appear to be strings:

```text
running
pending
created
success
failed
canceled
e2e-passed
blocked
waiting
released
```

Strings are flexible, but this system has enough complexity that typed enums would help:

```rust
JobStatus
PipelineStatus
ReleaseCanaryState
ProductionState
GateState
PoolState
CacheVerdict
AgentTaskState
```

The TUI can still render strings, but the domain model should reduce drift and inconsistent names.

# 3. What an agent would dream of being able to do

An agent would not dream of raw shell access. A good agent would dream of **high-bandwidth, low-risk, typed operational access**.

The ideal agent loop would be:

```text
Observe system
  -> identify blocker
  -> explain cause
  -> propose fix
  -> create constrained patch
  -> run selected validation
  -> watch pipeline
  -> fetch evidence
  -> repair if needed
  -> request merge
  -> monitor release
  -> escalate only when policy requires
```

## Dream agent controls

| Agent desire                        | Why it matters                             | Safe implementation                             |
| ----------------------------------- | ------------------------------------------ | ----------------------------------------------- |
| “Tell me what is blocking release.” | Lets agent focus on the actual constraint. | `ExplainReleaseBlockers`                        |
| “Show me allowed next actions.”     | Prevents invalid or unsafe operations.     | `ListAllowedActions(object)`                    |
| “Dry-run this action.”              | Lets agent reason before mutating state.   | `DryRunAction` with side-effect preview         |
| “Create a patch branch.”            | Enables real repair.                       | `ProposePatch` with file allowlist/size limits  |
| “Race three hypotheses.”            | Lets agent try multiple fixes safely.      | `RacePatches` with isolated branches and budget |
| “Run exactly the tests needed.”     | Reduces cost and latency.                  | VTI-backed `RequestValidation`                  |
| “Explain this failure.”             | Avoids blind retrying.                     | Capsule + trace + retry decision                |
| “Why was this cache missed?”        | Debugs performance and determinism.        | `ExplainCacheVerdict`                           |
| “Why is this job pending?”          | Finds runner/tag/capacity issues.          | `ExplainQueueDelay`                             |
| “Can I merge this?”                 | Policy-grounded merge readiness.           | `RequestMerge` through risk gate                |
| “Watch until state changes.”        | Avoids polling awkwardly.                  | `SubscribeEvents`                               |
| “Quarantine this suspicious thing.” | Responds to tripwire/cache compromise.     | `QuarantineObject` with high audit level        |
| “Escalate to human with context.”   | Keeps human approval efficient.            | `CreateApprovalRequest`                         |

## The key: action registry

Every object should expose actions. For example, a failed job might expose:

```json
{
  "object": { "type": "job", "id": 14445 },
  "state": "failed",
  "actions": [
    {
      "name": "retry",
      "allowed": true,
      "risk": "low",
      "reason": "failure classified as transient and no prior retry"
    },
    {
      "name": "fetch_capsule",
      "allowed": true,
      "risk": "read-only"
    },
    {
      "name": "open_trace",
      "allowed": true,
      "risk": "read-only"
    },
    {
      "name": "quarantine_runner",
      "allowed": false,
      "risk": "high",
      "reason": "no tripwire evidence"
    }
  ]
}
```

A release attempt might expose:

```json
{
  "object": { "type": "release_attempt", "version": "ci-fa51a52a7882" },
  "actions": [
    {
      "name": "reconcile",
      "allowed": true,
      "risk": "low"
    },
    {
      "name": "promote_prod",
      "allowed": false,
      "risk": "critical",
      "reason": "C validation artifact missing"
    },
    {
      "name": "show_missing_gates",
      "allowed": true,
      "risk": "read-only"
    }
  ]
}
```

This would transform the system. Agents and the TUI could share the same action model.

# 4. Issues I would prioritize first

## Priority 1: Fix the visible TUI/action mismatch

Current problem:

```text
Pool/Pipeline pane states exist,
but active Flow layout mostly exposes Jobs.
```

Fix:

```text
Either show Pools and Pipelines again,
or move to object/action selection.
```

This matters because invisible state controlling destructive actions is dangerous.

---

## Priority 2: Build a canonical object-action API

This would serve:

```text
TUI
agents
CLI
capability server
audit
future web UI
```

Start small:

```rust
ObjectRef {
    kind: Job | Pipeline | Pool | Release | SecretSet | CacheObject | AgentTask,
    id: String,
}

ActionDescriptor {
    name,
    label,
    allowed,
    reason,
    risk,
    requires_confirmation,
    dry_run_available,
}
```

Then implement:

```text
jeryu action list <object>
jeryu action dry-run <object> <action>
jeryu action run <object> <action>
```

The TUI can consume the same registry.

---

## Priority 3: Make Audit real

The Audit tab should become an event ledger browser with filters, object links, and action provenance.

This is especially important because agents, auto-retries, production promotion, secret rotation, cache tainting, and disk GC all need explainability.

---

## Priority 4: Improve release gate verification

Make production gate checks stronger than “artifact exists.”

Add:

```text
schema validation
SHA/version matching
artifact digest
source job/pipeline id
created_at freshness
SQLite gate verification record
```

Then render those checks in Release Watch.

---

## Priority 5: Multi-pipeline Flow Board

Render more than the first active pipeline.

At minimum, add a left pipeline stack:

```text
main      #530  running   62%  ETA 4m
release   #531  running   gate-canary-e2e
prod      --    waiting   blocked: validation
agent/a   #532  failed    regression
```

The center graph can still show one selected pipeline at a time.

---

## Priority 6: Historical ETA and queue diagnosis

Use existing data to answer:

```text
Why is this slow?
Is this queue wait normal?
Which runner pool is saturated?
Which job is on the critical path?
What changed from the previous successful run?
```

This would make the console feel intelligent rather than decorative.

# 5. Dream TUI redesign

## Design goal

The dream TUI should answer these questions instantly:

1. **Can we ship?**
2. **If not, what exactly blocks us?**
3. **What changed recently?**
4. **What is running right now?**
5. **What is slow?**
6. **What failed, why, and what should happen next?**
7. **What can I safely do from here?**
8. **What are agents doing?**
9. **Are caches, runners, secrets, Vault, GitLab, and disk healthy?**
10. **What will happen automatically if I do nothing?**

The current TUI is a dashboard. The dream version is a **mission control console**.

# 6. The most critical information to show

## A. Global health header

The top line should be a dense, high-signal health strip:

```text
GitLab ●
DB ●
Vault ●
Webhook ●
Runners ●
Cache ●
Mirror ●
Disk 71%
Jeryu ▲
Agents 3
Release ⏳
Prod —
```

Each chip should have:

| State            | Rendering              |
| ---------------- | ---------------------- |
| Healthy          | green dot              |
| Running / active | blue dot               |
| Waiting          | yellow dot             |
| Warning          | orange/yellow triangle |
| Failed           | red x                  |
| Blocked/security | magenta lock/alert     |
| Stale            | gray dim text          |

The header should also show source freshness:

```text
GitLab ● 2s
DB ● 0s
Flow ● 1s
Logs ● live
Release ● 3s
```

Freshness matters because the system merges multiple async data sources.

---

## B. Release status should dominate the top of the UI

For this system, the most important operational question is:

```text
Can this SHA reach production?
```

The release panel should show:

```text
Release ci-fa51a52
SHA fa51a52a7882
Upstream pipeline: success
Release-execution: running
Canary: e2e-running
Production: not triggered
Blocking gate: C validation artifact missing
Next automatic action: none until e2e gate passes
Manual action available: reconcile
Prod promote: denied — missing validation artifact
```

I would show release as a horizontal gate chain:

```text
Upstream ✔ → Eligible ✔ → Canary Deploy ● → E2E ○ → Handoff ✔ → Validation ⚠ → Prod ○
```

With each gate selectable.

## C. Pipeline and job critical path

The Flow Board should always identify:

```text
critical path job
current blocker
longest queue wait
most likely next completion
jobs eligible for retry
jobs requiring human attention
```

Example:

```text
Critical path: integration-e2e-live
Blocking release: yes
ETA: 5m ± 2m
Reason: historical p95 7m, runner queue normal, cache cold
```

## D. Runner pool pressure

The user needs to know whether jobs are slow because of code or capacity.

Show:

```text
default     3/4 managers   2 running   1 pending   queue p95 18s
build       4/4 managers   4 running   3 pending   saturated
untrusted   1/2 managers   1 running   0 pending   ok
```

A dream TUI should make saturation visually obvious.

---

## E. Failure triage

For every failed job, show:

```text
failure kind
classification
retry recommendation
prior retries
capsule availability
trace tail
suspected owner/module
changed files related to failure
VTI selector involvement
```

Example:

```text
FAIL test-release-gates
Classification: Regression
Retry: Do not retry
Likely owner: release.rs
Evidence: capsule #884
Changed paths: src/release.rs, src/state.rs
Suggested validation: release::tests
```

## F. Agent activity

The current architecture has agent spawn/list/merge, but the TUI does not appear to make agents a first-class operational view.

A dream TUI should show:

```text
Agent Tasks

  #18 race cache-miss hypotheses        3 branches   1 green
  #19 docs-only routing update          ready for merge, risk allow
```

For each agent task:

```text
goal
branch
MR
risk tier
current action
last evidence
validation status
merge gate result
human approval needed?
```

## G. Cache and taint explanations

Cache status should not just say hit/miss counts. It should answer:

```text
Why was this build cold?
Was the cache skipped because of trust tier?
Was there a taint?
Was the epoch invalidated?
Was the witness different?
What would make this cacheable next time?
```

A selected job inspector could show:

```text
Cache decision: cold execution
Reason: rust toolchain fingerprint changed
Epoch: rust-1.92.0
Witness: changed Cargo.lock
Taint: none
Promotion: allowed after success
```

## H. Security and supply-chain signals

Given the modules involved, the TUI should make these impossible to miss:

```text
honeypot breach
cache taint
detonation lane breach
untrusted runner escalation
admission hook denial
secret rotation/finalization
Vault health
production gate denial
custom executor quarantine
```

Security events should use a distinct visual language, probably magenta/purple plus lock/shield icons, not the same red/yellow used for normal CI failures.

Do not rely on color alone. Pair colors with icons and labels.

## Status icon language

Use compact icons with ASCII fallbacks:

| State      | Icon       | Fallback |
| ---------- | ---------- | -------- |
| success    | `✔`        | `OK`     |
| running    | `●`        | `RUN`    |
| pending    | `○`        | `PEND`   |
| failed     | `✖`        | `FAIL`   |
| canceled   | `⊘`        | `CANC`   |
| blocked    | `◆` or `⚠` | `BLOCK`  |
| security   | `🔒`       | `SEC`    |
| release    | `◆`        | `REL`    |
| production | `⬢`        | `PROD`   |
| stale      | `◌` dim    | `STALE`  |

## High-contrast pane rules

| UI element       | Visual rule                         |
| ---------------- | ----------------------------------- |
| Active pane      | cyan border, bold title             |
| Failed object    | red border/title                    |
| Security object  | magenta border/title                |
| Release object   | violet accent                       |
| Waiting object   | amber accent                        |
| Stale data       | dim gray, explicit age              |
| Selected row     | cyan background, black text         |
| Dangerous action | red/magenta label with confirmation |
| Disabled action  | dim gray with reason                |

## Avoid visual noise

The TUI should be colorful, but not rainbow-random. Color should encode meaning consistently:

```text
green = good/done
blue = active/running
cyan = selected/focus
yellow = waiting
orange = warning/capacity
red = failure
magenta = security/blocking
violet = release/prod
gray = stale/disabled
```

# 22. The dream north star

The dream version of `jeryu tui` is not merely a terminal dashboard. It is a **developer flight deck**.

It should let a power user look at one screen and know:

```text
We are on main@fa51a52.
The upstream pipeline passed.
The release pipeline is running.
Canary deploy succeeded.
E2E is the current critical path.
Production is blocked by C validation.
Runner capacity is healthy.
Cache is cold because the Rust epoch changed.
No security taints are active.
One agent is repairing a failed release test.
There are two safe actions and one denied action.
If nothing changes, the next automatic action will happen after E2E passes.
```

That is the bar.

The current system has enough primitives to get there. The main missing pieces are not raw power; the system already has power. The missing pieces are:

```text
unified object/action model
stronger explanations
real audit UI
multi-pipeline graph
historical ETA
agent cockpit
release gate matrix
streaming event model
```

Build those, and this becomes an exceptional agent-operable developer console.
