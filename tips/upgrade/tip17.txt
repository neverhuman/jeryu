## Overall read

JeRyu / `jeryu` is already much more interesting than a CI helper. It is a **local delivery control plane** with typed state, runner orchestration, release gates, failure evidence, Vault-backed secret lifecycle, VTI test selection, cache policy, and an operational TUI.

The biggest gap is not raw functionality. The biggest gap is that the system’s **power is spread across many surfaces**, while the agent and TUI surfaces still expose only a fraction of the operational intelligence that already exists internally.

The north-star improvement is:

> Turn `jeryu` into a real-time, typed, auditable **mission-control console** where humans and agents can see every pipeline, blocker, gate, action, ETA, risk, cache decision, test decision, release state, and remediation option in one place.

Right now, the internals are closer to that dream than the interface is.

---

# 1. The strongest parts of the current design

Several design choices are excellent and worth preserving.

## Release safety is properly mediated

The production path is not an ad hoc command. It is mediated through GitLab pipeline products:

```text
green main
  -> release-execution pipeline
  -> canary/E2E/handoff gates
  -> production-promotion pipeline
```

That is exactly the right shape. Agents should never get a raw “deploy prod” shell button. They should get a typed “request promotion” capability that returns a gate report and only acts when the release model says it is safe.

## RedlineDB as durable coordination state is the right move

The system has a durable state plane instead of relying only on live GitLab state. That enables:

* release attempt tracking
* retry history
* failure evidence
* job timing history
* VTI test plans
* cache verdicts
* secret audit events
* jeryu sync status
* TUI continuity under partial data

This is a very strong architectural foundation.

## The anti-blanking TUI behavior is smart

Keeping the last meaningful Flow Board visible and marking it stale is a very good operational UX choice. Production systems often have momentary empty states, stale API reads, or racey refresh windows. A console that blinks to empty destroys trust.

The current TUI correctly favors:

```text
stable but stale > fresh but misleadingly empty
```

That principle should be extended to every panel.

## The execution security model is conceptually strong

The combination of:

* trust-tiered runner pools
* custom executor for untrusted work
* honeypot tripwires
* taint/quarantine tracking
* cache witnesses
* BuildKit namespace isolation
* risk-gated merges

is the right overall shape for an agent-operable CI system.

## The VTI/test intelligence layer is a major differentiator

Smart test selection, external `.jeryu/testmap.toml`, selector audits, learning suggestions, test bottlenecks, and cache keys are exactly the kind of information agents need to avoid brute-force validation.

The current TUI does not yet make this feel central, but the underlying idea is very strong.

# 2. Biggest architectural gaps

## 2.1 The Capability API is too small for the system’s ambitions

The documented capability API currently has five intents:

```text
FetchCapsule
RunTests
ProposePatch
RacePatches
RequestMerge
```

But only `FetchCapsule` and `RunTests` have meaningful active behavior. The others deserialize and return success while being deferred.

That is dangerous for agents.

A capability that says “success” while not actually performing the named action can lead an agent to believe a patch, merge request, or merge request decision happened when it did not. Deferred actions should not look like successful actions.

Better response shape:

```json
{
  "success": false,
  "state": "deferred_unimplemented",
  "message": "RequestMerge is recognized but not active in this build.",
  "data": {
    "suggested_surface": "jeryu agent merge <project_id> <mr_iid>"
  }
}
```

Or more generally:

```text
completed
started
planned
blocked
denied
escalated
deferred_unimplemented
```

The agent-facing surface should distinguish these states very clearly.

## 2.2 Agents need a Plan → Act → Observe → Explain loop

Right now, agents can call commands, but the control plane does not yet expose a unified lifecycle for agent actions.

An ideal agent action should have:

```text
intent
target
dry-run plan
required permissions
expected side effects
risk class
approval requirements
execution id
event stream
final result
rollback/remediation hints
```

Example:

```text
Agent asks:
  "Run the minimal safe validation for this patch."

jeryu returns:
  - changed modules
  - selected tests
  - skipped tests
  - risk class
  - runner tags
  - estimated duration
  - command/pipeline plan
  - whether cache may skip anything

Agent confirms or policy auto-allows.

jeryu executes, streams progress, records evidence, and returns:
  - pass/fail
  - failure capsules
  - retry advice
  - next recommended action
```

That is the dream: not just “run tests,” but “run the right validation under policy and explain the result.”

## 2.3 The TUI has state/actions that are not visibly represented

The current active Flow layout visibly shows:

* Release Watch
* Flow Board
* Live Jobs
* Log Preview

But the `App` state and keybindings still include active panes for:

```text
Pools
Pipelines
Jobs
```

Only Jobs is visibly represented as a list in the active layout. Pools and Pipelines still affect actions and focus state, but they are not clearly shown.

That is a UX and safety issue.

For example, `p` toggles a selected pool, but the pool list is not visible in the active Flow layout. Even if the action only works when the internal active pane is Pools, the user cannot comfortably reason about what is selected.

This should be fixed one of two ways:

1. Reintroduce visible Pools and Pipelines panels.
2. Remove or disable invisible pane actions from the active layout.

Invisible operational focus is dangerous in a power-user console.

## 2.4 The Audit tab being a placeholder undercuts the system’s core promise

The architecture has a rich append-only event ledger, secret audit events, retry decisions, evidence capsules, release attempts, selector misses, and cache verdicts.

But the TUI Audit tab currently says:

```text
Audit CI Pipeline running...
Check Dashboard/Pipelines to see live progress.
```

That is much weaker than the system underneath it.

The Audit tab should become one of the most important screens. It should answer:

```text
What happened?
Who/what triggered it?
What policy allowed it?
What changed?
What evidence was recorded?
What did the system retry, cancel, quarantine, promote, or deny?
```

This is especially important if agents will operate the system.

## 2.5 Release visibility is good, but not yet decision-grade

The Release Watch pane exposes the latest release attempt and production status. That is useful, but a power user needs to know:

```text
Can production promotion happen?
If not, exactly why not?
Which gate is missing?
Which artifact is absent?
Which job produced or should have produced it?
Which command/action would fix or re-check it?
```

The production gate is currently described in the API as requiring:

* release attempt exists
* `canary_state == "e2e-passed"`
* C handoff artifact exists
* C validation artifact exists
* no existing production-promotion pipeline for same SHA/ref

The TUI should render that as a gate matrix.

Example:

```text
PRODUCTION GATES
✓ release attempt exists             ci-fa51a52a7882
✓ canary state                       e2e-passed
✓ C handoff artifact                 /path/to/handoff.json
✗ C validation artifact              missing
✓ no existing prod pipeline          none found

Result: BLOCKED
Next: rerun release validation job or inspect release-execution pipeline #1234
```

That would make the release pane dramatically more useful.

## 2.6 Flow Board only renders the first active pipeline

The Flow Collector can hold multiple active pipelines, but the current board renders the first active pipeline.

That is a limitation for real operations. A power user wants to see:

* latest main pipeline
* current release-execution pipeline
* production-promotion pipeline
* agent scratch test pipelines
* blocked pipelines
* superseded pipelines
* downstream child pipelines

At minimum, the TUI needs a pipeline selector or multi-pipeline swimlane view.

## 2.7 Graph edges are not computed

The Flow Graph has an `edges` field, but edges are currently empty.

That means the board is more of a columnar job map than a true dependency graph. For a dream console, edges matter because they reveal:

* critical path
* current blocker
* downstream impact
* jobs waiting on a failed dependency
* release gates depending on artifacts
* child pipeline relationships
* production promotion dependencies

Without edges, the visual graph cannot fully answer “why are we stuck?”

## 2.8 ETA is useful but too heuristic

The current ETA model uses lane defaults:

```text
Unit: 60s
Integration: 300s
Security: 120s
Build: 180s
ReleaseExecution: 120s
Other: 90s
```

That is fine for a first version, but the system already stores historical job timing in `ci_job_runs` and test bottlenecks. ETAs should eventually be based on:

* job name
* stage
* runner pool
* image
* recent median/p90 duration
* queue time
* active runner capacity
* cache hit/miss likelihood
* whether job is release-critical
* whether similar jobs are already running

The TUI should also show confidence:

```text
ETA 4m12s · p90 6m40s · confidence medium · queue risk high
```

## 2.9 `FetchCapsule` scans recent events instead of using the evidence table

The Capability API `FetchCapsule` scans the latest 500 event log rows for `failure_capsule`.

But the state API already has:

```text
evidence_capsules
latest_evidence_for_job
list_evidence_for_ref
```

The capability path should use the evidence table directly. A latest-500 event scan can miss older jobs, noisy periods, or high-volume event bursts.

This is a correctness and reliability issue.

## 2.10 Logs need redaction and structure

The TUI has syntax-highlighted logs and live polling, which is good.

But logs can leak sensitive values. Given that this system handles Vault, release handoff files, deployment variables, GitLab tokens, and secret rotations, log rendering should include a redaction layer.

The log viewer should detect and mask:

* GitLab tokens
* Vault tokens
* PAT-like strings
* secret env var values
* URLs with embedded credentials
* private keys
* obvious bearer tokens
* known release handoff secret paths if they include sensitive material

A console this powerful should assume logs are hostile until sanitized.

# 3. What an agent would dream of being able to do

I would not give an agent broad shell access, raw GitLab PATs, or direct Vault secret access. The dream is **more power through narrower typed capabilities**.

An agent would want these capabilities.

## 3.1 Ask for the current operational truth

The agent should be able to ask:

```text
What is currently blocking main?
What is currently blocking production?
Which pipelines are superseded?
Which jobs are on the critical path?
Which jobs are flaky versus true regressions?
Which runner pool is saturated?
Which release gate is missing?
Which cache taints affect this build?
Which tests were skipped, and why?
```

This should be available as structured JSON, not just text.

## 3.2 Run validation by intent, not by hand-written command

Instead of only:

```bash
jeryu test run --command "cargo test ..."
```

An agent wants:

```text
Validate this patch with the minimum safe proof lane.
Validate this release candidate.
Validate only affected subsystems.
Validate security-sensitive changes.
Validate after cache taint.
Validate this MR according to trust tier.
```

That would combine:

* `cargo-vrc plan`
* VTI selection
* witness diff
* risk tier
* changed paths
* affected modules
* historical failures
* required release gates

## 3.3 Get a structured failure diagnosis

For a failed job, an agent wants one call that returns:

```text
classification
failure kind
log excerpt
likely owning module
retry recommendation
whether this is known flaky
whether a retry already happened
related cache verdict
related taint state
related test history
suggested next action
```

The system already has most pieces. They need to be composed into a first-class “diagnose failure” capability.

## 3.4 Propose and update patches safely

The deferred `ProposePatch` capability should eventually become real.

Ideal behavior:

```text
Agent submits patch plan
  -> jeryu creates branch/worktree
  -> applies patch
  -> runs formatting/static checks
  -> creates commit
  -> opens or updates MR
  -> attaches validation plan
  -> returns MR, branch, changed files, risk gate preview
```

It should support dry-run mode:

```text
This patch would touch:
  - src/release.rs
  - src/state.rs
  - docs/API.md

Required proof lane:
  - cargo check -p jeryu
  - release::tests
  - state::tests::test_production_pipeline_tracking
  - repo audit-agent-surface

Risk:
  high, because release/state surfaces changed
```

## 3.5 Race hypotheses

`RacePatches` is a very agent-native concept. If implemented, it could be powerful.

An agent could submit several alternative fixes:

```text
Hypothesis A: fix retry race in engine
Hypothesis B: fix evidence capsule timing
Hypothesis C: fix state query freshness
```

Then `jeryu` could:

* create isolated branches
* run selected validation for each
* compare results
* rank by correctness evidence
* open the best MR
* archive the losing branches with notes

This is exactly the kind of control surface that would make agents feel superhuman without giving them unsafe direct access.

## 3.6 Ask for safe remediation options

For every problem, the agent wants to know:

```text
What can I safely do next?
```

Examples:

For failed job:

```text
Available actions:
  - retry job, allowed, confidence high, reason transient infra capsule
  - fetch full trace, allowed
  - quarantine cache key, requires elevated cache permission
  - open regression issue, allowed
```

For blocked release:

```text
Available actions:
  - rerun release validation job
  - reconcile release state
  - inspect missing artifact
  - request production promotion, blocked because C validation artifact missing
```

For disk pressure:

```text
Available actions:
  - host reclaim plan, allowed
  - cache GC dry-run, allowed
  - cache GC apply, requires approval
  - evict active manager caches, emergency only
```

This should become an “action registry” with policy metadata.

## 3.7 Control runner capacity under policy

Agents would love to ask:

```text
Scale capacity for this blocked pipeline.
Why is this job pending?
Which runner tags are missing?
Which pool should handle this job?
Can I safely drain this pool?
```

But this should be policy-bound:

* read-only diagnosis for most agents
* scale within budget for trusted agents
* drain/pause/delete requiring stronger permission
* privileged pools requiring explicit approval

## 3.8 Operate release without bypassing release gates

An agent should be able to do:

```text
Explain release eligibility.
Reconcile release attempts.
Show production gate failure.
Request canary launch.
Request production promotion.
```

But never:

```text
Run arbitrary deployment shell script.
Read raw production secrets.
Bypass canary/E2E/handoff gates.
```

The current architecture already points in this direction.

## 3.9 Work with secrets without seeing secrets

Agents should be able to:

```text
Rotate release secrets.
Finalize release secret set.
Regenerate handoff report.
Verify secret authority health.
Check whether required secret artifacts exist.
```

But should not receive raw secret values unless there is a very specific, audited break-glass path.

## 3.10 Subscribe to real-time state

A dream agent does not poll commands repeatedly. It subscribes.

Ideal capability:

```json
{
  "intent": "Subscribe",
  "payload": {
    "subjects": [
      "project:2/ref:main",
      "release:latest",
      "pipeline:123",
      "job:*",
      "cache:taints",
      "host:disk"
    ]
  }
}
```

The server streams events:

```text
job.started
job.failed
failure_capsule.inserted
retry_decision.inserted
release.gate_changed
pipeline.superseded
pool.scaled
cache.taint_added
disk_pressure.warning
```

That would make agents far more effective and reduce wasteful polling.

# 4. Clear issues I would address first

Here is the practical priority list.

## Priority 0: correctness and safety

| Issue                                                         | Why it matters                                        | Suggested fix                                                                         |
| ------------------------------------------------------------- | ----------------------------------------------------- | ------------------------------------------------------------------------------------- |
| Deferred capability intents return success                    | Agents may believe an action happened when it did not | Return `deferred_unimplemented` or `success:false`                                    |
| `FetchCapsule` scans latest 500 events                        | Can miss evidence                                     | Use `latest_evidence_for_job` / `evidence_capsules`                                   |
| Invisible TUI panes still have actions                        | Confusing and potentially unsafe                      | Make Pools/Pipelines visible or disable actions                                       |
| Logs may expose secrets                                       | High risk in a Vault/release system                   | Add redaction before display/storage/export                                           |
| Destructive/mutating TUI actions are too lightweight          | Power users can misfire                               | Add confirmation/action preview for pause, delete, cancel, prod-affecting actions     |
| Capability API lacks explicit auth model beyond socket access | Local socket is not enough for multi-agent trust      | Add peer credential checks, socket permissions, capability manifests, audit actor IDs |

## Priority 1: operator usefulness

| Issue                                           | Why it matters                             | Suggested fix                                           |
| ----------------------------------------------- | ------------------------------------------ | ------------------------------------------------------- |
| Audit tab is placeholder                        | Wastes one of the most valuable surfaces   | Render event ledger, filters, actor/action timeline     |
| Release pane lacks gate matrix                  | User cannot see why prod is blocked        | Show each production/canary gate with pass/fail reason  |
| Pipeline doctor/explain not integrated into TUI | Existing intelligence is hidden            | Add inspector panel or hotkeys for explain/doctor       |
| Flow Board only shows first pipeline            | Misses concurrent release/prod/agent flows | Add pipeline selector or swimlanes                      |
| No graph edges                                  | Hard to understand blockers                | Compute dependency/artifact/gate/downstream edges       |
| ETA lacks confidence/source                     | Can mislead users                          | Show confidence, p50/p90, queue component, stale marker |

## Priority 2: power-user and agent experience

| Issue                               | Why it matters                            | Suggested fix                                            |
| ----------------------------------- | ----------------------------------------- | -------------------------------------------------------- |
| No unified action console           | User must remember commands               | Add command palette and contextual actions               |
| VTI decisions are not central in UI | Test intelligence is a key differentiator | Show selected/skipped tests, risk, cache keys, misses    |
| Cache decisions not tied to jobs    | Cache behavior is hard to debug           | Link cache verdicts/taints/witnesses to job nodes        |
| No incident board                   | Failures are spread across panels         | Create active blockers/incidents view                    |
| No time-travel replay               | Hard to debug automation                  | Event timeline with correlation IDs                      |
| Polling logs only                   | Fine now, but not ideal                   | Add incremental trace fetch or streaming transport later |

# 5. The dream TUI redesign

I would redesign the TUI around a single idea:

> Every screen should answer: **What is happening, what is blocked, why, what can I safely do, and what will happen next?**

The current TUI is a dashboard. The dream version is a **command console**.

## 5.1 Overall layout

A high-density terminal layout could look like this:

```text
┌ JeRyu GOD MODE ─ GitLab ONLINE ─ main@fa51a52 ─ Release ci-fa51a52 ─ Prod BLOCKED ─ Disk 71% ─ Cache 84% ─ Taints 0 ┐
│ / command palette                                                                                                      │
├───────────────────────┬──────────────────────────────────────────────────────────────┬────────────────────────────────┤
│ NAV / INCIDENTS        │ DELIVERY FLOW GRAPH                                           │ SELECTED NODE INSPECTOR        │
│                       │                                                              │                                │
│ ! Prod blocked         │ Commit → Admission → Impact → Build → Tests → Security       │ Job: test-rust-nextest-1        │
│ ! build pool saturated │                         ↓                                    │ Status: running                 │
│ ✓ GitLab online        │ Package → Canary → E2E Gates → Handoff → Production          │ ETA: 2m14s p90 4m20s            │
│ ✓ Vault healthy        │                                                              │ Runner: build-2                 │
│ ! disk warning         │ [nodes, edges, critical path, live animation]                 │ Required: yes                   │
│                       │                                                              │ Actions:                       │
│ 1 Flow                 │                                                              │  [l] logs                       │
│ 2 Release              │                                                              │  [e] explain                    │
│ 3 Actions              │                                                              │  [r] retry if failed            │
│ 4 Agents               │                                                              │  [x] cancel                     │
│ 5 Tests                │                                                              │                                │
│ 6 Cache/Security       │                                                              │                                │
│ 7 Runners              │                                                              │                                │
│ 8 Audit                │                                                              │                                │
├───────────────────────┴──────────────────────────────────────────────────────────────┴────────────────────────────────┤
│ LIVE EVENT STREAM: job.started → cache.hit → test.failed → capsule.inserted → retry_decision.escalate                  │
└────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┘
```

This should feel like a mix of:

* CI dashboard
* release command center
* incident console
* terminal-native IDE
* agent supervision cockpit

## 5.2 The most important information to show

The console should prioritize these facts above everything else.

### 1. Current blockers

Always show:

```text
What is blocking forward progress right now?
```

Examples:

```text
Production blocked: missing C validation artifact
Pipeline blocked: failed release-critical job test-integration-live
Queue blocked: no available build,dind runner
Release blocked: extended lane failed
Agent MR blocked: pending security job
Cache blocked: tainted key requires cold execution
```

### 2. Release state

Always show:

```text
current ref
current SHA
version
upstream pipeline
release-execution pipeline
canary state
E2E gate
handoff artifacts
production pipeline
public canary URL
promotion eligibility
```

This is the heart of the system.

### 3. Critical path and ETA

Show not just “pipeline 43%,” but:

```text
Critical path:
  build-image -> integration-e2e -> canary-gate -> production-promotion

ETA:
  release ready in ~11m
  confidence medium
  risk: integration-e2e historically p90 18m
```

### 4. Job health

For each important job:

```text
status
required/optional
lane
runner/pool
queue time
elapsed time
ETA
flake history
failure classification
retry decision
capsule availability
logs
artifacts
```

### 5. Available actions

Every selected object should show contextual actions:

For a failed job:

```text
Retry
Explain
Fetch capsule
Open trace
Quarantine cache
Create issue
```

For a release attempt:

```text
Explain gates
Reconcile
Open canary evidence
Request production promotion
Regenerate release report
```

For a runner pool:

```text
Scale
Pause
Resume
Drain
Rotate token
Show logs
```

For disk/cache:

```text
GC dry-run
GC apply
Host reclaim plan
Host reclaim apply
Show taints
```

Every action should include risk color, required permission, and dry-run preview.

### 6. Data freshness

Each panel should show freshness:

```text
GitLab 1.2s ago
DB 0.4s ago
Docker 3.5s ago
Vault 10s ago
Flow stale 14s
Logs live
```

Power users need to know whether they are looking at fresh truth or retained stale truth.

### 7. Security and trust state

Always show:

```text
trust tier
taints
honeypot breaches
cold-execution downgrades
untrusted work
privileged runners
secrets authority health
release secret set state
```

Security should not be buried.

# 6. Dream screen set

## Screen 1: Flow Command Center

Purpose:

```text
See all active delivery flows and act on blockers.
```

Content:

* multi-pipeline graph
* release pipeline prioritized
* production-promotion pipeline visible
* agent test pipelines optionally collapsed
* critical path highlighted
* selected node inspector
* live log preview
* contextual actions

The graph should show columns like:

```text
Admission | Impact | Build | Unit | Integration | Security | Package | Canary | Gates | Production
```

But with actual edges.

Nodes should include compact metadata:

```text
● running  test-integration-live  3m/8m  p90 12m
✕ failed   canary-e2e             capsule ready
○ pending  prod-promote           blocked by handoff
✓ success  build-image            artifact ready
```

## Screen 2: Release Gates

Purpose:

```text
Answer: can we ship, and if not, why?
```

Layout:

```text
Release ci-fa51a52 on main@fa51a52

UPSTREAM
✓ pipeline #1002 success
✓ release-critical lane green
✓ extended lane green
✓ release-impacting change confirmed

CANARY
✓ release-execution pipeline #1005 success
✓ canary deployed
✓ public URL healthy
✓ E2E passed
✓ telemetry gate passed

HANDOFF
✓ C handoff artifact exists
✗ C validation artifact missing

PRODUCTION
✗ not eligible
○ production pipeline not triggered

Available:
  [r] reconcile
  [e] explain pipeline
  [h] open handoff artifact
  [v] rerun validation
  [p] request production promotion -- currently blocked
```

This screen should make production state impossible to misunderstand.

## Screen 3: Action Console

Purpose:

```text
Show all possible actions, their risk, and their current eligibility.
```

Actions grouped by risk:

```text
SAFE / READ-ONLY
  explain pipeline
  fetch job trace
  show capsule
  show release gates
  run test plan dry-run

LOW-RISK MUTATING
  retry failed transient job
  run selected tests
  reconcile release
  update jeryu remote

MEDIUM-RISK
  scale pool
  cancel superseded pipeline
  cache GC apply
  rotate runner token

HIGH-RISK
  drain pool
  host reclaim apply
  rotate release secrets
  trigger production promotion

BLOCKED
  production promotion: missing validation artifact
  merge MR: pending security job
```

Every action should have:

```text
risk
target
actor
required permission
dry-run summary
side effects
rollback hint
audit event id
```

## Screen 4: Agent Workbench

Purpose:

```text
Supervise autonomous agents.
```

Show:

* active agent tasks
* branches
* issues
* MRs
* current hypothesis
* validation status
* risk gate result
* changed files
* owning modules
* selected proof lane
* comments/actions taken
* next recommended action

Example:

```text
Agent Task #42: Fix release gate artifact detection

  ✓ cargo check
```


# 7. Color and visual language

The TUI should be high contrast, but not color-only. Use both color and symbols.

## Semantic color palette

| Meaning             | Color                  | Use                                         |
| ------------------- | ---------------------- | ------------------------------------------- |
| Healthy/success     | Bright green           | passed jobs, satisfied gates                |
| Running/live        | Bright blue or cyan    | active jobs, live logs, online services     |
| Waiting/pending     | Yellow                 | queued jobs, pending gates                  |
| Degraded/risky      | Orange                 | disk pressure, stale data, queue saturation |
| Failure             | Bright red             | failed jobs, denied gates                   |
| Security/escalation | Magenta                | taints, honeypot, untrusted escalation      |
| Release/prod        | Purple/violet          | release identity, promotion path            |
| Stale/disabled      | Gray                   | stale snapshots, canceled jobs              |
| Selected focus      | Cyan border/background | current pane/node                           |

## Icons

Use redundant symbols:

```text
✓ success
● running
○ pending
✕ failed
⚠ warning
◆ gate
⬢ release
🔒 secret/security
↻ retry
⏱ ETA
⛔ blocked
```

Even in plain terminal environments, users should understand state without relying only on color.

## Visual intensity

Use visual hierarchy:

* bright + bold for live critical information
* dim gray for stale or secondary context
* red/magenta only for things that need attention
* yellow/orange for “not broken, but watch this”
* green for completed/safe
* cyan for currently selected or actively streaming

Avoid making everything colorful. The dream console should be color-rich, but semantically disciplined.

Press `w` or `?` on any selected object.

```text
Likely affected:
```

The current schema is already rich, but I would add or formalize a few things.

## 9.1 Action ledger

A table for all human/agent/TUI actions:

```text
actions
  id
  actor
  actor_type: human | agent | engine | tui
  action_type
  target_type
  target_id
  risk_level
  dry_run
  status
  requested_at
  started_at
  finished_at
  policy_decision
  side_effect_summary
  result_summary
  correlation_id
```

This makes every operational action visible and auditable.

## 9.2 Gate evaluations

Instead of only deriving gate state ad hoc:

```text
gate_evaluations
  release_attempt_id
  gate_name
  status
  reason
  artifact_path
  evaluated_at
  source
```

Then the TUI can render gate history, not just current gate state.

## 9.3 Incidents/blockers

A normalized blocker model:

```text
incidents
  id
  kind
  severity
  subject_type
  subject_id
  status
  summary
  first_seen_at
  last_seen_at
  resolved_at
  suggested_action
```

Examples:

```text
release_blocked_missing_artifact
runner_pool_saturated
gitlab_unavailable
vault_unhealthy
cache_taint_active
disk_pressure_warning
pipeline_failed_required_job
```

This would make the TUI’s left-side incident list easy and powerful.

## 9.4 Capability leases

For agent governance:

```text
capability_leases
  id
  agent_id
  scope
  permissions
  expires_at
  budget
  trust_tier
  created_by
```

Agents should receive narrow, expiring capabilities, not ambient power.

## 9.5 Typed status enums

Several important statuses appear to be strings:

```text
canary_state
pipeline status
release status
manager state
```

Some of that is unavoidable because GitLab returns strings, but internal statuses should be typed as much as possible. The TUI and agents should not have to pattern-match arbitrary strings for safety-critical state.

# 10. TUI action model

The dream TUI should have a unified action registry.

Each action should know:

```text
label
hotkey
target
risk
allowed/blocked/escalated
reason
dry-run renderer
execute function
audit event type
confirmation requirement
```

Example:

```text
Action: Retry job
Target: job #14445
Risk: low
Allowed when:
  - job.status == failed
  - retry_decision != DoNotRetry
  - retry count below policy limit

Blocked when:
  - failure classification is Regression
  - retry already attempted
```

Another example:

```text
Action: Promote production
Risk: high
Allowed when:
  - canary_state == e2e-passed
  - handoff artifact exists
  - validation artifact exists
  - no existing production pipeline

Confirmation:
  type release version
```

This would make the TUI safe even while giving it enormous power.

# 11. Concrete improvements to the current TUI

If I were improving the current implementation without a total rewrite, I would do this sequence.

## Step 1: Make the active panes honest

Either render Pools/Pipelines again or remove their focus states from the active Flow screen.

A simple immediate fix:

```text
Flow screen bottom area:
  left: Live Jobs
  middle: Active Pipelines
  right: Log Preview
```

Then `Tab` between visible panes makes sense again.

## Step 2: Turn Audit into an event timeline

Use existing `events`, `retry_decisions`, `secret_audit_events`, and release attempt updates.

Even a simple chronological list would be a huge upgrade.

## Step 3: Add a Release Gates pane

Replace the current compact Release Watch body with a gate matrix. It should show:

```text
upstream eligibility
extended lane state
release-impact state
release-execution status
canary status
E2E gate
handoff artifact
validation artifact
production pipeline status
```

## Step 4: Add selected-object inspector

The right pane should adapt to the selection:

* selected job: status, logs, capsule, retry advice
* selected pipeline: explain/doctor summary, blockers
* selected release: gates, artifacts, actions
* selected pool: managers, queue, actions
* selected cache item: verdicts, taints, witness info

This would avoid cramming everything into the main graph.

## Step 5: Add command palette

A power user should not memorize every hotkey.

Press `/` or `Ctrl-P`:

```text
retry selected failed job
explain selected pipeline
run release reconcile
scale build pool
open selected logs
show failure capsule
run host reclaim plan
rotate release secrets
```

Searchable actions would make the TUI feel dramatically more powerful.

## Step 6: Add action preview and confirmation

Before mutating actions:

```text
You are about to pause pool build.

Impact:
  - GitLab runner #12 will be paused
  - pending build jobs may wait
  - local DB pool paused=true

Confirm? y/N
```

For high-risk actions, require typed confirmation.

## Step 7: Wire in existing intelligence

Bring these existing CLI concepts into the TUI:

* `pipeline explain`
* `pipeline doctor`
* `job explain`
* `test select`
* `test failed`
* `test bottlenecks`
* `release status`
* `release reconcile`
* `cache doctor`
* `host doctor`
* `repo audit-agent-surface`

The TUI should not duplicate all logic. It should call the same domain functions and render their structured reports.

# 12. What I would avoid

## Do not make production deployment a shell command

Keep production promotion pipeline-mediated.

The TUI can have a “request promotion” action, but it should only trigger the existing production-promotion path and show gates.

## Do not give agents raw secret access

Agents should operate secret lifecycle through typed actions:

```text
rotate
finalize
report
recover instructions
verify artifact exists
```

They should not receive raw Vault secret values as a normal capability.

## Do not trust color alone

Use icons, labels, and text. Some users have limited color perception, and terminal themes vary.

## Do not hide stale data

Retaining stale data is good. Failing to label it clearly would be bad.

Every retained panel should expose freshness.

## Do not let “success” mean “accepted but not done”

This is especially important for the capability API. Agents need precise execution semantics.

# 13. The single biggest conceptual upgrade

The best version of this system would introduce a first-class concept of an **operational object**:

```text

Gate
Agent Task

```

Each object should support:

```text

```

Then the TUI becomes an object browser plus command console. The capability API becomes an object action API. The audit log becomes a timeline of object state transitions.

That would unify the whole system.

# 14. My strongest recommendation

The internal architecture is strong enough that I would not start by adding more commands. I would first build a **unified action and explanation layer**.

Something like:

```rust
struct ActionDescriptor {
    id: String,
    label: String,
    target: ActionTarget,
    risk: RiskLevel,
    availability: ActionAvailability,
    dry_run: ActionPreview,
}
```

Then expose it through:

```text
TUI action palette
Capability API
CLI explain/action commands
Audit ledger
```

That one layer would make both humans and agents much more capable.

The current system already knows a lot. The next leap is making it answer, everywhere:

```text
What is happening?
Why?
What can be done?
Is it safe?
What evidence proves it?
What happens if I do it?
```

That is what would make `jeryu tui` a developer dream: not just a dashboard, but a real-time, high-trust, high-power command center for the entire delivery system.
