Absolutely. I read the system as a very capable local GitLab/Rust delivery control plane whose biggest weakness is not lack of primitives, but **lack of one unified, real-time operational brain** for humans and agents. The CLI/API surface is rich; the TUI currently exposes only a fraction of that richness.

Below is my best feedback, broken into: major gaps, agent “dream access,” issues to address, and a dream TUI redesign.

# Executive Take

`jeryu` is already much more than a CLI wrapper. It has the foundations of a serious agent-operable CI/CD control plane:

* Typed release gates.
* Smart test selection.
* Runner pool orchestration.
* Failure capsules.
* Cache taint/quarantine.
* Vault-backed release secrets.
* Webhook reconciliation.
* Risk-gated merges.
* A TUI with live jobs and logs.

The gap is that the system’s intelligence is spread across many surfaces:

```text
CLI has explain/doctor/progress/test intelligence.
Engine has live reconciliation and release behavior.
DB has evidence and events.
TUI shows jobs/logs/release/cache, but not the full decision model.
Capability API gives agents only a tiny active subset.
```

The dream version is not “a prettier dashboard.” It is a **mission-control command console** where every pipeline, release gate, failure, test decision, runner, cache decision, secret handoff, and agent action is visible, explainable, and actionable in real time.

The key design principle should be:

> Every action should answer three questions:
> **What is happening? Why is it happening? What can I safely do next?**

# 1. Biggest Architectural Gaps

## 1.1 The Capability API is far too weak for serious agents

Right now, the capability API has only two meaningful active behaviors:

* `FetchCapsule`
* `RunTests`

The most agent-useful intents are currently deferred:

* `ProposePatch`
* `RacePatches`
* `RequestMerge`

That means agents cannot yet use the socket as the primary safe operating surface. They still need to fall back to CLI commands, GitLab operations, or indirect workflows.

For a system that wants to be agent-operable, this is probably the single largest gap.

### What agents would want

Agents would dream of a capability API that can do this:

```text
Inspect state
  -> understand blockers
  -> propose patch
  -> create branch
  -> run selected tests
  -> compare hypotheses
  -> open/update MR
  -> request risk-gated merge
  -> monitor release
  -> explain failures
  -> continue repair loop
```

The capability API should become the canonical least-privilege agent surface.

Recommended new capability intents:

```rust
ExplainPipeline { project_id, pipeline_id }
DoctorPipeline { project_id, pipeline_id }
ProgressReport { project_id, ref_name }
ReleaseStatus { project_id, ref_name, sha }
TestSelect { repo_root, base, head }
RunFocusedTest { project_id, ref_name, command, timeout, tags }
FetchJobTrace { project_id, job_id, max_bytes }
FetchFailureCapsule { project_id, job_id }
ProposePatch { branch_name, files, commit_message }
ApplyPatch { branch_name, patch }
OpenMergeRequest { source_branch, target_branch, title, body }
RequestRiskGate { project_id, mr_iid, trust_tier }
RaceHypotheses { base_branch, hypotheses, validation_plan }
CancelPipeline { project_id, pipeline_id }
RetryJob { project_id, job_id }
ListAgentTasks { project_id }
EventStreamSubscribe { cursor, filters }
```

The dream API would not be a generic shell escape. It would be a typed, auditable, policy-aware command layer.

## 1.3 The active TUI pane model is confusing

The TUI still has active pane states for:

```text
Pools
Pipelines
Jobs
```

But in the active Flow layout, only the Jobs list is visibly represented. The old pool and pipeline renderers exist but are not wired into the active layout.

This creates UX hazards:

* `Tab` cycles through panes that are not all visible.
* `p` can pause/resume a selected pool, but there is no visible active pool pane in the current layout.
* Pipeline focus affects border color, but there is no standalone active pipeline list.
* `d` can delete selected pipeline/job records depending on active pane, but the user may not have a visible reminder of the active target.

This is an important issue. Hidden focus state plus destructive commands is dangerous.

### Fix

Either restore visible panes for Pools and Pipelines, or remove those active pane states from the current Flow layout.

A better version:

```text
Left rail:
  Flow
  Pipelines
  Jobs
  Runners
  Release
  Tests
  Cache
  Audit

Main area changes based on explicit selected mode.
No invisible active pane actions.
```

## 1.9 Log viewing is useful but incomplete

Missing features:

* Search within logs.
* Filter error/warning/panic lines.
* Jump to first failure.
* Preserve both head and tail of long logs.

# 2. Clear Issues That Need Addressing

## Issue 1: `/cache/summary` is unauthenticated and bound on `0.0.0.0`

The webhook server binds to:

```text
0.0.0.0:9777
```

`POST /hooks` is protected by `X-Gitlab-Token`, but `GET /cache/summary` has no auth.

That may be acceptable in a fully local trusted environment, but it is still a state leak. Cache counts, health, and object metadata can reveal operational information.

Recommendation:

* Bind non-webhook diagnostic routes to localhost only, or
* Require a token for `/cache/summary`, or
* Split webhook and diagnostics servers.

---

## Issue 2: `FetchCapsule` scans latest 500 event rows

The capability API scans recent events to find a `failure_capsule`.

That can miss capsules in a busy system.

Since there is already an `evidence_capsules` table and a `latest_evidence_for_job` method, `FetchCapsule` should use the indexed state API instead of scanning recent event log rows.

Also consider accepting:

```json
{
  "project_id": 2,
  "pipeline_id": 123,
  "job_id": 456
}
```

rather than only `job_id`.

---

## Issue 3: Capability payload limit is too small for patch/racing workflows

The Unix socket request is capped by a 64 KiB read buffer.

That is not enough for serious patch proposals, multi-file diffs, or hypothesis races.

Better options:

* Allow artifact/file references.
* Allow chunked requests.
* Allow git branch references.
* Allow patch bundles stored under a controlled workspace directory.
* Add request IDs and asynchronous polling/subscription.

---

## Issue 4: Deferred capability intents may create false confidence

The API accepts intents like `ProposePatch`, `RacePatches`, and `RequestMerge`, but returns success despite routing being deferred.

That is risky because an agent may believe the named operation happened.

Recommendation:

Return something explicit:

```json
{
  "success": false,
  "message": "Intent is recognized but not implemented",
  "data": {
    "implemented": false,
    "recommended_surface": "jeryu agent spawn"
  }
}
```

Or use:

```json
{
  "success": true,
  "message": "Deferred: no mutation performed"
}
```

But “success” should not imply action.

---

## Issue 5: `jeryu sync-now` is a placeholder

The CLI has:

```bash
jeryu jeryu sync-now --source <dir>
```

But it only tells the user to ensure `jeryu serve` is running.

That is confusing because the command name implies an immediate action.

Recommendation:

Either implement it or rename it to something like:

```bash
jeryu jeryu request-sync
```

with explicit behavior.

---

## Issue 6: Local TUI deletion can erase useful operational evidence

The TUI allows deleting local job/pipeline records from RedlineDB.

That can be useful for cleanup, but dangerous for debugging. A user might accidentally remove the evidence needed to explain a failure.

Better:

* Soft-delete records.
* Add undo.
* Move to “hidden” or “archived.”
* Require confirmation for deleting pipeline records.
* Record audit events for TUI deletions.

---

## Issue 7: Production gate reasons are not explicit enough

The system has strong production controls, but the UI should never make operators infer why production has not started.

For every release attempt, show a blocker list:

```text
Production blocked:
  ✗ canary_state is "running"; needs "e2e-passed"
  ✗ C handoff artifact missing
  ✓ no existing production pipeline
  ✓ release execution pipeline succeeded
```

This should appear in CLI, TUI, and capability responses.

---

## Issue 8: ETA is currently too heuristic

The docs say ETA is lane-based with fixed default durations.

That is fine initially, but the system already stores:

* `ci_job_runs`
* queued duration
* runtime duration
* test bottlenecks
* cache verdicts
* pool information

ETA should eventually use:

```text
historical job duration
+ queue estimate by pool
+ current runner capacity
+ cache hit probability
+ critical path
+ release gate history
```

The TUI should label estimates honestly:

```text
ETA: ~7m20s, medium confidence
Basis: last 12 runs, queue + runtime
```

---

## Issue 9: Audit, agent, release, and test intelligence are disconnected

Each subsystem is good individually. The dream system needs correlation.

A single pipeline should be able to answer:

```text
Push event
  -> impact decision
  -> VTI plan
  -> selected jobs
  -> actual jobs
  -> cache verdicts
  -> failures
  -> retry decisions
  -> release eligibility
  -> canary
  -> production
```

That requires consistent correlation IDs across:

* events
* tracked pipelines
* release attempts
* test plans
* evidence capsules
* cache verdicts
* secret sets
* agent tasks

This may already be partly possible through project/ref/sha/pipeline/job IDs, but the TUI would benefit from a first-class causal model.

# 3. What an Agent Would Dream To Be Able To Do

An agent’s dream is not “more shell access.” It is **more typed authority with better evidence**.

## 3.1 Dream capability: ask “what should I do next?”

The agent should be able to ask:

```json
{
  "intent": "RecommendNextAction",
  "payload": {
    "project_id": 2,
    "ref_name": "main"
  }
}
```

And get:

```json
{
  "state": "release-blocked",
  "blockers": [
    {
      "kind": "failed-job",
      "job_id": 123,
      "classification": "Regression",
      "retry_advice": "DoNotRetry"
    }
  ],
  "recommended_actions": [
    {
      "action": "FetchCapsule",
      "reason": "Need structured failure evidence"
    },
    {
      "action": "SpawnRepairTask",
      "reason": "Regression requires code change"
    }
  ]
}
```

This would let agents operate from system state, not guesswork.

---

## 3.2 Dream capability: repair loop

An agent should be able to do:

```text
Fetch failed job
Fetch capsule
Fetch relevant logs
Diagnose likely source
Propose patch
Run selected tests
Open MR
Request risk gate
Monitor result
```

As a typed transaction:

```json
{
  "intent": "RepairFailure",
  "payload": {
    "project_id": 2,
    "job_id": 123,
    "mode": "propose_only"
  }
}
```

The system could return a plan first:

```text
Plan:
  1. Create branch repair/job-123
  2. Apply patch to src/release.rs
  3. Run release::tests
  4. Run pipeline explain
  5. Open MR
Requires approval: yes
```

## 3.3 Dream capability: race hypotheses

This system already has the conceptual hook for `RacePatches`, which is exactly what agents want.

A dream implementation:

```json
{
  "intent": "RacePatches",
  "payload": {
    "base_branch": "main",
    "hypotheses": [
      {
        "name": "fix-release-gate-condition",
        "patch": "..."
      },
      {
        "name": "fix-state-tracking",
        "patch": "..."
      }
    ],
    "validation": {
      "tests": [
        "cargo test -p jeryu -- release::tests -- --nocapture"
      ]
    }
  }
}
```

Result:

```text
Hypothesis A:
  compile: pass
  tests: pass
  risk: low
  duration: 4m12s

Hypothesis B:
  compile: pass
  tests: fail
  failure: state::tests::...
  risk: medium

Winner:
  Hypothesis A
```

This would be extremely powerful.

## 3.4 Dream capability: semantic state queries

Agents should be able to ask:

```text
Why is production blocked?
What changed since last green main?
Which tests were skipped and why?
Which failures are probably infrastructure?
Which runner pool is saturated?
Which cache entries are tainted?
Which agent MRs are mergeable?
Which release attempt owns this pipeline?
```

These should not require raw SQL. They should be first-class capability intents or CLI/TUI query surfaces.

## 3.5 Dream capability: safe production/release stewardship

Agents should never deploy through shell commands, but they could steward the release pipeline safely.

Agent-accessible actions:

```text
ReleaseStatus
ReleaseExplainBlockers
ReleaseReconcile
CanaryStatus
PromoteProdWhenGatesPass
SecretStatus
SecretRotate
SecretFinalize
```

But always with explicit gate evidence.

An agent should be able to say:

```text
Production promotion cannot be triggered.

  canary_state is waiting
```

Or:

```text
Production promotion is safe to request.
```

## 3.6 Dream capability: event subscription

Agents should not have to poll blindly.

Add a capability/event stream:

```json
{
  "intent": "SubscribeEvents",
  "payload": {
    "cursor": 12345,
    "filters": {
      "project_id": 2,
      "ref_name": "main",

```

The system could use:

* Unix socket stream.
* Local event queue.
* Cursor-based polling.
* Server-sent events for local clients.

This would allow supervised agents to react quickly without broad access.


Every command should show a preview before dangerous mutation:

This would make the TUI a true command console, not just a viewer.

# 5. Critical Information That Matters Most

If the TUI can only prioritize a few things, these are the most important.

## 5.1 Production and release safety

The top priority:

```text
Can this SHA safely go to production?
If not, exactly why not?
```

Show:

* Current release attempt.
* Gate pass/fail matrix.
* Canary state.
* E2E/telemetry artifacts.
* Secret handoff status.
* Production pipeline status.
* Blockers.

---

## 5.2 Active blockers

A power user needs immediate blocker visibility:

```text
What is blocking progress right now?
```

Could be:

* Failed release-critical job.
* Pending job with no runner.
* Missing artifact.
* Failed gate.
* Secret set not finalized.
* Cache taint.
* Disk pressure.
* Untrusted MR escalation.
* Superseded pipeline.
* GitLab offline.

Blockers should be first-class objects.

---

## 5.3 Critical path and ETA

The user needs:

```text
What is the critical path?
What is the estimated time remaining?
How confident is that estimate?
```

ETA should include:

* Queue time.
* Runtime history.
* Runner capacity.
* Cache state.
* Release gates.

---

## 5.4 Failure classification and next action

For each failure:

```text
Is this infra, transient, regression, or unknown?
Should I retry, quarantine, escalate, or patch?
```

This should be visible without running a separate CLI command.

---

## 5.5 Test selection rationale

For smart CI to be trusted, the UI must show:

```text
Why did jeryu run these tests?
Why did it skip others?
What risk did it accept?
```

---

## 5.6 Runner capacity

For developer happiness:

```text
Am I waiting on compute or on code?
```

Show queue times, warm managers, saturated pools, and stuck jobs.

---

## 5.7 Audit and causality

For trust:

```text
Who did what, when, and why?
```

Especially for agents and production actions.

# 6. Color-Rich, High-Contrast Design

A brilliant TUI should use color as semantic structure, not decoration.

## Suggested palette

```text
Green      success, safe, deployed
Cyan       selected, active focus, interactive action
Blue       running, in progress
Yellow     pending, waiting
Orange     degraded, warning, stale-but-usable
Red        failed, blocked, dangerous
Magenta    security, secrets, taint, quarantine
Gray       inactive, unknown, historical
White      primary text
Black      high-contrast selected text background
```

## Status badges

Use badges everywhere:

```text
[PASS]
[RUN]
[WAIT]
[BLOCKED]
[FAILED]
[STALE]
[TAINT]
[READY]
[LOCKED]
[ESCALATE]
```

## Icons

Use icons, but always with text fallback:

```text
✓ PASS
● RUN
○ WAIT
✗ FAIL
! WARN
🔒 LOCKED
⚑ CRITICAL
```

Do not rely on color alone.

## Critical path styling

Critical path should be visually distinct:

```text
double border
bright white/cyan outline
CRIT badge
ETA shown
```

## Stale data styling

Do not blank stale panels. The current anti-blanking behavior is excellent.

But make stale state obvious:

```text
FLOW BOARD [STALE 42s]
```

Use dim/orange/gray styling.

## Production styling

Production should have a special visual language:

```text
Prod locked      gray + lock
Prod ready       green + READY
Prod running     blue + RUN
Prod blocked     red + BLOCKED
Prod deployed    green + DEPLOYED
```

Production actions should never look casual.

# 9. The Single Most Important Product Idea

The best version of this system should not just show data.

It should constantly produce a short operational sentence:

```text
main@fa51a52 is blocked from production because integration-e2e failed as a regression.
No retry is recommended. Suggested next action: spawn repair agent or inspect capsule.
```

Or:

```text
main@fa51a52 is ready for production promotion.
All release, canary, handoff, validation, and secret gates are satisfied.
```

Or:

```text
Pipeline #530 is healthy but delayed.
Critical path is integration-e2e. Estimated remaining time: 6m20s.
Runner pool untrusted is saturated.
```

That sentence should appear at the top of the TUI, in CLI `jeryu progress`, and in the capability API.

That is the “developer dream” experience: not just seeing the system, but understanding it instantly.

# Final Recommendation

The architecture is already strong. I would not radically change the core control plane yet. I would focus on **unifying visibility, action, and explanation**.

The highest-value improvements are:

1. Expand the TUI into a real command center.
2. Surface blocker reasons everywhere.
3. Wire `pipeline explain`, `pipeline doctor`, `job explain`, VTI plans, release gates, and secret state into the TUI.
4. Make the Audit tab real.
5. Implement the deferred capability intents.
6. Add safe action previews and confirmations.
7. Build a causal graph across events, jobs, tests, cache, release, secrets, and agents.
8. Make agents operate through typed capabilities, not broad shell access.

The system already has the bones of an incredible agent-native CI/CD platform. The dream version is a high-contrast, real-time, explainable mission-control console where every object has evidence, every blocker has a reason, and every action is gated, previewed, and auditable.
