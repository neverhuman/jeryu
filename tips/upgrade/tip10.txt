Based on the API, architecture, and TUI docs, `jeryu` already has the bones of a very serious agent-operable delivery control plane. The strongest design choice is that production, merge, cache, secrets, and executor behavior are mediated through typed surfaces, GitLab pipelines, state records, risk gates, and evidence—not loose shell commands. That is exactly the right direction.

The biggest opportunity is to make the whole system feel less like a collection of powerful commands and more like a **live operational command center**: one place where a human or agent can see what is happening, why it is happening, what is blocked, what is risky, and what safe action is available next.

Below is my best feedback.

---

# 1. Overall assessment

## What is already excellent

The system has several unusually strong foundations:

**The control surfaces are well documented.**
The API reference is clear about CLI, hidden commands, GitLab webhooks, capability socket, release orchestration, custom executor commands, state API, secrets, cache, and validation lanes. This matters because agents need exact surfaces, not vibes.

**The architecture correctly separates planes.**
Interface, control, state, execution, and intelligence planes are distinct. That gives you room to extend without turning `dispatch.rs` or the TUI into a god module.

**State is treated as an operational ledger.**
The SQLite model is not just configuration storage. It records jobs, pipelines, release attempts, cache events, test plans, selector misses, evidence capsules, retries, secrets, and audits. That is the right substrate for explainability.

**Release automation is gated instead of ad hoc.**
The green-main → release-execution → canary/E2E/handoff → production-promotion pipeline model is much safer than “run deploy shell script from TUI.”

**The TUI already solves an important real-world problem: unstable partial data.**
The anti-blanking behavior for flow snapshots is a very good choice. Keeping the last meaningful board visible and marking it stale is much better than making the operator watch the UI blink between useful and empty.

**The agent model is pointed in the right direction.**
Risk-gated merge, typed capability intents, failure capsules, test selection, and VTI are exactly the kind of primitives an agent needs.

The current system is not merely “GitLab with a dashboard.” It is close to being a **local autonomous CI/CD control tower**.

# 2. Biggest gaps for agents

If I were designing access for a high-performing coding/release agent, I would not give it raw shell power first. I would give it richer typed capabilities with strong visibility, dry runs, and audit trails.

## 2.1 The capability API is too small relative to the CLI

The CLI surface is huge and powerful, but the capability API currently only has a few intents:

```text
FetchCapsule
RunTests
ProposePatch      deferred
RacePatches       deferred
RequestMerge      deferred
```

That means the most agent-friendly surface is not yet the most capable surface.

An agent would dream of a capability API that exposes typed versions of the important CLI actions:

```text
ObserveSystem
ObservePipeline
ObserveRelease
ExplainBlocker
ExplainJob
FetchCapsule
PlanTests
RunTests
RetryJob
CancelPipeline
ScalePool
RunImpactAnalysis
RunVtiSelection
ProposePatch
RacePatches
OpenMergeRequest
RequestMerge
DryRunReleasePromotion
TriggerReleaseReconcile
RotateSecretsPlan
CacheGcPlan
HostReclaimPlan
```

Each mutating action should support:

```text
mode = "plan" | "apply"
risk = "read-only" | "low" | "destructive" | "production"
requires_confirmation = true/false
audit_event_id = ...
```

That would let agents reason before acting.

## 2.2 Deferred intents should not return generic success

This is a significant agent-footgun.

The docs say `ProposePatch`, `RacePatches`, and `RequestMerge` deserialize and return success, but active routing is deferred. For a human, that is mildly confusing. For an agent, it is dangerous because the agent may believe work happened.

Better response shape:

```json
{
  "success": false,
  "status": "not_implemented",
  "message": "RequestMerge is currently deferred. Use jeryu agent merge instead.",
  "data": {
    "recommended_surface": "jeryu agent merge <project_id> <mr_iid>"
  }
}
```

Or:

```json
{
  "success": true,
  "status": "accepted_noop_deferred",
  "message": "Intent recognized but no action was performed."
}
```

But plain “success” for a no-op should be avoided.

## 2.3 Agents need context packets, not scattered calls

The current surfaces expose many useful facts, but an agent often needs a packaged view.

For example, for a failed job, an agent wants one object containing:

```text
job metadata
pipeline metadata
runner/pool/trust tier
stage/lane classification
latest trace tail
failure capsule
retry decision
prior retries
cache verdicts
selected/skipped tests
related commits
recent similar failures
recommended next actions
```

A dream endpoint would be:

```bash
jeryu explain job <project_id> <job_id> --json
```

Or capability intent:

```json
{
  "intent": "ExplainJob",
  "payload": {
    "project_id": 2,
    "job_id": 14445,
    "include_trace_tail": true,
    "include_cache": true,
    "include_history": true,
    "include_actions": true
  }
}
```

The response should include both facts and possible actions:

```json
{
  "classification": "Infrastructure",
  "retry_advice": "RetryOnce",
  "blockers": ["runner unavailable", "cache taint active"],
  "safe_actions": [
    {
      "action": "RetryJob",
      "risk": "low",
      "reason": "First infrastructure-classified failure"
    }
  ],
  "unsafe_actions": [
    {
      "action": "RequestMerge",
      "risk": "blocked",
      "reason": "Pipeline has failed release-critical job"
    }
  ]
}
```

This would be incredibly useful for autonomous operation.

## 2.4 Agents need real-time subscriptions

The current capability API is one request, one response. That is fine for small actions, but agents need event streams.

Ideal additions:

```text
jeryu capability stream /tmp/jeryu-events.sock
jeryu events watch --json
GET /events/stream
```

Events should include:

```text
job_started
job_failed
job_retried
pipeline_created
pipeline_superseded
pipeline_blocked
release_claimed
canary_started
canary_e2e_passed
prod_promotion_triggered
pool_scaled
runner_oom
disk_pressure_gc_started
cache_taint_detected
secret_rotated
admission_rejected
agent_mr_ready
risk_gate_decision
```

Agents should be able to subscribe to specific scopes:

```json
{
  "project_id": 2,
  "ref": "main",
  "event_types": ["job_failed", "release_blocked", "risk_gate_decision"]
}
```

Right now, a supervising agent would need to poll a lot.

## 2.5 Agents need action manifests

A very powerful improvement would be for `jeryu` to expose its full action catalog:

```bash
jeryu actions list --json
jeryu actions describe job.retry --json
jeryu actions plan job.retry --payload ... --json
jeryu actions apply job.retry --payload ... --json
```

Every action should advertise:

```text
name
category
description
required inputs
side effects
risk level
auth requirements
whether dry-run exists
whether confirmation is required
related CLI command
related state tables
related audit events
```

This would let both the TUI and agents render a complete command console without hardcoding every action.

# 3. Clear issues and risks to address

## 3.1 TUI active pane model is currently misleading

The docs say the active pane can be:

```text
Pools
Pipelines
Jobs
```

But in the current Flow layout, only Jobs has a visible list. Pools and Pipelines still exist as state/actions, but their renderers are inactive.

That creates confusing behavior:

```text
Tab changes focus to panes the user cannot really see.
p pauses/resumes selected pool even though the pool list is not visible.
d deletes selected pipeline/job depending on hidden active pane state.
```

This should be fixed quickly.

Best options:

**Option A: restore visible Pools and Pipelines panes.**
Show them as collapsible side panels or alternate subpanes in Flow.

**Option B: remove Pools/Pipelines from active pane cycling in the current layout.**
Only expose their actions in dedicated tabs or a command palette.

**Option C: add a visible focus strip.**
For example:

```text
Focus: Jobs | Hidden panes: Pools, Pipelines available via Command Palette
```

Right now, hidden focus plus destructive actions is a UX risk.

## 3.2 Delete actions need confirmation or undo

The TUI allows `d` / `Delete` to delete selected local pipeline/job records from SQLite.

Even if this only deletes local records, it can still damage observability. It should not be a one-key irreversible action.

Better behavior:

```text
Press d once: mark pending delete, show confirmation.
Press d again or Enter: delete.
Press Esc: cancel.
```

Or provide an undo stack:

```text
Deleted local job record #12345. Press u to undo.
```

At minimum, destructive actions should show exactly what will be deleted.

## 3.3 Audit tab being a placeholder weakens trust

The system has a strong audit/event model, but the TUI Audit tab does not yet expose it. That is a mismatch.

The Audit tab should become one of the most important views.

It should show:

```text
event ledger
risk gate decisions
retry decisions
release attempts
secret audit events
admission rejects
cache taints
honeypot/tripwire events
host reclaim events
agent actions
webhook failures
```

This would make the system feel trustworthy and explainable.

## 3.4 Flow graph edges are not computed

The Flow Board is visually promising, but the graph is currently more of a columnar status board than a dependency graph.

Because `FlowGraph.edges` exists but is empty, the board cannot yet answer:

```text
What is blocking what?
Which job is on the critical path?
Which downstream pipeline depends on this bridge?
What happens if this job fails?
Which release gate is waiting for which artifact?
```

A dream TUI needs real edges.

Edges could come from:

```text
GitLab stages
needs:
child pipeline bridges
release attempt relationships
artifact dependencies
VTI plan dependencies
manual gate dependencies
production promotion prerequisites
```

## 3.5 Flow Board only renders the first active pipeline

This will break down for real power users.

A serious release flow may have:

```text
upstream main pipeline
release-execution pipeline
production-promotion pipeline
child pipelines
agent test pipelines
scratch jeryu-test pipelines
superseded pipelines
```

The TUI should support multiple active pipeline flows, with clear grouping:

```text
Main
Release
Production
Agent/Scratch
Superseded
Blocked
```

The first active pipeline is not always the most important pipeline.

## 3.6 ETA is too heuristic for a power-user console

The docs are honest that ETA is lane-based and simple. That is acceptable for an early dashboard, but for a “developer dream” TUI, ETA should use actual historical data.

You already have useful sources:

```text
ci_job_runs
test_executions
test_bottlenecks_avg
test_bottlenecks_latest
runner queue duration
job runtime duration
cache hit/miss data
pool capacity
```

A stronger ETA model would separate:

```text
queue ETA
execution ETA
critical path ETA
release ETA
production ETA
confidence
```

Example display:

```text
Pipeline ETA: 11m 40s ± 3m
Critical path: build-image → integration-e2e → canary-gate
Queue risk: high, untrusted pool saturated
Cache impact: cold execution likely, cache taint active
```

## 3.7 Storage tab displays approximate values without enough labeling

The docs say some Docker storage values are placeholders or approximations. The UI should make that explicit.

For example:

```text
Docker images: estimating...
Docker build cache: approximate
Root disk: exact from df
CAS: exact recursive scan
```

Operators need to know which numbers are reliable.

## 3.8 `/cache/summary` is unauthenticated on a server bound to `0.0.0.0`

The webhook server binds to:

```text
0.0.0.0:9777
```

Routes:

```text
GET /health          no auth
GET /cache/summary   no auth
POST /hooks          token auth
```

A public unauthenticated health endpoint may be fine. But unauthenticated cache summary might expose operational metadata. At minimum, this should be intentional and documented as safe. Better would be one of:

```text
bind only to localhost by default
require token for /cache/summary
serve /cache/summary only on an internal listener
```

## 3.9 Capability socket needs explicit permission model

The capability API is a major agent boundary. The docs do not mention Unix socket permissions, credential checks, or per-intent authorization.

This should be explicit.

For example:

```text
socket mode 0600 by default
optional group-readable mode for supervised agents
peer credential validation where supported
per-intent allowlist
audit event for every mutating capability call
payload schema version
max payload documented and enforced with error response
```

Without this, the capability socket could become a local privilege escalation surface.

## 3.10 `FetchCapsule` scans latest 500 event rows

The API says `FetchCapsule` scans latest 500 event log rows for a matching `failure_capsule`.

That is fragile.

You already have:

```text
evidence_capsules
Db::latest_evidence_for_job
```

Capability `FetchCapsule` should query the evidence table directly. Scanning event rows can fail when the event ledger is busy.

## 3.11 `jeryu sync-now` is a placeholder

A command named:

```bash
jeryu jeryu sync-now --source <dir>
```

that only tells the user to ensure `jeryu serve` is running is likely to disappoint both humans and agents.

Either implement it or rename it to something like:

```bash
jeryu jeryu request-sync
```

If implemented, it should append a sync request event that the jeryu loop consumes.

## 3.12 Release state naming needs canonical enums

The docs mention multiple canary/release states:

```text
green
released
in-flight
canary-authorized
waiting
ready-for-canary
blocked
blocked-by-upstream
failed
e2e-passed
running
```

The TUI color table does not obviously cover every important state, especially `e2e-passed`.

For safety, release state should be a strongly typed enum with canonical render metadata:

```rust
enum CanaryState {
    Waiting,
    ReadyForCanary,
    CanaryAuthorized,
    Running,
    E2ePassed,
    Blocked,
    BlockedByUpstream,
    Failed,
    Released,
}
```

Each state should define:

```text
color
icon
is_terminal
allows_prod_promotion
operator_message
next_expected_event
```

## 3.13 Agent merge risk gate needs richer evidence semantics

The risk gate currently considers successful, pending, and failed jobs plus trust tier. That is a good base, but for serious autonomous merging it should also consider:

```text
required jobs vs optional jobs
release-critical lanes
whether successful jobs are from current SHA
whether pipeline was superseded
whether jobs are stale
whether VTI skipped tests are justified
selector miss history
security/admission events
cache taint status
artifact integrity
untrusted/custom executor provenance
```

A clean pipeline should not mean “some successful jobs existed.” It should mean “the required evidence set for this change and trust tier is satisfied.”

## 3.14 Live log polling needs smarter transport/backoff

Polling every 650ms is reasonable, but a future power-user system should avoid hammering GitLab when logs are large or stale.

Possible improvements:

```text
incremental trace fetch when possible
ETag / range support if GitLab allows it
adaptive polling while job is active
slower polling for stale/finished jobs
pause polling when logs pane is hidden
multi-log tail only for pinned jobs
```

The current `LIVE_LOG_MAX_BYTES` cap is good.

## 3.15 Default project/repo assumptions should become profiles

Several defaults are hardcoded or semantically central:

```text
project_id = 2
ref = main
repo = dougx
release repo root = /home/ubuntu/dougx
```

This is fine for the current environment, but a dream console should support profiles:

```toml
[profiles.dougx]
project_id = 2
main_ref = "main"
release_repo_root = "/home/ubuntu/dougx"
vault_prefix = "veox"
```

Then the TUI can show:

```text
Profile: dougx / main / project 2
```

# 4. What an agent would dream to be able to do

A capable agent would want to operate like a disciplined release engineer, not like a shell script. It would want these abilities.

## 4.1 Observe the whole system

An agent should be able to ask:

```text
What is the current state of project 2/main?
Is anything blocked?
What is the latest release attempt?
What changed since last green?
Which jobs are on the critical path?
Which failures are likely infrastructure?
Which actions are safe?
```

Ideal command:

```bash
jeryu observe --project-id 2 --ref-name main --json
```

Response should include:

```text
system health
GitLab readiness
runner capacity
active pipelines
release state
production state
current blockers
failed jobs
pending jobs
risk gates
cache/taint status
disk pressure
recommended actions
```

## 4.2 Explain blockers

Agents need “why” more than raw status.

Ideal command:

```bash
jeryu explain blocker --project-id 2 --ref-name main --json
```

Example answer:

```text
Production promotion is blocked because:
1. release-execution pipeline #530 passed
2. canary_state is running, not e2e-passed
3. gate-canary-e2e.json does not exist yet
4. C validation artifact is missing
Next expected event: canary E2E job completion
Safe action: watch release
Unsafe action: trigger production promotion
```

## 4.3 Propose and race patches

The deferred intents should become real.

An agent would love:

```json
{
  "intent": "RacePatches",
  "payload": {
    "base_branch": "main",
    "hypotheses": [
      {
        "name": "fix cache taint classification",
        "files": [...]
      },
      {
        "name": "adjust release state enum",
        "files": [...]
      }
    ],
    "validation": {
      "plan": "vrc",
      "max_parallel": 3
    }
  }
}
```

The system should create isolated branches, run selected validation, compare results, and report the best candidate without merging automatically.

## 4.4 Run validation based on changed files

The VTI tooling is strong. Agents should be able to say:

```bash
jeryu validate changes --base origin/main --head HEAD --json
```

And get:

```text
changed subsystems
selected tests
skipped tests
confidence
historical selector misses
required proof lane
commands to run
```

Then:

```bash
jeryu validate apply <plan-id>
```

## 4.5 Request merge with complete evidence

A dream merge request flow:

```bash
jeryu agent request-merge \
  --project-id 2 \
  --mr-iid 17 \
  --trust-tier untrusted \
  --json
```

Response:

```text
decision: escalate
reason: untrusted tier requires human approval
evidence:
  successful release-critical jobs: 8
  failed jobs: 0
  pending jobs: 0
  selector misses in last 30d: 1
  cache taints: 0
  admission rejects: 0
suggested human review checklist:
  - inspect generated release state changes
  - verify secrets paths are untouched
  - verify custom executor behavior unchanged
```

## 4.6 Safely operate infrastructure

Agents should be able to request:

```text
pool scale plan
cache GC dry run
host reclaim plan
runner token rotation plan
release reconcile
pipeline cancel for superseded work
```

But every infrastructure action should have:

```text
dry-run
blast radius
audit event
reason
rollback or recovery note
```

## 4.7 Ask for the next best action

This would be extremely powerful:

```bash
jeryu next-action --project-id 2 --ref-name main --json
```

Example output:

```json
{
  "summary": "Pipeline #530 is blocked by one failed integration job.",
  "recommended_action": {
    "name": "RetryJob",
    "job_id": 14445,
    "confidence": 0.78,
    "reason": "Failure classified as infrastructure and no prior retry exists."
  },
  "alternatives": [
    {
      "name": "FetchCapsule",
      "reason": "Inspect failure evidence before retry."
    }
  ],
  "forbidden": [
    {
      "name": "RequestMerge",
      "reason": "Release-critical failed job exists."
    }
  ]
}
```

This is the difference between an agent having tools and an agent having operational judgment.

# 5. Dream TUI redesign: “jeryu Command Center”

The current TUI is useful. The dream version should feel like a **real-time mission control console for developers, release engineers, and agents**.

I would redesign it around five principles:

```text
1. Every object is selectable.
2. Every object has an explanation.
3. Every explanation has safe next actions.
4. Every action has plan/apply/audit.
5. Nothing important is hidden.
```

# 9. Color and visual language

A brilliant TUI needs strong color, but color should not be the only signal. Use icons, text, shapes, and border styles too.

Recommended palette:

```text
Black / near-black background
Bright white primary text
Dim gray metadata
Cyan selection/focus
Green success/safe
Blue running
Amber/yellow pending/waiting
Red failed/danger
Magenta blocked/security/escalation
Purple release/canary
Orange disk pressure/resource warning
Dark gray stale/canceled
```

Use consistent icons:

```text
✓ success
● running
○ pending/created
× failed
■ blocked
▲ warning
◆ manual gate
⧗ waiting/ETA
↻ retry
⚑ release gate
🔒 security/secrets, or textual LOCK if avoiding emoji
```

Example high-contrast job row:

```text
▶ ● RUNNING   43%  07:14  ETA 03:20  test/integration-e2e      pool default
  ▲ WAITING   00%  queue 06:10       deploy/canary-gate        needs artifact
  × FAILED    71%  04:22             security/secret-scan       regression
  ✓ SUCCESS  100%  01:02             build/image
```

For accessibility, never rely on color alone:

```text
BAD: red row only
GOOD: × FAILED red row
```

## Suggested visual hierarchy

```text
Bright + bold: selected entity, critical blocker
Bright color: active status
Dim gray: historical/completed metadata
Dashed/dim border: stale panels
Double border: production-impacting panels
Magenta border: security/escalation
Red border: failed release-critical blocker
```

# 12. High-impact implementation roadmap

If I were sequencing improvements, I would do this.

## Phase 1: fix confusing and risky UX

```text
Make hidden Pools/Pipelines focus visible or remove it.
Add confirmation/undo for TUI delete.
Turn Audit tab into real event ledger.
Clearly mark approximate storage values.
Add canonical release state rendering.
Make deferred capability intents return explicit not-implemented/deferred status.
Use evidence_capsules table for FetchCapsule.
```

These are relatively small changes with high trust impact.

## Phase 2: create an action manifest

Add:

```bash
jeryu actions list --json
jeryu actions describe <action> --json
jeryu actions plan <action> --payload ... --json
jeryu actions apply <action> --payload ... --json
```

Then use that manifest in:

```text
TUI command palette
agent capability API
docs
tests
audit
```

This would unify human and agent control.

## Phase 3: build context packets

Add rich explain endpoints:

```bash
jeryu explain system --json
jeryu explain pipeline <id> --json
jeryu explain job <id> --json
jeryu explain release --json
jeryu explain blocker --json
```

These become the foundation for agent reasoning and TUI inspector panels.

## Phase 4: upgrade Flow Board into a real DAG

Add:

```text
multi-pipeline rendering
real edges
downstream bridge support
release/prod pipeline linkage
critical path calculation
historical ETA
blocker propagation
```

The Flow Board then becomes the heart of the TUI.

## Phase 5: add event streaming

Add one or more:

```text
jeryu events watch --json
capability event stream
HTTP SSE /events
TUI internal event bus
```

This improves agents and reduces polling.

## Phase 6: implement real agent patch/race/merge capabilities

Turn deferred intents into actual workflows:

```text
ProposePatch
RacePatches
RequestMerge
```

But keep them risk-gated, branch-based, pipeline-validated, and audited.

# 13. My strongest recommendation

The most important conceptual improvement is this:

**Make every visible status in the TUI answer three questions:**

```text
Why is this the state?
What is the next expected event?
What actions are available, safe, blocked, or dangerous?
```

For example, instead of only:

```text
Prod: not-triggered
```

show:

```text
Prod: not-triggered
Reason: waiting for canary_state=e2e-passed and validation artifact
Next: canary E2E gate should emit gate-canary-e2e.json
Actions: promote-prod unavailable
```

Instead of only:

```text
Job failed
```

show:

```text
Job failed: integration-e2e
Classification: Infrastructure
Retry advice: RetryOnce
Safe action: retry job
Blocked action: merge
```

Instead of only:

```text
Pool default 2/4
```

show:

```text
Pool default 2/4
Pending jobs: 6
Queue p95: 5m40s
Recommended: scale to 4
```

That is how the TUI becomes not just beautiful, but operationally brilliant.

# 14. Summary of the clearest improvements

The highest-value fixes and enhancements are:

```text
1. Make the capability API honest and richer.
2. Add action manifests with plan/apply semantics.
3. Build context packets for jobs, pipelines, releases, blockers, and agents.
4. Make the TUI command palette expose every safe action.
5. Fix hidden pane focus in the Flow tab.
6. Add confirmation or undo for destructive local deletes.
7. Turn the Audit tab into a real live event ledger.
8. Upgrade the Flow Board from columns to a true multi-pipeline DAG.
9. Use historical data for ETA and critical path.
10. Add real-time event streaming for agents and the TUI.
11. Canonicalize release states and gate rendering.
12. Make every blocked state explain why and what comes next.
```

If those are implemented, `jeryu tui` could become an exceptional power-user console: not just a terminal dashboard, but a live, explainable, risk-aware command center where developers, release engineers, and agents can safely operate the entire CI/CD system.
