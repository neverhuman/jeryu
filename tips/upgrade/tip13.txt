## Overall read

This is already a serious control plane, not a toy CLI. The strongest design choices are:

1. **Production is pipeline-mediated**, not shell-mediated. That is exactly right.
2. **State is durable and typed enough** to support agents, TUI, retry logic, release gates, cache policy, and audit.
3. **The system separates operational surfaces**: CLI, TUI, webhooks, custom executor, capability socket, GitLab wrapper, RedlineDB state.
4. **The security model is explicit**: trust-tiered runners, custom executor sandboxing, taint, honeypots, Vault, risk gates, release gates.
5. **The TUI already has the right instinct**: keep useful state visible under partial data rather than flickering to empty.

The biggest opportunity is to turn `jeryu` from “many excellent commands plus a dashboard” into a **live, permissioned command-and-evidence cockpit** where every entity, action, failure, risk, and release gate is inspectable and actionable in real time.

# 1. Biggest gaps I see

## 1.1 There is no single “action model”

Right now actions exist in many places:

* CLI commands
* TUI keybindings
* capability intents
* engine automation
* release automation
* GitLab REST wrapper calls
* hidden executor/server-hook commands

But there does not appear to be one canonical registry that says:

```text
Action:
  name: retry_job
  subject: job
  required_state: failed
  risk: low
  auth_scope: gitlab-job-control
  preflight: available
  dry_run: yes
  audit_event: job_retry_requested
  rollback: none
  visible_in_tui: yes
  visible_to_agent: yes
```

That registry would be transformative.

Agents and power users need to know:

* What can I do here?
* Why is this action allowed or blocked?
* What evidence will be checked first?
* What will change?
* What event will be recorded?
* What is the rollback or recovery path?
* Is this safe for an untrusted agent?

Without that, the system has powerful surfaces, but the “operating grammar” is scattered.

**Recommendation:** create a first-class `ActionRegistry` / `ControlAction` layer. The CLI, TUI, capability API, and automation should all route through it when possible.

Example actions:

```text
job.retry
job.cancel
job.explain
pipeline.cancel
pipeline.explain
pipeline.ingest
release.reconcile
release.promote_prod
pool.pause
pool.resume
pool.scale
cache.gc
host.reclaim.plan
host.reclaim.apply
agent.merge
secret.rotate
secret.finalize
```

Each action should support some combination of:

```text
describe
preflight
dry-run
invoke
audit
```

That would make the TUI much more powerful and make agent access much safer.

## 1.2 Capability API has dangerous ambiguity

The capability API is promising, but the current deferred intents are risky:

* `ProposePatch`
* `RacePatches`
* `RequestMerge`

The docs say these deserialize and return success, but active routing is deferred.

That is a problem for agents. An agent may interpret `success: true` as “the patch was proposed” or “the merge was requested,” when in reality nothing happened.

**Recommendation:** deferred capabilities should return something like:

```json
{
  "success": false,
  "message": "Capability RequestMerge is recognized but not implemented",
  "data": {
    "status": "deferred",
    "implemented": false,
    "recommended_surface": "jeryu agent merge"
  }
}
```

Even better, expose:

```bash
jeryu capability describe --json
```

Or socket intent:

```json
{ "intent": "DescribeCapabilities" }
```

The response should say exactly which capabilities are active, deferred, deprecated, or admin-only.

---

## 1.3 The TUI has invisible-action mismatch

The TUI still tracks active panes for:

* Pools
* Pipelines
* Jobs

But the active Flow layout visibly renders only:

* Flow Board
* Live Jobs
* Log Preview

The old pool and pipeline renderers exist but are inactive. This creates confusing behavior:

* `Tab` can focus an invisible pane.
* `p` can pause/resume a pool even though the pool list is not visible.
* `d` can delete a selected pipeline record, but the user may not see the selected pipeline pane.
* Pipeline focus affects border color, but there is no standalone active pipeline list.

This is the kind of UX mismatch that makes a power tool feel haunted.

**Recommendation:** either restore visible Pools and Pipelines surfaces, or remove invisible pane focus from the active Flow layout.

A better version would make the Flow tab use visible subpanes:

```text
Flow | Jobs | Pipelines | Pools | Logs
```

or a left-side entity navigator:

```text
Refs
Pipelines
Jobs
Pools
Agents
Releases
Cache
```

The important rule: **any keyboard action should operate on a visible selected entity.**

---

## 1.4 “Delete” in the TUI can hide evidence too easily

The `d` / `Delete` action deletes local DB records for selected jobs or pipelines.

That is operationally useful, but in a system built around audit, evidence, failure capsules, retry decisions, and release tracking, deletion should be treated carefully.

Issues:

* It can make the TUI cleaner while hiding useful evidence.
* It may confuse GitLab state vs local state.
* It may erase context needed for debugging.
* It appears to be immediate and local-state-only.

**Recommendation:** change this to one of:

```text
archive local record
hide from dashboard
mark as dismissed
soft-delete with audit event
```

Then provide a separate dangerous action:

```text
purge local record
```

with confirmation and an audit event.

The TUI should show clearly:

```text
This deletes only local jeryu state. It does not cancel or delete the GitLab job/pipeline.
```

## 1.5 The Audit tab being a placeholder is a major gap

Given the architecture, the Audit tab should be one of the most important views.

This system has:

* release attempts
* secret rotations
* retry decisions
* evidence capsules
* impact decisions
* VTI plans
* selector misses
* cache taints
* host reclaim events
* production promotion decisions
* supersedence cancellations
* agent merges
* runner pool operations

A placeholder audit tab undersells the entire system.

**Recommendation:** make Audit a first-class event ledger explorer.

It should show:

```text
Time
Actor
Action/Event
Subject
Risk
Decision
Evidence
Correlation ID
Outcome
```

Filters:

```text
release
secret
agent
retry
cache
host
pool
pipeline
security
failed-only
human actions
agent actions
automation actions
```

The dream version is a **time-travel debugger** for the whole delivery system.

## 1.6 Flow graph edges are missing

The Flow Board currently classifies jobs into columns and lanes, but graph edges are not computed.

That limits the board’s ability to answer:

* What is blocked by what?
* What is the critical path?
* Which job is gating release?
* Which downstream pipeline is waiting?
* What happens after this job succeeds?
* What is safe to cancel?
* What is merely informational?

**Recommendation:** compute real graph edges from:

* GitLab stages
* `needs`
* bridge jobs
* downstream pipelines
* release attempt relationships
* production promotion relationship
* VTI selected jobs
* artifact dependencies
* gate dependencies

The Flow Board should become a true DAG:

```text
Push
  -> Admission
  -> Impact Plan
  -> Build
  -> Unit / Integration / Security
  -> Package
  -> Canary Gate
  -> E2E Evidence
  -> Production Promotion
```

Without edges, the board is a beautiful status matrix. With edges, it becomes a command center.

---

## 1.7 ETA is too heuristic for a “dream” experience

Current ETA is lane-based fallback:

* Unit: 60s
* Integration: 300s
* Security: 120s
* Build: 180s
* Release: 120s
* Other: 90s

That is good enough for a first pass, but a power user will quickly want:

* historical job duration by job name
* by runner pool
* by branch/ref
* by image/toolchain epoch
* by cache-hit vs cold execution
* queue time vs execution time
* critical-path ETA, not just lane ETA
* confidence interval
* “ETA worsened because runner capacity is exhausted”
* “ETA worsened because cache missed”
* “ETA unknown because no history exists”

**Recommendation:** build ETA from `ci_job_runs`, `test_executions`, cache verdicts, and runner pool queue data.

Dream display:

```text
Pipeline ETA: 7m 40s ± 1m 20s
Critical path: build-image -> integration-live -> gate-canary-e2e
Delay cause: integration-live waiting for untrusted runner
Confidence: medium, 12 historical samples
```

## 1.8 Release gate evidence needs stronger visibility

The release architecture is good: upstream success does not directly deploy production. Canary and production promotion are separate. That is the right safety model.

But the TUI should make the gate state extremely explicit.

For every release attempt, the user should see:

```text
Gate                         State        Evidence
Upstream pipeline green       passed       pipeline #123
Release eligibility           passed       explain report
Impact is release-affecting   passed       impact report
Canary pipeline triggered     running      pipeline #124
Canary E2E                    missing      gate-canary-e2e.json
Telemetry                     passed       telemetry-gate.json
C handoff artifact             present      path/hash/signature
C validation artifact          present      path/hash/signature
Secrets rotated                passed       secret set v...
Production promotion           blocked      waiting on canary_e2e
```

The key is that production should never appear as a mysterious status string. It should appear as a **gate matrix**.

---

## 1.9 Release artifact validation should be stronger than “exists”

The docs say production promotion requires:

* canary state is `e2e-passed`
* C handoff artifact exists
* C validation artifact exists
* no existing production-promotion pipeline for same SHA/ref

That is solid, but artifact existence is a lower bar than artifact integrity.

For production-grade confidence, promotion gates should validate:

* artifact path exists
* artifact JSON/schema is valid
* artifact is bound to the same SHA
* artifact is bound to the same release version
* artifact includes expected environment
* artifact includes timestamp
* artifact includes producing pipeline/job id
* artifact hash matches recorded value
* artifact was produced by trusted job/pipeline
* artifact was not modified after pipeline completion
* optional signature/provenance attestation

**Recommendation:** represent every release gate as structured evidence with schema validation and provenance.

---

## 1.10 Hidden/internal commands should be harder to misuse

The docs correctly warn that:

* custom executor commands are GitLab Runner protocol hooks
* server-hook reads pre-receive input from stdin
* production promotion is not a shell command
* secrets only support `dougx`
* capability routing is partial

But warnings are not as good as enforcement.

If an agent has shell access, it can still run hidden commands.

**Recommendation:** internal commands should validate caller context when possible.

For example:

```text
jeryu exec run
  requires CUSTOM_ENV_CI_JOB_ID
  requires CUSTOM_ENV_CI_PROJECT_DIR
  requires runner-specific context
  refuses direct invocation unless JERYU_ALLOW_EXEC_DRIVER_TESTING=1
```

For server hook:

```text
jeryu server-hook pre-receive
  validates hook install context or explicit test mode
```

The goal is not to make local shell access harmless. The goal is to reduce accidental misuse by agents and humans.

---

## 1.11 The system needs correlation IDs everywhere

The architecture has many event-producing components:

* webhook server
* engine reconciliation
* Docker event loop
* disk pressure sentinel
* release automation
* TUI actions
* CLI actions
* capability actions
* custom executor
* GitLab client wrapper

A dream agent/TUI setup needs causality.

Example:

```text
correlation_id: rel-20260425-fa51a52
caused_by:
  push event
  pipeline hook
  release reconciliation
  canary trigger
  release pipeline
  production promotion
```

Every major action/event should carry:

```text
event_id
correlation_id
causation_id
actor
surface
subject_type
subject_id
project_id
ref
sha
risk
decision
result
created_at
```

This would let the TUI show:

```text
Why did this happen?
What triggered it?
What did it trigger next?
Which agent/human/automation caused it?
```

# 2. What an agent would dream to be able to do

An agent does not really dream of unrestricted shell access. A good agent dreams of **typed authority, rich context, safe mutation, and immediate evidence**.

The ideal agent control surface would provide these capabilities.

---

## 2.1 Ask for a complete mission digest

An agent should be able to ask:

```text
What is currently wrong?
What is blocked?
What changed?
What is risky?
What needs human approval?
What can I safely fix?
```

Possible command:

```bash
jeryu agent context --project-id 2 --ref main --json
```

Output should include:

```text
current pipelines
active jobs
failing jobs
release status
production gate state
recent events
open agent tasks
MRs awaiting risk gate
test selector plan
cache/runner health
host pressure
recommended next actions
```

This is different from `jeryu status`. It is a task-oriented “what should an agent know before acting?” digest.

---

## 2.2 Preflight every action

Before mutating anything, an agent should be able to ask:

```bash
jeryu action preflight release.promote-prod --project-id 2 --ref main --json
```

Response:

```json
{
  "allowed": false,
  "risk": "high",
  "reason": "canary_e2e gate missing",
  "missing_evidence": [
    "gate-canary-e2e.json"
  ],
  "next_safe_actions": [
    "release.reconcile",
    "pipeline.explain",
    "job.trace"
  ]
}
```

This is ideal for agents because they can reason without guessing.

---

## 2.3 Get structured failure repair packets

The current failure capsule idea is excellent. Take it further.

For a failed job, an agent wants:

```text
failure kind
classification
likely owning module
changed files
relevant logs
minimal reproduction command
suggested validation command
related historical failures
last successful SHA
cache verdict
runner info
environment/toolchain fingerprint
retry policy
```

Possible command:

```bash
jeryu job repair-packet <project_id> <job_id> --json
```

This should combine:

* evidence capsule
* job trace
* VTI plan
* witness graph
* `cargo-witness diagnose`
* changed files
* historical failures
* retry decisions

That would be extremely useful.

---

## 2.4 Race hypotheses safely

The architecture mentions `RacePatches`, but it is deferred. This would be one of the most valuable agent features.

An agent could propose multiple candidate fixes:

```json
{
  "intent": "RacePatches",
  "payload": {
    "base_branch": "main",
    "commit_message": "Fix flaky release gate parsing",
    "hypotheses": [
      {
        "name": "strict-json-schema",
        "modifications": [...]
      },
      {
        "name": "artifact-path-normalization",
        "modifications": [...]
      },
      {
        "name": "timestamp-tolerance",
        "modifications": [...]
      }
    ]
  }
}
```

The system would:

1. Create isolated branches.
2. Apply patches.
3. Run selected tests.
4. Compare outcomes.
5. Report winner.
6. Optionally open one MR.
7. Preserve failed hypotheses as evidence.

This is a perfect fit for your GitLab/test/VTI architecture.

---

## 2.5 Use a safe patch proposal API

The agent should be able to propose a patch without raw repo write authority.

Possible flow:

```text
agent sends patch proposal
jeryu validates touched files
jeryu checks agent ownership/routing
jeryu creates branch
jeryu commits patch
jeryu runs selected validation
jeryu opens MR
risk gate evaluates
```

The patch proposal should support:

* unified diff
* file replacement
* structured modifications
* generated artifact updates
* validation plan
* explanation
* rollback branch

This would make the capability API genuinely agent-grade.

---

## 2.6 Ask “why is this blocked?”

For any entity, an agent should be able to ask:

```bash
jeryu explain blocked pipeline:123 --json
jeryu explain blocked release:ci-fa51a52 --json
jeryu explain blocked job:14445 --json
jeryu explain blocked pool:untrusted --json
```

The response should be causal:

```text
Production promotion is blocked because:
  - canary_state is running, required e2e-passed
  - gate-canary-e2e.json missing
  - release pipeline #1234 is still running job gate-canary-e2e
  - estimated completion 4m 20s
  - no action required unless job exceeds stale threshold
```

This is the difference between “dashboard” and “operator brain.”

---

## 2.7 Ask for a minimal validation plan

You already have VTI and proof-scoped tools. An agent would love a single command:

```bash
jeryu validate plan --changed-paths ... --risk target --json
```

It should unify:

* `jeryu test select`
* `jeryu test select-external`
* `cargo-vrc plan`
* `cargo-witness diff`
* release-impact analysis
* security-sensitive module detection
* required proof lanes

Output:

```text
Required:
  cargo check -p jeryu
  cargo test -p jeryu -- release::tests
  cargo run -p jeryu -- repo audit-agent-surface --json

Recommended:
  jeryu test run --command ...
  cargo-witness build

Skipped:
  TUI tests, because no tui files changed
```

Then:

```bash
jeryu validate run --plan vrc-plan.json
```

---

## 2.8 Let agents operate through “bounded autonomy”

A strong agent-access model would look like:

```text
Read access:
  all state summaries
  logs
  traces
  capsules
  event ledger
  release reports
  VTI plans
  cache verdicts
  test history

Low-risk write:
  run tests
  fetch capsules
  open issue
  propose patch branch
  create MR draft
  retry transient failed job once

Medium-risk write:
  cancel own scratch pipeline
  scale non-production runner pool within cap
  run cache GC dry-run/apply under threshold
  request merge through risk gate

High-risk write:
  release promotion
  secrets rotation
  host reclaim apply
  runner token rotation
  production-affecting merge

High-risk should require:
  explicit preflight
  audit event
  policy approval
  human confirmation or privileged agent identity
```

This would give agents meaningful power without turning them into uncontrolled shell operators.

# 3. Issues that likely need to be addressed

## 3.1 Multi-project assumptions

Several flows are hard-coded or defaulted around:

```text
project_id = 2
ref = main
repo = dougx
release root = /home/ubuntu/dougx
```

That is fine for the current system, but the architecture is otherwise general enough that these assumptions may become future friction.

**Improvement:** introduce a first-class project/release profile config.

Example:

```toml
[profiles.dougx]
project_id = 2
default_ref = "main"
repo_root = "/home/ubuntu/dougx"
release_enabled = true
external_testmap = "/home/ubuntu/dougx/.jeryu/testmap.toml"
```

Then commands can default to the active profile while still supporting explicit overrides.

---

## 3.2 Stale data needs stronger visual treatment

The anti-blanking behavior is good. Retaining the last useful Flow Board is the right call.

But stale data can become dangerous if it looks too normal.

**Improvement:** stale state should be visually unmistakable:

```text
STALE SNAPSHOT — last real update 2m 14s ago
GitLab unreachable since 14:03:12
Showing retained pipeline state; actions disabled except refresh/diagnose
```

Actions that depend on fresh state should either be disabled or require preflight.

---

## 3.3 TUI actions need confirmation levels

Not every action needs confirmation. Retrying a failed job is probably fine.

But these should have confirmation or preflight overlays:

* deleting local records
* canceling pipeline
* scaling pool to zero
* draining pool
* host reclaim apply
* cache GC with active manager eviction
* release promote-prod
* secrets rotate/finalize
* runner token rotation
* accepting MR

The TUI needs an action confirmation modal with:

```text
Action
Target
Risk
What changes
Why allowed
Evidence checked
Audit event to be recorded
Confirm key
```

For high-risk actions, require typed confirmation:

```text
type PROMOTE ci-fa51a52 to continue
```

---

## 3.4 Release and secret status should be linked

Release status and secret status are currently separate surfaces.

For production readiness, they should be linked in one view:

```text
Release version: ci-fa51a52
Secret set: rotated / finalized / missing
Vault authority: healthy
Runtime env: generated
Deploy env: generated
Recovery bundle: present
Finalization: pending production success
```

The TUI should never expose secret values, but it should expose secret lifecycle state.

---

## 3.5 Cache verdicts need to be explainable

The cache architecture is rich:

* epochs
* taints
* witnesses
* exact-hit skip
* cold execution downgrade
* promotions
* trust tier

But the TUI Cache tab is mostly aggregate metrics.

A power user needs per-job/per-build explanation:

```text
Cache decision for build-image:
  verdict: cold execution
  reason: epoch changed
  trust tier: untrusted
  witness: dockerfile hash changed
  taint: none
  promotion: allowed after success
```

This should be visible in the job inspector.

---

## 3.6 The job inspector exists but is not wired in

This is a missed opportunity.

A selected job should have an inspector pane with:

```text
Overview
Logs
Failure Capsule
Retry Decision
Artifacts
Cache Verdict
Runner/Pool
Timeline
Related Events
Actions
```

Right now, the log preview is useful, but a job is more than a log.

---

## 3.7 Live logs should eventually move behind a trace service

Polling GitLab every 650ms is acceptable for now, but the dream version should avoid every TUI instance independently hammering GitLab traces.

Better architecture:

```text
GitLab trace polling / streaming
  -> jeryu engine trace cache
  -> TUI subscribes via Unix socket/WebSocket/local event stream
```

Benefits:

* fewer duplicate requests
* better buffering
* multiple TUI clients can share state
* logs can be correlated with job lifecycle
* trace deltas can be pushed incrementally
* staleness is centrally tracked

---

## 3.8 Storage tab should distinguish exact vs approximate numbers

The docs say Docker detailed storage values are partly approximate.

That is fine, but the UI should label them:

```text
Docker image bytes       approximate
CAS objects              measured
jeryu DB                  measured
Root disk usage          measured
Docker build cache       unavailable / estimated
```

Power users need to know what is real before taking reclaim actions.

---

## 3.9 Production promotion should have an explicit “no direct deploy” banner

This is a strength of the architecture. Make it visible.

In the Release view:

```text
Production mode: pipeline-gated
Direct shell deploy: disabled
Promotion command: release.promote-prod
Required gates: 6/8 passed
```

This reinforces the safety model for humans and agents.

---

## 3.10 Agent tasks need a first-class TUI surface

The API has:

```text
jeryu agent spawn
jeryu agent list
jeryu agent merge
```

But the TUI design does not describe a real Agent tab.

That is a gap.

A dream TUI should show:

```text
Agent Tasks
  task
  branch
  issue
  MR
  trust tier
  risk gate status
  latest pipeline
  failed jobs
  proposed next action
  owner/agent identity
```

And actions:

```text
open logs
run selected tests
request merge
escalate
close task
race hypotheses
```

# 4. Dream TUI redesign

I would redesign the TUI around one concept:

## **JERYU Mission Control**

Not a dashboard. Not just a terminal UI. A live command console where every entity has:

```text
state
evidence
timeline
actions
risk
ETA
logs
explanation
```

The UI should answer four questions instantly:

1. **What is happening?**
2. **What is blocked?**
3. **Why?**
4. **What can I safely do next?**

## 5.2 The central Flow DAG

The current Flow Board should evolve into a true live deployment graph.

Example:

```text
┌ Admission ┐  ┌ Impact ┐  ┌ Build ┐  ┌ Tests ┐  ┌ Gates ┐  ┌ Canary ┐  ┌ Prod ┐
│ ✓ Hook    │→ │ ✓ VTI  │→ │ ● Img │→ │ ● Int │→ │ ○ E2E │→ │ ○ Run  │→ │ ⛔ Wait │
│ ✓ Policy  │  │ ✓ Full │  │ ✓ Bin │  │ ✓ Unit│  │ ✓ Sec │  │        │  │        │
└───────────┘  └────────┘  └───────┘  └───────┘  └───────┘  └────────┘  └──────┘
```

It should show:

* pipeline jobs
* bridge jobs
* downstream pipelines
* release-execution pipeline
* production-promotion pipeline
* gates
* artifacts
* blockers
* critical path
* ETA
* stale state
* runner assignment
* cache decision
* retry status

Visual rules:

```text
✓ passed
● running
○ pending
✕ failed
⛔ blocked
↻ retrying
⚠ degraded
◇ informational
```

Edges should indicate actual dependency:

```text
solid edge     required dependency
dashed edge    informational/extended lane
bold edge      critical path
dim edge       skipped/omitted
red edge       blocked dependency
```

# 7. Color-rich, high-contrast visual design

The UI should be dark, high-contrast, and status-dense. Color should help, but not be the only signal.

## 7.1 Semantic color palette

Use consistent meanings everywhere:

```text
Green       success / passed / eligible
Cyan        running / active automation / selected live entity
Blue        informational / neutral progress
Yellow      waiting / pending / attention
Orange      degraded / slow / stale soon
Red         failed / denied / hard blocker
Magenta     security / escalation / untrusted risk
Purple      agent activity / autonomous work
Gray        stale / skipped / inactive / historical
White       primary text
Black       selected text on bright backgrounds
```

Never use color alone. Pair with symbols:

```text
✓ pass
● running
○ pending
✕ failed
⛔ blocked
⚠ warning
↻ retrying
◆ agent
◇ informational
☠ tainted/quarantined
🔒 gated/locked
```

Even in terminals without emoji support, fallback glyphs can work:

```text
PASS
RUN
WAIT
FAIL
BLOCK
WARN
RETRY
AGENT
TAINT
LOCK
```

---

## 7.2 Visual hierarchy

The user’s eye should go here first:

1. Red blockers
2. Yellow pending gates
3. Cyan running critical path
4. Green completed lanes
5. Gray historical/skipped info

Critical path should be brighter than non-critical work.

Example:

```text
critical running job: cyan bold with progress pulse
failed blocker: red bold border
non-blocking failed research job: red text but dim border
skipped VTI job: gray with “vti-skipped”
```

---

## 7.3 Stale state treatment

Stale should not be subtle.

For retained flow snapshots:

```text
┌ FLOW BOARD — STALE 2m14s — GitLab trace unavailable ┐
```

Use:

* gray/orange border
* timestamp
* source health
* action limitations

If stale beyond a threshold:

```text
STALE > 5m: actions requiring fresh GitLab state disabled
```

# 8. Dream command console

The killer feature would be a universal command palette.

## 8.1 Ctrl-K action palette

Press `Ctrl-K`:

```text
Actions for selected job #14445:
  explain failure
  open full logs
  fetch artifacts
  retry job                    disabled: retry policy says regression
  run focused test
  create agent repair task
  open GitLab URL
  dismiss local record
```

For selected release:

```text
Actions for release ci-fa51a52:
  explain gates
  reconcile
  show artifacts
  show secret set
  promote production           disabled: canary E2E missing
  recover release bundle
```

For selected pool:

```text
Actions for pool build:
  scale up
  scale down
  pause
  resume
  rotate token                 high risk
  show manager logs
```

The palette should show disabled actions too, with reasons. Disabled actions are often more informative than enabled ones.

---

## 8.2 Natural-language command line

A dream power-user console could support typed questions:

```text
why is prod blocked?
show failed jobs on critical path
retry transient failures
explain pipeline 530
show cache misses for this pipeline
what changed since last green main?
create agent task to fix this failure
```

This does not need to be magical. It can map to typed commands and show the resolved action before execution.

Example:

```text
You asked: retry transient failures
Resolved action:
  job.retry for jobs [14445, 14446]
Preflight:
  14445: denied, regression
  14446: allowed, infrastructure
Proceed with 14446 only? [y/N]
```

# 11. Best “dream” features

If I were designing this as a developer’s dream console, these would be the signature features.

## 11.1 “Why?” on everything

Select anything and press `?`.

Examples:

```text
Why is this job running?
Why is this release blocked?
Why was this pipeline canceled?
Why did VTI skip this test?
Why was this cache hit rejected?
Why did the agent merge gate escalate?
```

Every answer should include evidence.

---

## 11.2 “What changed since last green?”

For any failed pipeline:

```text
Last green main: fa51a52
Current SHA: 4b92c11

Changed:
  src/release.rs
  src/state.rs
  docs/API.md

Likely affected:
  release gate tracking
  production pipeline status
  state migrations

Recommended validation:
  cargo test -p jeryu -- state::tests::test_production_pipeline_tracking release::tests
```

This would be incredibly valuable.

---

## 11.3 “Critical path mode”

One key toggles the TUI into critical-path-only view.

It hides noise and shows:

```text
current blocker
next required job
ETA
blocking gate
responsible lane
safe actions
```

For a power user, this matters more than seeing every job.

---

## 11.4 “Agent repair mode”

When a job fails:

```text
Create repair task?
  branch: agent/fix-release-gate-fa51a52
  likely files: release.rs, state.rs
  validation: release::tests
  risk: medium
  suggested prompt: ...
```

Then the agent can work through normal MR/risk gates.

---

## 11.5 “Release cockpit”

A release cockpit should feel like aircraft instrumentation:

```text
UPSTREAM       PASS
BUILD          PASS
SECURITY       PASS
CANARY         RUNNING
E2E            WAITING
TELEMETRY      WAITING
SECRETS        ROTATED
PRODUCTION     LOCKED
```

With one selected blocker:

```text
Production locked because Canary E2E evidence is missing.
Current job: gate-canary-e2e, running for 3m12s, ETA 1m40s.
```

---

## 11.6 “Policy simulator”

Before changing trust, runner pools, cache policy, or release gates:

```text
Simulate:
  What if untrusted MR !44 were treated as trusted?
  What if build pool max managers increased to 6?
  What if this pipeline were canceled?
  What if production promotion were requested now?
```

The output should be explicit:

```text
Promotion would be denied:
  missing canary E2E
  missing validation artifact
```

---

## 11.7 “Event replay”

The Audit tab should let users replay:

```text
push -> pipeline -> failed job -> retry decision -> agent task -> MR -> risk gate -> merge -> release
```

This is invaluable for debugging automation and building trust.

# 12. My strongest recommendation

The highest-leverage redesign is not visual first. It is architectural:

## Build a unified **Entity + Action + Evidence** model.

Everything in the TUI should revolve around selected entities:

```text
ReleaseAttempt
Pipeline
Job
AgentTask
Pool
Manager
CacheObject
SecretSet
TestPlan
SelectorMiss
AuditEvent
```

Each entity should have:

```text
summary
status
risk
timeline
evidence
available actions
blocked actions with reasons
related entities
logs/artifacts when applicable
```

Then the TUI becomes simple and powerful:

* left side selects entities
* center shows relationships/flow
* right side inspects evidence/actions
* bottom shows live events and command entry

That would make `jeryu tui` feel less like a dashboard and more like a **real-time operating system for delivery**.

# 13. Bottom line

The architecture is already unusually strong. The biggest remaining gaps are not missing commands; they are missing **unification, causality, and inspectability**.

The most important fixes are:

1. Add a canonical action/preflight/dry-run model.
2. Make capability responses unambiguous.
3. Wire TUI focus only to visible entities.
4. Promote Audit from placeholder to event/correlation explorer.
5. Turn the Flow Board into a real dependency DAG.
6. Make release gates a first-class matrix.
7. Add job inspectors with failure capsules, retry decisions, cache verdicts, and timelines.
8. Make stale state visually unmistakable.
9. Give agents bounded typed authority instead of broad operational ambiguity.
10. Redesign the TUI around entities, evidence, actions, ETAs, and blockers.

The dream version of this TUI is a high-contrast, real-time **Mission Control** where a developer can glance once and know:

```text
what is running
what is blocked
why it is blocked
how long it will take
what changed
what evidence exists
what actions are safe
what actions are forbidden
what the agent is doing
what production is waiting on
```

That would be genuinely exceptional.
