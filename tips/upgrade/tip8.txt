This is a **wildly ambitious and genuinely compelling system**. The strongest thing about JeRyu / `jeryu` is that it is not pretending CI/CD is just “run a command and hope.” It is building an actual **local delivery control plane** with typed state, release gates, runner orchestration, cache policy, failure evidence, agent workflows, Vault-backed secrets, and a supervisory TUI.

My biggest overall feedback:

**The architecture is powerful, but the UI and capability API have not yet caught up to the ambition of the control plane.**
The backend wants to be “mission control.” The current TUI is still partly “dashboard with logs.” The dream version should become a **real-time command cockpit** where every pipeline, gate, runner, cache decision, release action, blocker, agent proposal, and ETA is visible, explainable, and safely actionable.

Below is my best feedback, split into clear gaps, agent dream capabilities, issues to address, and a proposed dream TUI redesign.

---

# 1. High-level assessment

The architecture has several excellent instincts:

1. **Everything important is mediated through typed surfaces.**
   CLI, TUI, webhook server, capability socket, GitLab client, SQLite state, custom executor, release API, and secrets API are all documented as intentional surfaces.

2. **Production deployment is pipeline-mediated, not shell-mediated.**
   This is a very strong safety model. Canary and production promotion are GitLab pipeline products with explicit variables and gates, not ad hoc operator commands.

3. **State is durable and audit-oriented.**
   SQLite acts as a coordination layer, not just a cache. The event ledger, release attempts, evidence capsules, retry decisions, test plans, cache taints, and secret audit events are exactly the kind of state an agent needs.

4. **The execution model understands trust.**
   Trusted, privileged, and untrusted pools are separated. The custom executor, sandbox, honeypot, taint, witness, BuildKit, and cache-brain modules form a real supply-chain boundary.

5. **The TUI already has the right instinct: preserve meaningful state under partial failure.**
   The anti-blanking behavior for the Flow Board is excellent. A real control room should not flicker to emptiness just because GitLab hiccuped for one refresh cycle.

The core gap is that the system has many powerful primitives, but not yet a single unified “operator brain” surface that says:

> Here is what is happening.
> Here is what matters.
> Here is what is blocked.
> Here is why.
> Here is the safest next action.
> Here is what an agent could do, with evidence.

That is the dream TUI.

# 2. The clearest gaps I see

## 2.1 Capability API returns “success” for deferred actions

This is the biggest agent-facing correctness issue.

The Capability API has these intents:

* `ProposePatch`
* `RacePatches`
* `RunTests`
* `FetchCapsule`
* `RequestMerge`

But only `FetchCapsule` and `RunTests` are actively implemented. The others deserialize and return success while routing is deferred.

That is dangerous for agents.

An agent may interpret `success: true` as “the patch was proposed,” “the race ran,” or “the merge was requested.” Instead, the system should distinguish:

```json
{
  "success": false,
  "state": "deferred_not_implemented",
  "message": "RequestMerge is recognized but not currently routed.",
  "data": {
    "recommended_surface": "jeryu agent merge"
  }
}
```

Or:

```json
{
  "success": true,
  "state": "accepted_noop",
  "side_effects": []
}
```

The key is: **success must mean either the action happened or the response explicitly says no side effect occurred.**

For agents, ambiguous success is poison.

---

## 2.2 Capability `FetchCapsule` scans recent events instead of querying evidence directly

The system has an `evidence_capsules` table and `Db::latest_evidence_for_job`, but the capability behavior says it scans the latest 500 event rows for a matching `failure_capsule`.

That creates avoidable failure modes:

* older capsules can be missed
* noisy event ledgers can push relevant capsules out of the latest 500
* evidence lookup depends on event serialization shape
* the purpose-built evidence table is bypassed

For an agent, “fetch failure evidence” should be one of the most reliable operations in the whole system.

I would change `FetchCapsule` to query `evidence_capsules` first, probably through `latest_evidence_for_job`, then optionally fall back to the event ledger for  data.

---

## 2.3 The TUI has active state for panes that are not visibly rendered

The active Flow layout visibly shows:

* Release Watch
* Flow Board
* Live Jobs
* Log Preview

But the application still has active panes:

* Pools
* Pipelines
* Jobs

And keybindings still cycle through those panes. Some actions target invisible selections:

* `p` pause/resume selected pool
* `d` delete selected pipeline/job record
* `Tab`, `Left`, `Right` cycle focus through panes the user may not see

That is a UX and safety problem.

A power user should never be able to affect an invisible selected object unless the command palette explicitly says what object will be affected.

The immediate fix:

* Either restore visible Pools and Pipelines panes,
* or remove Pools/Pipelines from focus rotation in the current Flow layout,
* or add a compact visible “selection rail” showing the currently focused hidden object.

For destructive actions, the footer should say something like:

```text
Delete target: pipeline #530 local DB record
```

before `d` does anything.

---

## 2.4 The Audit tab being a placeholder is a major mismatch

This system is deeply audit-oriented:

* event ledger
* release attempts
* retry decisions
* secret audit events
* cache taints
* selector misses
* admission hooks
* production promotion gates

But the TUI Audit tab is currently a placeholder.

That is a huge missed opportunity.

The Audit tab should be one of the highest-value screens. It should answer:

* What happened?
* Who or what caused it?
* What side effects occurred?
* Which gates passed or failed?
* Which agent actions were attempted?
* Which retries were automatic?
* Which secrets were rotated or finalized?
* Which cache objects were quarantined?
* Which production promotions were triggered?
* Which admission decisions denied or allowed a ref?

Right now the system has the audit data; the TUI just does not expose it.

---

## 2.5 The Flow Board renders only the first active pipeline

For a system with upstream pipelines, release-execution pipelines, production-promotion pipelines, downstream bridges, child pipelines, and scratch test branches, showing only the first active pipeline is too limiting.

The release architecture is multi-pipeline by design. The UI should reflect that.

A dream Flow Board should show:

```text
main upstream pipeline
    -> release-execution pipeline
        -> canary gates
            -> production-promotion pipeline
```

Plus:

* child pipelines
* bridge jobs
* active agent test pipelines
* superseded pipelines
* canceled pipelines
* pending downstream pipelines

The current model already has `active_pipelines: Vec<PipelineFlow>`. The TUI should expose that vector instead of choosing only the first item.

---

## 2.6 Graph edges exist but are empty

The `FlowGraph.edges` model exists, but edges are not computed.

That means the TUI is visually a column board, not yet a true flow graph.

For power users, edges matter because they explain causality:

* this job unblocks that job
* this release gate depends on this artifact
* this production promotion depends on this canary result
* this failed test blocks this MR
* this cache taint forced this cold execution
* this selector decision skipped these tests

The dream TUI should use edges to show the actual dependency graph and critical path.

Even if exact GitLab DAG dependencies are not fully available, a partial edge model would be valuable:

* stage-order edges
* release-attempt edges
* upstream-to-release pipeline edge
* release-to-prod pipeline edge
* bridge/downstream edges
* artifact gate edges
* “blocked by” edges

---

## 2.7 Runner scaling appears too coarse if pending-job demand is global

The reconciliation loop is described as:

1. list pools
2. count pending jobs
3. compute target managers per pool as `min_warm + pending`, capped at `max_managers`
4. scale each unpaused pool to target

If `pending` is a global count rather than a per-pool, tag-aware, trust-aware demand calculation, this can over-scale all pools when only one pool has demand.

Example:

* 5 pending untrusted jobs
* default, build, and untrusted pools all see `pending = 5`
* every pool scales toward `min_warm + 5`

The system should have a **runner demand planner**:

```text
pending job -> tags -> matching pools -> trust tier -> executor type -> capacity target
```

It should compute pressure per pool:

```text
default:    min_warm 2 + demand 1
build:      min_warm 2 + demand 0
untrusted:  min_warm 1 + demand 5
```

The TUI should then show:

```text
untrusted pool: 5 queued, 2 max managers, saturated
```

That is exactly the kind of operational insight a power user or agent needs.

---

## 2.8 `/cache/summary` is unauthenticated while the server binds to `0.0.0.0`

`GET /health` without auth is fine.

But `GET /cache/summary` without auth may expose operational information:

* hit/miss behavior
* cache object count
* health state
* possible workload intensity

The server binds to `0.0.0.0:9777`, so unless there is external network isolation, this should probably be protected or explicitly limited.

Options:

* bind to localhost by default
* require a read-only token
* expose only coarse status unauthenticated
* move detailed cache summary behind auth
* document firewall assumptions clearly

For a local lab this may be acceptable; for a serious agent-operated control plane, unauthenticated operational telemetry should be treated carefully.

---

## 2.9 The capability socket needs explicit permission semantics

The Unix socket model is good, but agents need more than “socket path exists.”

I would want:

* filesystem permissions set explicitly
* peer credential checks where possible
* per-agent identity
* per-intent allow/deny policy
* audit event per request
* response includes side-effect IDs
* optional dry-run mode
* optional request nonce/idempotency key

A dream capability request should look like:

```json
{
  "intent": "RunValidationPlan",
  "actor": "agent.fix-regression.17",
  "request_id": "uuid",
  "dry_run": false,
  "payload": {
    "project_id": 2,
    "target_ref": "agent/fix-cache-taint",
    "plan": "vrc-plan.json"
  }
}
```

And the response should include:

```json
{
  "success": true,
  "side_effects": [
    {
      "type": "branch_created",
      "name": "agent/fix-cache-taint-ci-..."
    },
    {
      "type": "pipeline_triggered",
      "pipeline_id": 531
    }
  ],
  "audit_event_id": 9182
}
```

That would be beautiful for safe automation.

---

## 2.10 The 64 KiB capability payload cap limits patch workflows

A 64 KiB request cap is fine for small intents, but not for serious patch proposals, test plans, artifacts, or multi-hypothesis races.

Instead of sending large payloads over the socket, the agent capability API should support references:

```json
{
  "intent": "ProposePatch",
  "payload": {
    "patch_file": "/tmp/jeryu-agent/patches/123.diff",
    "manifest": "/tmp/jeryu-agent/patches/123.json"
  }
}
```

Or:

```json
{
  "intent": "ProposePatch",
  "payload": {
    "artifact_ref": "sqlite:agent_artifacts:123"
  }
}
```

Keep the socket for control messages, not bulk transfer.

---

## 2.11 `jeryu sync-now` is a placeholder

The API documents `jeryu sync-now` as a CLI trigger placeholder that tells the user to ensure `jeryu serve` is running.

That is surprising. A command named `sync-now` should either:

* actually request a sync through the daemon,
* enqueue a sync request in SQLite,
* call the sync implementation directly,
* or be renamed to something like `jeryu poke`.

This matters because agents will absolutely try to use `sync-now` as an action.

---

## 2.12 Release state is too stringly typed

Release status uses many strings:

* `green`
* `released`
* `in-flight`
* `canary-authorized`
* `waiting`
* `ready-for-canary`
* `blocked`
* `blocked-by-upstream`
* `failed`
* `e2e-passed`
* `not-triggered`

Some are canary states, some are production states, some are eligibility concepts, some are UI labels.

This should become a typed release state machine.

For example:

```rust
enum CanaryState {
    NotStarted,
    Claimed,
    ReleasePipelineTriggered,
    Running,
    WaitingForEvidence,
    E2ePassed,
    Failed,
    Blocked,
}

enum ProductionState {
    NotTriggered,
    Eligible,
    Triggered,
    Running,
    Succeeded,
    Failed,
    Canceled,
}

enum ReleaseEligibility {
    Eligible,
    BlockedByCriticalJobs,
    BlockedByExtendedLane,
    BlockedByImpact,
    BlockedByMissingArtifacts,
}
```

The UI can still render friendly labels, but the backend should not depend on string interpretation.

---

## 2.13 The release gate needs “why not production?” visibility

The production gate is intentionally strict:

* release attempt exists
* `canary_state == "e2e-passed"`
* C handoff artifact exists
* C validation artifact exists
* no existing production-promotion pipeline for same SHA/ref

That is excellent.

But the TUI should expose these as checklist rows:

```text
Production promotion readiness
✓ Release attempt exists
✓ Canary state: e2e-passed
✓ Handoff artifact found
✗ C validation artifact missing
✓ No existing production pipeline for this SHA
```

Right now, the Release Watch shows some of the state, but not the full gate explanation.

An operator or agent should never have to guess why production did not trigger.

---

## 2.14 Auto-retry depends on evidence quality

The failed-job auto-retry path is elegant:

```text
failed job -> evidence capsule -> classification -> retry decision -> maybe retry once
```

But it depends on timely evidence capsule creation.

The system should explicitly handle:

* failed job with capsule
* failed job without capsule
* capsule exists but malformed
* capsule classification unknown
* retry decision already exists
* retry requested but GitLab retry failed

Each state should be visible in the TUI and event ledger.

A failed job without a capsule should become a first-class problem:

```text
FAILED: test-rust-nextest-1
Evidence: missing
Retry: blocked, no capsule
Suggested action: fetch trace and synthesize capsule
```

---

## 2.15 Cache decisions need first-class explainability

The cache system is one of the most sophisticated parts of the architecture:

* exact-hit skip
* cold execution
* promotions
* taints
* epochs
* witnesses
* trust tiers
* force-refresh rules

But the TUI summary is still mostly metrics.

For a power user, the key question is:

> Why did this job run cold, skip, promote, or quarantine?

Every job node should be able to show:

```text
Cache verdict
Mode: exact-hit skipped
Reason: input_signature matched
Epoch: rust-2026-04-25
Trust: untrusted -> no cross-tier promotion
Witness: docker+rust signatures present
Taint: none
```

Or:

```text
Cache verdict
Mode: cold execution
Reason: active taint rule matched crate-cache
Taint id: 44
Downgrade: yes
```

This turns cache from invisible magic into operational evidence.

# 3. What an agent would dream of being able to do

The dream agent should **not** get broad shell access or a raw GitLab PAT.

The dream agent should get a rich, typed, least-privilege capability surface with strong auditability.

## 3.1 Agent dream: observe everything in real time

Agents need a streaming state feed, not just one-shot commands.

I would add:

```rust
AgentIntent::SubscribeEvents {
    project_id: Option<i64>,
    ref_name: Option<String>,
    event_types: Vec<String>,
}
```

Or a separate local event stream:

```text
jeryu capability stream /tmp/jeryu-events.sock
```

The stream should include:

* job status changes
* pipeline changes
* release attempt changes
* cache verdicts
* taint events
* retry decisions
* selector decisions
* agent task events
* admission decisions
* secret lifecycle events
* disk pressure events
* runner capacity changes

This is what would let an agent become situationally aware instead of polling.

---

## 3.2 Agent dream: ask “what is blocking release?”

A beautiful agent intent would be:

```json
{
  "intent": "ExplainBlockers",
  "payload": {
    "project_id": 2,
    "ref_name": "main",
    "scope": "release"
  }
}
```

Response:

```json
{
  "success": true,
  "data": {
    "release_eligible": false,
    "blockers": [
      {
        "kind": "failed_job",
        "job_id": 14445,
        "name": "integration-live",
        "classification": "Regression",
        "capsule_id": 88,
        "suggested_actions": [
          "inspect_trace",
          "spawn_fix_agent",
          "run_vti_plan"
        ]
      },
      {
        "kind": "missing_artifact",
        "path": "gate-canary-e2e.json",
        "phase": "production_promotion"
      }
    ]
  }
}
```

This would be enormously useful.

---

## 3.3 Agent dream: run validation plans, not just individual tests

`RunTests` currently takes a simple `test_scope`.

The agent really wants:

```rust
RunValidationPlan {
    project_id,
    target_ref,
    changed_paths,
    max_parallel,
    budget,
    include_security,
    include_release_gates,
}
```

The system already has the pieces:

* VTI selector
* external testmap
* cargo-vrc plan
* cargo-witness diff
* cargo-aer scan
* dynamic CI test runner
* batch mode

The dream agent action is:

```text
Given this patch, choose the minimal safe validation plan, explain skipped tests, run the selected tests, and return release risk.
```

That is much more powerful than `unit`, `integration`, or `lint`.

---

## 3.4 Agent dream: active `ProposePatch`

The deferred `ProposePatch` should eventually become real.

A good version would:

1. accept a patch or file modifications
2. create a branch
3. commit changes
4. run VTI selection
5. trigger validation
6. open or update MR
7. attach evidence
8. return branch, MR, pipeline, validation plan

Response should include:

```json
{
  "branch": "agent/fix-flaky-cache-verdict",
  "merge_request_iid": 17,
  "pipeline_id": 531,
  "validation_plan_id": 77,
  "risk": "trusted",
  "next_required_action": "wait_for_pipeline"
}
```

---

## 3.5 Agent dream: active `RacePatches`

`RacePatches` could be one of the coolest features in the system.

Imagine an agent can propose three hypotheses:

```text
Hypothesis A: fix cache taint pruning
Hypothesis B: adjust witness signature
Hypothesis C: bypass stale verdict path
```

`jeryu` would:

* create three branches
* run the same validation plan on each
* compare pass/fail, runtime, cache behavior, and code diff size
* return the best candidate with evidence

This is exactly the kind of task an agent would love.

But the key is safety:

* each hypothesis gets its own branch
* validation is bounded
* untrusted work goes through untrusted pool
* merge still requires risk gate

---

## 3.6 Agent dream: automatic failure repair loop

An ideal agent workflow:

```text
Fetch failed job
  -> fetch capsule
  -> fetch trace
  -> map failure to module owner
  -> inspect recent changes
  -> propose patch
  -> run focused validation
  -> open MR
  -> request risk-gated merge
```

The system already has almost all of the ingredients.

Missing capability intents could be:

```rust
AnalyzeFailure { project_id, job_id }
SuggestRepair { capsule_id }
SpawnRepairTask { project_id, job_id, trust_tier }
RunRepairValidation { branch_name, plan_kind }
```

The response should always include evidence and uncertainty.

---

## 3.7 Agent dream: safe operational controls with preflight

Agents should be able to request operational actions, but not blindly perform them.

Good examples:

```rust
PreflightAction {
    action: ScalePool { pool: "untrusted", count: 2 }
}

PreflightAction {
    action: ReconcileRelease { project_id: 2, ref_name: "main" }
}

PreflightAction {
    action: PromoteProduction { version: "ci-fa51a52a7882" }
}
```

Preflight response:

```text
Action: production promotion
Allowed: no
Reason: C validation artifact missing
Required evidence:
  ✓ canary e2e passed
  ✓ handoff artifact exists
  ✗ validation artifact missing
```

Then `ApplyAction` should only work when preflight passes.

---

## 3.8 Agent dream: access to proof-scoped tools through `jeryu`

The workspace tools are excellent:

* `cargo-witness`
* `cargo-vrc`
* `cargo-aer`
* `arc-bench`
* `witness-rt`

But they sit beside `jeryu`.

An agent would benefit from a unified surface:

```text
jeryu proof witness build
jeryu proof witness diff
jeryu proof vrc map
jeryu proof vrc plan
jeryu proof aer scan
jeryu proof diagnose
```

Or capability intents:

```rust
RunWitnessBuild
RunWitnessDiff
RunVrcPlan
RunAerScan
DiagnoseCompilerErrors
```

This would make structural validation part of the same control plane.

# 5. Dream TUI redesign: “JERYU Mission Control”

The current TUI is good as a foundation. The dream version should be a **high-contrast, command-driven, real-time operational cockpit**.

The design principle:

> Every visible object should answer three questions:
> What is happening?
> Why does it matter?
> What can I safely do next?

I would organize the dream TUI around four zones:

```text
┌─────────────────────────────────────────────────────────────────────────────┐

│               │                                               │ ACTIONS     │
```

# 16. Suggested architecture improvements for the TUI backend

To make the dream TUI possible, I would add a stronger TUI-facing model layer.

## 16.1 Create a unified `OperationalSnapshot`

Instead of separate snapshot pieces, define a higher-level model:

```rust
struct OperationalSnapshot {
    health: HealthSummary,
    release: ReleaseSummary,
    blockers: Vec<Blocker>,
    pipelines: Vec<PipelineFlow>,
    jobs: Vec<JobView>,
    pools: Vec<PoolView>,
    agents: Vec<AgentTaskView>,
    cache: CacheSummary,
    security: SecuritySummary,
    storage: StorageSummary,
    audit_tail: Vec<EventView>,
    generated_at: DateTime<Utc>,
}
```

The current `TuiStateSnapshot` is already close, but it is more data-bag than decision model.

The dream TUI needs precomputed:

* blockers
* actions
* warnings
* stale sources
* confidence
* gate checklists
* suggested next steps

## 16.2 Add `ActionDescriptor`

Every actionable thing in the TUI should be represented as data:

```rust
struct ActionDescriptor {
    id: String,
    label: String,
    target: ActionTarget,
    risk: ActionRisk,
    requires_confirmation: bool,
    preflight: Vec<PreflightCheck>,
    enabled: bool,
    disabled_reason: Option<String>,
}
```

Then the UI renders actions and the backend enforces them.

This prevents keybindings from becoming hidden logic.

## 16.3 Add `Blocker` as a first-class type

```rust
enum BlockerKind {
    FailedJob,
    PendingRunnerCapacity,
    MissingReleaseArtifact,
    VaultUnavailable,
    DiskPressure,
    MissingEvidenceCapsule,
    CacheTaint,
    AdmissionHookMissing,
    GitLabUnavailable,
}
```

A power console needs blockers as first-class data.

## 16.4 Add source freshness metadata

Every panel should know where its data came from:

```rust
struct SourceFreshness {
    source: String,
    generated_at: DateTime<Utc>,
    last_success_at: Option<DateTime<Utc>>,
    stale: bool,
    error: Option<String>,
}
```

This would make retained stale data safer.

# 17. Things I would be careful not to do

A few tempting ideas would be risky.

## Do not give agents broad shell access

The architecture is strongest when agents use typed actions, MRs, pipelines, and capability intents.

Broad shell access would bypass the safety model.

## Do not make production promotion a TUI hotkey without gate display

Even if there is a keybinding, it should be impossible to trigger without showing the full checklist.

## Do not hide stale data behind pretty visuals

The anti-blanking behavior is great, but retained data must be loudly marked stale.

## Do not over-animate

A terminal UI should feel alive, but not chaotic. Use motion sparingly:

* spinners for running jobs
* pulsing only for urgent blockers
* stable layout always

## Do not bury the boring health checks

GitLab, Docker, DB, Vault, webhook, hook, cache, disk, and runner capacity are the nervous system. They belong in the top strip.

# 20. Final take

This system has the bones of something exceptional.

The backend architecture already has many of the right primitives: typed controls, durable state, evidence capsules, release gates, trust-tiered execution, Vault, SmartCache, VTI, risk gates, agent workflows, and a TUI. The most important next step is to make the control surface **truthful, complete, and explainable**.

The biggest fixes are:

1. make capability responses precise,
2. expose audit data,
3. remove invisible-action ambiguity in the TUI,
4. show exact release/prod blockers,
5. make pipeline flow truly multi-pipeline,
6. add per-job evidence/cache/retry explanations,
7. make actions preflighted and visible.

The dream TUI is not just prettier. It is a **real-time, evidence-aware command console** where a developer can see every moving part of CI, release, cache, runners, agents, secrets, and production promotion — and take safe, preflighted action from one place.

That would be genuinely incredible.
