Absolutely — this is a genuinely exciting system. The architecture already has the hard parts that many CI/control-plane projects never reach: typed operational surfaces, release gates, test intelligence, runner orchestration, failure evidence, cache policy, Vault integration, and a TUI that is already trying to be more than a dashboard.

My strongest feedback is this:

**JeRyu / `jeryu` is close to becoming an agent-native delivery operating system, but today it still feels split between “rich orchestration exists internally” and “humans/agents can only see or operate parts of it.”** The biggest opportunity is to turn all the hidden intelligence — release gates, risk decisions, cache decisions, runner pressure, VTI plans, failure capsules, retries, secret state, agent tasks — into one coherent real-time command console.

Below is my best assessment.

---

# 1. Highest-level assessment

The strongest architectural decisions are excellent:

* **Production is pipeline-mediated**, not shell-mediated.
* **Agents are intended to operate through typed capabilities**, not broad arbitrary access.
* **RedlineDB acts as a durable operational ledger**, not just a cache.
* **Runner pools have trust tiers**, which is essential for agent-generated work.
* **Failure capsules and retry decisions create evidence**, which is much better than blind auto-retry.
* **VTI smart test selection and release impact analysis are first-class**, not bolted on.
* **The TUI already prioritizes stability under partial data**, especially with anti-blanking flow snapshots.

The biggest gap is that these pieces are not yet unified into a fully legible operational brain.

Right now, the system has a lot of powerful commands, but the operator still has to know which command to run, why a gate did or did not fire, which pipeline matters, which job is blocking, whether an agent can act safely, and whether a release is stuck because of test failure, missing artifact, missing telemetry, stale GitLab data, runner pressure, secret state, or policy escalation.

The dream version should answer one question instantly:

> **What is happening, what is blocked, why is it blocked, what can I safely do next, and what will happen if I do it?**

That should be the central design principle.

# 2. Critical issues and gaps I would address first

## P0 / P1: The capability API is too small for the system’s ambition

The Capability API is conceptually one of the most important surfaces, but it currently only has two truly active behaviors:

* `FetchCapsule`
* `RunTests`

The other major agent intents are deferred:

* `ProposePatch`
* `RacePatches`
* `RequestMerge`

That creates a mismatch. The system wants least-privilege typed agent control, but agents needing real work may still need to fall back to CLI, GitLab direct operations, or broad shell access.

That is dangerous because the best safety model here is not “give the agent a shell.” It is:

> **Give the agent many narrow, typed, audited capabilities.**

I would make Capability API v2 a major priority.

Suggested new active intents:

```rust
QuerySystemStatus
QueryPipeline
ExplainPipeline
ExplainRelease
ExplainJobFailure
ProposePatch
CreateMergeRequest
CommentOnMergeRequest
RunTestPlan
RunSelectedTests
RetryJob
CancelJob
CancelPipeline
RequestMerge
RequestReleaseReconcile
RequestProductionPromotion
ScalePool
PausePool
ResumePool
FetchJobTrace
FetchArtifact
SubscribeEvents
```

Every intent should include:

```json
{
  "agent_id": "...",
  "task_id": "...",
  "correlation_id": "...",
  "reason": "...",
  "dry_run": true,
  "approval_token": "optional-for-high-risk-actions"
}
```

The response should include not just success/failure, but also:

```json
{
  "success": true,
  "decision": "allow|deny|escalate|dry_run_only",
  "reason": "...",
  "risk": "low|medium|high|critical",
  "actions_taken": [],
  "next_suggested_actions": [],
  "links": {},
  "audit_event_id": 123
}
```

This would let agents act powerfully without becoming unsafe.

## P1: The TUI’s visible panes and internal pane state are inconsistent

The current TUI still has `Pools`, `Pipelines`, and `Jobs` pane state, but the active Flow layout visibly exposes only the Live Jobs list. Pool and pipeline renderers exist but are dead or inactive.

That creates UX confusion:

* `Tab` cycles through panes that are not all visible.
* `p` can pause/resume a pool, but the active layout does not show a pool pane.
* pipeline focus affects the Flow Board border, but there is no visible pipeline list.
* old renderers exist but are not part of the current command surface.

This is a very fixable gap. Either:

1. **Reintroduce visible Pools and Pipelines panels**, or
2. **Remove them from active navigation until they are visible**, or
3. **Convert them into a command palette / side inspector model.**

For a power-user TUI, I would not keep hidden pane concepts. Every focusable thing should be visible, searchable, or reachable from a command palette.

A real Audit tab would make this system much more trustworthy.

## P1: Release gates need visible “why not” explanations

Release automation is strong, but the operator needs to know exactly why something did or did not happen.

For example, `maybe_trigger_production_promotion(...)` returns `None` unless conditions pass. That is good internally, but in the UI it can feel like silence.

The TUI and CLI should expose a gate ledger like:

```text
Production Promotion Gate
  Canary state e2e-passed             PASS
  C handoff artifact exists           PASS
  C validation artifact exists        FAIL
  Existing prod pipeline absent       PASS
  Secret set finalized?               WAITING
  Decision                            BLOCKED
  Reason                              Missing validation artifact
```

Similarly for canary launch:

```text
Canary Launch Gate
  Ref is main                         PASS
  Latest successful pipeline          PASS
  Not release-execution pipeline      PASS
  Release eligible                    PASS
  Extended lane green                 FAIL
  Release-impacting change            PASS
  Atomic claim available              PASS
  Decision                            BLOCKED
  Reason                              Extended lane failed: integration-e2e
```

This should be persisted, not only calculated live. Add tables such as:

* `release_gate_evaluations`
* `release_gate_items`
* `promotion_gate_evaluations`
* `policy_decisions`

That would make the TUI and agents dramatically smarter.

The Flow Board currently renders the first active pipeline in `active_pipelines`.

* upstream pipeline

```text
main@abc123
```

The current single-pipeline board is a good foundation, but the dream console needs multi-pipeline release lineage.

## P1: Graph edges are not computed

`FlowGraph.edges` exists but is empty. That is a missed opportunity.

Edges matter because they let the UI show:

* critical path
* blocked-by relationships
* stage dependencies
* bridge/downstream relationships
* release lineage
* artifact dependency chains
* “this job failed because that prior job did not produce X”
* “prod is waiting on canary E2E evidence”

Without edges, the graph is a visual grouping. With edges, it becomes an operational map.

Even approximate edges would help:

* stage order edges
* `needs:` edges from GitLab CI YAML when available
* downstream bridge edges
* release attempt edges
* artifact gate edges
* retry edges
* supersedence edges

This would make the Flow Board far more powerful.

## P1: Job/lane classification by name is brittle

The Flow graph classifies columns and lanes using job-name matching:

* `test`, `unit`, `integration`
* `security`, `secret`, `guard`
* `canary`, `prod`, `deploy`
* etc.

That is useful as a fallback, but brittle as a primary mechanism.

A stronger design would allow CI jobs to declare metadata:

```yaml
variables:
  JERYU_LANE: "release-execution"
  JERYU_COLUMN: "canary"
  JERYU_REQUIRED: "true"
  JERYU_CRITICAL_PATH: "true"
  JERYU_RISK: "high"
```

Or through artifacts:

```json
{
  "jeryu": {
    "lane": "integration",
    "column": "tests",
    "required": true,
    "blocks_release": true,
    "produces": ["gate-canary-e2e.json"]
  }
}
```

Keep name inference as a fallback, but let important jobs self-identify.

## P1: Capability `RunTests` branch naming conflicts with `jeryu test run`

This is a subtle but important issue.

`jeryu test run` creates scratch branches named:

```text
jeryu-test-*
```

The Push Hook intentionally skips branches starting with `jeryu-test-*`.

But Capability API `RunTests` creates branches like:

```text
<target_ref>-ci-<timestamp_ms>
```

That means capability-triggered test branches may not get the same webhook skip behavior as CLI-triggered test branches.

I would align these immediately.

Use one scratch branch naming policy:

```text
jeryu-test-<target-ref>-<timestamp>
```

or:

```text
jeryu-agent-test-<task-id>-<timestamp>
```

Then make the Push Hook skip all known ephemeral prefixes:

```text
jeryu-test-*
jeryu-agent-test-*
jeryu-ci-scratch-*
```

Also add scratch branch TTL cleanup.

## P1: The Capability API request limit is too small for patch workflows

The current Unix socket read buffer caps payloads around 64 KiB. That is fine for `FetchCapsule`, but too small for realistic patch proposals, multiple file edits, or hypothesis racing.

For patch workflows, use one of these:

1. Chunked protocol.
2. Patch files written to a staging directory, with the capability request referencing a staged patch ID.
3. Git tree / branch operations where the agent submits structured file actions.
4. Streaming socket protocol.

A dream agent surface needs to support large diffs safely.

## P1: `FetchCapsule` should use indexed evidence state, not scan recent events

`FetchCapsule` currently scans the latest 500 event rows for `failure_capsule`.

But the state API already has:

* `evidence_capsules`
* `latest_evidence_for_job`

The capability should use the indexed evidence table. Scanning the event ledger can miss older but still relevant capsules, and it couples behavior to event ordering.

## P1: Webhook event ordering and deduplication need explicit treatment

The engine handles GitLab webhooks and updates state. But webhook systems can deliver:

* duplicate events
* delayed events
* out-of-order events
* events after manual cancellation
* pipeline status changes racing with reconciliation

The architecture likely already handles some of this implicitly, but I would make it explicit.

Add event identity and monotonic handling:

* GitLab event UUID if available
* project ID
* pipeline ID
* job ID
* status
* commit SHA
* observed timestamp
* source timestamp
* previous known state
* transition validity

Persist webhook receipts separately:

```text
webhook_receipts
webhook_event_transitions
```

Then the TUI can show:

```text
Last webhook: 4s ago
Webhook lag: 1.2s
Dropped duplicate events: 12
Out-of-order transitions ignored: 2
```

That would be phenomenal for operations.

## P1: `/cache/summary` is unauthenticated and the server binds to `0.0.0.0`

`GET /cache/summary` has no auth. That may be fine on a fully local trusted network, but since the engine binds to:

```text
0.0.0.0:9777
```

it deserves a second look.

The endpoint may only expose cache metrics, but in a security-conscious system, even operational metadata can be sensitive.

Options:

* bind engine to localhost by default
* require token for non-health endpoints
* split public health from private operational APIs
* expose `/health` unauthenticated, but gate `/cache/summary`
* allow config override for trusted LAN operation

## P1: The custom executor remains a major security boundary

The custom executor is powerful and high risk. It has:

* sandbox creation
* honeypots
* taint handling
* cache decisions
* BuildKit integration
* Docker/Rust witnesses
* failure capsules

That is great, but the docs also say it may assume Docker socket access for some flows. That is always a major risk, especially for untrusted work.

I would harden this area continuously:

* rootless BuildKit where possible
* explicit network egress policy
* per-trust-tier filesystem mounts
* seccomp/AppArmor profiles
* no long-lived broad tokens in executor environment
* job-scoped credentials only
* strict artifact boundaries
* tripwire alerts surfaced in TUI
* cache promotion disabled by default for untrusted jobs unless policy explicitly allows
* visible “why cache hit was accepted” evidence

The TUI should make executor security visible, not hidden.

## P2: The system is too project-2 / `dougx` centered

A lot of release behavior assumes:

```text
DEFAULT_RELEASE_PROJECT_ID = 2
default release repo root = /home/ubuntu/dougx
```

That is okay for the current target, but the architecture wants to be broader.

I would add a project registry:

```text
projects
  id
  gitlab_project_id
  name
  repo_root
  default_ref
  release_enabled
  secrets_repo
  testmap_path
  trust_policy
```

Then commands can still default to project 2, but the TUI and engine can reason about multiple projects cleanly.

## P2: The general TUI snapshot worker may do expensive work too frequently

The general snapshot worker runs about every 1500ms and refreshes many things, including filesystem sizes.

Recursive size checks for CAS, crate cache, runner dirs, and DB size can become expensive. Docker state checks can also become expensive. Over time, this can make the TUI feel heavy.

I would split refresh rates:

```text
Fast:      jobs, logs, pipeline states            500ms-1500ms
Medium:    pools, release status, cache metrics   2s-5s
Slow:      disk scans, recursive dir sizes         15s-60s
Manual:    expensive Docker storage details        on demand
```

The TUI should display freshness per panel:

```text
Jobs: fresh 0.8s
Storage: fresh 42s
Cache: fresh 3s
Release: fresh 2s
```

That makes approximations acceptable.

## P2: The release and production path needs rollback visibility

The release path is strong on promotion, but I do not see a first-class rollback path in the docs.

A production-grade console should show:

* current production version
* previous production version
* rollback eligibility
* rollback artifact availability
* rollback secret compatibility
* rollback pipeline trigger
* canary rollback status
* post-rollback validation

Even if rollback is implemented in `dougx`, `jeryu` should be able to surface and mediate it.

## P2: Scratch branch lifecycle needs cleanup visibility

`jeryu test run` cleans up scratch branches “when possible.” Capability tests create ephemeral branches too. Agents may create many branches.

Add:

```text
jeryu test scratch-list
```


## P2: TUI log polling should support incremental fetch or smarter caching

Polling every 650ms is workable, but full trace fetching can become expensive.

Improvements:

* use GitLab trace range requests if available
* keep per-job trace offsets
* cache log chunks
* show log fetch latency
* show dropped/failed polls
* allow “pin logs” for multiple jobs
* support side-by-side log compare for retries

The current 160 KB tail cap is sensible, but the UI should show when truncation occurred:

```text
Log truncated to last 160 KB. Press F to fetch full artifact.
```

# 3. What an agent would dream of being able to do

A great agent does not want raw power. It wants safe, typed, inspectable leverage.

The dream agent loop is:

```text
Observe -> Explain -> Plan -> Act -> Verify -> Learn -> Report
```

Today, `jeryu` has many pieces of that loop. The opportunity is to make it explicit.

* gate states
* active blockers
* failed jobs
* failure capsules
* flaky tests
* runner pressure
* cache health

# 4. Issues that may need to be addressed

Here is a sharper list of specific issues I would investigate.

## API / architecture consistency issues

### 1. `jeryu init` docs are slightly inconsistent around Vault

The API lifecycle section says `jeryu init` starts GitLab, but the architecture says it starts GitLab CE and Vault containers.

This should be reconciled. Since Vault is central to secrets, the lifecycle docs should be precise.

### 2. `jeryu sync-now` is a placeholder

The command exists but tells users to ensure `jeryu serve` is running. That is surprising for a command named `sync-now`.

Either implement it or rename it to something like:

```text
jeryu jeryu request-sync
```

Better: add a DB-backed sync request queue that the daemon consumes.

### 3. The Capability API has no session model

A one-request/one-response Unix socket API is simple, but agents will want:

* sessions
* subscriptions
* streaming logs
* action progress
* cancellation
* large payload handling
* retries
* correlation IDs

A v2 protocol should support both request/response and streaming.

### 4. Socket security should be explicit

The docs do not mention socket permissions or peer credential checks.

For a capability API, this is crucial.

Recommendations:

* place sockets under `data_dir()/run`, not arbitrary `/tmp`, by default
* chmod socket directory to `0700`
* verify Unix peer credentials where possible
* require an agent identity envelope
* audit every request

### 5. CLI JSON outputs should have stable schemas

Many commands support `--json`, but the docs do not mention schema versions.

Every machine-facing JSON output should include:

```json
{
  "schema_version": "jeryu.release.status.v1",
  "generated_at": "...",
  "data": {}
}
```

This would help agents enormously.

### 6. GitLab client is intentionally narrow, but agent workflows may need more MR visibility

The GitLab client supports MRs, jobs, pipelines, files, etc. For the dream system, it may also need:

* MR pipelines
* MR discussions
* approval state
* changed files
* diff refs
* pipeline variables
* environments/deployments
* releases
* protected branches
* job artifact metadata

Keep it typed, but expand where needed.

---

## State/data issues

### 1. Event ledger needs first-class event taxonomy

The `events` table is central, but the docs do not show a formal event taxonomy.

I would define event families:

```text
pipeline.*
job.*
release.*
production.*
agent.*
pool.*
cache.*
secret.*
admission.*
executor.*
host.*
jeryu.*
tui.*
```

Then every event should have:

```text
event_type
entity_type
entity_id
project_id
ref_name
sha
actor_type
actor_id
correlation_id
severity
summary
payload_json
created_at
```

This makes the Audit tab, agent reasoning, and incident reconstruction much easier.

### 2. Need action request/result tables

The system has commands and events, but I would add explicit action tracking:

```text
action_requests
action_results
action_approvals
```

This would allow the TUI to show:

```text
Action Queue
  retry job #123              succeeded
  scale pool build to 4        running
  promote prod ci-abc123       blocked: missing validation artifact
```

### 3. Need release gate evaluation tables

As mentioned above, release gates should be persisted.

This is one of the most valuable improvements.

### 4. Need stale-data tracking

The TUI already marks Flow stale. Extend this concept everywhere:

* GitLab state stale
* Docker state stale
* Vault state stale
* release evidence stale
* cache metrics stale
* logs stale
* jeryu sync stale

A dream console makes freshness obvious.

---

## Execution/security issues

### 1. Docker socket access in custom executor flows is dangerous

Even with tripwires, Docker socket access can be equivalent to host-level control. Keep tightening this area.

### 2. Cache promotion from untrusted work needs maximum clarity

The system has taint and trust boundaries, but the UI should show:

```text
Cache decision: cold execution
Reason: untrusted job cannot promote exact-hit artifact
Witness: rust-build sha256:...
Taint: none
Epoch: rust-1.92.0
```

Cache correctness is a supply-chain security topic here, not just a performance topic.

### 3. Honeypot/tripwire events should be highly visible

A tripwire breach should light up the TUI. It should not be buried in logs.

Dream display:

```text
SECURITY EVENT
  Job: untrusted/test-sandbox
  Evidence: honeypot token touched
  Action: cache quarantine + job failure + escalation
  Time: 12s ago
```

### 4. Vault state should be integrated into release readiness

Secrets are release-critical. The TUI should show whether the latest release has:

* secret set created
* handoff files generated
* Vault refs valid
* finalization pending
* recovery report available
* stale/expired rotation

# 5. The dream agent control surface

The agent dream is not “let me do everything.” It is:

> **Let me understand everything, propose safe actions, execute narrow actions, and produce evidence.**

I would design the agent capability model around these action classes.

## Class A: Read-only capabilities

Always safe, no approval:

```text
QueryStatus
QueryPipeline
QueryRelease
QueryJob
FetchCapsule
FetchTraceTail
ExplainFailure
ExplainPipeline
ExplainRelease
ExplainRiskGate
PlanTests
PlanValidation
GetAuditEvents
```

## Class B: Low-risk write capabilities

Allowed for trusted agents, audited:

```text
CreateBranch
CommitPatchToAgentBranch
CreateMergeRequest
CommentOnIssue
CommentOnMergeRequest
RunFocusedTest
RunTestPlan
RetryFailedJobOnce
```

## Class C: Operational capabilities

Need policy checks:

```text
CancelPipeline
ScalePool
PausePool
ResumePool
DrainPool
RunCacheGc
RunHostDoctor
RunReleaseReconcile
```

## Class D: High-risk capabilities

Require explicit human approval or strong gate evidence:

```text
AcceptMergeRequest
RotateSecrets
FinalizeSecrets
RequestProductionPromotion
HostReclaimApply
RotatePoolToken
ClearTaint
```

## Class E: Forbidden direct capabilities

Agents should not get these as raw powers:

```text
Arbitrary shell on host
Raw GitLab PAT
Raw Vault token
Direct production deploy command
Direct database write
Direct Docker socket access
```

This is exactly the kind of safety model `jeryu` seems built for.

# 12. Dream color system

Use color as semantic language.

## Core status colors

```text
Green       success / healthy / eligible
Blue/Cyan   running / live / streaming
Yellow      waiting / pending / attention
Orange      degraded / slow / nearing threshold
Red         failed / blocked / critical
Magenta     security / policy / trust escalation
Purple      release / canary / production
Gray        stale / inactive / unknown
White       normal foreground
Black       selected text on bright background
```

## Use bold and background sparingly

High contrast matters more than lots of colors.

Examples:

```text
[BLOCKED]   red background, white bold
[RUNNING]   blue/cyan foreground
[WAITING]   yellow foreground
[SECURITY]  magenta background
[STALE]     gray foreground
[PROD]      purple bold
```

## Do not rely on color alone

Every important state should also have:

* icon
* text
* reason
* age

Example:

```text
✗ BLOCKED  missing validation artifact  age 2m13s
● RUNNING  canary deploy                 ETA 1m20s
○ WAITING  prod promotion                waiting for e2e
⚠ DEGRADED cache mirror                  last healthy 4m ago
```

# 24. Real-time architecture improvements for the TUI

The TUI currently has multiple polling loops and channel merge semantics. That is workable, but a dream console should have an event spine.

I would add an internal event bus from `engine.rs`:

```text
/job-events
/pipeline-events
/release-events
/cache-events
/security-events
/host-events
/log-events
```

Transport options:

* Unix socket stream
* local websocket
* SSE
* tokio broadcast channel for in-process TUI
* DB tailing fallback

The TUI can still poll as fallback, but primary updates should be event-driven.

The dream architecture:

```text
engine event bus
  -> TUI live subscriptions
  -> capability SubscribeEvents
  -> audit ledger
  -> optional notification integrations
```

This would make the TUI feel instant and would reduce duplicated polling logic.

# 25. Information architecture: separate “state” from “actions”

A common TUI problem is mixing display logic with operational logic.

I would create a command/action registry:

```rust
Command {
    id,
    label,
    description,
    applies_to,
    risk_level,
    dry_run,
    required_permission,
    execute,
    explain,
}
```

Then the TUI command palette, CLI commands, and capability server can share the same action definitions.

That would prevent drift like:

* CLI has command
* TUI has partial action
* capability has deferred intent
* docs mention behavior but implementation differs

A unified action registry would be very powerful.

The dream version of `jeryu tui` is not just a TUI.

It is a **live operational cockpit for a local autonomous delivery system**.

It should feel like this:

* The top bar tells me whether the system is healthy.
* The main graph tells me where code is in the path to production.
* The blocker banner tells me why we are not shipping.
* The inspector tells me what the selected thing means.
