I reviewed the repo through its GitHub source/docs and raw code. I wasn’t able to do a local `git clone` inside the container because DNS resolution to GitHub failed, so I did **not** compile or run the binary. I still reviewed the core docs, source layout, TUI docs, architecture docs, VTI docs, capability layer, engine, agent flow, executor/sandbox, cache/security primitives, and API surface.

My headline take: **JeRyu / `jeryu` is not literally a GitLab fork yet; it is a Rust control plane that orchestrates a local GitLab/GitLab Runner/Docker/Vault stack.** That is not a bad thing. In fact, it may be the better architecture for an agent-first Git platform, because GitLab remains the substrate while `jeryu` owns policy, runners, evidence, CI decisions, agent capabilities, cache, release gates, and the TUI. The README and architecture docs describe exactly that: a single-binary Rust control plane over local GitLab, with runner pools, webhooks, custom executor policy, release/canary automation, SmartCache, VTI, Vault secrets, evidence, settings, and TUI support. ([GitHub][1])

## Executive verdict

This repo has the **right ambition** and a lot of the right primitives. The best thing about it is that it already thinks in the correct agent-native units: **capabilities, evidence, intent, runner pools, cache trust, test impact, replayable state, release gates, and operator visibility**. The project’s own mission doc is also refreshingly honest: infrastructure and agent substrate are more real than the decision layer, while semantic CI, supersedence, evidence-driven gating, policy authority, and causal explainability are still the big opportunity. ([GitHub][2])

The main gap is that the system currently feels like a **powerful orchestration/control plane with early intelligence scaffolding**, not yet a fully autonomous proof-carrying software factory. To become the world’s best agent-first Git option, the next leap should be:

> Every agent action should become an auditable chain: **Intent → Plan → Capability Grant → Execution → Evidence Capsule → Risk Decision → Merge/Deploy/Reject → Explanation**.

That chain should be machine-readable, streamable, replayable, policy-bound, and visible in the TUI.

# What it already does well

## 1. The project has a clear agent-first mission

The mission is not just “run CI faster.” It explicitly frames the problem as replacing brittle, human-babysat Git workflows with an agent-governed Rust control plane where Git is the ledger, GitLab is the substrate, Rust enforces policy/safety, and agents act within capability bounds. The mission doc also identifies the right pain points: superseded pipelines, brittle selective testing, flaky infra, coarse retry/resume, poor explainability, and broad human/API tokens. ([GitHub][2])

That is exactly the right thesis. GitHub/GitLab today are still mostly designed around humans clicking buttons and CI systems reacting to commits. Agents need something more like an **autonomy OS for code**.

## 2. The module surface is broad and ambitious

The root `src` layout includes modules for admission, agents, agent surface, cache, cache brain, cache proxy, capability, decision, dispatch, engine, exec, explain, GitLab client, honeypot, impact, policy, pool, release, sandbox, secrets, jeryu, state, taint, telemetry, test runner, witness, plus gateway/test intelligence/TUI submodules. ([GitHub][3])

That breadth matters. A real agent-first Git platform cannot be “just an MCP server,” “just a CI wrapper,” or “just a bot.” It needs to control the whole loop: repo state, CI state, test selection, evidence, capability boundaries, merge gates, release gates, and human visibility.

## 3. It has a serious CLI/API surface already

The docs expose lifecycle commands, pool/job/pipeline/cache commands, agent commands, jeryu sync, VTI commands, release commands, secrets, progress, next-action, explain, action registry, repo audit, host ops, hidden executor commands, webhook APIs, and capability intents. The capability API includes intents like `ProposePatch`, `RacePatches`, `RunTests`, `FetchCapsule`, `RequestMerge`, `ExplainBlockers`, `GetSystemSnapshot`, `ListAllowedActions`, and `PlanValidation`. ([GitHub][4])

That is a strong foundation. The key improvement is to make this surface more formal, safer, streaming, and eventually agent-consumable through REST/gRPC/MCP-style protocols.

## 4. The project has the right proof/validation culture

The repo includes proof-scoped crates like `cargo-witness`, `cargo-vrc`, `cargo-aer`, `witness-rt`, and `arc-bench`, plus AGENTS guidance around proof lanes, module ownership, guardrails, diagnostics, and security-sensitive modules. ([GitHub][5])

That is excellent. The best agent-first systems will be built around **evidence**, not vibes. This repo already has the vocabulary for that.

## 5. The TUI is real, not an afterthought

The TUI is built with Ratatui/crossterm and has a documented architecture: polling loops, a flow board, release watch, live jobs, logs, cache/storage/audit/tests tabs, action side effects, stale-state handling, and capture/smoke commands. Current docs describe tabs for Mission/Release/Jobs/Agents/Tests/Pools/Cache/Evidence/Secrets, with a newer Flow-focused screen documented as the most important view. ([GitHub][6])

This is a big deal. For autonomous coding, the TUI should become the “mission control” where a human can see not only *what* is happening, but *why the agent believes it is safe*.

## 6. VTI / test intelligence is pointed in the right direction

The VTI docs describe smart test selection from changed files to minimal tests, internal/external planners, generated plans, GitLab CI runtime interception, cache keys, nightly oracle behavior, miss detection, learning suggestions, explanations, and fallback-to-full behavior. ([GitHub][7])

The implementation is conservative, which is good. The planner falls back to full runs for empty diffs, global invalidators, unknown subsystems, low confidence, and force-full subsystems. ([GitHub][8])

That is the right safety posture for agent-led coding: accelerate only when evidence is strong, default to full validation when confidence is weak.

## 7. The state/evidence model is a strong base

The architecture docs describe RedlineDB primary with RedlineDB fallback, DB schema groups for pools/managers/jobs/pipelines/releases/cache/evidence/agents/policies/jeryu/progress/test intelligence, and invariants around append-heavy events and typed DB access. ([GitHub][9])

For agents, this is critical. A good agent platform needs a **durable memory of what happened**, not just transient logs. Every flaky test, retry decision, failed release, selector miss, cache taint, and merge gate should become part of the system’s future intelligence.

## 8. Cache trust and tainting are the right primitives

`CacheBrain` checks action-cache validity, epoch, taint state, and trust namespace before returning exact hits or denials. The taint manager can propagate taints through dependent cache verdicts. ([GitHub][10])

This is exactly the kind of thing agentic CI needs. Agents will generate more branches, more experiments, and more speculative work than humans. Without cache trust and taint propagation, the platform will either become slow or unsafe.

## 9. The project has early but valuable security tripwires

The executor/honeypot layer seeds fake AWS and npm credentials, watches for touches, kills the job process, and writes a quarantine marker. ([GitHub][11])

That is not sufficient isolation by itself, but it is a clever “detonation lane” primitive. It should evolve into a broader deception and egress-detection system.

## 10. It already thinks about agent identities

The agent code creates ephemeral project bots, branches, issues, MRs, pipelines, and race branches. The README and API also talk about ephemeral Matrix/GitLab identities and constrained agent control. ([GitHub][12])

This is directionally excellent. The next step is making every identity/grant narrowly scoped, auditable, short-lived, and tied to exact refs, SHAs, and allowed actions.

# Obvious gaps and inconsistencies

## 1. It is not actually a GitLab fork

The repo positions `jeryu` as a Rust control plane for local GitLab, not as a modified GitLab codebase. The architecture docs explicitly describe GitLab/GitLab Runner/Docker/Vault/RedlineDB/RedlineDB as components driven by the Rust binary. ([GitHub][9])

That is fine, but the product language should be precise:

> “Agent-first GitLab control plane” is accurate.
> “GitLab fork” is not currently accurate.

If the ambition is to become a true GitLab/GitHub alternative, this should eventually grow its own repo browser, API, permission model, code search, review UI, issue/MR UX, and hosted multi-tenant model. But the current control-plane-over-GitLab architecture is probably the best starting point.

## 2. The decision layer is still too heuristic

`decision.rs` has useful enums and policies, but failure classification is mostly string matching over kind/summary/log snippets; retry logic is simple; merge gate evaluation is based on summarized counts, selector misses, cache taints, trust tier, and evidence-policy booleans. ([GitHub][13])

That is okay for a first version, but not enough for full autonomy. The platform needs a richer decision engine that consumes actual evidence graphs: test history, dependency impact, coverage, branch protection, approvals, cache provenance, runtime traces, flake probability, release blast radius, and security policy.

## 3. Supersedence is currently cancel-only

The decision model includes `Cancel`, `Preserve`, `Degrade`, and `Ignore`, but the engine’s supersedence handler cancels older pending/running/created pipelines for the same ref. ([GitHub][13])

For agent-first Git, this is one of the biggest opportunities. Supersedence should not mean “kill old CI.” It should mean:

* reuse unchanged job results,
* preserve still-valid artifacts,
* degrade obsolete long-running jobs,
* resume partial DAGs,
* transfer evidence from old commits when proven safe,
* keep expensive tests alive if their input set did not change,
* cancel only truly obsolete work.

## 4. Capability `RequestMerge` is advisory and underpowered

The capability layer’s `RequestMerge` path checks selector misses and cache taints, but then evaluates merge risk with hardcoded successful/pending/failed job counts, hardcoded trust tier, and evidence policy fields set false. It returns a proof but does not actually merge. ([GitHub][14])

This is a major gap for autonomous coding. If agents are expected to request merges, the system must query the real MR, real head SHA, real required checks, real approvals, real pipeline state, real branch protection, real VTI selector misses, real cache taints, real security findings, and real release risk.

## 5. Agent race support is not complete enough yet

The capability layer can create racing patches and trigger pipelines, and the agent module can detect race winners in some cases, but the code still has TODO-style behavior around listing agents and fully merging winners/cleaning losers. `list_agents` currently returns an empty list. ([GitHub][12])

Patch racing is one of the highest-potential features here. It should become a flagship workflow, not a partial scaffold.

## 6. The sandbox claims are stronger than the implementation

The `sandbox.rs` implementation says strict network isolation currently launches `bash` with environment variables and relies on detonation lanes/proxy egress, while Docker pools provide container-level isolation. It explicitly notes that full namespace isolation via `bwrap`/`unshare` would require additional networking setup. ([GitHub][15])

For an agent-first system, this is the most important security gap. Agents will run untrusted generated code. “Strict mode” must actually mean strict isolation: namespaces, cgroups, seccomp, no host Docker socket, no ambient credentials, controlled DNS, controlled egress, read-only mounts where possible, process-tree killing, and durable forensics.

## 7. There appears to be a webhook mismatch

The engine has push-hook behavior for supersedence, impact planning, and VTI planning, but the GitLab client’s group webhook creation payload sets `push_events: false`. ([GitHub][16])

Maybe another path creates project push hooks, but as written this is suspicious. If push events are not enabled, the core “on push, cancel superseded pipelines and plan impact/VTI” behavior will not fire reliably.

## 8. The capability socket needs stronger access control

The capability server binds a Unix socket, reads up to 64 KiB, deserializes an `AgentIntent`, and handles it. I did not see strong peer credential verification, framing, per-agent auth, or permission hardening in the reviewed path. ([GitHub][14])

For local dev this is fine. For serious agent autonomy, every capability call needs authenticated identity, grant lookup, nonce/replay protection, audit event, policy decision, and preferably signed intent/proof records.

## 9. GitLab client TLS behavior should be dev-only

The GitLab client uses a reqwest client configured with `danger_accept_invalid_certs(true)`. ([GitHub][17])

That is understandable for local GitLab bootstrapping, but it should be gated behind an explicit dev/self-signed setting and made impossible in production mode.

## 10. VTI is still mostly path/subsystem heuristic

The internal planner maps changed paths to subsystems/jobs and uses confidence/fallback rules. That is a good start, but it is not yet a semantic impact engine. ([GitHub][8])

The dream version needs AST/build graph awareness, coverage mapping, historical failure correlation, mutation-test confidence, branch-protection mapping, and required-check synthesis.

## 11. The test runner has some rough edges

The dynamic test runner creates scratch branches, commits `.gitlab-ci.yml`, triggers pipelines, cancels spurious branch-create pipelines, waits for results, deletes branches, and records executions. ([GitHub][18])

That is useful, but generated CI needs careful YAML escaping, command safety, artifact capture, structured result extraction, and exact duration accounting. In the reviewed code, result duration appears tied to queued duration in at least one path, which may confuse performance analysis.

## 12. The TUI is good but still mostly observational

The TUI docs list current limitations: logs are polled instead of streamed, old pool/pipeline lists are inactive, Flow renders the first active pipeline, graph edges are empty, ETA is heuristic, storage is approximate, Audit is placeholder-like, job inspector is inactive, and mouse capture exists but the app is keyboard-driven. ([GitHub][6])

For the world’s best agent-first Git platform, the TUI must become an **interactive command center**, not just a monitor.

# The biggest product leap: proof-carrying autonomy

The key upgrade is to make the platform think in **proof packs**.

Every important action should produce a signed, replayable, machine-readable proof pack:

```text
intent.json
plan.json
capability-grant.json
diff.patch
impact-graph.json
selected-tests.json
skipped-tests.json
cache-provenance.json
sandbox-manifest.json
job-results.json
logs.sarif / junit.xml / traces.otlp
risk-decision.json
merge-decision.json
release-decision.json
human-readable-explanation.md
```

The proof pack answers:

* Who/what requested this?
* What was changed?
* What could be affected?
* What tests were run?
* What tests were skipped, and why?
* What cache was reused, and why was it trusted?
* What failed?
* What was flaky?
* What policy allowed or denied the next step?
* What exact command reproduces the result?
* What changed between previous failed and current passing evidence?
* Why is this safe to merge or deploy?

This would make JeRyu meaningfully different from GitHub/GitLab.

# Best improvements by area

## Agent features

### 1. Add a real Agent Work Queue

Create first-class tasks with:

* task ID,
* owner agent,
* branch,
* issue/MR,
* goal,
* constraints,
* budget,
* allowed tools,
* required evidence,
* current hypothesis,
* blockers,
* retries,
* final proof pack.

Agents should not just spawn branches. They should operate against durable task objects.

### 2. Implement full patch racing

Make `RacePatches` a flagship feature:

1. Generate N hypotheses.
2. Create N isolated branches/workspaces.
3. Run impact-specific tests.
4. Compare evidence.
5. Pick winner using deterministic policy.
6. Merge or request approval.
7. Delete/quarantine losers.
8. Preserve all evidence.
9. Show race tree in TUI.

This is one of the clearest “agent-first” features GitHub/GitLab do not have.

### 3. Add agent leases and file/semantic locks

Agents need coordination. Add:

* file locks,
* symbol locks,
* subsystem locks,
* MR dependency edges,
* conflict prediction,
* stale branch detection,
* merge-train awareness,
* “do not touch” policy zones.

The system should prevent ten agents from thrashing the same module.

### 4. Add agent memory, but make it evidence-bound

Store memories like:

* “this test flakes on runner pool X,”
* “this module requires integration test Y,”
* “this dependency often breaks cache,”
* “this reviewer rejected pattern Z,”
* “this release gate failed because metric Q regressed.”

But every memory should link to evidence, not just free-form notes.

### 5. Add “agent-readable repo context capsules”

Before an agent starts coding, it should fetch a compressed capsule:

* repo map,
* module ownership,
* coding conventions,
* active incidents,
* flaky tests,
* recent failures,
* relevant prior patches,
* allowed tools,
* branch protection,
* known risk zones,
* commands likely needed.

This would make agents faster and less destructive.

## CI and test intelligence

### 6. Build a true semantic impact graph

Current VTI is path/subsystem-based. Upgrade it to combine:

* file path rules,
* Cargo crate graph,
* Rust module graph,
* function/symbol graph,
* test coverage,
* changed public APIs,
* dependency lockfile changes,
* build script changes,
* generated code,
* migration/schema changes,
* previous selector misses,
* historical failure correlation.

Then produce a confidence score and a human-readable explanation.

### 7. Required-check synthesis

The system should answer:

> “Given this diff and branch protection, what is the minimum evidence needed to merge?”

Not just “what tests should run,” but:

* which checks are required by policy,
* which are required by branch protection,
* which are required by changed files,
* which are required by risk,
* which are optional,
* which can be skipped,
* which previous results can be reused.

### 8. Nightly oracle as a learning engine

The VTI doc already has nightly oracle concepts. Make it central:

* Every skipped test eventually gets sampled.
* Selector misses become training data.
* Confidence is automatically adjusted.
* Policy can say “this subsystem has had two misses this week; force full.”
* Agents can see why test selection changed.

### 9. Failure classifier 2.0

Move from string heuristics to structured classification:

* compiler error,
* type error,
* lint,
* unit regression,
* integration regression,
* snapshot mismatch,
* dependency outage,
* runner failure,
* resource exhaustion,
* flake,
* secret/policy failure,
* sandbox violation,
* network/registry failure,
* cache corruption,
* release/canary failure.

Attach suggested next action:

* retry,
* rerun on clean runner,
* quarantine runner,
* rerun full suite,
* ask agent to patch,
* escalate to human,
* block merge,
* rollback release.

### 10. Test economics

Agents need to reason about cost. Add:

* expected test duration,
* expected confidence gain,
* flake probability,
* historical failure probability,
* cache hit probability,
* queue delay,
* dollar/token/time cost.

Then the agent can choose the best next validation step.

## Supersedence and pipeline performance

### 11. Replace cancel-only supersedence with DAG reuse

The current engine cancels older pipelines. The dream system should classify each job in an older pipeline:

* obsolete: cancel,
* reusable: keep and attach evidence to new commit,
* still useful: let finish,
* degraded: stop logs/artifacts but preserve result,
* resumable: restart from checkpoint,
* duplicate: dedupe with singleflight.

### 12. Pipeline time travel

For every new commit, compute:

> “What evidence from previous commits is still valid?”

Use:

* input digests,
* dependency graph,
* test selection graph,
* cache manifests,
* environment digest,
* commit diff,
* toolchain digest.

If unchanged, do not rerun. Reattach proof.

### 13. Runner smart routing

The runner pool should route jobs based on:

* job type,
* expected duration,
* required isolation,
* trust tier,
* flake history,
* cache locality,
* GPU/CPU/memory needs,
* runner health,
* agent priority.

Expose this in the TUI as “why this job ran there.”

### 14. Predictive prewarming

Before an agent commits, the system can infer likely tests and prewarm:

* containers,
* dependencies,
* cargo registry,
* sccache,
* buildkit layers,
* branch workspace,
* test database.

This would make the platform feel magical.

## Security

### 15. Make sandboxing real

Priority one: implement real isolation.

For untrusted/agent code:

* rootless container or microVM,
* user namespaces,
* mount namespaces,
* PID namespace,
* network namespace,
* seccomp profile,
* cgroup limits,
* read-only host mounts,
* no Docker socket,
* controlled DNS,
* explicit egress allowlist,
* process-group kill,
* syscall/audit logs,
* artifact quarantine.

The current honeypot system is clever, but honeypots should be additive, not the primary isolation boundary.

### 16. Capability grants should be exact and tiny

Every grant should specify:

* agent identity,
* allowed action,
* repo/project,
* branch/ref,
* exact SHA if possible,
* allowed file globs,
* allowed commands,
* allowed network hosts,
* allowed token scopes,
* expiration,
* max cost,
* required evidence,
* whether human approval is required.

No broad “api + write_repository” token unless absolutely necessary.

### 17. Protected branch admission should be default-deny

The admission hook should enforce:

* only known capability grants can update agent refs,
* no direct protected branch writes,
* MR merge only through gate,
* no secret-touching diffs without special policy,
* no CI config changes without elevated validation,
* no dependency lockfile changes without supply-chain checks,
* no generated code mass change without extra review.

### 18. Signed evidence and provenance

Use Sigstore/in-toto/SLSA-style attestations:

* signed job result,
* signed artifact manifest,
* signed cache hit,
* signed test selection proof,
* signed merge decision,
* signed release decision.

Agents should not be able to fake “tests passed.”

### 19. Secrets should become evented, scoped, and revocable

Vault integration is a good start. Next:

* one secret lease per job,
* automatic revocation,
* no plaintext in logs,
* per-command secret access,
* honey secrets,
* secret egress detection,
* redaction verification,
* secret access appears in proof pack.

### 20. Add supply-chain policy

Agents often modify dependencies. Add:

* dependency diff risk,
* license policy,
* vulnerability scan,
* maintainer reputation,
* package age,
* typosquat detection,
* lockfile provenance,
* registry allowlist,
* “new transitive dependency” review.

# Dream TUI behavior

The current TUI foundation is good, but the world-class version should feel like **mission control for autonomous software engineering**.

## 1. Flow graph as the home screen

Instead of only a list of jobs, show a live DAG:

```text
Commit → Impact Plan → Selected Tests → Jobs → Evidence → Merge Gate → Release Gate
```

Each node has:

* status,
* confidence,
* duration,
* cost,
* owner agent,
* blockers,
* reusable evidence,
* “why” explanation.

Edges should show causality: “this file caused this test,” “this cache taint blocked this job,” “this failure blocked this merge.”

## 2. Agent command center

A dedicated Agents view should show:

* all active agents,
* their task,
* branch,
* MR,
* current hypothesis,
* last action,
* next planned action,
* tokens/grants held,
* budget spent,
* confidence,
* blockers,
* risk tier.

For a selected agent, show a timeline:

```text
Plan created
Capability granted
Patch proposed
Tests selected
Pipeline started
Failure classified
Patch revised
Merge requested
Gate denied: missing integration evidence
```

## 3. “Why can’t this merge?” panel

For every MR:

```text
Merge decision: DENY

Blockers:
- integration_tests pending
- cache artifact cargo-build-abc tainted by failed job 812
- VTI skipped db_migration_tests but schema.rs changed
- agent token lacks protected-branch merge capability

Next best action:
Run db_migration_tests and regenerate impact proof.
```

This should be one keystroke from anywhere.

## 4. Evidence capsule viewer

For a job/test/failure:

* root cause summary,
* failing lines,
* related previous failures,
* flake probability,
* exact repro command,
* environment digest,
* artifact links,
* logs,
* trace,
* screenshots if available,
* agent-readable JSON.

The TUI should let a human copy a repro command or hand the capsule to an agent.

## 5. Test impact view

For a commit/MR:

```text
Changed file              Reason              Selected tests        Confidence
src/cache_brain.rs        cache subsystem     cache::*, taint::*    0.91
src/policy.rs             trust policy        policy::*, gate::*    0.88
docs/README.md            docs-only           skipped              0.99
```

Also show skipped tests and why they are safe to skip.

## 6. Supersedence savings view

Show:

* canceled obsolete jobs,
* reused jobs,
* preserved artifacts,
* minutes saved,
* dollars saved,
* confidence of reuse.

This becomes a killer demo.

## 7. Security cockpit

Show live:

* active capability grants,
* expiring tokens,
* sandbox violations,
* honeypot touches,
* secret accesses,
* egress attempts,
* quarantined jobs,
* untrusted cache entries,
* protected branch attempts.

Make it obvious when an agent is operating safely.

## 8. Cache anatomy view

For any cache hit/miss:

```text
Cache decision: DENIED
Reason: namespace untrusted cannot satisfy trusted job
Input digest: ...
Toolchain digest: ...
Epoch: valid
Taint: none
Producer: agent-17
```

This is crucial for trust.

## 9. Action palette with dry-run

A user should press `:` or `Ctrl-P` and type:

* `rerun failed`
* `merge if safe`
* `quarantine runner`
* `explain blocker`
* `spawn race`
* `rollback prod`
* `run full validation`
* `show proof`

Every action should preview exact side effects before execution.

## 10. Human/agent paired mode

The TUI should support a mode where the agent asks for approval inline:

```text
Agent proposes:
- Modify src/cache_brain.rs
- Run cache + policy tests
- Request merge if green

Approve? [y] [n] [edit policy] [ask why]
```

This would be much better than agents acting invisibly through opaque GitHub/GitLab comments.

# The highest-leverage “epic” features

If I were trying to make this the world’s #1 agent-first Git option, I would build these as headline features.

## 1. `jeryu why`

Ask why anything happened:

```bash
jeryu why job 812 failed
jeryu why mr 44 cannot merge
jeryu why tests were skipped
jeryu why cache was denied
jeryu why pipeline 91 was canceled
jeryu why release was blocked
```

Agents and humans both need this.

## 2. `jeryu next`

The system should always know the best next action:

```text
Next best action:
Run integration_tests::db because schema.rs changed and VTI confidence is 0.62.
Estimated time: 4m 20s.
Merge gate will likely pass after this evidence.
```

## 3. `jeryu race`

Parallel hypotheses should be native:

```bash
jeryu race "fix flaky cache invalidation" --agents 5 --budget 30m --merge-if-safe
```

This is exactly where agents can outperform humans.

## 4. `jeryu proof`

Every merge/release should have a proof pack:

```bash
jeryu proof mr 123 --format json
jeryu proof release prod-2026-04-27 --open
```

This becomes the trust layer.

## 5. `jeryu replay`

Every failure should be reproducible:

```bash
jeryu replay job 812
jeryu replay capsule cap_abc123 --sandbox
```

Agents need reproducibility more than humans do.

## 6. `jeryu autopilot`

A controlled autonomy mode:

```bash
jeryu autopilot --repo . --max-risk medium --budget 2h --require-human-for prod
```

Autopilot should operate inside explicit policy, not free-form.

## 7. `jeryu simulate merge`

Before merging:

```bash
jeryu simulate merge mr 123
```

Output:

* likely conflicts,
* required checks,
* affected services,
* stale evidence,
* release risk,
* dependent branches,
* cache invalidations.

## 8. `jeryu safe-run`

A universal agent-safe command runner:

```bash
jeryu safe-run "cargo test -p cache" --trust untrusted --egress none
```

This should be the only way untrusted agents execute code.

# Final recommendation

The repo should lean into this identity:

> **JeRyu is not another Git host. It is an agentic software delivery control plane.**

The strongest current pieces are the Rust control plane, state ledger, GitLab orchestration, runner pool management, VTI scaffolding, cache trust/taint concepts, evidence capsules, capability intents, and TUI. The weakest pieces are the incomplete decision layer, incomplete merge/capability enforcement, shallow sandboxing, heuristic impact selection, partial patch-race lifecycle, and TUI limitations.

My recommended strategic direction:

1. **Make security real first.** Real sandboxing, real capability grants, real admission enforcement.
2. **Make decisions evidence-based.** No hardcoded merge trust; everything reads from proof.
3. **Make the TUI the command center.** Every object gets “why,” “next,” “proof,” and “risk.”
4. **Make patch racing the killer feature.** This is the most agent-native workflow.
5. **Make VTI semantic and self-learning.** Test less, but prove why it is safe.
6. **Make every merge/deploy proof-carrying.** That is the trust breakthrough.

The repo already has the bones of something genuinely different. The next level is turning the scaffolding into a **proof-carrying, policy-bound, real-time autonomous coding platform** where agents can move fast because every action is constrained, observable, reproducible, and explainable.
