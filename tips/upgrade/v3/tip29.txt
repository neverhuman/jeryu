I reviewed it closely. The exciting part is that **JeRyu/jeryu is already aimed at exactly the right missing layer**: not “AI writes YAML,” but an agent-governed delivery system that decides what work matters, proves why, and gives agents bounded authority. One important framing correction: this is **not really a GitLab fork yet** in the normal sense. It is a **Rust control plane around a local GitLab substrate**: GitLab CE, GitLab Runner, Docker, Vault, RedlineDB/RedlineDB, custom executor hooks, a TUI, cache intelligence, VTI test selection, and agent capability APIs. The README calls it a single-binary Rust control plane for local GitLab, and the architecture doc says GitLab is one part of a broader agent-operable delivery system. ([GitHub][1])

I could not complete a normal `git clone` from the execution container because DNS resolution for GitHub failed in that environment. I did, however, inspect the repo through GitHub and raw source views, including the README, architecture/API docs, mission doc, TUI doc, `AGENTS.md`, `Cargo.toml`, and key source modules such as `agent.rs`, `admission.rs`, `engine.rs`, and `gitlab_client.rs`.

## Executive verdict

**This is a very strong foundation, but it is currently more “agent-first GitLab control plane” than “world’s #1 autonomous Git platform.”** The strongest parts are infrastructure control, runner orchestration, custom executor control, event/state ledgers, cache/test-intelligence scaffolding, release gates, and TUI observability. The biggest missing piece is the actual autonomous decision/proof kernel: the thing that safely decides what to run, what to skip, what to reuse, what to retry, what to quarantine, what an agent may do, and what must escalate.

The repo’s own mission doc is impressively honest about that: it says the infrastructure and agent substrate are “mostly real,” while the decision-making layer is only partially scaffolded and the intelligent CI control plane is the main opportunity. It also explicitly calls out missing commit supersedence with partial DAG reuse, finished semantic impact analysis, flake memory, structured causal explanations, and policy-driven agent authority. ([GitHub][2])

## What it already does well

**1. The product thesis is excellent.**
The best idea in the repo is the framing: Git remains the ledger, GitLab is the first substrate, Rust owns orchestration/policy/safety, and agents operate inside explicit capability bounds. That is exactly the right conceptual split. GitHub/GitLab are still primarily human-first systems with automation bolted on; JeRyu is trying to make the agent a first-class actor. ([GitHub][2])

**2. It has a surprisingly complete control-plane map.**
The architecture is not just README fantasy. The system is split into interface, control, state, execution, and intelligence planes. It has modules for admission control, agent workflow, custom executor, sandboxing, cache brain, taint tracking, release automation, secrets, state, telemetry, test intelligence, TUI, and witness generation. ([GitHub][3])

**3. The runner/custom-executor direction is a big differentiator.**
GitHub Actions and GitLab CI are painful for agents because the actual execution substrate is opaque. JeRyu goes after that directly: it has runner pool management, custom executor prepare/run/cleanup commands, sandbox preparation, honeypot tripwires, cache decisions, witness computation, and failure capsule capture. That is the right primitive for autonomous coding because the agent needs to understand and control the execution environment, not just wait for a web UI status bubble. ([GitHub][4])

**4. The state ledger is the right backbone.**
The state API tracks pools, managers, jobs, pipelines, release attempts, evidence capsules, retry decisions, cache objects, cache taints, test executions, test plans, selector misses, and more. This is exactly what an agent-first Git system needs: a durable memory of decisions, evidence, and execution context. ([GitHub][4])

**5. The capability API is the right shape, even if incomplete.**
The Unix-socket JSON API exposes agent intents like `ProposePatch`, `RacePatches`, `RunTests`, `FetchCapsule`, `RequestMerge`, `ExplainBlockers`, `GetSystemSnapshot`, `ListAllowedActions`, and `PlanValidation`. That is very promising. Agents should not be scraping logs or clicking UI; they should operate through typed capabilities with durable grants and audit trails. ([GitHub][4])

**6. The action registry is a very good design choice.**
A single source of truth for actions, risk tiers, surfaces, grants, dry-run status, and side effects is exactly how agent tools should be exposed. The API doc already defines risk tiers such as ReadOnly, Low, High, and Production, and maps actions across CLI, TUI, and capability surfaces. ([GitHub][4])

**7. Ephemeral bot identity is a strong agent-auth primitive.**
The agent code provisions project bot identities with scoped permissions, creates tracking issues, creates branches, and uses labels like `agent:pending` and `agent:running`. The GitLab client creates project access tokens and logs bot user IDs rather than tokens. That is much better than handing every agent one shared god token. ([GitHub][5])

**8. The proof-scoped workspace tooling is unusually agent-friendly.**
`AGENTS.md` defines proof routing, validation lanes, module ownership, witness tooling, VRC mapping, AER structural scans, and required proof commands. That is excellent. Most repos force agents to infer validation strategy from tribal knowledge; this repo is trying to make validation machine-readable. ([GitHub][6])

**9. VTI/test intelligence is the right strategic battlefield.**
The architecture includes VTI smart test selection, subsystem-based planning, `.jeryu/testmap.toml`, CI child-pipeline YAML generation, deterministic test cache keys, nightly selector audit, learning, and selector-miss persistence. Selective testing is one of the places where autonomous coding lives or dies, so this is a great area to keep pushing. ([GitHub][3])

**10. The TUI foundation is strong.**
The TUI already has mission, release, jobs, agents, tests, pools, cache, evidence, and secrets tabs; a flow board; live jobs; log preview; screenshot capture; background workers; live log polling; and state hydration. This is exactly the right interface for supervising autonomous systems: a command center, not a CRUD dashboard. ([GitHub][7])

## Obvious gaps and risks

**1. The “GitLab fork” claim should be tightened.**
The repo is GitLab-first, but GitLab is explicitly described as the substrate, not the product itself. That is good architecture, but it means the positioning should be “agent-native Git/CI control plane built on GitLab first,” not “fork of GitLab,” unless the project actually intends to vendor/modify GitLab itself. ([GitHub][2])

**2. There appears to be a hard compile-breaking typo in `engine.rs`.**
In the release failure path, the code creates a `note` string and then calls `finish_release_canary(..., Some(¬e), ...)`. That `¬e` token looks like a paste/corruption typo and should fail compilation. This is a top-priority fix because it sits in the engine’s release pipeline handling. ([GitHub][8])

**3. TLS verification is disabled in the GitLab client.**
`GitlabClient::new` builds a reqwest client with `.danger_accept_invalid_certs(true)`. That may be acceptable for a disposable local GitLab dev stack, but it should be impossible in production unless an explicit local/dev setting is enabled. This matters a lot for an agent-control plane because agents will be moving code, secrets, release gates, and tokens through this client. ([GitHub][9])

**4. Admission control is too permissive by default for a world-class agent system.**
The docs say agent refs without matching ledger proof are audit-only by default, and `JERYU_ADMISSION_ENFORCE=1` is required to deny them. For local experimentation that is friendly; for agent-first production, the default should be the opposite: deny unproven agent writes unless a repo is explicitly in learning/audit mode. ([GitHub][4])

**5. Merge Request webhooks are logged but not acted on.**
That is a huge missing area. In an agent-first Git system, MR events should be central: review, proof freshness, required evidence, risk gates, stale branch detection, human comments, agent rebuttals, patch suggestions, approval policy, and merge readiness. The API docs currently say MR hooks are logged with no active behavior yet. ([GitHub][4])

**6. The merge gate is still advisory/conservative.**
`RequestMerge` returns a proof record with blockers, selector-miss count, cache-taint count, branch identity, and decision, but the docs say the current merge proof is conservative/advisory and the roadmap moves it into the shared merge gate. That shared merge gate should become the product’s beating heart. ([GitHub][4])

**7. The autonomous agent loop is mostly scaffolded.**
`agent.rs` describes the dream loop: create branch, perform task, commit, open MR, watch CI, fix failures, force-push, and merge or flag for review. The inspected spawn path creates the identity, issue, and branch, and race support can fan out hypotheses, but the true coding/debugging loop needs to become an explicit state machine with patch attempts, proofs, failure analysis, retries, and authority checks. ([GitHub][5])

**8. Sandbox security defaults need to be more aggressive.**
The settings schema shows `sandbox.strict_network_isolation` defaulting to `false`. For an agent-first system, especially one expected to run untrusted or semi-trusted code, strict network isolation should default to true, with explicit allowlists for package registries, GitLab, cache gateways, and release endpoints. ([GitHub][4])

**9. Some core files look too large/centralized.**
The repo already recognizes mega-file risk through `cargo-aer`, and `AGENTS.md` says all errors currently use `anyhow` with typed errors as a future migration. For safety-critical areas like admission, release, state, secrets, and sandboxing, typed errors and smaller bounded modules will pay off quickly. ([GitHub][6])

**10. The current API surface is good but not yet “agent-native enough.”**
A Unix socket with one JSON payload per response is a start, but serious autonomous agents need streaming logs, event subscriptions, structured diffs, long-running operation handles, cancellation, leases, grants, proof artifacts, and machine-readable schemas. The existing capability API is a great seed, but it should evolve into a full agent protocol. ([GitHub][4])

## Highest-impact improvements

### P0: correctness and security hardening

1. **Fix the `Some(¬e)` engine typo immediately**, then run `cargo check --workspace --message-format=json`, `cargo nextest`, and release-path integration tests.

2. **Make TLS behavior environment-aware.**
   Default to strict TLS. Allow invalid certs only when `settings.gitlab.allow_invalid_certs = true` or `JERYU_DEV_INSECURE_TLS=1`, and show a giant TUI warning when enabled.

3. **Flip admission defaults for production.**
   Make audit-only mode explicit: `JERYU_ADMISSION_MODE=audit`. Default should be `deny` for agent refs without a matching unexpired grant tied to exact ref and SHA.

4. **Set `sandbox.strict_network_isolation` to true by default.**
   Provide explicit network profiles: `no-network`, `package-readonly`, `gitlab-only`, `release-egress`, and `debug-open`.

5. **Add test cases for the most dangerous paths.**
   Required tests: release failure update, admission DB unavailable, grant expired, grant wrong SHA, capability branch write, revoked bot token, failed canary, production promotion blocked, honeypot tripwire, cache taint.

6. **Move security/release errors from `anyhow` to typed errors.**
   Keep `anyhow` at the CLI boundary, but not inside policy-critical internals. Agents need to distinguish “permission denied,” “proof missing,” “GitLab unavailable,” “sandbox breach,” and “release gate failed.”

### P1: build the agent proof kernel

The system needs one canonical lifecycle:

**Intent → Plan → Grant → Patch → Validate → Evidence → Merge/Reject → Release/Replay**

Every agent action should create a durable record with:

* agent identity
* task prompt/summary
* repo/ref/SHA
* permission grant
* files touched
* patch bundle
* tests selected
* tests skipped and why
* artifacts reused and why
* cache taint status
* failure capsules
* human approvals
* final decision
* rollback/replay instructions

This should be queryable from CLI, TUI, and API. The agent should never merely “do work.” It should produce a **proof-carrying change**.

### P2: make “merge proof” the central product object

GitHub and GitLab give you many little green checks. JeRyu should give you one canonical object:

**`jeryu/merge-proof`**

It should answer:

* Is this branch safe to merge?
* Which evidence proves it?
* Which tests ran?
* Which tests were skipped?
* Why were they skipped?
* What evidence was reused?
* What changed since the evidence was produced?
* Are there selector misses?
* Is cache tainted?
* Are secrets involved?
* Is the agent authorized?
* Is human approval required?
* What is the blast radius?
* What rollback exists?

This lets branch protection depend on one meaningful proof instead of dozens of brittle CI jobs.

### P3: turn VTI into a real Impact Graph

The current VTI direction is good, but to become best-in-world it needs to go beyond path globs and subsystem maps. Build a live **Impact Graph**:

* files → symbols
* symbols → modules
* modules → packages
* packages → tests
* tests → CI jobs
* CI jobs → release gates
* services → runtime dependencies
* ownership → approval policy
* historical failures → flake confidence
* prior fixes → suggested remediation

For Rust, use `cargo metadata`, `rustdoc JSON`, `syn`, public API witness graphs, call/dependency approximations, feature flags, and test history. For other languages, support plugins. The goal is not “minimal tests.” The goal is **minimal safe proof**.

### P4: make failure capsules world-class

A failure capsule should not just be a log snippet. It should include:

* failure kind: compiler, unit test, integration, infra, timeout, OOM, network, sandbox, secret, release gate
* exact command
* exact environment/toolchain
* dependency/cache signature
* changed files likely involved
* first failing frame or assertion
* relevant log excerpt
* reproducer command
* whether it is new, flaky, or known
* prior similar failures
* recommended next action
* confidence score
* safe retry policy
* whether an agent may attempt a fix

Agents should be able to ask: **“What is the smallest useful next action?”**

### P5: add a real flake/reliability memory layer

Autonomous releases cannot trust binary red/green. Add durable per-test and per-job reliability:

* historical pass/fail rate
* recent volatility
* infra vs product failure classification
* quarantine state
* retry budget
* owner
* last known good SHA
* correlated failures
* “confidence green” vs “technical green”
* “confidence red” vs “probably transient”

This unlocks intelligent retry, quarantine, and escalation instead of blind reruns.

## Dream features for the world’s #1 agent-first Git system

### 1. Agent Run Ledger

Every autonomous task gets a canonical run page/object:

```text
Agent: fix-auth-timeout-17
Goal: Reduce auth test flake in OAuth refresh path
Branch: agent/fix-auth-timeout-17
Authority: write_repository, run_tests(unit,integration), no_secrets
Budget: 45 min, 8 CPU-hours, $2.00 model/tool budget
Current state: validating hypothesis 2 of 4
Confidence: 78%
Next action: run targeted integration test auth_refresh_retry
Blocked by: none
```

This should be available in TUI, CLI, and API.

### 2. Parallel Hypothesis Racing done all the way

The repo already has the right instinct with `RacePatches`. Take it further:

* spawn N patch hypotheses
* run minimal tests on all
* kill losers early
* promote the first safe candidate
* compare diff complexity
* prefer smaller safer patch
* preserve useful failed hypotheses as evidence
* show a “why this patch won” explanation

This is agent-native development. Humans do not work this way naturally; agents can.

### 3. Proof-preserving cache reuse

Cache hits should not be “trust me.” They should be proof-bearing:

* input Merkle root
* toolchain fingerprint
* env fingerprint
* dependency lock hash
* sandbox profile
* trust namespace
* taint status
* prior job/test proof
* expiration
* promotion policy

If a result is reused, the merge proof should say exactly why reuse was safe.

### 4. Required-check virtualization

Instead of trying to satisfy branch protection with many noisy checks, JeRyu should publish a single final check:

```text
jeryu/merge-proof: PASS
- Unit: reused 812, ran 42, skipped 0 unsafe
- Integration: ran 11 impacted, skipped 97 with proof
- Security: required, passed
- Cache taint: none
- Selector misses last 30d: 0
- Agent authority: valid
- Human approval: not required
```

If branch protection wants individual check names, JeRyu can synthesize them, but internally the proof object should be canonical.

### 5. Agent-aware code review

A normal MR review says “looks good.” An agent-first review should say:

* what changed semantically
* what risk class it belongs to
* which invariants it touches
* which tests are necessary
* which tests were skipped
* whether the patch is simpler than alternatives
* whether generated code/docs are stale
* whether secrets or release gates are implicated
* whether an agent is allowed to merge it

### 6. Repository memory for agents

Add a persistent “repo brain” that is not just embeddings:

* symbol graph
* ownership graph
* test graph
* failure graph
* release graph
* API change history
* prior bug/fix pairs
* “why this file exists”
* “what breaks when this changes”
* “which humans/agents usually own this area”

The repo already has `agent-index` and proof tools; this is the next leap.

### 7. Policy-as-code for autonomous authority

Agents need explicit authority boundaries:

```toml
[agent.tiers.untrusted]
can_write_refs = ["refs/heads/agent/*"]
can_run_tests = ["unit", "lint"]
can_read_secrets = false
can_merge = false
network = "package-readonly"

[agent.tiers.trusted]
can_write_refs = ["refs/heads/agent/*", "refs/heads/jeryu/*"]
can_run_tests = ["unit", "integration", "e2e"]
can_merge = "low_risk_only"
requires_human_for = ["prod", "secrets", "schema", "auth"]

[agent.tiers.release]
can_trigger_canary = true
can_trigger_prod = false
requires_threshold_approval = true
```

Use OPA/Rego, Datalog, or a Rust-native policy DSL, but make it inspectable and enforced everywhere: CLI, TUI, capability API, hooks, release automation.

### 8. Cross-repo and release-train intelligence

Agent-first Git cannot stop at one repo. Add:

* multi-repo change bundles
* dependency impact propagation
* atomic release trains
* downstream breakage prediction
* shared evidence across repos
* “this library change requires these service tests”
* rollback plans across repos

### 9. Secure package/diff detonation

For suspicious dependency changes or agent-generated code:

* run in no-network sandbox
* compare syscall/network/file behavior
* inject honeytokens
* watch for exfiltration
* quarantine cache on breach
* require human approval for suspicious artifacts

The repo already has honeypot/taint concepts; make them central to supply-chain safety.

### 10. Agent-native protocols

Add:

* MCP server
* JSON-RPC/gRPC streaming API
* OpenAPI for admin/webhook surfaces
* event subscription API
* operation handles
* structured log streams
* proof artifact download
* cancellation/rollback endpoints
* typed schemas for every decision

A serious coding agent should never need to scrape terminal output.

## Performance upgrades

1. **Make every expensive thing content-addressed.**
   Build outputs, test results, dependency downloads, Docker layers, generated CI YAML, failure capsules, and proof artifacts should all be keyed by material inputs.

2. **Use singleflight everywhere.**
   The repo already has a `singleflight` cache gateway module. Extend that concept to package downloads, Git object fetches, test planning, log hydration, and repeated impact calculations. ([GitHub][3])

3. **Predict warm pools.**
   Use job history and branch activity to pre-warm runner pools before agents launch heavy validation.

4. **Cancel obsolete work instantly.**
   A 300-second reconciliation loop is fine for maintenance, but supersedence decisions should happen within seconds via event-driven paths.

5. **Make test planning incremental.**
   Cache impact graph results by diff base/head, toolchain, package graph, and test map version.

6. **Expose “CI minutes saved” as a first-class metric.**
   Users will love seeing: “jeryu saved 42 CPU-hours today by canceling superseded jobs and reusing safe evidence.”

## Security upgrades

1. **No long-lived tokens for agents.**
   Use OIDC/SPIRE-style workload identity or short-lived Vault dynamic credentials. PATs should be last resort and extremely scoped.

2. **Per-tool capability broker.**
   Do not give an agent a shell and hope. Let it request: read file, propose patch, run test, fetch capsule, request merge. Each tool action gets a policy decision.

3. **Default egress deny.**
   Allow only GitLab, Vault, package mirrors, cache gateways, and explicitly approved endpoints.

4. **Rootless isolation by default.**
   Use rootless containers, user namespaces, seccomp, AppArmor, cgroups, no-new-privileges, read-only mounts, and per-job tmpfs where possible.

5. **Provenance for releases.**
   Add SLSA/in-toto style attestations, SBOM generation, Sigstore/cosign signing, and immutable release evidence bundles.

6. **Secret redaction by construction.**
   Logs, capsules, traces, TUI panes, and API responses should all pass through one redaction layer. Honeytokens should prove the layer works.

7. **Cache quarantine must be aggressive.**
   If an untrusted job trips a honeypot or violates policy, quarantine its cache namespace and every artifact lineage descendant.

## Product positioning

I would position this as:

> **jeryu is the agent-native Git/CI control plane: a local-first Rust system that turns GitLab into a bounded, explainable, autonomous software delivery substrate.**

Not “better GitLab UI.” Not “AI CI wrapper.” Not “agent writes YAML.” The killer promise is:

> **Every autonomous code change carries its own proof: what changed, what ran, what was skipped, what was reused, who/what approved it, and why it is safe.**

That is the north star.

## Final take

This repo has unusually good instincts. The runner fleet, custom executor, state ledger, VTI direction, capability API, action registry, release gates, and TUI are all pointing toward something much more interesting than a GitLab clone. The path to “world’s #1 agent-first Git option” is to make the **decision/proof layer** absolutely central: every agent action bounded, every CI decision explainable, every skipped test justified, every cache reuse proven, every merge gated by evidence, and every release reversible.

The magic sentence should be:

> **JeRyu lets agents code autonomously because it can prove what they did, why it was safe, and when they must stop.**
