I reviewed the current source with the scope narrowed to **API surfaces + TUI**. My strongest conclusion: **JeRyu already has the skeleton of a serious agent operations cockpit, but its current TUI is still a dashboard, not a command center.** The biggest upgrade is not just prettier panels. It needs a dedicated **TUI/API view-model layer**, streaming events, structured action execution, better entity models, and a redesigned cockpit around “what is blocked, why, what changed, what should I do next?”

## Current API surface, as implemented

JeRyu currently exposes four main “API” surfaces:

1. **CLI API** through `jeryu` subcommands.
2. **Engine HTTP API** for webhooks/health/cache summary.
3. **Capability Unix socket API** for structured agent intents.
4. **Action registry** that partially unifies CLI/TUI/capability actions.

The CLI surface is broad. It includes `init`, `serve`, `tui`, `down`, `status`, pool/job/pipeline/cache/log/agent/jeryu/test/release/secrets/progress/repo/host commands, hidden custom executor commands, hidden server-hook commands, hidden capability commands, `next`, `explain-blocker`, and `action list`. The TUI command supports `--once`, `--capture`, `--tab`, output path, width, and height, which is useful for deterministic UI smoke tests and screenshots. ([GitHub][1])

The test CLI is one of the more agent-relevant areas: it supports single test runs, planning, batch parallel runs, pipeline results, retrying failed jobs, showing failed traces, impact analysis, smart test selection, explanation of plans, external workspace selection, selector audit/learning, and test cache status. ([GitHub][1])

The release/secrets/host/repo CLI surface is also fairly rich: release status/watch/reconcile/promote/preflight/doctor, Vault-backed secret init/status/rotate/finalize/report/recover, host storage/doctor/reclaim, and repo agent-surface/index commands. ([GitHub][1])

The engine HTTP API is much smaller. The router currently exposes `/health`, `/hooks`, and `/cache/summary`; `/hooks` handles GitLab webhook events, while `/cache/summary` requires `X-Jeryu-Token` and returns bytes served, hits, object count, and health status. ([GitHub][2])

The capability API is a Unix domain socket server that accepts a tagged JSON `AgentIntent`. Supported intents include `ProposePatch`, `RacePatches`, `RunTests`, `FetchCapsule`, `RequestMerge`, `ExplainBlockers`, `GetSystemSnapshot`, `ListAllowedActions`, and `PlanValidation`. The server reads one request into a fixed 65,536-byte buffer and returns a `CapabilityResponse { success, message, data }`. ([GitHub][3])

The action registry is one of the best current pieces. It defines risk tiers, surfaces, side-effect classes, grant requirements, action metadata, dry-run flags, descriptions, and JSON contracts. It is explicitly consumed by the TUI command palette, `jeryu action list`, and capability `ListAllowedActions`. Registered actions include open logs, retry job, delete local record, pause pool, explain blockers, system snapshot, propose patch, race patches, request merge, plan validation, run tests, next action, tab navigation, audit toggle, and quit. ([GitHub][4])

## Current TUI, as implemented

The TUI has nine tabs: **Mission, Release, Jobs, Agents, Tests, Pools, Cache, Evidence, Secrets**. The top-level draw path renders a two-line header/tab bar, main content, a bottom event console, and footer; maximized logs replace the main content with a full log view. ([GitHub][5])

The app state pulls together pools, GitLab readiness, active containers, recent jobs, tracked pipelines, flow snapshots, live logs, jeryu summary, cache metrics, proxy/registry health, taint/breach counts, storage breakdown, release status, test bottlenecks, evidence capsules, secret audit events, agent pipelines, and recent audit events. It runs background sync loops, including a flow collector, live log polling around every 650ms, and broader snapshot refresh around every 1500ms. ([GitHub][5])

The header shows GitLab status, active container count, active/total pools, release state, stale-data warning, and numbered tabs. The event console currently emphasizes recent evidence capsules. Keyboard help includes `Ctrl-K` for palette, tab cycling, number-key tab switching, job log opening, retry, delete, pause, and quit. ([GitHub][6])

The Mission tab is a 2x2 health overview: system health, pipeline summary, release status, and cache health. The Release tab is a gate matrix with main green/release identity/release pipeline/canary/E2E/prod rows and a right-hand inspector. The Jobs tab has a pipeline nav, release banner, flow board, jobs list, live logs, and job inspector. ([GitHub][6])

The Agents tab currently shows “agent pipelines” and a capability-call history filtered from audit events. It does not yet look like an agent task/work-order cockpit; it is closer to a pipeline/audit dashboard for branches that look agent-related. ([GitHub][6])

The Tests tab shows bottlenecks by average/latest duration and lets the user load execution history. The Pools tab shows runner pools and allows pause/resume. The Cache dashboard shows storage overview, gateway health, singleflight analytics, and trust/taint boundaries. ([GitHub][6])

The Evidence tab has two modes: capsule list/detail and audit ledger. Capsule detail includes job, ref, SHA, stage, exit code, failure kind, causal links like superseded/retry-of when present, and a log snippet. The Secrets tab shows secret audit events and a Vault-status panel. ([GitHub][6])

The flow model is promising. It has pipeline flows, graph columns, nodes, edges, ETA estimates, required/critical-path flags, backend refs, and column/lane enums such as Commit, Admission, Impact, Pipeline, Build, Tests, Security, Package, Gates, Canary, Production, Unit, Integration, Security, Research, and ReleaseExecution. ([GitHub][7])

The flow graph builder classifies jobs into columns/lanes by name, creates stable node IDs from pipeline ID + job name, estimates progress/ETA, creates stage-ordering edges, and computes a critical path over the generated DAG. ([GitHub][8])

The flow widget draws columns, lane groups, icons, selected nodes, and critical-path badges using text graphics. Status icons include success/running/failed/pending/canceled style markers, and critical nodes get `[CRIT]`. ([GitHub][9])

## The biggest thing missing

The TUI is currently **stateful, useful, and broad**, but it is not yet **decision-oriented**.

It tells me:

> “Here are jobs, pipelines, releases, cache stats, evidence, secrets, and agent-ish branches.”

It should tell me:

> “Here is the engineering mission. These agents are working. This exact thing is blocked. This is why. This is the next safest action. This action will cost X, save Y, affect Z, require approval Q, and produce evidence P.”

That shift requires both API and UI redesign.

# API redesign: build a real TUI/agent control API

## 1. Create a dedicated TUI Backend API

Right now the TUI directly combines DB, Docker, GitLab, release state, cache metrics, and logs. That is fine for a prototype, but it makes the UI hard to evolve. Create a dedicated **TUI Backend-for-Frontend** layer.

Recommended endpoints:

```text
GET  /v1/tui/snapshot
GET  /v1/tui/stream
GET  /v1/tui/entities/{kind}/{id}
POST /v1/actions/invoke
GET  /v1/actions
GET  /v1/actions/{id}/schema
GET  /v1/actions/{id}/preview
```

The TUI should not have to know how to query GitLab jobs, local DB records, Docker containers, release attempts, cache metrics, and evidence capsules separately. It should consume a coherent model:

```json
{
  "generated_at": "...",
  "freshness": {
    "state_age_ms": 400,
    "gitlab_age_ms": 1200,
    "log_age_ms": 200
  },
  "missions": [],
  "agents": [],
  "pipelines": [],
  "work_orders": [],
  "events": [],
  "alerts": [],
  "recommended_actions": []
}
```

## 2. Replace polling-first UI with event streaming

The TUI currently redraws frequently and has background polling/sync loops. That works, but it is not world-class realtime. Keep polling as a fallback, but make the primary transport event-driven.

Use one of:

```text
/v1/tui/stream        Server-Sent Events
/v1/tui/ws            WebSocket
Unix socket stream    for local-only mode
```

Events should be typed:

```json
{
  "event_id": "evt_...",
  "seq": 48102,
  "type": "job.status_changed",
  "entity": {"kind": "job", "id": "job:2:9981"},
  "severity": "info",
  "timestamp": "...",
  "patch": [
    {"op": "replace", "path": "/jobs/job:2:9981/status", "value": "failed"}
  ],
  "recommended_actions": ["open_failure_capsule", "retry_clean_runner"]
}
```

The TUI should animate deltas:

```text
job moved pending → running
runner acquired
capsule generated
merge proof invalidated
policy changed
secret lease opened
cache tainted
```

## 3. Unify IDs across API/TUI

Everything should have a stable typed entity ID:

```text
repo:{id}
branch:{project_id}:{ref}
commit:{sha}
pipeline:{project_id}:{pipeline_id}
job:{project_id}:{job_id}
agent:{agent_id}
workorder:{id}
mr:{project_id}:{iid}
capsule:{id}
action:{id}
grant:{id}
secret_lease:{id}
release:{version}
```

Then every screen, action, log line, capsule, and event can deep-link to the same object. The current TUI uses separate selected indices for jobs/tests/evidence and sometimes reuses selection state in places where a typed selected entity would be cleaner. Replace that with:

```rust
enum SelectedEntity {
    Mission(String),
    Agent(String),
    WorkOrder(String),
    Pipeline { project_id: i64, pipeline_id: i64 },
    Job { project_id: i64, job_id: i64 },
    Evidence(String),
    SecretLease(String),
    CacheObject(String),
}
```

## 4. Redesign the action API around preview → approve → execute → observe

The action registry is good, but actions need a richer lifecycle.

Current registry fields are good:

```text
id
label
key_hint
risk_tier
surfaces
dry_run
description
side_effect_class
required_grant
```

Add:

```text
input_schema
output_schema
preview_schema
confirmation_policy
undo_policy
idempotency_required
expected_duration
expected_cost
requires_selected_entity
danger_reasons
success_event_types
failure_event_types
```

Action lifecycle:

```text
discovered
→ parameterized
→ previewed
→ policy_checked
→ confirmed / denied
→ executing
→ succeeded / failed / rolled_back
→ evidence_attached
```

Example:

```json
POST /v1/actions/invoke
{
  "action_id": "retry_job",
  "selected_entity": "job:2:9981",
  "idempotency_key": "ui-...",
  "mode": "preview"
}
```

Preview response:

```json
{
  "allowed": true,
  "risk_tier": "low",
  "summary": "Retry failed job test-unit-core on a clean runner",
  "will_touch": ["pipeline:2:812", "job:2:9981"],
  "will_create": ["job:new"],
  "estimated_cost": {"runner_minutes": 3.5},
  "required_grants": [],
  "confirm": false
}
```

For dangerous actions:

```json
{
  "allowed": false,
  "blocked_by": [
    {
      "kind": "policy",
      "message": "Merge proof is stale: MR head SHA changed after pipeline success."
    }
  ],
  "next_actions": ["rerun_required_checks", "generate_merge_proof"]
}
```

## 5. Make “why blocked?” structured, not string-based

The capability API already has `ExplainBlockers`, but it returns mostly string explanations. Make blocker explanations first-class objects:

```json
{
  "entity": "mr:2:42",
  "blocked": true,
  "blockers": [
    {
      "id": "stale_pipeline",
      "severity": "critical",
      "title": "Latest successful pipeline is not for current MR head",
      "evidence": ["pipeline:2:991", "commit:abc123"],
      "fix_actions": ["rerun_required_checks"]
    },
    {
      "id": "cache_taint",
      "severity": "high",
      "title": "Cache taint active for dependency layer",
      "fix_actions": ["run_cold_validation", "inspect_taint"]
    }
  ]
}
```

The TUI should render this as the right-side inspector everywhere.


# Ground-up TUI redesign

## The new mental model

The current tabs are organized by subsystem:

```text
Mission / Release / Jobs / Agents / Tests / Pools / Cache / Evidence / Secrets
```

That is logical for engineers who already know the internals. A world-class TUI should be organized by **operator questions**:

```text
What is happening?
What is blocked?
What changed?
What is risky?
What should I do?
What are agents doing?
Can this merge/release?
What evidence supports that?
```

I would redesign the TUI into this shell:

```text
┌────────────────────────────────────────────────────────────────────────────┐
│ JeRyu Mission Control  repo:dougx  main@abc123  🟢 healthy  ⚡ live     │
├───────┬──────────────────────────────────────────────────────┬─────────────┤
│ NAV   │ MAIN CANVAS                                          │ INSPECTOR   │
│       │                                                      │             │
│ Home  │ Agent / pipeline / proof / timeline / graph view      │ Why?        │
│ Work  │                                                      │ Next action │
│ Agents│                                                      │ Evidence    │
│ Flow  │                                                      │ Risk        │
│ Tests │                                                      │ Approvals   │
│ Merge │                                                      │ Logs        │
│ Sec   │                                                      │             │
│ Cache │                                                      │             │
│ Rel   │                                                      │             │
├───────┴──────────────────────────────────────────────────────┴─────────────┤
│ Event stream: live deltas, alerts, capsules, approvals, commands           │
├────────────────────────────────────────────────────────────────────────────┤
│ ^K command  / search  / ? help  / Enter inspect  / a act  / space select   │
└────────────────────────────────────────────────────────────────────────────┘
```

The key change: **right inspector is always contextual**. Whatever is selected, it answers:

```text
What is it?
Why is it in this state?
What evidence supports that?
What can I safely do next?
What is risky?
```

Visual rules:

* Critical path nodes are bold and connected with a bright line.

The inspector for a selected node should show:

# Visual design upgrades

## Semantic color palette

The current TUI uses mostly standard Ratatui colors: green success, blue running, red failed, yellow pending, cyan selected, magenta blocked/risk, dark gray inactive. That is a reasonable start. Make it a formal theme system.

Recommended palette:

```text
Success        green
Running        cyan/blue
Pending        amber
Failed         red
Blocked        magenta
Risk           orange/red
Security       purple/magenta
Cached/reused  teal
Stale          gray/yellow
Human needed   violet
Agent action   cyan
Production     red with bold
```

Also add non-color indicators for accessibility:

```text
✓ success
● running
○ pending
✗ failed
⊘ canceled
! blocked
◆ cached/reused
S secret
N network
H human approval
P production
```

## Better density modes

Support:

```text
compact
normal
presentation
incident
```

Compact mode: more rows, fewer descriptions.
Presentation mode: big readable cards for demos.
Incident mode: highlights failures, security, production, and actions.

## Better command palette

The current palette filters actions and executes mostly navigation/toggle actions. Make it the main interaction model.

Palette should support:

```text
natural search
typed command forms
recent commands
preview pane
risk badge
required approval
keyboard shortcuts
selected-entity context
dry-run previews
```

Example:

```text
> retry

Retry failed job
  entity: job:2:9981
  risk: low
  effect: creates new GitLab job
  confirmation: not required

Retry failed job on clean runner
  risk: low
  effect: creates new job, avoids same runner
```

For production actions:

```text
> promote

Promote release v1.4.2 to production
  risk: production
  requires: production approval
  blockers: canary telemetry incomplete
  action unavailable
```

## Better inspector

Right side inspector should always have this structure:

```text
Summary
State
Why
Evidence
Risk
Next actions
Related
Raw details
```

For any selected thing, the user should not need to guess.

## Better footer

Footer should be contextual. Current global help is useful but too static.

Example:

```text
Enter inspect  a actions  l logs  e evidence  r retry  f filter  / search  ? help
```

On Merge Proof tab:

```text
g generate proof  r rerun checks  a approve  m merge  e evidence  ? help
```

On Agent tab:

```text
p pause  R redirect  k kill  f fork  d diff  e evidence  c capabilities
```

# What the current TUI/API is missing most

## Missing from API

| Missing                           | Why it matters                                                                       |
| --------------------------------- | ------------------------------------------------------------------------------------ |
| Dedicated TUI snapshot/stream API | UI should not compose raw DB/GitLab/Docker state itself.                             |
| Typed event stream                | “Realtime” should be event-driven, not mostly polling.                               |
| Unified entity IDs                | Makes deep links, inspector, actions, and timelines much cleaner.                    |
| Work-order model                  | Agents need objectives, states, plans, budgets, blockers, and proof.                 |
| Structured blockers               | The UI needs machine-readable reasons and fix actions.                               |
| Action preview/confirmation       | Critical for safe UX around merge/release/security operations.                       |
| Full action execution lifecycle   | Users need to see executing/succeeded/failed/evidence states.                        |
| Merge proof API                   | Agent-first merge UX needs exact evidence, not generic green checks.                 |
| Failure capsule navigation        | Capsules need similar failures, reproducers, suspected cause, fix actions.           |
| Graph API                         | Pipeline graph should come from real dependency/stage data, not job-name heuristics. |
| User preferences/layouts          | World-class TUI needs saved filters, themes, layouts, and keymaps.                   |

## Missing from TUI

| Missing                   | Why it matters                                                                          |
| ------------------------- | --------------------------------------------------------------------------------------- |
| True agent cockpit        | Current Agents tab is pipeline/audit oriented, not task/agent oriented.                 |
| Work-order dashboard      | Users need to manage autonomous missions, not just jobs.                                |
| Contextual next actions   | Every screen should suggest safe actions.                                               |
| Structured “why blocked?” | This is the killer UX for agent-led work.                                               |
| Merge-proof screen        | Required for autonomous coding trust.                                                   |
| Failure triage workflow   | Logs are not enough; need diagnosis and reproducer.                                     |
| Security cockpit          | Agent tokens, capabilities, taints, egress, secrets, and prompt attacks need one place. |
| Timeline/replay           | Users need “what changed since I last looked?”                                          |
| Rich charts/heatmaps      | Current visuals are useful but still mostly text/list/card views.                       |
| Action preview modals     | Dangerous actions need preview, policy, blast radius, and confirmation.                 |
| Search/filter everywhere  | Large repos will overwhelm fixed lists.                                                 |
| Deep linking/bookmarks    | Operators need jumpable entities and saved views.                                       |

# Specific redesign proposal

## New primary navigation

I would replace the current nine tabs with:

```text
1 Home
2 Work Orders
3 Agents
4 Flow
5 Failures
6 Merge Proof
7 Tests
8 Security
9 Cache
0 Release
```

Keep old tab names as aliases.

## Home

Purpose: “What needs my attention?”

Cards:

```text
System health
Active work
Blockers
Merge/release readiness
Security alerts
Performance/cost
Recent changes
```

## Work Orders

Purpose: “What are agents trying to accomplish?”

Shows objectives, plan progress, confidence, risk, budget, blockers, touched files, next action.

## Agents

Purpose: “What is each agent doing right now?”

Shows live step, tool call, branch, diff summary, capabilities used, budget, last heartbeat.

## Flow

Purpose: “Where is this pipeline/release stuck?”

Shows DAG, lanes, critical path, ETA, current bottleneck, node inspector.

## Failures

Purpose: “Why did it fail and what do I do?”

Shows capsules, suspected cause, reproducer, similar failures, next actions, log slice.

## Merge Proof

Purpose: “Can this merge safely?”

Shows exact SHA proof, required checks, skipped tests, cache/secret/sandbox state, policy decision.

## Tests

Purpose: “Are we testing the right things?”

Shows selected/skipped tests, confidence, bottlenecks, flakes, coverage/impact.

## Security

Purpose: “Are autonomous agents behaving safely?”

Shows capabilities, token leases, secret leases, network egress, taints, honeypots, prompt-injection alerts.

## Cache

Purpose: “Are we wasting compute?”

Shows hit ratios, cache taints, top invalidators, runner queue, disk pressure, proof reuse.

## Release

Purpose: “Can we ship?”

Shows release candidate, gates, canary metrics, prod promotion, rollback readiness, evidence bundle.

# Highest-impact quick wins

## 1. Make command palette execute every registered action

Right now the registry contains many action definitions, but `execute_palette_action` only handles tab switching and audit toggling; other actions are effectively placeholders from the palette path. The fastest UX win is to make every registry action executable through a shared action dispatcher with preview and result feedback. ([GitHub][4])

## 2. Add a persistent right-side inspector to every tab

Jobs already has an inspector, Release has an inspector, Evidence has detail. Standardize it everywhere:

```text
selected entity
status
why
evidence
next actions
raw JSON
```

## 3. Add a “Top Blockers” panel to Mission

Use structured blockers from jobs, release, cache taints, selector misses, stale pipelines, failed capsules, and missing approvals.

## 4. Turn Agents tab into Work Orders + Agents

Replace “agent pipelines” as the primary view with:

```text
work order
agent
objective
state
confidence
risk
blocker
next action
```

Agent pipelines can become a lower detail panel.

## 5. Add action preview modals

For `retry`, `delete`, `pause`, `merge`, `promote`, `rotate secret`, show:

```text
What will happen
Risk tier
Required grant
Affected entities
Undo path
Confirm/cancel
```

## 6. Upgrade live events

The event console should include all high-value event types, not mostly evidence capsules:

```text
agent.step_changed
job.status_changed
pipeline.blocked
capsule.created
merge_proof.invalidated
secret.lease_opened
cache.tainted
release.gate_passed
approval.required
```

## 7. Add search/filter everywhere

Use `/` for search and `f` for filters.

Examples:

```text
status:failed
agent:agent-7
risk:high
has:capsule
branch:agent/*
kind:security
```

## 8. Add stale/stream quality indicators

Current stale indicators are good. Extend them:

```text
State live: 240ms
GitLab: 1.2s
Logs: 650ms
Events: streaming
DB: ok
```

## 9. Add a “What changed since last view?” overlay

This would be a killer feature:

```text
Since 12:04:
- Agent 7 generated patch for auth/session.rs
- Pipeline #991 failed integration-auth
- Failure capsule cap_123 created
- Cache taint cleared
- Merge proof invalidated by new commit abc123
```

## 10. Make screenshots beautiful by design

The TUI already supports PNG capture. Use that as a product feature:

```text
jeryu tui --capture --tab merge-proof
jeryu tui --capture --tab agent --entity agent-7
jeryu tui --capture --tab release --theme presentation
```

This is great for demos, incident reports, and release artifacts.

# Suggested API contract for world-class TUI

A minimal v1:

```text
GET /v1/tui/snapshot
GET /v1/tui/stream
GET /v1/entities/{id}
GET /v1/entities/{id}/timeline
GET /v1/entities/{id}/evidence
GET /v1/entities/{id}/blockers
GET /v1/entities/{id}/actions
GET /v1/actions
POST /v1/actions/preview
POST /v1/actions/execute
GET /v1/actions/{execution_id}
GET /v1/logs/{job_id}/stream
GET /v1/pipelines/{id}/graph
GET /v1/mrs/{id}/merge-proof
GET /v1/workorders
GET /v1/agents
```

Core response shape:

```json
{
  "ok": true,
  "request_id": "req_...",
  "generated_at": "...",
  "data": {},
  "errors": [],
  "warnings": [],
  "next_actions": []
}
```

Action result shape:

```json
{
  "execution_id": "actx_123",
  "action_id": "retry_job",
  "status": "running",
  "risk_tier": "low",
  "actor": "human:taylor",
  "target": "job:2:9981",
  "started_at": "...",
  "events_url": "/v1/actions/actx_123/stream",
  "evidence": []
}
```

# Product-level North Star

The TUI should feel like this:

> “I am supervising a team of autonomous engineers, and the system is showing me exactly what they are doing, why they are doing it, what is blocked, what is risky, what evidence exists, and what I can safely approve next.”

Right now, JeRyu has a good foundation: broad CLI, structured capability intents, a useful action registry, a multi-tab TUI, live-ish logs, flow graph modeling, evidence capsules, cache/security/release surfaces, and screenshot capture. The next leap is to make it **action-centered, entity-centered, and evidence-centered**.

The world-class version is not “terminal GitLab.” It is:

**an agent mission-control cockpit with a streaming API, proof-aware actions, contextual inspectors, beautiful DAG/timeline graphics, structured blockers, and one-click safe next actions.**
