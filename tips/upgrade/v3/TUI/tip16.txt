## Executive take

The current API/TUI has the right ambition: it exposes CLI, TUI, webhook, capability socket, Git hook, custom executor, action registry, state, GitLab client, release, secrets, and test-intelligence surfaces, which is exactly the kind of multi-surface control plane an agent-first Git system needs. ([GitHub][1])

But the TUI currently feels more like an **operator/debug dashboard** than a **world-class human+agent mission-control cockpit**. The biggest issue is not just layout or colors. The deeper issue is that the API does not yet deliver the high-level objects the TUI needs: live event streams, entity relationships, action previews, risk explanations, blocker graphs, agent run states, race results, and “next best action” data.

My redesign would make the TUI the canonical frontend for one question:

> **What is happening, what is blocked, why, what should happen next, who/what is allowed to do it, and what proof exists?**

---

## What the current API/TUI already has

The current API reference documents a large control surface: CLI, TUI, webhook server, Unix-socket capability API, Git server hook, custom executor, action registry, state API, settings, GitLab REST wrapper, release API, and secrets API. That is a strong starting point because the TUI can eventually become a unified cockpit instead of a passive terminal dashboard. ([GitHub][1])

The TUI itself has moved toward a nine-tab product shape: Mission, Release, Jobs, Agents, Tests, Pools, Cache, Evidence, and Secrets. The CLI supports `jeryu tui --once` plus deterministic PNG capture with `--capture`, `--tab`, width, and height options, which is excellent for visual regression tests, papers, and demo automation. ([GitHub][2])

The application state is also richer than a normal terminal app. The `TuiStateSnapshot` includes pools, GitLab readiness, containers, jobs, pipelines, flow snapshot, live log state, jeryu sync, cache metrics, taints, storage, release status, test bottlenecks, recent evidence, secret audit events, agent pipelines, and recent audit events. ([GitHub][3])

The flow system is promising. It has a `FlowSnapshot`, `PipelineFlow`, `FlowGraph`, columns, lanes, nodes, and edges. The graph builder classifies jobs into stages like Admission, Impact, Build, Tests, Security, Package, Gates, Canary, and Production; it also computes stage-ordering edges and a critical path. ([GitHub][4])

The action registry is a very good idea. It defines action IDs, labels, key hints, risk tiers, surfaces, dry-run status, descriptions, side-effect class, grant requirements, and JSON contracts. That is exactly the right direction for making CLI, TUI, and agent APIs converge on one policy-aware action model. ([GitHub][5])

## Biggest current TUI/API gaps

### 1. The TUI has tabs, but not yet a cockpit-level mental model

The current screenshots show a lot of empty bordered panels, tiny text, and heavy box-drawing noise. Mission, Jobs, Agents, Evidence, and Tests all render, but much of the visual space is either empty or placeholder-like when data is sparse. 

A world-class TUI should not feel empty when there are no jobs. It should still answer:

* Is the system safe?
* Are agents allowed to work?
* Are release gates blocked?
* What is the next recommended action?
* What is stale, missing, degraded, or risky?
* What changed in the last minute?

Right now, empty states often say “No events yet” or “Waiting,” but they do not explain whether that is good, bad, expected, stale, or actionable.

### 2. The docs and current code are inconsistent

`JERYU_TUI.md` says the old keybindings are `1: Flow`, `2: Storage`, `3: Cache`, `4: Audit`, `5: Tests`, while the current source and CLI expose nine tabs: Mission, Release, Jobs, Agents, Tests, Pools, Cache, Evidence, and Secrets. ([GitHub][6])

That matters because agents and humans rely on docs as an API. For an agent-first system, stale docs are not just annoying; they cause tool-use errors.

### 3. The capability API is too low-level and too synchronous

The current capability API is a single JSON request over a Unix socket with one JSON response and a 64 KiB read buffer. It supports intents like `ProposePatch`, `RacePatches`, `RunTests`, `FetchCapsule`, `RequestMerge`, `ExplainBlockers`, `GetSystemSnapshot`, `ListAllowedActions`, and `PlanValidation`. ([GitHub][1])

That is useful, but it is not enough for a world-class TUI. The TUI needs streaming progress, partial results, cancellation, correlation IDs, entity links, grant status, action previews, and live updates. A one-shot request/response protocol forces the TUI to poll and reconstruct state.

### 4. The command palette exists, but it barely executes actions

The registry lists many actions, including retry job, pause pool, explain blockers, propose patch, race patches, request merge, plan validation, run tests, and next action. ([GitHub][5])

But the current palette executor only switches tabs and toggles the audit ledger; other actions fall through to no-op comments. ([GitHub][7])

That means the most important UX primitive — “hit Ctrl-K, type what you want, preview it, run it safely” — is not yet real.

### 5. The TUI owns too much operational logic directly

The current `App` owns handles to the DB, Docker controller, and GitLab client, and it directly performs actions such as retrying jobs, deleting records, and pausing/resuming pools. The TUI docs explicitly describe `App` as owning these operational handles, not just view state. ([GitHub][6])

For a serious agent-first product, that should be inverted. The TUI should call a single action API that handles dry-run, permission checks, grant checks, execution, result streaming, audit logging, and rollback hints. Otherwise, CLI/TUI/capability behavior will drift.

### 6. Realtime is polling-based, not event-driven

The current TUI runs a 250 ms draw/input/tick loop; the general snapshot and flow collector refresh roughly every 1500 ms; live logs poll GitLab every 650 ms. The docs also explicitly say there is no websocket transport. ([GitHub][6])

Polling is okay for a prototype, but world-class realtime insight needs a stream:

* job status changed,
* log line appended,
* agent action requested,
* grant approved/denied,
* pipeline superseded,
* release gate failed,
* cache tainted,
* selector miss found,
* evidence capsule created.

The TUI should render from an event log and snapshot deltas, not repeatedly scrape GitLab/DB/Docker itself.

### 7. The log preview is misleading

The UI has a log preview pane, but the current `update_log_target()` only sets a log target when `maximize_logs` is true. When not maximized, it clears the log target. ([GitHub][3])

So the bottom-right “Log Preview” concept is not truly a live selected-job preview unless the user opens the maximized log view. That makes the Jobs screen less useful than it looks.

### 8. Multiple pipelines are collected, but the flow board renders only the first

The flow collector can collect multiple active pipelines, and the Jobs tab has a pipeline navigation pane. But the flow board renderer uses `active_pipelines.first()`, so the visible graph is only the first active pipeline. ([GitHub][8])

A world-class agent cockpit must support multiple concurrent branches, races, MRs, release pipelines, and agent runs. The first pipeline is rarely enough.

### 9. The graph has edges, but the widget does not visually render them

The builder creates stage-ordering edges and marks critical-path nodes, but the widget renders columns, lane groups, and nodes only. The result is more of a stage board than a true graph. ([GitHub][9])

That is a missed opportunity. A true visual flow should show dependencies, blocked downstream nodes, critical path, skipped optional lanes, and release-gate causality.

### 10. The Agent tab is not yet an agent cockpit

The current Agent tab shows agent pipelines and capability call history. The empty state says branches starting with `agent/` appear there, and the screenshot shows no agent pipelines or capability calls. ([GitHub][10])

That is not enough. A real agent tab should show agent objectives, current step, branch, MR, grants, risk tier, token expiry, confidence, cost, last tool call, next action, blockers, and intervention controls.

# Ground-up redesign: “jeryu Mission Control”

I would redesign the TUI around a stable four-zone layout:

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ jeryu Mission Control  SAFE/DEGRADED  main@58dcf4d  agents:3  gates:2 blocked │
│ Next: fix release artifact contract → Run: explain release blocker           │
├───────────────┬───────────────────────────────────────────────┬──────────────┤
│ NAV           │ WORKSPACE                                     │ INSPECTOR    │
│ Mission       │ Flow / Agents / Release / Evidence / Tests    │ selected     │
│ Work Queue    │                                               │ entity       │
│ Flow          │                                               │ actions      │
│ Agents        │                                               │ blockers     │
│ Evidence      │                                               │ proof        │
│ Tests         │                                               │ grants       │
│ Release       │                                               │ history      │
│ Infra         │                                               │              │
├───────────────┴───────────────────────────────────────────────┴──────────────┤
│ Timeline: job#144 failed → capsule created → agent repair proposed → gate ... │
└──────────────────────────────────────────────────────────────────────────────┘
```

The current top-tab layout is okay for a small app, but it becomes cramped and low-context once agents, pipelines, releases, cache, evidence, and secrets are all active. A vertical left nav plus persistent right inspector would make the TUI much more intuitive.

The right inspector is the most important UX change. Every selected thing — job, pipeline, agent, MR, capsule, test, release gate, secret set, cache taint — should show:

```text
What is this?
Current state
Why it matters
Why it is blocked
Required proof
Allowed actions
Dangerous actions
Recent events
Related entities
```

That turns the TUI from “panels of metrics” into “click/select anything and understand it.”

## Redesigned screens

### 1. Mission screen: the answer-first dashboard

Current Mission has four boxes: system health, pipeline summary, release status, and cache health. That is a good start. But the dream version should be a prioritized command center.

It should show:

```text
SYSTEM POSTURE
SAFE TO AUTONOMOUSLY CODE: YES
SAFE TO MERGE:             NO — 2 blockers
SAFE TO RELEASE:           NO — artifact contract broken
AGENT WRITE GRANTS:         3 active, 0 expired, 0 suspicious
CACHE TRUST:                DEGRADED — 1 taint
TEST SELECTOR CONFIDENCE:   91%, 2 misses this week
```

Then immediately below:

```text
TOP BLOCKERS
1. Release ci-58dcf4d: artifact contract broken
   Next: inspect gate artifact / rerun release doctor / open capsule
2. Cache taint: tripwire lane breach
   Next: quarantine cache / run cold proof / explain taint
3. Agent race #42: no winner after 18m
   Next: cancel losers / promote best partial / request human
```

This should be the default tab, not Jobs. The first thing a user sees should not be raw jobs; it should be operational truth.

### 2. Work Queue: the missing “what should I do now?” page

Add a new Work Queue screen that merges all actionable items:

```text
Priority  Type       Entity          Why                         Action
P0        Release    ci-58dcf4d      prod blocked                explain / doctor
P0        Merge      !17             missing security proof       run proof
P1        Job        #14445          compile failure              open capsule
P1        Agent      agent/fix-auth  grant expires in 8m          extend / pause
P2        Cache      crate-index     low hit ratio                inspect misses
```

This page should be computed by the API, not by the TUI. It should expose a ranked, typed `ActionableItem` list with severity, confidence, recommended action, and entity refs.

### 3. Flow screen: from job board to causality graph

Current Flow is conceptually good but visually weak. It should become the “CI/release subway map.”

Use lanes:

```text
Git ── Admission ── Impact ── Build ── Tests ── Security ── Package ── Gates ── Canary ── Prod
        ✓ hook       ✓ VTI     ● img    ✗ unit    ○ scan      ○ pkg      ○ e2e     ○ canary   ○ prod
                                      └── blocked: unit failed job#14445
```

Features I would add:

* multiple pipeline selector with branch/MR/agent labels;
* visible edges between stages;
* critical path highlighted;
* optional jobs visually distinct from required jobs;
* superseded pipelines dimmed;
* stale pipelines marked and collapsible;
* downstream child pipelines nested;
* failed node shows capsule badge;
* selected node lights up related log, capsule, agent, MR, and action options;
* “why blocked” overlay.

The API should provide this as a precomputed `FlowGraphView`, not force the widget to infer everything from job names.

### 4. Agents screen: real autonomous-run supervision

The Agent tab should not center on “agent pipelines.” It should center on **agent runs**.

```text
Agent        Objective              State        Risk   Branch              MR    Next
wrath-17     fix release contract    repairing    high   agent/fix-contract  !18   run selected proof
wrath-18     reduce test flake       racing       low    race/flake-a        !19   promote winner
wrath-19     update deps             blocked      high   agent/deps          —     needs approval
```

Right inspector:

```text
Agent wrath-17
Objective: Fix artifact contract broken in release ci-58dcf4d
State: diagnosing → patching → proof
Confidence: 0.73
Grant: grant-abc, expires in 21m
Allowed paths: src/release/**, docs/API.md
Last tool call: FetchCapsule(job#14445)
Next planned action: ProposePatch
Risk: touches release gate logic
Human options:
  [a] approve patch attempt
  [r] race 3 alternatives
  [p] pause
  [k] revoke grant
  [q] quarantine
```

The API needs first-class `AgentRun`, `AgentStep`, `AgentGrant`, and `AgentDecision` objects. Branch names are not enough.

### 5. Race board: the killer agent-first screen

For autonomous coding, this should be one of the signature experiences.

```text
Race #42: fix release artifact contract
┌─────────────┬──────────────┬─────────┬─────────┬────────────┬────────────┐
│ Hypothesis  │ Diff         │ Unit    │ Release │ Risk       │ Score      │
├─────────────┼──────────────┼─────────┼─────────┼────────────┼────────────┤
│ A           │ +34/-7       │ PASS    │ RUN     │ low        │ 82         │
│ B           │ +91/-12      │ FAIL    │ —       │ medium     │ 41         │
│ C           │ +12/-3       │ PASS    │ PASS    │ low        │ 94 WINNER  │
└─────────────┴──────────────┴─────────┴─────────┴────────────┴────────────┘
```

Actions:

* promote winner,
* cancel losers,
* compare diffs,
* inspect failure capsules,
* request human review,
* open MR,
* run deeper proof.

The current `RacePatches` returns launched branches and says the user must poll pipelines and manually clean up or implement `PollRaceResult`. ([GitHub][11])

That is exactly where the TUI should shine.

### 6. Evidence screen: from logs to repair intelligence

The current Evidence tab has capsule list/detail and an audit-ledger toggle. That is a good base. It already shows job id, ref, SHA, stage, exit code, failure kind, superseded/retry links, and log snippet when payload parsing succeeds. ([GitHub][10])

The dream Evidence tab should show:

```text
Capsule #cap-14445
Failure kind: compile_failure
Likely root cause: missing trait bound in release::artifact
Confidence: 0.81
First bad commit: 58dcf4d
Similar failures: 3
Replay:
  jeryu capsule replay cap-14445
Repair options:
  [1] ask agent to repair
  [2] race 3 repairs
  [3] run local proof
  [4] attach to MR
Redaction: clean
Superseded by: none
```

Also add:

* failure timeline,
* “same root cause” clustering,
* capsule search,
* replay status,
* redaction status,
* “was this fixed?” detection,
* agent repair attempts linked to capsule.

### 7. Tests screen: VTI confidence, not just bottlenecks

The current Tests tab shows average/latest bottlenecks and a selected test history drilldown. ([GitHub][6])

That is useful, but the agent-first value is **proof selection confidence**.

Redesign it around:

```text
VTI PLAN FOR branch agent/fix-contract
Changed files: 4
Selected tests: 12 / 187
Skipped tests: 175
Confidence: 93%
Selector misses: 0 in 24h, 2 in 7d
Global invalidators: none
Required lanes:
  unit       selected 8
  security   selected 2
  release    selected 2
```

Add a heatmap:

```text
Subsystem       Selected  Misses  Flake  Avg time  Risk
release          4        0       low    2m10s     high
artifact         3        1       med    1m40s     high
cache            0        0       low    0m        low
```

The user should instantly see whether the agent is under-testing, over-testing, or testing exactly enough.

### 8. Release screen: from matrix to release command center

The current Release tab has a release gate matrix and inspector. The screenshot shows useful rows like main branch green, release identity, release pipeline, canary health, E2E gate, and prod promotion. 

Make it more actionable:

```text
Release ci-58dcf4d
Status: BLOCKED
Blocking gate: artifact contract
Reason: release pipeline produced invalid handoff file
Safe actions:
  [d] doctor
  [e] open evidence
  [r] rerun release proof
  [a] assign agent repair
Unsafe actions:
  [p] promote prod — disabled: missing E2E gate
Rollback:
  previous green ci-a91be33
```

Add a visual release timeline:

```text
main green ✓ ── identity ✓ ── release pipeline ● ── canary ○ ── e2e ○ ── prod ○
                         blocked here
```

### 9. Cache/Infra screen: less raw counters, more decisions

Current Cache shows storage overview, gateway health, singleflight analytics, and trust/taint boundaries. ([GitHub][6])

The upgrade is to convert metrics into decisions:

```text
Cache Trust: DEGRADED
Reason: 1 active taint, 0 breaches
Impact: agent proofs use cold execution for release lane
Recommended action: inspect taint → quarantine or clear
Savings today: 2.1 GB / 18m
Top misses:
  cargo index        41 misses
  rust:1.92 image    7 misses
```

Show sparklines for hit ratio, queue depth, disk pressure, and cache ROI.

### 10. Secrets screen: show lifecycle and blast radius

The current Secrets tab shows secret audit events and a static Vault status pane. ([GitHub][10])

Make it answer:

* Which release secret sets exist?
* Which are active?
* Which are expired?
* Which agents had access?
* Which deployments depend on them?
* Which rotations are pending?
* What is the blast radius if compromised?

# API redesign to make the TUI world-class

## 1. Add a dedicated TUI service API

Do not make the TUI assemble state by directly polling DB, Docker, GitLab, release files, and filesystem. It currently uses all of those data sources. ([GitHub][6])

Create a local service over Unix socket or localhost HTTP:

```text
GET  /v1/tui/snapshot
GET  /v1/tui/stream
GET  /v1/entities/{type}/{id}
GET  /v1/entities/{type}/{id}/timeline
GET  /v1/entities/{type}/{id}/related
GET  /v1/actions
POST /v1/actions/{id}/dry-run
POST /v1/actions/{id}/execute
POST /v1/actions/{id}/cancel
GET  /v1/search?q=...
```

The TUI renders. The service reasons.

## 2. Introduce event envelopes

Every realtime update should use a common envelope:

```json
{
  "seq": 182391,
  "time": "2026-04-27T12:35:41Z",
  "kind": "job.status_changed",
  "severity": "info",
  "entity": {"type": "job", "id": "2/14445"},
  "parent": {"type": "pipeline", "id": "2/530"},
  "summary": "unit-test changed running → failed",
  "patch": {
    "status": "failed",
    "finished_at": "2026-04-27T12:35:40Z"
  }
}
```

This lets the TUI show a live timeline, update panels incrementally, replay state, and avoid flicker.

## 3. Add a canonical `MissionSnapshot`

The Mission tab should not infer health from many raw counters. Give it one object:

```json
{
  "overall": "degraded",
  "safe_to_code": true,
  "safe_to_merge": false,
  "safe_to_release": false,
  "top_blocker": {
    "entity": "release:ci-58dcf4d",
    "reason": "artifact contract broken",
    "recommended_action": "release.doctor"
  },
  "counts": {
    "active_agents": 3,
    "blocked_agents": 1,
    "running_jobs": 8,
    "failed_jobs": 2,
    "open_capsules": 4,
    "active_grants": 5
  }
}
```

This becomes the top of the TUI, CLI `jeryu status`, and agent `GetSystemSnapshot`.

## 4. Make actions first-class and typed

The action registry is close, but it should become a real executable action contract. Today, side-effect class and required grant are computed by matching `entry.id`, which is brittle. ([GitHub][5])

Each action should include:

```json
{
  "id": "retry_job",
  "label": "Retry job",
  "context_types": ["job"],
  "risk_tier": "low",
  "side_effect_class": "ci_execution",
  "required_grant": "agent_task",
  "input_schema": {
    "type": "object",
    "required": ["project_id", "job_id"]
  },
  "preconditions": [
    "job.status in ['failed', 'canceled']"
  ],
  "dry_run_supported": true,
  "undo": "cancel_job"
}
```

The palette should query actions valid for the selected entity. That means if a failed job is selected, the palette should prioritize:

* open logs,
* explain blocker,
* open capsule,
* retry job,
* race repair,
* run selected proof,
* cancel pipeline.

## 5. Add dry-run previews everywhere

Before any mutating action, the TUI should show:

```text
Action: Retry job #14445
Risk: low
Will:
  - call GitLab retry endpoint
  - create audit event
  - attach retry to capsule cap-14445
Will not:
  - modify source branch
  - merge code
  - touch production
Required grant: none / agent_task / merge_approval
```

Production actions should require a typed confirmation phrase and display exact SHA, policy version, evidence, and rollback.

## 6. Add entity graph APIs

Every object should link to related objects:

```text
agent run → branch → MR → commits → pipeline → jobs → capsules → tests → release gates
```

API:

```text
GET /v1/entities/job/2:14445/related
```

Response:

```json
{
  "entity": "job:2/14445",
  "related": [
    {"type": "pipeline", "id": "2/530", "relation": "belongs_to"},
    {"type": "capsule", "id": "cap-14445", "relation": "failure_evidence"},
    {"type": "agent_run", "id": "run-17", "relation": "repair_owner"},
    {"type": "test", "id": "unit::artifact_contract", "relation": "executed"}
  ]
}
```

That single API would make the right inspector dramatically more useful.

## 7. Replace `GetSystemSnapshot` with layered snapshots

The current `GetSystemSnapshot` returns pools, cache, selector misses, recent failures, and generated time. ([GitHub][11])

Keep that, but add specialized snapshots:

```text
GET /v1/snapshots/mission
GET /v1/snapshots/agents
GET /v1/snapshots/flow
GET /v1/snapshots/release
GET /v1/snapshots/evidence
GET /v1/snapshots/tests
GET /v1/snapshots/infra
```

The TUI can subscribe only to what is visible.

## 8. Add log streaming and log intelligence

Current logs poll the GitLab job trace endpoint for the selected job. ([GitHub][6])

Add:

```text
GET /v1/jobs/{id}/logs/stream
GET /v1/jobs/{id}/logs/search?q=panic
GET /v1/jobs/{id}/logs/failure-span
GET /v1/jobs/{id}/logs/annotations
```

The TUI should not show giant raw logs first. It should show:

* failure span,
* compiler/test error,
* suggested root cause,
* artifact links,
* full log available on demand.

## 9. Add `ExplainBlocker` as a universal API

The current capability API has `ExplainBlockers`, but it is basic: job explanations come from capsules, release explanations summarize recent failures, and merge is listed as an entity type but not deeply modeled. ([GitHub][11])

Upgrade it to:

```text
POST /v1/explain
{
  "entity": "merge_request:2/17",
  "depth": "full",
  "format": "human+json"
}
```

Response:

```json
{
  "blocked": true,
  "summary": "MR !17 cannot merge because security proof is missing for auth/token.rs.",
  "blockers": [
    {
      "code": "missing_required_proof",
      "severity": "high",
      "required_action": "run_tests",
      "suggested_command": "jeryu test run --scope security --ref agent/fix-auth"
    }
  ],
  "evidence": [...],
  "safe_actions": [...]
}
```

This object should power the TUI inspector, CLI, and agent API.

# Visual/UI upgrades

## 1. Fix the screenshot glyph renderer

The generated PNG screenshots currently show lots of square placeholder-like border characters. The raw screenshots make the UI look noisier and less polished than the actual terminal may be. 

The capture renderer uses a small custom 5x7 glyph table and falls back for unknown characters, which likely explains why box-drawing symbols render poorly. ([GitHub][7])

Fixes:

* support Unicode box-drawing glyphs in the PNG renderer;
* or add an ASCII capture mode;
* or render via a real monospace font renderer;
* or avoid heavy borders in screenshots.

This is a high-impact polish fix.

## 2. Reduce border density

Current screens use many full borders. Borders help separation, but too many create visual noise. Use:

* one outer frame,
* subtle pane separators,
* fewer nested boxes,
* colored title bars,
* whitespace,
* badges.

The terminal should look like a command center, not a wall of rectangles.

## 3. Use a semantic color system

Current status colors are simple: green success, blue running, red failed, yellow pending, gray canceled. ([GitHub][10])

Keep that, but introduce theme tokens:

```text
OK:          green
RUNNING:     cyan/blue
WAITING:     amber
FAILED:      red
BLOCKED:     magenta
PRODUCTION:  purple/red
STALE:       gray → amber → red by age
AGENT:       violet/cyan
SECURITY:    magenta
CACHE:       blue
EVIDENCE:    orange/red
```

Never rely on color alone. Pair color with badges:

```text
[OK] [RUN] [WAIT] [FAIL] [BLOCKED] [STALE] [PROD] [RISK]
```

## 4. Add sparklines and heatmaps

Ratatui can represent rich data compactly. Add:

* queue depth sparkline,
* job duration sparkline,
* cache hit-ratio sparkline,
* selector confidence trend,
* flake-rate heatmap,
* release timeline,
* agent cost/latency gauge,
* proof coverage heatmap.

Example:

```text
Queue:  ▁▂▅█▇▃▁  p95 wait 41s
Cache:  ▆▆▇▇█▅▃  hit 73%, taint 1
Tests:  release ███░  auth ██░░  cache ░░░░
```

## 5. Add responsive layouts

For 100-column terminals, show one main panel and inspector. For 160+ columns, show nav/workspace/inspector. For giant wall displays, show TV mode with mission, flow, agents, release, and timeline.

API should expose “layout hints,” but TUI should own rendering.

## 6. Make mouse real

The current TUI enables mouse capture, but the docs say active interactions are keyboard-driven. ([GitHub][6])

Add:

* click tab/nav item,
* click job,
* click graph node,
* scroll logs,
* click action button,
* hover/select inspector rows,
* right-click context actions if supported.

Keyboard should remain first-class, but mouse should not be fake.

# The single biggest product shift

The current TUI mostly shows **state**.

The world-class version should show **state + causality + next action + authority**.

For every selected thing, the user should see:

```text
What happened?
Why did it happen?
What is blocked?
What proof exists?
What is the safest next action?
Who/what is allowed to do it?
What will happen if I approve it?
How do I undo it?
```

That requires API changes more than UI changes. The TUI cannot invent good insight from raw counters and job lists. It needs structured, typed, realtime, policy-aware objects.

## Final design principle

Do not make the TUI “GitLab in a terminal.”

Make it the **agent operations cockpit** GitLab/GitHub do not have:

> A place where humans supervise autonomous coding by risk, evidence, policy, and intent — not by manually watching CI logs.
