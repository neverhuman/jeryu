## Focused review: current API + current TUI

I reviewed the current API docs, TUI docs, CLI surface, TUI source, flow model, action registry, and capability API. I did **not** run the binary locally, so this is a source/API/UX review rather than a runtime usability test.

My overall take: **JeRyu already has a strong skeleton for an agent-first Git TUI, but the current UX still feels like a powerful operator dashboard rather than a world-class agent mission-control interface.** The biggest missing pieces are realtime streaming, typed action execution from the TUI, richer agent state, true dependency graphs, better evidence drilldowns, and a more opinionated “what should I do next?” layer.

---

# What exists today

The current API docs describe a broad surface: CLI, TUI, webhook HTTP API, capability API, Git server hook/admission API, executor API, action registry, state API, settings, GitLab REST client wrapper, release operations, and secrets API. The docs say API.md is versioned `5.0.0` and was verified against the code on April 26, 2026. ([GitHub][1])

The CLI is large and agent-oriented. It supports lifecycle commands, TUI launch/capture, status, pools, jobs, pipelines, cache, logs, agents, jeryu sync, tests/VTI, releases, secrets, progress, repo/host helpers, server-hook, capability API, next-action, explain-blocker, and action-list commands. The TUI command supports `--once`, `--capture`, `--tab`, output path, width, and height, which is great for regression testing and demos. ([GitHub][2])

The current TUI is Ratatui/crossterm-based, has nine tabs — Mission, Release, Jobs, Agents, Tests, Pools, Cache, Evidence, Secrets — and uses a draw/input/tick loop with a 250ms interval. It keeps its own `App` state, clients, snapshots, flow state, selected job/log state, command palette state, and background refresh workers. ([GitHub][3])

The Jobs view is now more advanced than a simple list: the source shows a three-column layout with pipeline navigation on the left, release banner / flow board / jobs / logs in the center, and an inspector on the right. The top-level chrome includes the tab strip, GitLab status, pool/release indicators, event console, and footer shortcuts. ([GitHub][4])

The TUI has a flow model with phases, lanes, nodes, blockers, ETA, active pipelines, pools, active containers, cache metrics, and release state. However, graph construction is still mostly inferred from job names and lanes, not from a true dependency DAG. The docs describe columns, lanes, icons, and stale-state behavior; older graph code classifies jobs into phases such as Source, Admission, Impact, Build, Test Matrix, Release Gates, and Prod using heuristics. ([GitHub][3])

The log UX has useful pieces: ANSI/syntax highlighting, manual/follow scrolling, maximized log view, and a `LIVE_LOG_MAX_BYTES` cap of 160,000 bytes. But the current log transport polls GitLab roughly every 650ms, which is a pragmatic implementation rather than a true realtime stream. ([GitHub][3])

The action registry is one of the better design choices. It is intended as a single source of truth for the TUI command palette, CLI `action list`, and capability API `ListAllowedActions`. Actions carry metadata such as risk tier, side-effect class, grant requirement, surface, docs, and dry-run support. ([GitHub][5])

The capability API is also promising. It exposes typed agent intents such as `ProposePatch`, `RacePatches`, `RunTests`, `FetchCapsule`, `RequestMerge`, `ExplainBlockers`, `GetSystemSnapshot`, `ListAllowedActions`, and `PlanValidation`. However, the API docs describe several mutating intents as grant-bound or advisory, and `RequestMerge` is currently framed as conservative/advisory rather than a complete merge-execution contract. ([GitHub][1])

The state/settings API is rich enough to support a much better TUI than the current one: it stores jobs, events, release attempts, evidence capsules, secret audits, capability grants/intents, admission decisions, test executions/plans, selector misses, cache records, taints, leases, verdicts, promotions, and more. Settings also include TUI-related fields such as sync interval, recent job limits, pool display limits, and cache warning thresholds. ([GitHub][1])

# What the current TUI does well

## 1. It already thinks in “operator surfaces,” not just CI lists

The nine-tab structure is the right general shape. Mission, Release, Jobs, Agents, Tests, Pools, Cache, Evidence, and Secrets are exactly the domains an autonomous coding platform needs. The problem is not the tab names; the problem is that some tabs still behave like dashboards rather than decision surfaces.

## 2. It has good bones for a live control center

The TUI already has background workers, stale-state handling, stable selections, a log reader, capture mode, and flow snapshots. This means the architecture is close to being upgraded into a realtime system without replacing everything.

## 3. The action registry is a great foundation

Most TUIs fail because keyboard actions are hardcoded and disconnected from the API. JeRyu’s action registry can become the backbone of command palette UX, permission previews, dry-runs, audit logs, and agent capability grants.

## 4. The TUI source already has useful visual concepts

The flow board, release gate matrix, inspector panels, mission health cards, pipeline navigation, event console, and job detail inspector are all good ingredients. The next step is to make them more connected, more explanatory, and more action-oriented.

# Biggest missing pieces

## 1. The TUI needs to become realtime, not polling-first

Polling logs every 650ms and refreshing snapshots periodically is workable, but a world-class agent control plane should stream state changes. The TUI should receive typed events: job started, job failed, log chunk appended, capability granted, agent transitioned, test plan updated, selector miss detected, cache tainted, release gate passed, admission denied.

The current API already has webhook ingestion and state tables, but the TUI needs a dedicated local event stream: ideally Unix-domain-socket SSE/WebSocket-style streaming for local use, plus HTTP SSE/WebSocket for remote use.

**Target behavior:** when a job fails, the UI should update instantly, highlight the blocking lane, generate a failure capsule, show the next recommended action, and let the user spawn a fix agent or retry from the same screen.

## 2. The command palette is not yet a true action system

The action registry has excellent metadata, but the TUI command palette should do much more than navigation. It should be the central UX for every safe operation.

A world-class command palette should support:

* fuzzy search;
* action arguments;
* risk preview;
* dry-run preview;
* required capability grant;
* affected refs/jobs/pipelines;
* expected side effects;
* rollback/undo option;
* confirmation only for risky actions;
* execution progress;
* result toast;
* audit ledger entry.

For example, selecting **“Retry failed jobs”** should show:

```text
Action: Retry failed jobs
Risk: Low
Affects: 3 jobs in pipeline #184
Requires: GitLab API write
Dry run: available
Expected result: new job attempts attached to same pipeline
Audit: yes
```

Then the user hits Enter to run, or `d` to dry-run.

## 3. The Agents tab is not agent-native enough

The current Agents area appears to center on agent-related pipelines and capability-call history. That is useful, but not enough. An agent-first Git system needs the Agents tab to be the primary product surface.

Each agent should have a visible lifecycle:

```text
Created → Planning → Editing → Testing → Debugging → Awaiting Grant → Merge Ready → Released → Archived
```

Each agent card should show:

* goal;
* owner;
* branch/MR;
* current state;
* autonomy level;
* files touched;
* tests run;
* latest failure;
* next planned action;
* budget/time spent;
* risk score;
* capability grants;
* evidence completeness;
* whether it is blocked by human, CI, policy, cache, secrets, or admission.

The user should not have to infer what an agent is doing from jobs and logs.

## 4. The flow graph is visually useful but not yet trustworthy

The current flow board is a good start, but a world-class system needs a real execution graph, not mostly job-name classification. The UI should represent actual dependencies from GitLab jobs, child pipelines, release pipelines, VTI-selected tests, cache dependencies, and gate conditions.

The graph should show:

* real DAG edges;
* required vs optional jobs;
* skipped jobs with reasons;
* retried jobs grouped by attempt;
* critical path;
* queue bottlenecks;
* estimated time remaining with confidence;
* blockers;
* jobs that are irrelevant to the current merge proof;
* jobs that are release-critical.

A great graph would not just say “job failed.” It would say:

```text
Blocked by: integration:api-auth
Reason: failed on required release lane
Likely cause: migration fixture mismatch
Next action: spawn fix agent or open failure capsule
Merge impact: blocks MR
Release impact: blocks canary
```

## 5. The TUI needs an omnipresent “next best action”

The repo has a `next-action` command and blocker explanation APIs, but the TUI should make this unavoidable. Every tab should have a small “Recommended next action” region.

Examples:

```text
Recommended:
Run VTI minimal proof for MR !42
Reason: changed files only touch cache + secrets modules
Risk: Medium
Shortcut: a → Plan validation
```

```text
Recommended:
Do not merge
Reason: selector miss detected after head SHA a81f…
Blocking evidence: test selector oracle failed
Shortcut: e → Open evidence
```

```text
Recommended:
Pause untrusted pool
Reason: 2 cache taints originated from runner pool untrusted-linux-2
Shortcut: p → Pause pool
```

This is the difference between a dashboard and a control plane.

## 6. Logs need search, structure, and failure extraction

Current live logs are useful, but logs should become structured evidence.

Missing log features:

* search inside current log;
* jump to first error;
* jump to last failure;
* collapse install/setup noise;
* classify error type;
* show related historical failures;
* extract repro command;
* show failing test name/file/line;
* show environment/container/image digest;
* show whether failure is new, flaky, retried, or known;
* copy failure capsule;
* open artifacts directly;
* spawn fix agent from selected error.

The log pane should not just display text. It should help the user understand and act.

## 7. Evidence needs to become a first-class timeline

The current TUI includes Evidence, and the docs/source mention capsules, audits, action registry, and event console pieces. But evidence should be the core trust layer.

A world-class Evidence tab should be searchable and structured:

```text
Timeline
12:03:11 capability.granted    agent/cache-fix may write refs/agents/cache-fix/*
12:04:02 patch.applied          7 files changed, diff hash …
12:04:40 test.plan.created      14 selected, 96 skipped, confidence 0.87
12:06:18 job.failed             integration:cache-taint
12:07:01 failure.capsule        suspected cause: stale fixture
12:08:45 patch.updated          2 files changed
12:10:03 tests.green            required proof complete
12:10:12 merge.blocked          selector miss unresolved
```

Every item should open an inspector with JSON, human summary, related logs, and safe actions.

## 8. Cache/provenance needs better visuals

The Cache tab already has cache, taint, gateway, and storage ideas. It should evolve into a supply-chain/provenance graph.

Useful views:

* cache hit/miss sparkline;
* cache savings estimate;
* taint propagation graph;
* trusted/untrusted boundary crossings;
* cache object lineage;
* “why this cache hit was accepted” explanation;
* “why this cache object was rejected” explanation;
* BuildKit/sccache/registry health;
* top cache invalidators;
* cache objects blocking merge/release.

An agent-first system must explain cache decisions. “Cache hit” is not enough; the UI needs “cache hit accepted because producer trust tier ≥ consumer requirement and lockfile/toolchain/source hashes match.”

## 9. Tests/VTI need a full planning cockpit

The Tests tab should become one of the most important screens. It should show not only slow tests, but the current validation plan.

It should answer:

* What changed?
* Which tests were selected?
* Which tests were skipped?
* Why were they skipped?
* What confidence do we have?
* What historical misses exist?
* What flakes exist?
* What would force full CI?
* Which proof lane is cheapest and sufficient?
* Which tests are required for merge?
* Which tests are required for release?

A dream Tests tab would have four panes:

```text
Changed files        Selected tests        Skipped tests        Confidence/risk
src/cache.rs         cache_unit::*         release_e2e          0.88
src/taint.rs         taint_integration     ui_snapshot          selector miss: none
```

Pressing Enter on a skipped test should explain exactly why it was skipped.

## 10. The TUI needs theme and accessibility systems

The current UI uses status colors and risk colors, but it needs a cohesive theme system.

Themes should include:

* dark default;
* light;
* high contrast;
* colorblind-safe;
* monochrome;
* presentation/demo mode;
* dense mode;
* spacious mode.

Color should encode consistent meaning:

```text
Green       safe/pass
Blue/Cyan   running/in progress
Yellow      warning/needs attention
Orange      degraded/risky
Red         failed/blocked/destructive
Magenta     policy/security/capability
Gray        stale/skipped/unknown
```

Risk colors should align with the action registry risk tiers: read-only, low, high, production. The registry already has risk metadata, so the TUI can derive colors and confirmation behavior from it. ([GitHub][5])

# Ground-up redesign

I would redesign the TUI around four principles:

1. **Every screen answers “what matters now?”**
2. **Every visual element has a direct action.**
3. **Every action has preview, proof, and audit.**
4. **Every agent operation is shown as a state machine, not a pile of logs.**

## New top-level navigation

I would keep nine tabs, but slightly reorganize them:

```text
1 Mission
2 Work Graph
3 Agents
4 Reviews
5 Tests
6 Release
7 Evidence
8 Security
9 Infra
```

Where:

* **Mission** = executive overview, blockers, next action, system health.
* **Work Graph** = pipelines/jobs/MRs/branches as a live DAG.
* **Agents** = autonomous agents, goals, state, budget, capabilities.
* **Reviews** = MRs, patch diffs, risk, comments, proof completeness.
* **Tests** = VTI, selected/skipped tests, flakes, coverage, confidence.
* **Release** = gates, canary, prod, rollback, release evidence.
* **Evidence** = capsules, audit ledger, capability grants, admission.
* **Security** = secrets, admission, permissions, taints, policy.
* **Infra** = pools, runners, cache, storage, host, Docker/GitLab status.

The current tab set is close, but “Jobs” should become “Work Graph,” “Pools/Cache” should be merged into “Infra” for normal users, and “Secrets” should broaden into “Security.” Evidence should stay first-class.

# API changes that would make the TUI world-class

## 1. Add a dedicated TUI snapshot API

The TUI should not need to assemble everything from many DB/client calls. Give it a purpose-built snapshot endpoint.

```http
GET /api/v1/tui/snapshot?view=mission
GET /api/v1/tui/snapshot?view=work-graph&mr=42
GET /api/v1/tui/snapshot?view=agent&id=cache-fixer
```

Response shape:

```json
{
  "api_version": "1.0",
  "request_id": "req_...",
  "generated_at": "2026-04-27T12:00:00Z",
  "ttl_ms": 1000,
  "stale": false,
  "source_status": {
    "gitlab": "ok",
    "db": "ok",
    "docker": "degraded"
  },
  "recommended_actions": [],
  "entities": {},
  "view": {}
}
```

This makes rendering simpler, faster, and more stable.

## 2. Add an event stream API

Polling should be replaced by typed streaming events.

```http
GET /api/v1/events/stream
```

Events:

```json
{
  "type": "job.updated",
  "id": "evt_...",
  "time": "2026-04-27T12:00:00Z",
  "entity": {
    "kind": "job",
    "id": "123"
  },
  "payload": {
    "status": "failed",
    "pipeline_id": "184",
    "required": true
  }
}
```

Important event types:

```text
pipeline.updated
job.updated
job.log_chunk
agent.transitioned
agent.blocked
capability.granted
capability.denied
admission.decision
test.plan.created
test.selector_miss
cache.verdict
cache.tainted
release.gate_updated
evidence.created
action.started
action.completed
```

This would immediately make the TUI feel alive.

## 3. Add log streaming with offsets

Current log polling should become offset-based streaming.

```http
GET /api/v1/jobs/{job_id}/logs/stream?offset=12345
```

Each chunk should include:

```json
{
  "job_id": "123",
  "offset_start": 12345,
  "offset_end": 14002,
  "ansi": true,
  "redacted": true,
  "chunk": "...",
  "detected_errors": []
}
```

This enables live logs, redaction, search indexing, and failure extraction.

## 4. Make the action registry executable

The action registry should not only list actions. It should define schemas and execution contracts.

```http
GET /api/v1/actions
GET /api/v1/actions/{id}/schema
POST /api/v1/actions/{id}/preview
POST /api/v1/actions/{id}/execute
POST /api/v1/actions/{id}/cancel
```

Action preview response:

```json
{
  "action_id": "job.retry_failed",
  "risk": "low",
  "side_effect": "gitlab_write",
  "requires_grant": false,
  "dry_run_available": true,
  "affected_entities": [
    { "kind": "job", "id": "123" }
  ],
  "warnings": [],
  "expected_events": [
    "job.retry_requested",
    "job.updated"
  ]
}
```

This would turn the command palette into a safe, powerful control surface.

```http
GET /api/v1/release-proof/{release_id}
```

Evidence should be first-class, not scattered across state tables.

## 9. Add typed errors and source-status metadata everywhere

Every API response should include:

```json
{
  "request_id": "req_...",
  "ok": true,
  "data": {},
  "errors": [],
  "warnings": [],
  "source_status": {
    "gitlab": {
      "status": "ok",
      "latency_ms": 42
    },
    "db": {
      "status": "ok",
      "latency_ms": 3
    }
  }
}
```

Generic `success/message/data` responses are convenient early on, but they limit UI quality. A great TUI needs typed errors, warnings, source freshness, and recommended actions.

## 10. Add pagination, cursors, and stable entity IDs

Any endpoint returning jobs, events, evidence, logs, tests, or agents should support cursors:

```http
GET /api/v1/evidence?cursor=...&limit=100
```

The TUI should never silently truncate important data.

```text

Cache hit rate:     ▁▂▃▆▇█▇▆ 82%
```

Use simple density blocks for risk and churn:

```text
src/cache.rs      ████████ high churn, medium risk
```

Every estimate should show confidence:

```text
ETA: 7m ±3m     confidence: 63%
```

# The single most important redesign idea

Make the TUI center on this object:

```text
Work Item
```

A Work Item could be an MR, agent task, release, failed pipeline, cache incident, or security event. Every Work Item should have:

```text
Goal
Owner / agent
State
Risk
Current blocker
Required proof
Evidence
Next action
Timeline
Affected resources
Available actions
```

Then the entire TUI becomes coherent. Instead of jumping between jobs, pools, agents, tests, and evidence as separate worlds, the user sees one mission and drills into the relevant supporting systems.

That is the path from “good Rust CI dashboard” to **world-class agent-first engineering cockpit**.
