I reviewed the **current API docs, TUI docs, CLI definitions, TUI app/rendering code, action registry, flow graph model/builder/widget, capability API, and TUI event loop**. I could not locally clone/run the repo from this environment because GitHub DNS failed here, so this is a **source/docs review**, not a runtime screenshot/UX test.

## Executive take

The current API/TUI is already pointed in the right direction, but it is still mostly an **operator status dashboard**. To be world-class, it needs to become an **agent mission-control cockpit**: real-time, decision-oriented, explainable, action-safe, keyboard-first, visually clear, and built around “what should happen next?” rather than “here are raw jobs/events.”

The biggest redesign principle:

> Every TUI screen should answer: **What is happening, why does it matter, what is blocked, what should I or the agent do next, and what proof/risk is attached?**

Right now, the system has many strong primitives: a broad CLI, a Unix-socket capability API, a webhook API, an action registry, release status, VTI/test controls, cache metrics, evidence capsules, and a Ratatui dashboard. The API docs identify the current surfaces as CLI, TUI, webhook server, capability API, Git server hook, custom executor, action registry, state API, settings API, GitLab REST client, release API, and secrets API. ([GitHub][1])

But the current UX is still fragmented: the TUI pulls from many sources, logs poll GitLab, the command palette mostly navigates instead of executing meaningful actions, the flow graph is still heuristic, the agent view is not yet a full agent-session view, and the API lacks a unified real-time TUI data contract.

# What exists now

## Current TUI shape

The current CLI supports `jeryu tui`, `--once`, and deterministic PNG capture. The accepted capture tabs are `mission`, `release`, `jobs`, `agents`, `tests`, `pools`, `cache`, `evidence`, and `secrets`. ([GitHub][1])

The current source also maps capture aliases such as `jobs|flow`, `tests|vti`, and `evidence|audit` into nine active tabs: Mission, Release, Jobs, Agents, Tests, Pools, Cache, Evidence, and Secrets. ([GitHub][2])

The TUI app state tracks pools, GitLab readiness, active containers, recent jobs, tracked pipelines, flow snapshots, live logs, cache metrics, taints, disk/storage metrics, release status, test bottlenecks, evidence, secret audit events, agent pipelines, and audit events. It also keeps selection state, command-palette state, log state, and test/evidence view modes. ([GitHub][3])

The app has three main background data streams: a general snapshot worker, a flow snapshot worker, and a live log worker. The TUI docs say the general worker refreshes pools, Docker containers, GitLab jobs, DB job events, tracked pipelines, jeryu sync, release state, cache health, storage, and test bottlenecks; the flow collector emits snapshots about every 1500 ms, and the live log worker polls GitLab traces about every 650 ms. ([GitHub][4])

## Current tab behavior

The top-level renderer draws a header, content region, event console, footer, and optional command palette. It dispatches the content region to Mission, Release, Jobs, Agents, Tests, Pools, Cache, Evidence, or Secrets. ([GitHub][5])

The Mission tab is a four-panel system overview: system health, pipeline summary, release status, and cache health. The Release tab renders a release gate matrix and an inspector. The Jobs tab has pipeline nav, release banner, flow board, jobs list, logs, and a job inspector. The Agents tab currently shows agent pipelines and a capability/audit-event history rather than full agent sessions. ([GitHub][5])

The Tests tab shows average/latest bottlenecks and drill-down history. The Pools tab shows pool state/detail. The Cache tab shows storage overview, gateway health, singleflight analytics, and taint/trust boundaries. The TUI docs also note Docker storage values are partly approximations and that cache/taint counts come from local DB queries. ([GitHub][4])

## Current flow graph

The flow model has a good foundation: `FlowSnapshot`, `PipelineFlow`, `FlowGraph`, `FlowColumn`, `LaneGroup`, `FlowNode`, `FlowEdge`, lane kinds, column kinds, ETA estimates, and backend job references. ([GitHub][6])

The current graph builder classifies jobs mostly by job-name heuristics: names containing build/compile go to Build, test/unit/integration/e2e/lint/fmt go to Tests, secret/security/guard go to Security, canary/prod/deploy/gate go to release lanes, and so on. It also builds stage-ordering edges by connecting jobs in column N to jobs in column N+1, then computes a critical path over that synthetic DAG. ([GitHub][7])

The current flow widget renders columns, lane headers, status icons, selected nodes, and critical-path badges using Ratatui buffer drawing. ([GitHub][8])

ETA is explicitly heuristic: lane defaults such as Unit 60s, Integration 300s, Security 120s, Build 180s, ReleaseExecution 120s, and Other 90s. The code comment says a real implementation should query historical bottleneck data. ([GitHub][9])

## Current command/action model

The action registry is one of the best pieces. It defines risk tiers, side-effect classes, grant requirements, surfaces, dry-run flags, labels, descriptions, and contract JSON. It is consumed by the TUI command palette, CLI action list, and capability `ListAllowedActions`. ([GitHub][10])

The registry includes useful actions such as open logs, retry job, delete local record, pause/resume pool, explain blockers, system snapshot, propose patch, race patches, request merge, plan validation, run tests, next action, tab navigation, evidence toggle, and quit. The API docs also list risk tiers and surfaces for these actions. ([GitHub][1])

But the current command palette execution path only handles tab switching and toggling the evidence/audit view; other actions are left to direct keybindings or external capability/CLI paths. ([GitHub][2])

## Current agent/capability API

The capability API is a Unix-domain-socket JSON API with a tagged `AgentIntent` enum and one request/one response. Current intents include `ProposePatch`, `RacePatches`, `RunTests`, `FetchCapsule`, `RequestMerge`, `ExplainBlockers`, `GetSystemSnapshot`, `ListAllowedActions`, and `PlanValidation`. ([GitHub][1])

Mutating intents such as `ProposePatch`, `RacePatches`, and `RunTests` create durable capability-intent/grant rows, and branch-writing grants are scoped to `refs/heads/` with a 24-hour lifetime. ([GitHub][1])

The docs also call out a key limitation: capability routing is partial, not all intents fully perform the named action, and secrets commands are still repo-specific to `dougx`. ([GitHub][1])

# The biggest gaps

## 1. The TUI is still showing state, not driving outcomes

The Mission tab shows health, pipelines, release, and cache, but it does not yet give a ranked operational agenda. It should not merely say “2 failed jobs.” It should say:

```text
NEXT BEST ACTION
1. Retry job #1842? Classified transient infra failure. Low risk. Press r.
2. Release blocked: missing C validation artifact. Run release doctor.
3. Cache taint active in test lane. Disable cache for MR !42 or clear taint.
4. Agent patch-race-12 has winner candidate. Review proof, request merge.
```

The API already has `jeryu next`, `explain-blocker`, and an action registry, but the TUI does not yet make those concepts the primary UX. The API docs describe `jeryu next` as checking recent job failures, active pipelines, release gate state, and selector misses to print suggested commands. ([GitHub][1])

**Redesign:** make “Next Best Action” the top-left pane of the entire app, always visible.

---

## 2. No unified real-time event API

The TUI currently merges DB reads, GitLab reads, Docker reads, release reports, cache checks, logs, and flow snapshots. That works for a prototype, but world-class realtime UX needs a single event spine.

Right now, the webhook server supports `/health`, `/hooks`, and `/cache/summary`; it processes Job Hook, Pipeline Hook, Push Hook, and Merge Request Hook, but MR hooks are currently only logged and not actively acted on. ([GitHub][1])

The TUI still polls logs through GitLab job trace every 650 ms, with no websocket transport. ([GitHub][4])

**Redesign:** add a first-class event stream:

```text
GET /v1/events/stream
```

Events should include:

```json
{
  "event_id": "evt_01...",
  "seq": 184223,
  "time": "2026-04-27T18:42:10Z",
  "kind": "job.failed",
  "project_id": 2,
  "entity": {"type": "job", "id": 14445},
  "severity": "high",
  "correlation_id": "pipe_8821",
  "summary": "integration:e2e failed after 11m",
  "recommended_actions": ["fetch_capsule", "retry_job", "explain_blocker"]
}
```

Then the TUI consumes one event stream plus occasional snapshots instead of directly stitching everything together.

---

## 3. Flow graph is useful but too heuristic

The current flow graph uses job-name classification and synthetic edges between columns. That means it can look like a DAG without truly being the GitLab DAG. It should ingest GitLab `needs`, bridge jobs, child pipelines, artifacts, release gates, and VTI decisions as real edges.

**Current problem:** a job called `cargo check` is inferred as Build, a job called `secret guard` as Security, and columns are connected by generated stage-order edges. That is clever, but world-class UI needs actual causality. ([GitHub][7])

**Redesign:** the Flow tab should render a **real causality graph**:

```text
Commit → Admission → Impact/VTI → Build → Unit → Integration → Security → Package → Canary → Prod
```

Each node should know:

```json
{
  "node_id": "job:14445",
  "kind": "ci_job",
  "label": "integration:e2e",
  "status": "failed",
  "required": true,
  "critical_path": true,
  "blocked_by": ["job:14412"],
  "blocks": ["gate:canary"],
  "evidence": ["capsule:14445"],
  "actions": ["open_logs", "fetch_capsule", "retry_job"]
}
```

---

## 4. ETA and progress are not trustworthy enough yet

Current ETA is mostly lane defaults, and docs say progress is heuristic rather than GitLab-native. The docs describe pipeline progress as distinct-job counts with running jobs counted as 0.5, job row progress as elapsed time over an assumed 120-second window, and flow ETA as lane-based defaults. ([GitHub][4])

**Redesign:** use historical runtime distributions:

```text
ETA: 7m20s remaining
confidence: medium
reason: same job passed 14 times in median 9m, current run at p72
critical path: build-image → integration:e2e → canary-smoke
```

For each ETA, show:

* confidence
* historical basis
* queue-time vs run-time split
* critical path
* “why this estimate changed”

---

## 5. The command palette is underpowered

The command palette is a huge opportunity, but current execution only handles navigation and evidence-view toggle. ([GitHub][2])

**Redesign:** make it the central interaction model:

```text
> retry
Retry selected job #14445
Risk: low
Will call: jeryu job retry 2 14445
Why: failed job, retry advice says transient infra
Grant: not required / or agent_task if invoked by agent
[Enter] execute  [d] dry run  [e] explain
```

For production actions:

```text
> promote
Promote release ci-a1b2c3 to production
Risk: production
Blocked: missing C validation artifact
Next: jeryu release doctor --version ci-a1b2c3
```

Every action should have:

```json
{
  "id": "retry_job",
  "enabled": true,
  "risk_tier": "low",
  "dry_run_available": true,
  "requires_grant": "agent_task",
  "blockers": [],
  "preview": "Retry GitLab job #14445",
  "command": "jeryu job retry 2 14445"
}
```

## 6. Agent tab is not yet an agent-control center

The current Agents tab shows agent pipelines and capability call history. That is useful, but an “agent-first Git” TUI needs full agent sessions: objective, branch, MR, current step, attempt count, budget, tool grants, blockers, proof, and next action.

**Dream Agent tab:**

```text
AGENT FLEET
> patch/cache-epoch-17     testing v3      MR !42     risk: high
  objective: fix cache epoch invalidation
  current: integration tests running
  next: inspect capsule if failed
  grants: run_tests approved, merge pending
  budget: 4/8 attempts, 31m used
  proof: unit green, integration pending, cache taint none
```

Right panel:

```text
SESSION TIMELINE
18:02 plan generated
18:04 patch v1 committed
18:07 unit failed: borrow checker
18:12 patch v2 committed
18:17 unit green
18:18 integration started
18:31 failure capsule captured
18:34 patch v3 committed
```

Bottom actions:

```text
[a] approve next grant   [k] kill   [h] handoff   [m] request merge   [f] force full tests
```

## 7. Evidence is not yet the center of trust

The system has evidence capsules and an evidence tab, but the UX should treat evidence as the backbone of autonomy. Every risky action should link to proof.

Evidence objects should be cross-linked:

```text
MR !42
 ├─ capability grant grant-123
 ├─ admission decision allow
 ├─ pipeline #911 green
 ├─ VTI plan selected 18 tests
 ├─ cache verdicts clean
 ├─ selector misses 0
 └─ merge proof allow
```

The API should expose a universal evidence graph:

```text
GET /v1/evidence/graph?entity=mr:42
```

---

## 8. Color and visual language need a design system

Current colors are basic Ratatui status colors: success green, running blue, failed red, pending yellow, canceled dark gray. ([GitHub][4])

That is fine, but world-class TUI colors need semantic consistency and accessibility:

```text
Green       verified/safe/completed
Blue/Cyan   active/running/selected
Yellow      waiting/attention
Orange      risk/degraded
Red         failed/danger
Magenta     trust/security/cache-taint
Gray        inactive/stale/unknown
White       primary text
Dim gray    metadata
```

Add themes:

```text
jeryu theme set dark
jeryu theme set high-contrast
jeryu theme set colorblind-safe
jeryu theme set minimal
```

And make every color meaningful. No decorative color. Color should always encode status, risk, confidence, freshness, or focus.

---

## 9. Documentation drift is already visible

The TUI doc says capture accepts the nine tabs, but the same doc still describes a five-tab header with `1: Flow 2: Storage 3: Cache 4: Audit 5: Tests`. ([GitHub][4])

The source/API clearly expose the nine-tab model. ([GitHub][2])

**Redesign:** generate TUI docs from the action registry and tab registry, or at least add a `jeryu tui schema --json` command that emits current tabs, keys, panes, actions, and data sources.

# Ground-up TUI redesign

## New top-level structure

I would replace the current nine equal tabs with a more intuitive hierarchy:

```text
1 Mission
2 Work
3 Flow
4 Agents
5 Evidence
6 Security
7 Release
8 Infra
9 Settings
```

### 1. Mission

Purpose: “What needs attention right now?”

Panels:

```text
┌ Next Best Action ┐ ┌ System Health ┐
│ 1 Retry job...   │ │ GitLab OK     │
│ 2 Release block  │ │ Runners 5/6   │
│ 3 Cache taint    │ │ Disk 72%      │
└──────────────────┘ └───────────────┘
┌ Active Incidents ┐ ┌ Agent/Risk Summary ┐
│ failed jobs      │ │ 4 agents active    │
│ blocked release  │ │ 2 grants pending   │
└──────────────────┘ └───────────────────┘
```

This should be the default tab, not Jobs.

### 2. Work

Purpose: “What work is in flight?”

Combine MRs, branches, issues, agent tasks, releases, and pipelines into one queue.

```text
WORK QUEUE
> MR !42  agent/cache-epoch       tests running   risk high
  MR !41  docs/update-readme      ready merge     risk low
  REL ci-a1b2c3                  canary wait     prod blocked
```

Right side shows selected work item:

```text
objective
owner/agent
branch/MR
status
blockers
proof
next actions
```

### 3. Flow

Purpose: real DAG and critical path.

Show current selected pipeline/release as a DAG with lanes:

```text
Commit → Admission → Impact → Build → Tests → Security → Package → Canary → Prod
                           └─ Unit ✓
                           └─ Integration ● 7m left [CRIT]
                           └─ Flake quarantine ⚠
```

Features:

* real GitLab `needs` edges
* child pipeline/bridge expansion
* critical path highlight
* zoom into lane
* compare current vs previous pipeline
* show queue time separately from runtime
* “why pending?” explanations
* VTI overlay: selected/skipped/forced tests

### 4. Agents

Purpose: supervise autonomous coding.

Show:

* agent sessions
* objectives
* attempt count
* current tool
* capability grants
* risk
* pending human approvals
* proof status
* budget/time

Agent session detail:

```text
PLAN
PATCHES
TESTS
FAILURES
GRANTS
MR
MERGE GATE
```

### 5. Evidence

Purpose: trust, replay, audit.

Two modes:

```text
Capsules | Ledger | Graph | Replay
```

Show evidence objects attached to entities:

```text
job:14445
mr:42
agent:cache-epoch-17
release:ci-a1b2c3
grant:grant-abc
```

Every evidence item should support:

```text
[e] explain  [r] replay  [c] copy repro  [o] open related entity
```

### 6. Security

Purpose: “Can I trust what is happening?”

Show:

* active capability grants
* denied/admitted refs
* secret rotations
* Vault health
* cache taints
* honeypot/tripwire events
* sandbox breaches
* untrusted agent actions
* production-impacting pending approvals

This should replace burying security details in cache/evidence/secrets.

### 7. Release

Purpose: ship-room.

Show release as a vertical gate ladder:

```text
Main green              ✓
Release identity        ✓
Release pipeline        ● running
Canary deploy           waiting
Canary health           waiting
E2E gate                waiting
Production promotion    blocked
Rollback ready          ✓
```

For each gate:

```text
status
required proof
blocking entity
recommended action
owner/agent
artifact path
```

### 8. Infra

Purpose: runners/cache/storage/performance.

Merge Pools, Cache, Storage, host doctor, and reclaim into one tab:

```text
Runner Pools
Cache
Storage
Network/Gateway
Bottlenecks
```

Visuals:

* runner saturation bars
* queue depth by pool
* disk usage gauge
* cache hit-rate sparkline
* taint map
* reclaim plan preview

### 9. Settings

Purpose: project/profile/control-plane settings.

Show:

* active project/ref
* GitLab URL/project ID
* release repo
* webhook bind
* cache ports
* Vault mount/prefix
* TUI refresh settings
* theme
* keymap

The current settings schema includes TUI refresh limits, release defaults, cache ports, Vault settings, GitLab ports, and sandbox settings. ([GitHub][1])

# API redesign to make the TUI world-class

## 1. Add a unified TUI snapshot API

Right now, the TUI constructs snapshots internally from DB/GitLab/Docker/release/cache/test sources. That logic should become a reusable API.

```text
GET /v1/tui/snapshot?project_id=2&ref=main
```

Response:

```json
{
  "schema_version": "tui.snapshot.v1",
  "generated_at": "2026-04-27T18:42:00Z",
  "freshness": {
    "db_lag_ms": 42,
    "gitlab_lag_ms": 600,
    "docker_lag_ms": 300,
    "stale": false
  },
  "mission": {
    "health_score": 87,
    "top_blockers": [],
    "next_actions": []
  },
  "work": [],
  "agents": [],
  "pipelines": [],
  "release": null,
  "security": {},
  "infra": {},
  "evidence": []
}
```

This gives TUI, CLI, agents, and web clients the same source of truth.

## 2. Add streaming deltas

```text
GET /v1/events/stream?after=184220
```

Event kinds:

```text
job.created
job.running
job.failed
job.succeeded
pipeline.created
pipeline.superseded
pipeline.blocked
agent.started
agent.step.completed
agent.grant.requested
grant.approved
grant.denied
release.gate.blocked
cache.taint.created
cache.taint.cleared
secret.rotated
admission.denied
evidence.capsule.created
recommendation.created
```

The TUI should subscribe to events and update panels incrementally. Snapshots are for startup/recovery; events are for live UX.

## 3. Add action dry-run/execute APIs

The action registry already has the right metadata. Now make it executable.

```text
GET  /v1/actions
POST /v1/actions/{id}/dry-run
POST /v1/actions/{id}/execute
```

Dry-run response:

```json
{
  "action": "request_merge",
  "enabled": false,
  "risk_tier": "production",
  "required_grant": "merge_approval",
  "blockers": [
    {
      "code": "selector_miss_recent",
      "message": "1 selector miss in the last 24h",
      "fix": "Run full validation or update VTI map"
    }
  ],
  "would_call": "jeryu agent merge 2 42 --trust-tier trusted"
}
```

This makes the command palette safe and useful.

## 4. Add agent-session APIs

The current capability API has agent-like intents, but it lacks a durable session model.

```text
GET  /v1/agent-sessions
GET  /v1/agent-sessions/{id}
POST /v1/agent-sessions/{id}/pause
POST /v1/agent-sessions/{id}/resume
POST /v1/agent-sessions/{id}/cancel
POST /v1/agent-sessions/{id}/grant
POST /v1/agent-sessions/{id}/handoff
```

Session shape:

```json
{
  "agent_id": "agent_cache_epoch_17",
  "objective": "Fix cache epoch invalidation",
  "branch": "agent/cache-epoch-17",
  "mr_iid": 42,
  "state": "testing",
  "current_step": "integration:e2e",
  "attempt": 3,
  "budget": {"attempts_used": 3, "attempts_max": 8},
  "risk": "high",
  "grants": [],
  "proof": {
    "unit": "green",
    "integration": "running",
    "selector_misses": 0,
    "cache_taints": 0
  },
  "next_action": {
    "id": "fetch_capsule",
    "why": "integration failed on previous attempt"
  }
}
```

## 5. Add real flow graph API

```text
GET /v1/pipelines/{pipeline_id}/flow
```

Return actual nodes/edges, not name-classification only:

```json
{
  "pipeline_id": 911,
  "graph": {
    "nodes": [],
    "edges": []
  },
  "critical_path": [],
  "eta": {
    "remaining_secs": 441,
    "confidence": "medium",
    "reason": "historical p70 for same jobs"
  }
}
```

The current GitLab client wrapper already has methods for pipeline jobs, bridges, downstream jobs, and pipeline graph-ish data. ([GitHub][1])

## 6. Add log streaming, search, and semantic folding

Current live logs are polling-based and keep only a capped tail. The source caps live log text at 160,000 bytes. ([GitHub][4])

Add:

```text
GET /v1/jobs/{job_id}/logs/stream
GET /v1/jobs/{job_id}/logs/search?q=error
GET /v1/jobs/{job_id}/logs/summary
```

The TUI log pane should support:

* regex search
* error folding
* warnings/errors counters
* “first failure” jump
* ANSI color preservation
* structured test output
* copy repro command
* capsule link
* previous-run diff

## 7. Add recommendation API

```text
GET /v1/recommendations?scope=mission
```

Recommendation object:

```json
{
  "id": "rec_01",
  "priority": 95,
  "severity": "high",
  "title": "Release blocked by missing C validation",
  "why": "prod promotion requires c-validation.json",
  "entity": {"type": "release", "id": "ci-a1b2c3"},
  "actions": [
    {"id": "release_doctor", "label": "Run doctor"},
    {"id": "open_evidence", "label": "Show evidence"}
  ],
  "confidence": "high"
}
```

This is the most important API for making the TUI feel intelligent.

## 8. Add typed errors everywhere

Current capability responses are basically:

```rust
CapabilityResponse { success, message, data }
```

That is not enough for a great TUI. Add codes, blockers, next actions, and correlation IDs:

```json
{
  "success": false,
  "code": "merge_gate_denied",
  "message": "Merge denied",
  "correlation_id": "req_abc",
  "blockers": [
    {
      "code": "cache_taint_active",
      "severity": "high",
      "explain": "1 active cache taint affects this branch",
      "fix_actions": ["show_cache_taints", "run_cold_validation"]
    }
  ],
  "next_actions": ["explain_blockers", "run_tests"]
}
```

## 9. Add idempotency and pagination

Agent and TUI calls need to be safely retryable:

```text
Idempotency-Key: action_retry_job_14445_20260427
```

Every list endpoint should support:

```text
limit
cursor
sort
filter
```

Every mutating response should include:

```text
request_id
action_id
ledger_event_id
grant_id
entity_refs
```

## 10. Add schema introspection

```text
GET /v1/schema
GET /v1/tui/schema
GET /v1/keymap
GET /v1/theme
```

This lets the TUI, agents, docs, tests, and external clients stay consistent.

```text
e2e              ○░░░░░░░░
prod             ⛔ blocked
```

## 7. Add “why” labels everywhere

Every blocked or red/yellow thing should be explainable inline:

```text
blocked: cache taint active
stale: GitLab unreachable for 42s
waiting: runner pool untrusted has no warm managers
pending: previous stage build-image still running
```

## 8. Add compare mode

For pipelines:

```text
Current #912 vs Previous green #908
+ integration:e2e duration +48%
+ new security job added
- docs job skipped by VTI
failure signature seen 3 times before
```

For agent patches:

```text
Patch A small diff, green
Patch B larger diff, failed compile
Patch C green but introduced perf regression
```

## 9. Add approval modal UX

For risky actions:

```text
REQUEST MERGE !42

Risk: production
Required proof:
✓ pipeline green
✓ selector misses 0
✓ cache taints 0
✓ release impact none
⚠ codeowner approval missing

Decision: blocked

[e] explain blocker   [f] force full validation   [Esc] cancel
```

Never let a high-risk command execute from a single ambiguous keypress.

# What should be removed or consolidated

## Merge Pools, Storage, Cache into Infra

Pools, cache, storage, host doctor, reclaim, and gateway health are all infrastructure. Splitting them makes sense for implementation, but not UX. Users care about:

```text
Can work run?
Is it fast?
Is it safe?
Will disk explode?
```

## Merge Evidence, Audit, Secrets into Trust/Security plus Evidence

Secrets lifecycle belongs in Security/Trust. Evidence should be universal and entity-based.

## Keep Release separate

Release is important enough to remain its own ship-room.

## Make Jobs a detail view, not the default

Jobs are implementation detail. The default should be Mission or Work.

# Concrete TUI implementation changes

1. **Move event handling out of `mod.rs` into `events.rs`.** The current `events.rs` is a stub and event handling lives in the run loop. ([GitHub][11])

2. **Make command palette execute all registered actions.** It should call action dry-run/execute and show blockers/risk.

3. **Make `Mission` the default tab.** The current default active tab is Jobs in the app state. ([GitHub][3])

4. **Replace raw polling with event subscriptions where possible.** Keep polling fallback.

5. **Replace heuristic graph edges with real GitLab DAG/bridge edges.**

6. **Add an entity inspector component used by every tab.**

7. **Add global search/filter.**

8. **Add theme abstraction.** No hard-coded colors scattered through render functions.

9. **Add generated docs/schema from current tab/action registries.**

10. **Add project/ref selector.** Too much current API/TUI behavior defaults to project `2`, ref `main`, and repo-specific release/secrets assumptions. ([GitHub][1])

# Final verdict

The current API/TUI has a strong foundation: typed capability intents, broad CLI coverage, an action registry with risk metadata, release/test/cache/evidence state, and a Ratatui dashboard with multiple background streams. But the current UX is still too much like a status board.

To make it world-class, redesign around five primitives:

1. **Recommendations** — what should happen next.
2. **Entity inspector** — why the selected thing matters.
3. **Evidence graph** — what proof exists.
4. **Action dry-run/execute** — safe control from the TUI.
5. **Real-time event stream** — no stale stitched-together polling as the primary UX.

The most important API upgrade is a unified `TuiSnapshot + EventStream + Action API`. The most important TUI upgrade is a Mission Control view where every line is actionable, explainable, and tied to proof.
