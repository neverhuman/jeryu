I reviewed the current API docs, TUI docs, CLI definitions, TUI source, action registry, flow model/builder/widget code, and capability API. I did not run the repo locally in this pass; this is a source/API/TUI design review.

## Bottom line

The current TUI is already much better than a generic CI dashboard: it has mission/release/job/agent/test/cache/evidence/security-ish surfaces, deterministic capture mode, live logs, flow visualization, command palette hooks, and a shared action registry. But it still feels like an **operator dashboard bolted onto backend state**, not yet a **world-class command center for agent-led development**.

The biggest opportunity is to redesign the API and TUI around one question:

> “What is happening, why does it matter, what is blocked, and what is the safest next action?”

Right now, the TUI has many panes, but the API does not yet provide a clean realtime, typed, decision-oriented state model. The result is that several screens show raw status, partial approximations, or placeholder summaries instead of deeply actionable insights.

# 1. What the current API/TUI already does well

## Current API strengths

The API surface is broad and ambitious. `docs/API.md` describes the project as having CLI, hidden hook commands, a custom executor, webhook HTTP API, Unix-socket capability API, GitLab client wrapper, state API, release controls, test intelligence, secrets, action registry, settings, and operational hooks. That is the right shape for an agent-first Git system. ([GitHub][1])

The TUI can be launched normally, smoke-tested once, or rendered deterministically to PNG with selectable tabs including `mission`, `release`, `jobs`, `agents`, `tests`, `pools`, `cache`, `evidence`, and `secrets`. That capture mode is genuinely valuable for CI artifacts, demos, audits, and agent-visible snapshots. ([GitHub][1])

The capability API is especially promising. It exposes typed `AgentIntent` requests over a Unix socket, including `ProposePatch`, `RacePatches`, `RunTests`, `FetchCapsule`, `RequestMerge`, `ExplainBlockers`, `GetSystemSnapshot`, `ListAllowedActions`, and `PlanValidation`. That is far better than giving agents raw shell access or forcing them to scrape CLI output. ([GitHub][1])

The action registry is also a strong foundation. It defines actions with IDs, labels, key hints, risk tiers, surfaces, dry-run support, descriptions, side-effect classes, and required grants. It already separates read-only, low-risk, high-risk, and production-risk actions, and exposes actions across CLI, TUI, and capability surfaces. ([GitHub][1])

## Current TUI strengths

The TUI is built with Ratatui/crossterm, has an event loop, polls input, and keeps a state snapshot from GitLab, Docker, the DB, release state, cache state, tests, jeryu sync, logs, and other internal state. The docs describe nine tabs: Mission, Release, Jobs, Agents, Tests, Pools, Cache, Evidence, and Secrets. ([GitHub][2])

It already has background workers for flow collection, live log polling, and general snapshot sync. The docs describe a general refresh around 1500ms, flow refresh around 1500ms, and live log polling around 650ms. It also has anti-blanking behavior so stale flow/log state is preserved rather than flashing empty. ([GitHub][2])

The source has moved beyond the older docs in useful ways. The current draw path has a header, content area, event console, footer, maximized log mode, tab switching, and command palette rendering. ([GitHub][3])

The Mission tab already tries to summarize system health: GitLab status, containers, pools, proxy/registry, tracked pipelines, running/failed jobs, ETA, release state, and cache health. That is the correct default concept. ([GitHub][3])

The Release tab has a gate matrix with rows for main branch, release identity, release pipeline, canary health, E2E gate, and prod promotion, plus an inspector showing release detail. This is the right pattern: a release screen should be a gate board, not just a log viewer. ([GitHub][3])

The Jobs tab source now has a strong layout direction: left pipeline nav, center release banner/flow board/jobs/logs, and right inspector. That is much closer to a true control room than the older documented flow-only layout. ([GitHub][3])

The flow model has useful building blocks: `PipelineFlow`, `FlowGraph`, `FlowNode`, columns, lanes, critical path fields, ETA fields, backend labels, elapsed time, and progress. The builder classifies jobs by name into columns/lanes, creates stage-ordering edges, and computes a critical path. The widget renders status icons, colors, selected-node markers, and critical-path badges. ([GitHub][4])

# 2. The biggest current gaps

## Gap 1: The TUI still shows “state,” not enough “judgment”

The current TUI can tell me jobs are running, logs are streaming, release state exists, cache has metrics, and agents have pipelines. But the core product question should be:

> “What should I do next?”

The API already has `next`, `explain-blocker`, action listing, and capability intents, but the TUI does not yet make “next best action” the center of every screen. The current footer exposes shortcuts like retry, delete, pause, logs, and quit; the event console shows recent evidence events; but there is no universal **Action Panel** that ranks safe next actions based on the selected object. ([GitHub][3])

The redesign should make every selected object produce:

```text
Status
Why it matters
Blockers
Confidence
Evidence
Recommended next action
Available actions
Risk of each action
```

## Gap 2: The API is command-rich but not state-model-rich

The CLI has many commands, the capability API has typed intents, and the webhook server receives job/pipeline/push hooks. But the TUI still has to assemble its own view from DB, Docker, GitLab, filesystem, and release evidence. The docs explicitly say the app owns DB, Docker, and GitLab clients. ([GitHub][2])

That creates a design problem: the TUI is doing too much collection and interpretation. A world-class TUI should consume a normalized API:

```text
/v1/snapshot
/v1/events/stream
/v1/jobs/:id
/v1/pipelines/:id/graph
/v1/agents/:id
/v1/merge-gates/:mr
/v1/evidence
/v1/actions
/v1/actions/:id/dry-run
/v1/actions/:id/execute
```

The TUI should not be the place where GitLab state, Docker state, DB state, release state, and agent state are reconciled. That should live in a `jeryud` or internal state service.

## Gap 3: Realtime is polling-heavy

The current docs say live logs are polled, not streamed through a websocket. They also describe polling for flow and general snapshots. ([GitHub][2])

Polling is fine for an MVP, but a world-class control room needs event streams:

```text
job.status_changed
job.log_appended
pipeline.graph_changed
agent.state_changed
capability.granted
capability.denied
merge_gate.changed
release.stage_changed
cache.taint_added
secret.scan_failed
admission.denied
```

The TUI should render from a durable event stream with cursors, not periodically reconstruct the universe.

## Gap 4: The API has some agent-critical gaps that directly weaken the TUI

The capability `RequestMerge` path currently appears advisory and under-evidenced. In the reviewed source, it checks selector misses and cache taints, then calls the merge gate with `head_sha: None`, fake successful/pending/failed job counts, and disabled evidence requirements. That makes it impossible for the TUI to present a trustworthy “agent requested merge” state from that API alone. ([GitHub][5])

`ExplainBlockers` supports job/release-style entities, but the implementation path shown does not actually implement a real merge blocker explanation even though the response text mentions `merge` as an option. ([GitHub][5])

`PlanValidation` appears to count selector misses globally over a recent time window rather than being truly project/ref/diff bound. That limits how useful it is for a TUI test-planning panel. ([GitHub][5])

`RacePatches` returns branch/pipeline information but says the user should poll pipeline status to determine the winner and notes that losing branches need manual cleanup or a future `PollRaceResult` implementation. That means the API does not yet give the TUI a complete “race board” lifecycle. ([GitHub][5])

## Gap 5: The Agents tab is not yet an agent command center

The Agents tab source describes itself as placeholder-like and mostly renders agent pipelines plus capability/audit history. It shows refs, SHAs, statuses, pipeline IDs, timestamps, and recent capability-related events, but not a full agent task lifecycle. ([GitHub][3])

A world-class agent TUI needs much more:

```text
Agent objective
Current hypothesis
Current branch
MR
Diff summary
Budget used
Tokens/tool calls/runtime
Allowed capabilities
Denied capabilities
Current blocker
Next planned action
Confidence
Risk
Test plan
Evidence generated
Human intervention needed?
```

## Gap 6: The flow graph is useful but still heuristic

The flow builder classifies columns and lanes by job-name matching. It creates a graph from job stage ordering and computes a critical path from that constructed graph. ([GitHub][4])

That is a good prototype, but it can be misleading. A production-grade pipeline graph should use real GitLab DAG information:

* stages
* `needs`
* bridges
* child pipelines
* downstream pipelines
* artifacts dependencies
* manual jobs
* skipped jobs
* allow-failure jobs
* environment/deployment jobs
* retry lineage
* cancellation lineage

The TUI should visually separate “actual DAG” from “inferred grouping.”

## Gap 7: The docs and source appear out of sync in places

The current API/CLI/source use nine TUI tabs, while parts of the TUI doc still describe older numeric keybindings like `1 Flow`, `2 Storage`, `3 Cache`, `4 Audit`, `5 Tests`. The docs also describe some inactive renderers and old panes. ([GitHub][2])

For world-class UX, docs, keybindings, source, API, capture tabs, screenshots, and help overlays need to be perfectly aligned.

## Gap 8: The Audit/Evidence experience is not yet deep enough

The docs describe the Audit tab as placeholder-like with no interactive audit drill-down. ([GitHub][2])

The source has an event console and evidence-related panels, which is good, but the system needs a first-class immutable ledger view:

```text
Capability requested
Capability granted
Patch proposed
Branch pushed
Pipeline started
Job failed
Evidence capsule created
Selector miss recorded
Cache taint detected
Merge gate blocked
Release promoted
Admission denied
```

For agents, this is not just UI. It is trust infrastructure.

# 3. Ground-up redesign: the TUI should be “Mission Control”

I would redesign the TUI around a three-pane mental model:

```text
┌────────────────────────────────────────────────────────────────────┐
│ Global status: GitLab ●  Runners ●  Agents ●  Release ⚠  Cache ✓   │
├───────────────────┬───────────────────────────────┬────────────────┤
│ Navigation / Now  │ Main visual workspace          │ Inspector      │
│                   │                               │                │
│ Mission           │ Pipeline graph / agent map /   │ Selected item  │
│ Pipelines         │ merge gate / test map          │ status         │
│ Agents            │                               │ blockers       │
│ Merge Gate        │                               │ evidence       │
│ Tests             │                               │ actions        │
│ Evidence          │                               │                │
│ Security          │                               │                │
│ Cache/Fleet       │                               │                │
│ Release           │                               │                │
├───────────────────┴───────────────────────────────┴────────────────┤
│ Event stream: newest critical events, denials, failures, next steps │
└────────────────────────────────────────────────────────────────────┘
```

The default screen should not be Jobs. It should be **Mission**.

The Mission screen should answer, in this order:

1. Is the system healthy?
2. What is currently active?
3. What is blocked?
4. What is risky?
5. What action should I take next?
6. What are agents doing?
7. Is anything unsafe?

A rough mockup:

```text
JERYU Mission Control   GitLab ✓  Runners 8/10  Agents 3  Release BLOCKED  Cache ✓
───────────────────────────────────────────────────────────────────────────────

NEXT ACTION
  Fix selector miss blocking !42
  Why: agent/cache-race has green unit tests, but validation selector missed integration/cache_eviction
  Suggested: Run missing proof lane
  Action: [r] run missing tests   [x] explain   [a] open actions

ACTIVE WORK
  !42 agent/cache-race     validating   92%   blocked by selector miss
  !43 agent/tui-redesign   testing      44%   unit tests running
  rel 2.0.1 canary         watching     73%   telemetry gate pending

CRITICAL SIGNALS
  Failures: 1 new, 0 unretriable
  Security: 0 secret leaks, 2 admission denials
  Cache: 3 taints active, no trusted-lane contamination
  Fleet: queue pressure medium, rust-build pool saturated

EVENTS
  12:41 job 8821 failed: integration/cache_eviction
  12:40 capability denied: RequestMerge !42 missing merge proof
  12:38 agent a-19 proposed patch on agent/cache-race
```

That is the difference between a dashboard and a command center.

# 4. TUI redesign by screen

## A. Header: make it semantic, not decorative

The current header shows tabs, GitLab status, containers, pools, release, and stale status. ([GitHub][3])

I would change it to a compact, always-visible system strip:

```text
GitLab ✓  API 42ms | Runners 8/10 | Queue 14 | Agents 3 | Release BLOCKED | Sec ✓ | Cache ⚠ | Stream live
```

Rules:

* Green means verified healthy.
* Blue means running.
* Amber means waiting/risky.
* Red means blocked/failing.
* Purple means security/taint.
* Gray means stale/unknown.
* Flashing or bold should be used only for state changes, not constant decoration.
* Never rely on color alone: use icons, words, and shapes.

The header should also show event stream freshness:

```text
Stream live
Stream stale 8s
GitLab degraded
DB offline
```

## B. Mission tab: become the primary homepage

The Mission tab should be the first tab and default tab.

It should have four blocks:

```text
1. Next Action
2. Active Work
3. Risk & Blockers
4. Recent Events
```

The current Mission tab already collects health, pipeline, release, and cache information. ([GitHub][3]) The redesign should make it judgment-driven:

```text
NEXT ACTION
  [HIGH] Merge blocked: !42 missing integration proof
  Press x to explain, r to run missing proof, a for actions

BLOCKERS
  !42 selector miss unresolved
  Release 2.0.1 canary telemetry pending
  rust-build pool queue > 10m

SYSTEM
  GitLab online
  Runner pools 4/5 healthy
  Cache healthy but 3 active taints
  Security no active leaks
```

## C. Jobs/Pipelines tab: make the graph real and interactive

The current source has a good three-column Jobs layout: pipeline nav, center flow/jobs/logs, and right inspector. ([GitHub][3]) I would keep that structure but make it much richer.

### Pipeline nav

Left column:

```text
PIPELINES
● #881 main             running   73%  ETA 4m
✕ #880 agent/cache      failed    1 blocker
✓ #879 main             passed    8m ago
○ #878 release/2.0.1    pending   manual gate
```

Add filters:

```text
f filter   s sort   / search   space pin   enter inspect
```

### Flow board

The center should show an actual DAG, not only inferred stage columns:

```text
prepare       build             test                 release
  ✓ env  ──▶  ✓ cargo build ──▶  ● unit       ──▶     ○ package
             └──────────────▶   ✕ integration
                                ⚠ security scan
```

Each node should display:

```text
status icon
job name
duration / ETA
retry count
cache state
required/optional marker
critical-path marker
```

The current widget already has status icons, colorization, selection, and `[CRIT]` badges. ([GitHub][6]) Build on that, but make the graph data real.

### Inspector

The current job inspector shows selected job details, evidence capsule summary, snippet, and actions like retry/delete. ([GitHub][3])

Upgrade it into:

```text
JOB INSPECTOR
  Job: integration/cache_eviction
  Status: failed
  Failure kind: regression
  First bad line: src/cache.rs:244
  Repro: cargo nextest run cache_eviction
  Flake probability: 4%
  Blocks: !42 merge gate, release 2.0.1
  Evidence: capsule #abc123
  Cache: trusted, no taint
  Suggested action:
    1. Ask agent a-19 to patch
    2. Retry not recommended
    3. Run related tests after patch

ACTIONS
  x explain
  r retry
  p propose agent fix
  c copy repro
  e open evidence
```

Current Agents tab is still too close to “pipelines related to agents.” It needs to become the place where human operators supervise autonomous work.

Agent list:

Agent detail:


## E. Merge Gate tab: add this as a first-class tab

This is the most important missing screen.

Autonomous coding lives or dies by safe merges. The TUI needs a dedicated **Merge Gate** screen.

```text
MERGE GATE  !42  agent/cache-race → main

DECISION: BLOCKED

HEAD
  MR head SHA:       abc123
  Tested SHA:        abc123
  Base SHA:          def456
  Branch protected:  yes

EVIDENCE
  ✓ unit tests passed
  ✓ lint passed
  ✓ no pending required jobs
  ✗ integration/cache_eviction missing
  ✓ no failed required jobs
  ✓ no secret scan findings
  ⚠ cache taints active but unrelated
  ✗ selector miss unresolved
  ✓ capability grant valid
  ✗ merge proof not signed

NEXT ACTION
  Run missing validation lane: integration/cache_eviction
```

Actions:

```text
x explain blocker
r run missing proof
a ask agent to fix
p print proof JSON
m request merge
```

This requires API improvements. `RequestMerge` must be real, exact-SHA-bound, and evidence-bound. The current advisory merge path is not enough for a trustworthy TUI gate. ([GitHub][5])

## F. Tests tab: turn VTI into a visual test-decision system

The current Tests tab is primarily bottlenecks/history oriented, with duration coloring. ([GitHub][2])

The world-class version should show:

```text
TEST INTELLIGENCE

Diff: abc123 → def456
Confidence: 87%
Runtime saved: 34m
Risk: medium

SELECTED TESTS
✓ unit/cache_core             reason: src/cache_brain.rs changed
● integration/cache_eviction  reason: previous selector miss
○ security/secrets            reason: auth-adjacent file changed

SKIPPED TESTS
docs/build                    reason: docs unaffected
frontend/e2e                  reason: no frontend paths changed

MISSES
1 unresolved selector miss from last 24h

ACTIONS
r run selected
f force full
l learn from failure
x explain selection
```

Every selected/skipped test should have a reason. Agents should be able to query the same explanations through API.

## G. Evidence tab: make it the trust ledger

Evidence should not be a simple recent-events display. It should be a searchable ledger.

```text
EVIDENCE LEDGER

Type          Entity        Result     Actor      Time
capability    patch         granted    agent-19   12:38
job           unit          passed     runner-4   12:40
capsule       integration   failed     runner-2   12:41
merge-gate    !42           blocked    jeryu       12:42
admission     ref-update    denied     hook       12:43
```

Selecting an event:

```text
EVIDENCE DETAIL
  Hash: sha256:...
  Entity: job 8821
  Bound SHA: abc123
  Project: 2
  Branch: agent/cache-race
  Summary: integration/cache_eviction failed
  Repro command: ...
  Related blockers: !42 merge gate
  Related actions: retry, agent fix, run related tests
```

The current docs say Audit is placeholder-like with no interactive drill-down. ([GitHub][2]) This should become one of the product’s strongest features.

## H. Security tab: split it from “Secrets”

The current capture tabs include `secrets`, and the source has secret/audit-related state, but agent-first systems need a broader security command center. ([GitHub][1])

Security tab:

```text
SECURITY

  ✓ no escapes
  ✓ network policy active
  ⚠ 2 denied egress attempts

  ✓ no leaks in latest diff

  ⚠ 4 audit-only ref updates
```

This screen should explain whether the system is safe for autonomous execution.

## I. Cache/Fleet tab: make cache actionable

The current Cache tab has storage overview, gateway health, singleflight analytics, and trust/taint panels. It estimates some values and counts DB-backed taints/verdicts. ([GitHub][2])

Make it answer:

```text
Is cache safe?
Is cache causing failures?

```

Example:

```text
  purge tainted
  explain taint
  promote trusted
  cold-run selected proof
```

# 5. API redesign needed to make the TUI world-class

## A. Create a normalized TUI/state API

Right now the TUI aggregates from multiple systems. Instead, create a single normalized state service:

```text
GET /v1/tui/snapshot
GET /v1/tui/events?cursor=...
GET /v1/tui/events/stream
GET /v1/tui/logs/:job_id/stream
GET /v1/tui/entities/:kind/:id
```

Snapshot shape:

```json
{
  "version": "1",
  "observed_at": "2026-04-27T12:42:00Z",
  "freshness": {
    "gitlab": "live",
    "db": "live",
    "docker": "stale",
    "release": "live"
  },
  "system": {},
  "mission": {},
  "pipelines": [],
  "agents": [],
  "merge_gates": [],
  "tests": {},
  "evidence": [],
  "security": {},
  "cache": {},
  "fleet": {},
  "release": {}
}
```

The key is that every field must include freshness and source confidence. The TUI should never silently mix live GitLab state with stale DB state.

## B. Add a durable event stream

Event envelope:

```json
{
  "cursor": "evt_000001923",
  "type": "job.failed",
  "severity": "error",
  "entity": {
    "kind": "job",
    "id": 8821,
    "project_id": 2
  },
  "summary": "integration/cache_eviction failed",
  "blocked_entities": [
    {"kind": "merge_request", "id": 42}
  ],
  "next_actions": [
    {
      "action_id": "explain_blockers",
      "label": "Explain blocker",
      "risk_tier": "read_only"
    },
    {
      "action_id": "run_missing_tests",
      "label": "Run missing proof",
      "risk_tier": "low"
    }
  ],
  "observed_at": "2026-04-27T12:41:03Z"
}
```

The TUI should subscribe to topics:

```text
jobs
pipelines
agents
merge_gates
release
security
cache
logs
```

This would replace most polling and make the interface feel alive.

## C. Make logs incremental

The docs say live logs are currently polled from GitLab job traces every 650ms. ([GitHub][2])

Add:

```text
GET /v1/jobs/:id/log?offset=12345
GET /v1/jobs/:id/log/stream
```

Return:

```json
{
  "job_id": 8821,
  "offset": 12345,
  "next_offset": 13888,
  "complete": false,
  "chunks": [
    {
      "kind": "stderr",
      "text": "...",
      "annotations": [
        {"range": [120, 144], "kind": "error"}
      ]
    }
  ]
}
```

The TUI can then preserve scrollback, annotate failures, and avoid re-fetching entire traces.

## D. Make the action registry executable through one API

The action registry is already a strong foundation. ([GitHub][7])

Expose it as:

```text
GET  /v1/actions
POST /v1/actions/:id/dry-run
POST /v1/actions/:id/execute
```

Invocation:

```json
{
  "target": {
    "kind": "job",
    "id": 8821
  },
  "actor": "tui:user",
  "idempotency_key": "uuid",
  "reason": "Retry after transient network failure"
}
```

Response:

```json
{
  "request_id": "act_123",
  "decision": "requires_confirmation",
  "risk_tier": "high",
  "side_effects": ["rerun_job", "consume_runner_minutes"],
  "required_grant": null,
  "preview": "Retry job integration/cache_eviction on pipeline #881",
  "confirmation_text": "Retry failed job 8821?"
}
```

The TUI command palette should be generated from this API. No duplicated shortcut logic.

## E. Add real merge-gate API

Add:

```text
GET /v1/merge-gates/:project_id/:mr_iid
POST /v1/merge-gates/:project_id/:mr_iid/explain
POST /v1/merge-gates/:project_id/:mr_iid/request-merge
```

Response:

```json
{
  "decision": "blocked",
  "mr_iid": 42,
  "source_branch": "agent/cache-race",
  "target_branch": "main",
  "head_sha": "abc123",
  "tested_sha": "abc123",
  "required_evidence": [
    {
      "id": "unit",
      "status": "passed",
      "bound_sha": "abc123"
    },
    {
      "id": "integration/cache_eviction",
      "status": "missing",
      "reason": "selector miss requires proof"
    }
  ],
  "blockers": [
    {
      "kind": "missing_test",
      "severity": "high",
      "summary": "integration/cache_eviction proof missing",
      "next_action": "run_missing_tests"
    }
  ],
  "proof": null
}
```

The current `RequestMerge` path needs to become exact-SHA-bound and based on real MR/job evidence before the TUI can safely present it as anything more than advisory. ([GitHub][5])

# 8. Specific API changes that would improve the TUI immediately

## P0 API changes

1. **Add `/v1/tui/snapshot`**
   One normalized snapshot, with freshness, confidence, source timestamps, and typed entities.

2. **Add `/v1/events/stream`**
   SSE/WebSocket/local stream with durable cursors and event topics.

3. **Add incremental log streaming**
   Replace repeated whole-trace polling with offset-based chunks.

4. **Add `/v1/actions`, `/dry-run`, `/execute`**
   Drive the command palette from the action registry.

5. **Fix `RequestMerge`**
   Require real MR data, real head SHA, real pipeline/job evidence, real policy, and return a merge-gate object.

6. **Implement merge-aware `ExplainBlockers`**
   The TUI needs blocker explanations for jobs, releases, agents, and merge requests.

7. **Make `PlanValidation` diff-bound**
   It should be project/base/head/path aware, not only global miss aware.

8. **Create first-class agent task endpoints**
   The Agents tab should consume agent tasks, not infer from pipelines.

9. **Expose real pipeline DAGs**
   Use GitLab stages/needs/bridges/downstreams, not only name-based classification.

10. **Add evidence ledger endpoints**
    Make audit/capsule/proof data queryable and drillable.

## P1 API changes

1. **Add subscription filters**
   The TUI should subscribe only to visible/selected entities when needed.

2. **Add idempotency keys to all mutating actions**
   Essential for retries and flaky network conditions.

3. **Add action correlation IDs**
   Every TUI action should produce events tied to one request ID.

4. **Add entity relationships**
   Job → pipeline → MR → agent → release → evidence.

5. **Add freshness/confidence metadata everywhere**
   The TUI should never hide stale state.

6. **Add structured failure analysis**
   Every failed job should expose failure kind, root-cause guess, repro command, flake probability, and blocker impact.

7. **Add permission/grant previews**
   Before rendering an action, the TUI should know whether the selected actor can perform it.

# 10. The highest-impact redesign idea

The TUI should stop being a set of tabs and become an **entity/action/evidence system**.

Every visible thing should be an entity:

```text
job
pipeline
agent
merge_request
release
test_plan
cache_object
secret_event
capability_grant
evidence_capsule
runner_pool
```

Every entity should have:

```text
status
freshness
risk
blockers
relationships
evidence
available actions
recommended next action
```

Every action should have:

```text
risk tier
side effects
dry run
confirmation requirements
grant requirements
result event
```

Every result should generate evidence.

That creates a loop:

```text
State → Insight → Action → Evidence → Updated State
```

That is the product loop for a world-class agent-first TUI.

# 11. Final recommended roadmap

## First 2 weeks: make current TUI coherent and actionable

* Make Mission default.
* Fix tab/keybinding/doc mismatch.
* Add top-level next-action panel.
* Wire command palette to action registry.
* Add confirmation modal system.
* Improve Jobs inspector with blocker/evidence/action sections.
* Add freshness labels everywhere.
* Make Evidence tab interactive.
* Start normalized snapshot API.

## Next 4–6 weeks: make it realtime and agent-native

* Add event stream with cursors.
* Add incremental log streaming.
* Add real agent task API.
* Add Merge Gate tab and API.
* Fix `RequestMerge`.
* Implement merge-aware `ExplainBlockers`.
* Add validation-plan object for Tests tab.
* Expose real pipeline DAG data.

## Next 2–3 months: make it world-class

* Full event-sourced TUI.
* Agent lifecycle supervision.
* Agent race board.
* Evidence/proof ledger.
* Security/admission command center.
* Cache taint and trust map.
* Release telemetry/rollback cockpit.
* Theme system and screenshot/presentation mode.
* Replay and comparison modes.

The current TUI has strong bones. The best next move is not more panels; it is a better state/action/evidence model. If the API starts giving the TUI typed realtime facts, blockers, risks, and recommended actions, the interface can become genuinely world-class: not just a prettier CI dashboard, but an autonomous software delivery cockpit.
