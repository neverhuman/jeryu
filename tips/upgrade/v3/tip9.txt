# Executive verdict

This repo has a **very strong agent-first thesis** and several unusually good ideas already implemented or scaffolded: ephemeral GitLab bot identities, agent task branches/issues, custom executor interception, evidence capsules, cache tainting, subsystem-aware test planning, release gates, action registry, TUI, and capability-based agent intents.

But it is currently closer to a **prototype / research control plane** than a “world’s #1 agent-first Git option.” The most important gaps are:

1. **Security boundaries are weaker than advertised.** The sandbox invariant says strict isolation uses `unshare`/`bwrap`, but the code path currently launches `bash` and sets `JERYU_SANDBOXED=1`; the comments admit full network namespace isolation is not implemented there. ([GitHub][2])
2. **Capability/admission control is not strong enough for autonomous agents.** The capability server accepts JSON over a Unix socket with a one-shot read and no evident per-agent authentication, framing, replay protection, rate limiting, or signed grants. ([GitHub][3])
3. **There are likely compile-breaking raw source issues.** I saw `Some(¬e)` in `engine.rs`; if that is not a GitHub rendering artifact, it is a hard build failure. ([GitHub][4])
4. **The “semantic CI/CD / AST intelligence” promise is ahead of the code.** The current `impact.rs` mostly classifies changed paths into coarse lanes such as full/unit/integration/docs-only. That is useful, but it is not yet real semantic dependency analysis. ([GitHub][5])
5. **The agent lifecycle is incomplete.** The code can spawn bots, issues, branches, race hypotheses, and pipelines, but `list_agents` is explicitly TODO/empty, race winner merge is TODO, and much of the “autonomous coder” behavior is still orchestration rather than a complete agent work loop. ([GitHub][6])

The repo is exciting because it is already thinking in the right primitives: **proof, evidence, capabilities, task identity, sandboxes, cache trust, release gates, and TUI-driven operator control.** To become world-class, it should double down on being an **Agent Operating System for Git**, not just “GitLab but Rustier.”

# What JeRyu already does well

## 1. Strong mission framing

The mission doc is unusually clear: Git remains the ledger of truth, GitLab is substrate rather than product, Rust owns orchestration/policy, and agents operate inside bounded capabilities. It also honestly names the real problem space: superseded pipelines, brittle selective testing, flaky tests, coarse retry/resume, poor explainability, and human-centric credentials. ([GitHub][7])

That is the right framing. Most GitHub/GitLab automation treats agents as fancy users. JeRyu is aiming to treat agents as **governed actors with contracts, evidence, and scoped authority**.

## 2. Good Rust control-plane stack

The README and `Cargo.toml` show a coherent async Rust architecture: Tokio, Axum, Bollard, SQLx, Git2-RS, Ratatui, crossterm, notify, dashmap, cargo metadata, and custom witness/runtime crates. ([GitHub][1])

The workspace is also more than one binary. It includes `cargo-witness`, `witness-rt`, `cargo-vrc`, `cargo-aer`, and `arc-bench`, which points toward a bigger “agent-native Rust workspace” toolchain. ([GitHub][8])

## 3. Very rich CLI surface

The CLI already exposes a broad operations surface: init/bootstrap, serve, TUI capture, pool management, job trace/play/cancel/retry/explain, pipeline explain/doctor/bottlenecks, cache controls, logs, agent spawn/race/list/merge, jeryu sync, tests, release, secrets, host reclaim/doctor, executor hooks, capability server, next action, explain-blocker, and action registry commands. ([GitHub][9])

That is a real strength. Agent-first platforms need **low-friction machine-callable operations**, and this repo is already moving beyond a web UI.

## 4. Agent identity is headed in the right direction

`spawn_agent` creates a task branch, creates a GitLab project bot token with `api` and `write_repository` scopes, creates a tracking issue assigned to the bot, creates a branch, and labels the issue as running. That is much better than giving every agent the same long-lived PAT. ([GitHub][6])

The race path is also conceptually good: generate several hypothesis branches, commit patches, trigger pipelines, and let them compete. That is exactly the kind of thing agent-first Git should make native. ([GitHub][6])

## 5. Failure capsules are a big win

The `FailureCapsule` model captures job ID, pipeline ID, project ID, stage, exit code, commit SHA, ref, working directory, log snippet, repro script, environment, failure kind, summary, supersedence, and retry ancestry. It also redacts obvious token/key/secret/password environment values when constructing repro scripts. ([GitHub][10])

This is one of the most agent-useful pieces in the repo. Agents should not parse raw CI logs forever. They should receive **structured failure packets** with enough context to reproduce and fix the issue.

## 6. Test intelligence is pragmatic and explainable

The VTI planner maps changed paths to subsystems, selected tests, skipped subsystems, confidence, rationale, fallback reasons, and sentinel tests. It conservatively falls back to full validation for empty diffs, global invalidators, unknown paths, low confidence, and test-intelligence changes. ([GitHub][11])

That is the correct shape. A world-class agent Git system needs selective testing, but every skip must be explainable, auditable, and reversible.

## 7. Cache trust and taint are good primitives

The cache brain refuses hits when the action key is absent, epoch is invalid, the object is tainted, or trust namespaces do not match. The policy module also prevents trusted consumers from using untrusted artifacts without explicit promotion. ([GitHub][12])

That is exactly the right instinct. Autonomous agents will generate a lot of speculative artifacts; cache reuse must be trust-aware.

## 8. The TUI is already ambitious

The TUI is not just a terminal list of jobs. It has mission/system health, release gate matrix, jobs/flow/logs, agents, tests, pools, cache, evidence/audit, and secrets tabs through the action registry. ([GitHub][13])

The UI code includes a Mission tab with GitLab health, containers, pools, proxy, registry, pipeline/job summaries, release status, and cache health. ([GitHub][14]) It also includes a Release Gate Matrix and a Jobs tab with pipeline navigation, flow board, jobs, logs, and an inspector panel. ([GitHub][14])

That is a strong foundation for “God-mode” human/agent co-piloting.

## 9. Action registry is exactly the right abstraction

The action registry defines action IDs, labels, risk tiers, surfaces, dry-run flags, side-effect classes, and required grants. It includes read-only actions, local state actions, Git writes, CI execution, merge, and production classes. ([GitHub][13])

This should become one of the central contracts of the product: every agent-visible operation should have **risk, permission, dry-run semantics, expected evidence, rollback semantics, and audit output**.

# Obvious gaps and risks

## P0: likely build correctness issues

The raw `engine.rs` contains `Some(¬e)` in a release failure path. If that is really in the repository, it will not compile. ([GitHub][4]) I also saw odd characters in `release.rs` output around an aggregate status merge expression. I would treat this as a first-day fix: run `cargo fmt`, `cargo check --all-targets --all-features`, `cargo clippy`, and a local smoke test before adding more features.

The repo has no published releases on GitHub, despite a `2.0.1` changelog entry and version metadata. ([GitHub][1]) For a tool that wants to control CI/CD and releases, that weakens confidence.

## P0: sandboxing currently overpromises

The README says each agent task is isolated in a Copy-on-Write sandbox and tightly constrained Docker container with scoped PATs. ([GitHub][1]) But `sandbox.rs` says the invariant is `unshare`/`bwrap`, while the implementation for strict mode spawns `bash` with an environment flag and relies on Docker pools / proxy controls elsewhere. ([GitHub][2])

That gap is dangerous because “agent-first” means untrusted code execution is the default case. A malicious or compromised agent should be expected.

Minimum fix: rename the current mode to `proxy_sandbox`, add a real `namespace_sandbox`, and fail closed when strict mode cannot be established.

## P0: capability server needs authentication and framing

The capability server accepts `AgentIntent` JSON over a Unix socket and dispatches powerful actions such as patch proposals, test branch creation, race patches, merge requests, blocker explanations, and system snapshots. ([GitHub][3]) That is the right API shape, but the server needs a stronger security model:

* Per-agent signed requests.
* Capability grant ID required for mutating actions.
* Nonce / timestamp replay prevention.
* JSONL or length-prefixed framing.
* Request size limits per intent type, not one global buffer.
* Socket permissions and peer credential checks.
* Audit event for every accepted and rejected intent.
* Rate limits and budget limits per agent/task.

Without that, a local process can become an agent with broad powers.

## P0: admission control is too heuristic

The admission hook is promising: it installs a GitLab pre-receive hook and evaluates agent refs, capability grants, and enforcement mode. ([GitHub][15]) But the current actor detection is based heavily on branch/ref namespace, not verified GitLab actor identity, bot token identity, signed commit identity, project scope, MR target, and capability grant provenance.

For autonomous coding, admission should enforce:

* “This exact agent identity may push this exact SHA to this exact branch in this exact project.”
* “The diff may only touch the granted paths.”
* “The grant expires after N minutes or N pushes.”
* “Protected branches cannot be modified by agents except through risk-gated merges.”
* “Every accepted push links to task, issue, capsule, plan, and prior grant.”

## P1: GitLab API client lacks full production robustness

The GitLab client supports a lot: runners, jobs, traces, artifacts, project bots, issues, MRs, branches, commits, pipelines, and webhooks. ([GitHub][16]) But several list methods appear to request `per_page=100` and not robustly paginate all results. That will bite as soon as jobs/pipelines/projects exceed one page. GitLab and GitHub REST APIs both expose pagination mechanisms; integrations must follow next-page/link/keyset semantics rather than assuming one page is complete. ([GitLab Docs][17])

Also: `GitlabClient::new` uses `danger_accept_invalid_certs(true)` according to the raw file. That should only be allowed in explicit local-dev mode, loudly surfaced in TUI/status, and blocked for production.

## P1: agent lifecycle is incomplete

The agent code has strong beginnings, but the gaps are obvious:

* `list_agents` is TODO/empty. ([GitHub][6])
* Race winner merge is TODO. ([GitHub][6])
* There is not yet a full “agent work loop”: plan → patch → validate → observe → repair → rebase → merge → release.
* There is not enough branch cleanup, stale bot cleanup, issue status reconciliation, or abandoned-task garbage collection.
* Merge gating exists, but capability `RequestMerge` appears to evaluate a proof rather than actually merge, while the action registry describes it as merging through the risk gate. ([GitHub][13])

## P1: semantic CI is mostly path-based today

The README claims real-time semantic CI and AST intelligence. ([GitHub][1]) The `impact.rs` implementation is currently path heuristics: Rust files select unit, tests select integration, broad config selects full, docs selects docs-only, and unknown changes fall back to full. ([GitHub][5]) The VTI system is better and more structured, but still mostly manually mapped subsystem/path rules. ([GitHub][11])

That is fine for a prototype, but the product should not overclaim “AST intelligence” until it has real language-aware dependency graphs, symbol-level impact, package-level build graph awareness, and historical skip-miss auditing.

## P1: evidence capsules need stronger redaction and provenance

Failure capsules redact sensitive values from the generated repro script, but the stored `environment` map still captures `CUSTOM_ENV_CI_*` and `GITLAB_*` variables. ([GitHub][10]) That needs a formal redaction policy before storing to DB, transmitting to agents, or rendering in TUI.

Dream version: every capsule should include a signed provenance envelope, redaction report, artifact digests, toolchain snapshot, container image digest, network policy, cache inputs, and “safe-to-share-with-agent” field-level labels.

## P1: cache archive risks

The custom executor stores action cache archives and can restore exact hits. It also captures failures into capsules, injects sccache/BuildKit settings, uses CoW workspaces, and archives sandbox outputs. ([GitHub][18])

The risk: archiving an entire sandbox can capture secrets, temporary tokens, honeypot files, local env output, build credentials, or huge irrelevant directories. This needs a manifest-based cache format:

* Explicit include/exclude rules.
* Secret scanner before cache commit.
* CAS object signing.
* Trust tier and taint metadata.
* Toolchain/image digest binding.
* Max size and TTL.
* Per-path restoration, not “extract everything.”

## P1: release subsystem is powerful but too repo-specific

The release module is rich: release status, canary gates, production promotion, CI schema, pipeline explanation, doctor/preflight, progress report, and artifact syncing. ([GitHub][19]) But it is currently tied to `/home/ubuntu/dougx`, a default release project ID, `veox-testctl`, and repo-specific artifact paths. ([GitHub][19])

To become a platform, release logic needs a declarative `jeryu.release.toml` contract per repo.

## P1: secrets handling should not bootstrap Vault with easy-mode material

The secrets module initializes Vault via Docker, uses a one-share/one-threshold setup, writes root/unseal/bootstrap material locally, and writes operational env files, even though it tries to keep files restricted. ([GitHub][20]) For local experiments that may be fine; for an autonomous release platform, this needs hardening:

* KMS/HSM-backed unseal where possible.
* No local root token after bootstrap.
* Per-agent dynamic secret leases.
* OIDC/JWT/SPIFFE identity.
* Secret-use telemetry.
* Automatic revocation on job end.
* Egress broker detects secret exfil attempts.

# The biggest product insight

The winning product is not “GitLab with agents.”

The winning product is:

> **A proof-routing, capability-governed, evidence-native Git operating system where agents can safely modify, test, repair, merge, and release software with minimal human intervention and maximal auditability.**

That means the system’s core object should not be just a repo, branch, MR, issue, job, or pipeline. The core object should be an **Agent Work Contract**.

Add a first-class `WorkContract` table/API/schema:

```text
work_contract_id
repo/project
objective
allowed_paths
forbidden_paths
allowed_actions
risk_budget
```

Every agent action should attach to a contract. Every push, test, cache hit, secret access, merge request, release promotion, and rollback should say: **which contract allowed this?**

This is the missing bridge between “agent as bot user” and “agent as autonomous engineer.”

# My best roadmap to make this world-class

## Phase 1: make the current system trustworthy

1. **Fix build correctness first.** Resolve raw source typos such as `Some(¬e)`, run format/check/clippy/tests, and add GitHub/GitLab CI that proves the repo itself is green. ([GitHub][4])
2. **Make docs match reality.** Where sandbox/network isolation is proxy-only, say so. Where semantic CI is path/subsystem-based, say so.
3. **Lock down capability server.** Add authenticated signed requests, grant IDs, peer credential checks, nonce/timestamp, framing, audit, and rate limits.
4. **Lock down admission.** Verify actor identity, project, ref, SHA, diff scope, grant, expiry, and branch protection.
5. **Implement real strict sandbox.** Use rootless containers, `bubblewrap`, `unshare`, gVisor, Firecracker, Kata, or a pluggable isolation backend. Fail closed when strict isolation is requested but unavailable.
6. **Add API pagination.** Every list method should exhaust pages or explicitly expose pagination.
7. **Harden cert handling.** `danger_accept_invalid_certs(true)` should never be default outside local dev.

## Phase 2: complete agent autonomy loop

Build the canonical flow:

```text
objective
→ plan
→ request capabilities
→ create branch
→ patch
→ run selected validation
→ ingest capsule if fail
→ repair
→ rerun
→ rebase
→ produce merge proof
→ merge
→ release/canary
→ monitor
→ rollback if needed
→ close task
```

The repo has many pieces, but not yet the whole autonomous loop.

Add commands/APIs:

```bash
jeryu agent start --objective ...
jeryu agent plan --contract ...
jeryu agent step --contract ...
jeryu agent repair --capsule ...
jeryu agent rebase --contract ...
jeryu agent prove-merge --mr ...
jeryu agent merge --mr --proof ...
jeryu agent release --sha ...
jeryu agent close --contract ...
```

## Phase 3: Evidence Capsules 2.0

Every failure/success/release proof should become a durable, signed, queryable object.

A capsule should include:

* Failure/success classification.
* Full command, normalized command, and working directory.
* Toolchain versions.
* Container image digest.
* Dependency lockfile digests.
* Cache verdicts and cache object IDs.
* Network egress summary.
* Secret lease summary without secret values.
* Test selection rationale.
* Skipped tests and why.
* Logs split into structured sections.
* Minimal repro script.
* Artifact links/digests.
* Suggested next action.
* Confidence score.
* Redaction manifest.
* Agent-readable JSON and human-readable markdown.

Then agents stop wasting context on raw logs.

## Phase 4: real semantic impact engine

The current VTI is a good baseline. Next level:

* Rust symbol graph using `rustdoc` JSON / `cargo metadata` / `ra_ap_*`.
* Function/type/module dependency graph.
* Test-to-symbol coverage mapping.
* Historical failure correlation.
* Flake probability model.
* “Blast radius” score per diff.
* Required checks generated from impact graph.
* Automatic “selector miss” learning when skipped tests would have failed.
* Nightly full-run reconciliation to calibrate skip safety.
* Cross-language adapters: TS/JS, Python, Go, Java, C/C++.

The output should not just be “run unit/integration/full.” It should be:

```json
{
  "changed_symbols": ["crate::pool::scale_pool_to"],
  "affected_symbols": ["crate::engine::check_scale_up"],
  "required_tests": ["pool_tests", "job_tests"],
  "skipped_tests": [{"name": "release_tests", "reason": "no dependency path"}],
  "confidence": 0.93,
  "risk": "medium",
  "fallbacks": [],
  "proof": "selector-v4:sha256:..."
}
```

## Phase 5: merge gate as formal proof

The merge gate already has `MergeGateInput` and `MergeGateProof` with blockers, job counts, selector misses, cache taints, trust tier, and policy version. ([GitHub][21]) Make that the center of the product.

A merge should require:

* Head SHA exactly matches proof SHA.
* Required checks complete.
* No pending jobs.
* No failed release-blocking jobs.
* No unrepaired selector misses.
* No active cache taints.
* No unresolved security findings.
* Agent’s capability grant permits merge.
* Human approval if trust tier/risk requires it.
* Signed proof stored in repo notes or MR comment.
* All evidence linked.

The TUI should show a one-line answer: **“Can this merge?”** with exact blockers.

# Features GitHub/GitLab still do not give agents well enough

The real differentiator is to make these first-class:

## 1. Agent-native repository API

Agents need APIs like:

```json
{
  "objective": "...",
  "base_sha": "...",
  "allowed_paths": ["src/pool.rs"],
  "proposed_changes": [...],
  "desired_validation": "minimal_safe",
  "risk_tolerance": "low"
}
```

The platform should respond with:

```json
{
  "branch": "agent/...",
  "patch_status": "applied",
  "validation_plan": "...",
  "required_capabilities": [...],
  "merge_blockers": [...]
}
```

Do not make agents compose 20 human-centric REST calls.

## 2. Agent memory tied to code

The system should remember:

* Which fixes worked for similar failures.
* Which tests catch which bugs.
* Which files are flaky.
* Which reviewers reject certain patterns.
* Which changes often break release.
* Which dependency upgrades are risky.
* Which cache hits later caused failures.

This memory must be repo-scoped, permissioned, and evidence-backed.

## 3. Autonomous race orchestration

JeRyu already has a race concept. Make it world-class:

```text
hypothesis A: minimal patch
hypothesis B: refactor
hypothesis C: revert + small fix
hypothesis D: dependency pin
```

The platform should:

* Allocate runner budget.
* Run all hypotheses.
* Cancel losers.
* Merge or propose the winner.
* Preserve evidence for why it won.
* Learn which strategy class tends to win.

## 4. Proof-carrying commits

Each agent commit should carry a machine-readable proof bundle:

```text
commit sha
contract id
agent id
diff scope
tests run
tests skipped
cache policy
security scan
review status
merge gate proof
```

Store it as Git notes, MR comments, artifacts, and DB records.

## 5. Agent-safe secrets

Agents should never receive raw long-lived secrets. They should request actions:

```text
deploy preview
query staging db schema
run migration dry-run
rotate canary secret
```

The platform should broker secret use and return results, not expose secret values.

## 6. CI as resumable computation

Instead of “rerun pipeline,” agents need:

* Resume from failed job.
* Reuse previous green jobs if inputs unchanged.
* Cancel superseded jobs.
* Merge DAGs across commits.
* Replay only affected downstream jobs.
* Explain every reuse decision.

The mission doc already identifies this as a key missing opportunity: supersedence with partial DAG reuse, semantic impact tied to required checks, flake memory, causal explanations, and policy-driven authority. ([GitHub][7])

# Security improvements

For an agent-first Git platform, security is not a feature. It is the product.

## Isolation

Implement pluggable isolation backends:

```text
local-bash: dev only
rootless-docker: baseline
bubblewrap/unshare: Linux local
gVisor/Kata: higher isolation
Firecracker: untrusted high-risk agents
remote-exec: enterprise
```

Strict mode must fail closed.

## Policy

Use policy-as-code:

```rego
allow_merge if
  proof.head_sha == mr.head_sha
  proof.failed_jobs == 0
  proof.pending_jobs == 0
  proof.selector_misses == 0
  proof.cache_taints == 0
  actor.trust_tier in ["trusted", "privileged"]
```

Surface policy decisions in TUI and capsules.

## Supply chain

Add:

* Sigstore signing for artifacts.
* SLSA provenance.
* SBOM generation.
* Dependency review.
* `cargo deny` integration.
* Secret scanning.
* Container image signing.
* Reproducible build mode.
* Cache object signatures.
* Tamper-evident event log.

## Agent abuse controls

Add:

* Per-agent action budget.
* Per-agent network egress policy.
* File path ACLs.
* CPU/memory/io limits.
* Human approval thresholds.
* Escalation on anomalous behavior.
* Automatic revocation on honeypot tripwire.

The honeypot idea is good: it seeds decoy AWS/npm credentials and kills/quarantines a process on access. ([GitHub][24]) But it should be one layer, not the core security boundary.

6. Add VTI selector-miss learning loop.
7. Add flake memory.
8. Add repo-generic release config.
9. Add TUI action enable/disable explanations.
10. Add “why blocked?” everywhere.

## P2 — epic differentiators

1. Symbol-level Rust impact graph.
2. Multi-hypothesis autonomous race engine.
3. Agent memory backed by evidence.
4. Proof-carrying commits.
5. Remote execution / CAS protocol.
6. Firecracker/gVisor untrusted lanes.
7. Policy marketplace for agent autonomy levels.
8. Cross-repo dependency graph and release trains.
9. “Autonomous incident rollback” mode.
10. Enterprise audit export.

---

# Suggested product positioning

Do **not** position this as “a fork of GitLab” unless you actually fork GitLab.

Position it as:

> **JeRyu / jeryu: an agent-first GitLab control plane that turns self-hosted GitLab into an autonomous coding, testing, evidence, and release OS.**

That is much more credible and much more exciting.

The repo’s own mission doc already says GitLab is the substrate, not the product. ([GitHub][7]) Lean into that.

---

# Final take

JeRyu has the bones of something genuinely differentiated. The strongest parts are the **agent-oriented mental model**, **Rust control plane**, **custom executor**, **evidence capsules**, **test intelligence**, **cache trust/taint model**, **release gates**, and **TUI/action registry**.

The weak parts are not lack of ambition. The weak parts are **security hardening, compile/build confidence, incomplete agent lifecycle, overclaimed semantic intelligence, incomplete admission/capability enforcement, and repo-specific release/secrets assumptions**.

My highest-conviction recommendation: make the next milestone boring and brutal: **green build, real sandbox, signed capabilities, strong admission, complete agent lifecycle, and honest docs.** After that, build the epic stuff: Work Contracts, proof-carrying commits, semantic test selection, autonomous race orchestration, and a TUI that answers “what is blocked, why, what can an agent safely do next?”
