## What it already does well

### 1. The core thesis is strong: GitLab as substrate, Rust as control plane

The repoŌĆÖs best idea is not ŌĆ£fork GitLab.ŌĆØ It is: **leave Git as the durable ledger, use GitLab as a replaceable substrate, and build the agent-native brain in Rust.** That is the right architectural bet. GitLab/GitHub are optimized around human review and human navigation. Agents need a lower-latency, more structured, more deterministic control surface.

The mission doc identifies real pain points: superseded pipelines, brittle selective testing, flaky tests and infra, coarse retry/resume, bad logs/status, and human-centered token/approval models. ([GitHub][2]) That diagnosis is excellent. It means the repo is trying to solve the right problem: not ŌĆ£make Git prettier,ŌĆØ but ŌĆ£make code change, test, review, merge, release, and rollback machine-operable.ŌĆØ

### 2. The repo has an unusually complete systems map

The repository is not just a toy CLI. It has modules for admission, agents, capability handling, engine/webhooks, execution, gateway/cache layers, GitLab client, honeypots, impact analysis, policy, pools, release, sandboxing, secrets, taint, telemetry, test intelligence, TUI, and witness/proof tooling. ([GitHub][3]) The generated agent index also shows TUI submodules for actions, app state, event handling, flow building, flow inspection, ETA, graph, and widgets. ([GitHub][3])

That breadth matters. A world-class agent-first Git platform cannot be just a code editor plugin or CI runner. It needs an operating system around code: state, evidence, identities, runners, logs, policy, capabilities, caches, releases, and recovery. This repo is pointed in that direction.

### 3. The Rust workspace/proof discipline is a good cultural foundation

The workspace uses Rust 2024, Rust version 1.85, and contains multiple proof/control crates such as `cargo-witness`, `witness-rt`, `cargo-vrc`, `cargo-aer`, and `arc-bench`. ([GitHub][4]) The `AGENTS.md` file defines proof lanes such as `leaf-bugfix`, `state-change`, `api-change`, `release-change`, `security-relevant`, and `cross-module`, and maps them to increasing levels of proof. ([GitHub][5])

That is exactly the kind of culture an agent-first system needs. Agents should not just ŌĆ£make changes.ŌĆØ They should make changes with attached proof obligations. The repo is already thinking in that direction.

### 4. The CLI surface is broad and agent-relevant

The CLI exposes commands for init/bootstrap, serving, TUI, pool management, jobs, pipelines, cache, logs, agents, jeryu sync, tests, releases, secrets, progress, repo, host, capability, next actions, blocker explanation, and action execution. ([GitHub][6]) The job and pipeline commands include list, trace, play, cancel, retry, explain, doctor, ingest jobs, and bottlenecks. ([GitHub][7]) The test command family includes run, plan, batch, results, retry failed tests, impact selection, plan explanation, audit, learn, and cache status. ([GitHub][7])

That is a good sign. Agent-first platforms need the boring operational verbs. The CLI is already shaped like something a human, an agent, or a higher-level orchestrator could drive.

### 5. It has real event/state machinery

The state layer supports SQLite/Postgres and uses an append-only event-log style for system activity. ([GitHub][8]) It also defines records for release attempts, evidence, retries, jeryu sync, capability admission, test UI tracking, cache metrics, build records, and image signatures. ([GitHub][8]) ([GitHub][8])

The engine exposes webhook handling for GitLab job, pipeline, push, and merge-request events, records events, retries failed jobs based on failure classification, scales pools on pending jobs, and performs supersedence by canceling older in-flight pipelines for the same ref. ([GitHub][9]) ([GitHub][9]) ([GitHub][9])

That is a meaningful base. Autonomous agents need a durable memory of what happened, why it happened, and what should happen next. The repo already has the skeleton for that.

### 6. Ephemeral bot identities are the right primitive

`spawn_agent` creates a timestamped branch, a project bot/access token with `api` and `write_repository` scopes, assigns a tracking issue to the bot, and labels the issue as agent-running. ([GitHub][10]) ([GitHub][10]) The GitLab client has explicit project-access-token creation support. ([GitHub][11])

This is directionally correct. Agents should not share one god token. They should operate with ephemeral identities, scoped authority, branch leases, traceable actions, and revocable capabilities.

### 7. The test/impact direction is right

The impact module has conservative fallback behavior and distinguishes docs-only changes, Rust source changes, test changes, broad config changes, and unknown changes. ([GitHub][12]) ([GitHub][12]) The test planner produces `Full`, `Selected`, or `DocsOnly` plans with selected tests, affected subsystems, rationale, confidence, and skipped subsystem reasons. ([GitHub][13]) It also has batch execution with concurrency via semaphores and join sets. ([GitHub][14])

This is one of the biggest areas of upside. Agents are bottlenecked by slow feedback. Intelligent test selection plus honest evidence is the killer feature.

### 8. CoW workspace cloning is practical

The executor prepares per-job sandboxes and uses fast clone behavior: APFS `cp -c -r` on macOS, Linux `cp --reflink=auto -r`, and fallback `cp -r`. ([GitHub][15]) ([GitHub][15]) This is a good pragmatic performance move. Agents need cheap speculative branches, cheap reproductions, cheap bisections, and cheap ŌĆ£try 10 hypothesesŌĆØ workflows.

### 9. Failure capsules/honeypot/quarantine are exactly the right concepts

The executor records traces, captures log tails, writes failure/quarantine capsules, starts honeypot tripwires, and preserves sandboxes when quarantine markers are present. ([GitHub][15]) ([GitHub][15])

This is the kind of thing GitHub/GitLab still do poorly. A failed job should not be ŌĆ£a wall of logs.ŌĆØ It should become a structured, replayable failure object with suspected cause, reproducer, environment, diff, artifacts, risk, and next actions.

## The biggest gaps and risks

### 1. The repo currently overstates ŌĆ£agent autonomyŌĆØ

The module docs for `agent.rs` describe an ambitious autonomous flow: create branch, perform task, commit/push, open MR, watch pipeline, read traces, fix failures, force-push, auto-merge or request review. ([GitHub][10]) But the actual `spawn_agent` path mostly provisions a bot, issue, and branch. ([GitHub][10]) ([GitHub][10]) `list_agents` is still a TODO that returns an empty vector. ([GitHub][10]) The race flow has TODOs/manual cleanup around polling race results and winner handling. ([GitHub][16])

That gap matters. To become agent-first, JeRyu needs a true **agent runtime**, not just agent scaffolding.

Minimum needed:

* durable agent work orders;
* task planning and decomposition;
* tool execution sessions;
* patch generation/apply/verify loops;
* failure recovery loops;
* resumability after process restart;
* human interruption and redirection;
* agent-to-agent handoff;
* evidence-bound finalization;
* cost/time/budget tracking;
* ŌĆ£why did the agent do this?ŌĆØ replay.

### 2. Merge safety is not yet evidence-bound enough

`merge_agent_mr` does have a risk gate, but it lists project jobs and filters by source branch, which risks counting stale jobs from an older branch head unless it binds everything to the exact MR head SHA/latest pipeline. ([GitHub][10]) The GitLab clientŌĆÖs `list_jobs` endpoint is project-wide. ([GitHub][11]) Even more concerning: the capability `RequestMerge` path builds a merge gate with `head_sha: None`, `successful_jobs: 1`, zero failed/pending jobs, and evidence-policy requirements all set to false. ([GitHub][16])

For autonomous coding, this is the most dangerous class of bug. Merge decisions must be tied to:

* exact repository;
* exact MR;
* exact source branch;
* exact head SHA;
* exact latest pipeline ID;
* exact required jobs;
* exact job attempt numbers;
* exact artifacts/digests;
* exact testmap version;
* exact policy version;
* exact agent identity and capabilities used;
* exact approval/risk decision.

Anything else creates a path for stale-green merges, accidental merges, or adversarial bypasses.

### 3. The sandbox security claims currently exceed the implementation

`sandbox.rs` declares a Workload Sandbox with network namespace isolation and says strict network isolation uses `unshare`/`bwrap`. ([GitHub][17]) But the actual strict isolation branch starts `bash`, sets `JERYU_SANDBOXED=1`, and comments that full `bwrap`/`unshare` would require more setup; it relies instead on Docker pools, honeypot, and proxy behavior. ([GitHub][17]) The executor does useful CoW cloning and honeypot handling, but that is not the same as strong process/network/filesystem isolation. ([GitHub][15])

For an agent-first coding system, this is a critical issue. Agents will run untrusted code, dependencies, test scripts, generated scripts, and maybe adversarial repo content. The sandbox should be upgraded before broad autonomous operation.

### 4. Capability handling is promising but too loose

The capability layer defines useful typed intents: propose patch, race patches, run tests, fetch capsules, request merge, explain blockers, get snapshot, list allowed actions, and plan validation. ([GitHub][16]) But the socket server reads up to 65,536 bytes, which is small for real patches or structured evidence bundles. ([GitHub][16]) Some flows ignore pipeline trigger failures with `.ok()`, continue through branch/commit failures, or return manual-cleanup style messages. ([GitHub][16]) ([GitHub][16]) ([GitHub][16])

A capability system for autonomous agents should be boringly strict: every request has an ID, lease, scope, actor, expiration, policy version, idempotency key, audit trail, success/failure result, and revocation semantics.

### 5. Test intelligence is still mostly heuristic

The impact model is currently path/category driven: docs-only, Rust source, tests, broad config, unknown, empty diff, and so on. ([GitHub][12]) ([GitHub][12]) The planner has the right shape and confidence fields, but the system still needs real semantic evidence: coverage maps, dependency graphs, runtime traces, flaky history, ownership, prior failure correlation, and proof reuse. ([GitHub][13])

The repo knows this. The mission doc explicitly calls out missing commit supersedence with partial DAG reuse, semantic impact tied to required checks, flake memory, structured causal explanations, and policy-driven agent authority. ([GitHub][2])

### 6. The CI orchestration is clever but branch-heavy

The test runner creates ephemeral `jeryu-test-*` branches, dynamically writes `.gitlab-ci.yml`, triggers pipelines, waits, cancels older branch-create pipelines, deletes branches, and records execution. ([GitHub][14]) ([GitHub][14]) ([GitHub][14]) That works, but it is not the ideal agent-native interface.

For world-class performance, agents should be able to request a test run as a structured job spec without committing temporary CI YAML into branches. GitLab can remain the substrate, but JeRyu should expose a faster internal execution path.

### 7. Security policy is thin

The repo has a `SECURITY.md`, but it is mostly a vulnerability reporting process: only latest `main` supported, private reporting, acknowledge within 48 hours, investigate, publish patch. ([GitHub][18]) That is fine as a project hygiene baseline, but it is not a threat model.

An agent-first Git platform needs a real threat model for malicious repos, malicious dependencies, prompt injection through logs/issues/MRs, compromised agents, secret exfiltration, dependency confusion, cache poisoning, stale evidence, fake artifacts, privilege escalation, rogue runners, and poisoned release bundles.

### 8. Secrets are ambitious but currently too environment-specific

The secrets module has Vault support, local Vault initialization, KV-v2 mount configuration, policies, ops tokens, and release secret rotation/recovery commands. ([GitHub][19]) ([GitHub][19]) But parts of it are coupled to environment-specific assumptions, including repo-root defaults and external release scripts. That is okay for a lab, but the world-class version needs portable secret backends and strict per-action leases.

### 9. The TUI exists, but it needs to become the cockpit

The CLI supports TUI running, one-shot rendering, and PNG capture. ([GitHub][20]) The module map has TUI actions, app, event handling stubs, flow builder, collector, ETA, inspector, graph, and widgets. ([GitHub][3]) That is a strong start, but the TUI should become the central ŌĆ£agent mission control,ŌĆØ not merely a dashboard.

## The highest-impact improvements

### P0: Make merge safety mathematically boring

This is the first thing I would fix.

Every merge should require a `MergeProof` object:

```text
MergeProof {
  repo_id,
  mr_id,
  source_branch,
  target_branch,
  head_sha,
  base_sha,
  latest_pipeline_id,
  required_checks[],
  successful_jobs[{job_id, name, attempt, sha, artifact_digest}],
  failed_jobs[],
  pending_jobs[],
  selected_tests[],
  skipped_tests[{reason, confidence, policy}],
  impact_graph_version,
  testmap_version,
  policy_version,
  agent_id,
  capability_grants[],
  sandbox_attestations[],
  secret_access_summary,
  cache_inputs[],
  cache_taints[],
  human_approvals[],
  risk_score,
  decision,
}
```

Then enforce:

* no `head_sha: None`;
* no stale pipeline;
* no branch-only job matching;
* no fake `successful_jobs: 1`;
* no ignored pipeline-trigger failures;
* no merge if the policy version changed after the proof was generated;
* no merge if cache/test evidence has taint;
* no merge if required checks were skipped without policy-approved rationale.

This should be the core differentiator: **GitHub/GitLab say ŌĆ£checks passed.ŌĆØ JeRyu should say ŌĆ£this exact change is proven safe enough under this exact policy.ŌĆØ**

### P0: Build a real agent runtime

Right now, the system provisions agent workspaces. It needs a durable runtime.

I would introduce:

```text
AgentWorkOrder
  id
  repo
  objective
  constraints
  allowed_paths
  denied_paths
  max_budget
  max_runtime
  risk_tier
  required_evidence
  branch_strategy
  release_strategy
  human_escalation_policy
```

Lifecycle:

```text

ŌåÆ planned

ŌåÆ merge_gate
ŌåÆ merged

ŌåÆ monitored
ŌåÆ closed
```

Each transition should be append-only, replayable, and inspectable in the TUI. The agent should be able to pause, resume, fork hypotheses, abandon, checkpoint, or hand off to another agent.

### P0: Stop swallowing errors in autonomous paths

Anywhere that currently uses `.ok()`, `let _ =`, or ŌĆ£manual cleanupŌĆØ in agent/capability/test flows should become structured failure evidence. For human tools, ignored errors are annoying. For autonomous systems, ignored errors become silent false confidence.

Replace ignored results with:

```text
ActionResult {
  action_id,
  idempotency_key,
  attempted_at,
  actor,
  inputs_digest,
  result: Success | Failed | Skipped | Retried,
  error_kind,
  error_message,
  cleanup_required,
  next_recommended_action,
}
```

### P0: Make the sandbox real

The current sandbox should be treated as a convenience workspace, not a security boundary. The production-grade version should support:

* rootless containers by default;
* no host Docker socket exposure;
* read-only source mounts unless explicitly writable;
* per-job ephemeral filesystem;
* seccomp profile;
* AppArmor/SELinux profile where available;
* no network by default;
* egress broker with allowlist and DNS logging;
* syscall/network/file access telemetry;
* secret broker injection only at execution time;
* automatic secret redaction in logs;
* signed sandbox attestation;
* quarantine on suspicious behavior;
* Firecracker/gVisor option for high-risk repos.

The TUI should visibly distinguish:

```text
Workspace clone: safe for speed
Container sandbox: medium isolation
MicroVM sandbox: high isolation
```

Do not let users think `JERYU_SANDBOXED=1` is isolation.

### P1: Create an agent-native API, not just a CLI

The CLI is useful, but agents need fast structured APIs.

Add:

* local Unix socket API;
* gRPC or JSON-RPC API;
* streaming event subscriptions;
* idempotent command IDs;
* typed schemas for all actions;
* stable action registry;
* MCP-compatible tool surface;
* batch endpoints for logs/jobs/evidence;
* ŌĆ£next best actionŌĆØ endpoint;
* ŌĆ£why blocked?ŌĆØ endpoint;
* ŌĆ£what changed since checkpoint?ŌĆØ endpoint.

Example agent API:

```text

GET /v1/workorders/{id}/events?since=...
```

Agents hate scraping logs, polling pages, and guessing state. JeRyu should make those unnecessary.

### P1: Make semantic impact analysis the killer feature

Path-based test selection is a start. The next level is a persistent impact graph:

```text
file ŌåÆ module ŌåÆ symbol ŌåÆ package ŌåÆ binary ŌåÆ tests ŌåÆ historical failures ŌåÆ required checks
```

Inputs:

* `cargo metadata`;
* Rust module graph;
* symbol-level dependency map;
* test coverage;
* runtime traces;
* previous test outcomes;
* flaky history;

The key rule: **a skipped test needs stronger explanation than a run test.**

### P1: Evidence capsules should become first-class artifacts

Failure capsules are a great idea. Expand them into general evidence capsules:


Every important decision should point to evidence capsules. The TUI should be able to open any decision and show its proof.

### P1: Flake memory and retry intelligence

Flaky tests are one of the biggest blockers for autonomous coding. Add a `FlakeBrain` that tracks:

* test name;
* failure signature;
* stack trace cluster;
* environment;
* branch;
* runner;
* dependency versions;
* historical pass/fail rate;
* recent regressions;
* suspected owner;
* retry success rate;
* quarantine status;
* policy exceptions.

Then retry decisions should be evidence-based:

```text
Regression: do not retry blindly; investigate.
Known flake: retry once/twice under policy; mark evidence.
Infra failure: retry on different runner.
Security-sensitive failure: no auto-retry merge path.
Unknown: escalate or run broader validation.
```

### P1: Supersedence with proof reuse

The mission doc already identifies supersedence with partial DAG reuse as a missing area. ([GitHub][2]) This can be a standout feature.

When commit `C3` supersedes `C2`, do not simply cancel everything and restart. Build a proof DAG:

```text
C1 passed full suite
C2 changed files A,B; ran tests T1,T2,T3
C3 changed only file B relative to C2
Reuse T1 if inputs unchanged
Invalidate T2/T3 if dependency graph affected
Run T4 because new impact discovered
```

This is where JeRyu can beat GitHub/GitLab. Humans think in pipelines. Agents need proof reuse.

### P1: Replace branch-mutating CI test requests with out-of-band jobs

The current ephemeral branch + dynamic `.gitlab-ci.yml` approach is clever but heavy. It creates extra branch churn, pipeline noise, cleanup concerns, and latency.

Build a native execution planner:

```text
jeryu test run --spec test-spec.json
```

Where `test-spec.json` includes:

* base SHA;
* patch/diff;
* command;
* environment;
* cache keys;
* sandbox tier;
* required artifacts;
* timeout;
* network policy;
* secret policy.

GitLab runners can still execute under the hood, but agents should not need temporary branch gymnastics for every test.

### P2: Capability tokens should be narrower and shorter-lived

Ephemeral project bots are good, but `api` + `write_repository` + Developer access for two days is broader than ideal for high-autonomy agents. ([GitHub][10]) ([GitHub][11])

Use a broker model:

```text
Agent has no raw GitLab token by default.
Agent requests capability: "update branch X at SHA Y with patch Z."
Policy evaluates.
Broker performs the action or issues a minute-long, action-scoped token.
Every action is audited.
Token dies immediately after use.
```

The platform should prefer action-specific capabilities over broad credentials.

### P2: Add prompt-injection defenses for code agents

Agent-first Git has a security class GitHub/GitLab barely model: adversarial text influencing the agent.

Treat these as untrusted input:

* issue descriptions;
* MR comments;
* commit messages;
* README files;
* test logs;
* dependency install output;
* generated stack traces;
* code comments;
* documentation;
* files changed by another agent.

Add:

* source labels: trusted/untrusted/system/user/tool;
* prompt firewall;
* secret-leak detector;
* instruction hierarchy;
* log sanitizer;
* ŌĆ£untrusted text attempted to instruct agentŌĆØ alerts;
* TUI indicator when agent reasoning was exposed to untrusted content.

### P2: Make policy-as-code central

Add a `jeryu-policy.toml` or Rego/Cedar-style policy layer.

Examples:

```toml
[merge]

require_head_sha_match = true

```

Agents should not ŌĆ£knowŌĆØ their permissions by convention. They should query them.

### P2: Add supply-chain and release proof

To own autonomous releases, JeRyu needs:

* SBOM generation;
* SLSA-style provenance;
* signed artifacts;
* signed container images;
* dependency diff risk;
* secret access summary;
* test evidence;
* policy decision;
* human approvals;
* rollout plan;
* canary metrics;
* rollback trigger;
* release capsule.

A release should not be ŌĆ£pipeline green.ŌĆØ It should be a signed proof bundle.

### P2: Make cache proof-aware

Caching is dangerous unless tied to exact inputs. SmartCache should track:

* command;
* environment;
* source digest;
* dependency lockfile digest;
* toolchain digest;
* secrets used or not used;
* network access used or not used;
* sandbox profile;
* previous outputs;
* taint status.

A cache hit should say:

```text

  source digest unchanged
  dependency digest unchanged
  toolchain unchanged
  sandbox profile unchanged

  policy allows reuse for this check
```

### P3: Add multi-agent racing as a first-class product feature

The repo already has a race-patches concept. ([GitHub][10]) Make it great.

A race should support:

* hypotheses;
* agent assignment;
* branch leases;
* shared evidence pool;
* duplicate-work suppression;
* winner criteria;
* merge candidate selection;
* loser cleanup;
* patch synthesis from multiple winners;
* ŌĆ£why winner wonŌĆØ explanation.

Example:

```text
Race: fix flaky auth test

  A: timing issue
  B: database isolation issue
  C: mocked token expiry issue

  performance impact
```

This would be genuinely agent-first and better than todayŌĆÖs GitHub/GitLab workflows.

## The ŌĆ£GitHub/GitLab donŌĆÖt do thisŌĆØ wishlist

To become the worldŌĆÖs best agent-first Git option, JeRyu should focus on primitives GitHub/GitLab do not expose well.

### Agent work orders, not just issues

Issues are human-native. Agents need work orders with objective, constraints, permissions, budgets, evidence requirements, and lifecycle state.

### Branch leases

Agents should not just create branches. They should lease branches:

```text
branch lease:
  owner agent
  base SHA
  expires
  allowed paths
  max force-push policy
  cleanup policy
  merge eligibility
```

### Evidence-backed required checks

Required checks should not be static names. They should be generated from impact/policy:

```text
This change requires:
  unit/auth
  integration/session
  security/secrets-scan
  release/preflight
because:
  auth/session.rs changed
  token expiry path affected
  policy requires integration auth validation
```

### Structured MR comments for agents

Do not make agents parse human text blobs. Add machine-readable review events:

```json
{
  "kind": "review_instruction",
  "severity": "must_fix",
  "path": "src/auth/session.rs",
  "line": 142,
  "request": "preserve backward compatibility for expired tokens",
  "author": "human",
  "trusted": true
}
```

### ŌĆ£Why blocked?ŌĆØ as a first-class endpoint

An agent should be able to ask:

```text
Why can I not merge?
```

And receive:

```text
- latest pipeline is stale relative to head SHA
- required check integration/auth missing
- cache evidence tainted by network access
- security policy requires human approval for secrets path
```

### Autonomous release proof

GitHub/GitLab releases are too human/page-centric. JeRyu should have:

```text
ReleaseCandidate
  source MR proofs
  artifact provenance
  SBOM
  signatures
  deployment plan
  canary metrics
  rollback evidence
  policy decision
```

### Speculative parallel coding

Let agents race solutions and synthesize the best one. This should be first-class, not a hack.

### Agent-native logs

Logs should be summarized, chunked, embedded, searchable, classified, and linked to code/test/evidence. Agents should never scroll 10,000 lines.

### Agent memory with safety boundaries

Memory should include:

* known flaky tests;
* previous fixes;
* subsystem owners;
* style preferences;
* risk patterns;
* release incidents;
* known false positives;
* dependency quirks.

But it must separate trusted memory from untrusted repo/log/comment content.

### Built-in rollback thinking

Agents should not only ask ŌĆ£can I merge?ŌĆØ They should ask:

```text
How would I detect this release is bad?
How would I roll it back?
What metrics should I watch?
What is the blast radius?
```

## Performance upgrades

1. **Proof reuse over rerun everything.** Track exact input digests and invalidate only what changed.

2. **Avoid temporary branch CI for every test.** Use out-of-band job specs where possible.

3. **Batch GitLab API calls.** Agents should not poll repeatedly for jobs, traces, pipelines, MRs, and artifacts one object at a time.

4. **Stream logs and summarize incrementally.** Do not wait for a full trace to classify failure.

5. **Warm workspace pools.** Keep pre-cloned, dependency-warmed, toolchain-warmed sandboxes.

6. **Content-addressed build/test cache.** Tie cache to source/toolchain/env/policy digests.

7. **Predictive runner scaling.** Scale before queue spikes based on agent plan fanout.

8. **Speculative execution with budget caps.** Let agents race likely fixes, but cap compute and auto-kill losers.

9. **Slow-test profiler.** Continuously recommend splitting, parallelizing, or quarantining expensive tests.

10. **Supersedence-aware cancellation.** Cancel obsolete work, but preserve reusable evidence.

## Final take

The repo is **directionally excellent**. It already thinks about the things normal Git platforms under-serve: agent identity, capability boundaries, test impact, CI waste, failure capsules, runner pools, cache, release control, and TUI supervision.

The biggest weakness is that the project currently feels like **a powerful Rust orchestration substrate with agent-shaped scaffolding**, not yet a complete autonomous coding platform. The path to ŌĆ£worldŌĆÖs #1 agent-first Git optionŌĆØ is clear:

**Make every autonomous action evidence-bound, every capability scoped, every sandbox real, every merge tied to exact proof, every failure replayable, and every TUI screen actionable.**

If JeRyu does that, it should not merely imitate GitHub/GitLab. It should become the thing agents actually want: **a proof-carrying, policy-aware, sandboxed, low-latency engineering operating system built around Git.**
