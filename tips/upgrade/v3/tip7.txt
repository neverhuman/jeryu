## Review scope

I attempted to clone the repo locally, but the sandbox could not resolve `github.com`, so I could not run the code or tests. I reviewed the live GitHub repository, raw source/docs, module maps, CLI definitions, architecture docs, and official GitHub/GitLab API docs. My read is: **JeRyu/jeryu is an impressive Rust control plane around a local GitLab instance, not yet a true GitLab fork.** The README positions it as a “Headless GitLab & Autonomous Agent Hypervisor,” a single-binary Rust control plane for local GitLab orchestration, CI/CD, sandboxes, semantic test planning, runner pools, cache control, and agent operations. ([GitHub][1])

## Executive verdict

The project has the right instincts for an agent-first Git platform: typed agent capabilities, durable state, proof lanes, runner pools, smart test selection, release/canary gates, cache tainting, sandbox/tripwire concepts, and a TUI-centered operational model. The architecture is much more agent-aware than GitHub/GitLab’s normal “REST endpoint + CI logs + PR/MR” workflow.

But the current code appears to be **foundation plus scaffolding**, not yet the world-class autonomous coding/release system implied by the README. The main gaps are that the autonomous agent loop is not end-to-end, merge proof is partially stubbed, several safety checks are advisory or default-off, the GitLab API wrapper needs pagination/retry/backoff/TLS hardening, and the TUI is still more polling dashboard than live agent mission control. The core idea is strong enough to keep building; the next phase should be about making every agent action a verifiable transaction: **intent → capability grant → patch → validation DAG → evidence proof → gated merge → release proof**.

## What JeRyu already does well

**1. The product thesis is unusually good.** It treats Git not just as source control, but as an execution substrate for autonomous coding: local GitLab, CI orchestration, branch/MR operations, runner pools, cache, release gates, and agent capabilities are all inside one Rust binary. The README’s feature list includes autonomous hypervisor control, CoW sandboxes, ephemeral Matrix identities, a custom executor, semantic CI/CD, and memory tuning, while the CLI covers lifecycle, agent, pool, cache, release, jeryu sync, and test operations. ([GitHub][1])

**2. The module boundaries are agent-friendly.** `lib.rs` exposes separate modules for engine, release, state, exec, secrets, honeypot, sandbox, admission, taint, decision, test intelligence, TUI, and gateway. That is exactly the kind of separation an agent needs to reason locally, avoid mega-file edits, and select proof lanes. ([GitHub][2])

**3. The repo has unusually strong “agent routing” metadata.** `AGENTS.md` tells agents to use `proof-lanes.toml`, maps change types to proof lanes, identifies module ownership, lists invariants, and says state changes should go through `state::Db` rather than raw SQL callers. That kind of machine-readable ownership/proof system is one of the best parts of the repo. ([GitHub][3])

**4. The CLI/API surface is broad and deliberately agent-facing.** The CLI enum includes lifecycle, TUI, status, pool, job, pipeline, cache, logs, agent, jeryu, test, release, secrets, progress, repo, host, hidden executor, hidden server-hook, capability, next-action, explain-blocker, and action commands. The test subcommands include run/plan/batch/results/retry/failed/impact/select/explain/audit/learn/cache-status, which is exactly the kind of surface autonomous agents need. ([GitHub][4])

**5. The state layer is ambitious and appropriately durable.** The state model includes pools, managers, job events, CI timing, event logs, tracked pipelines, release attempts, evidence capsules, retry records, jeryu sync, secret authorities, secret audit events, capability intents/grants, admission decisions, test executions, cache objects, cache requests, cache metrics, build/image signatures, force refresh rules, cache taints, leases, verdicts, promotions, material objects, action cache, epochs, toolchain fingerprints, test plans, and selector misses. That is the right direction: agents need persistent evidence, not vibes. ([GitHub][5])

**6. The capability API is a strong design choice.** Instead of letting an agent execute arbitrary shell payloads, `AgentIntent` has typed operations like `ProposePatch`, `RacePatches`, `RunTests`, `FetchCapsule`, `RequestMerge`, `ExplainBlockers`, `GetSystemSnapshot`, `ListAllowedActions`, and `PlanValidation`. This is much closer to an agent-safe control surface than a generic CLI wrapper. ([GitHub][6])

**7. Admission control is the right safety primitive.** The pre-receive hook design records admission decisions and can require agent refs to match active capability grants. That is exactly the kind of ledger-backed guardrail an autonomous coding platform should have. ([GitHub][7])

**8. VTI/test intelligence is one of the most valuable subsystems.** The docs describe subsystem-aware smart test selection, docs-only skipping, full-test fallbacks for invalidators, external `.jeryu/testmap.toml` support, child pipeline YAML emission, selector-miss auditing, and a nightly oracle that records cases where VTI would have skipped a failing test. This is a real differentiator versus ordinary GitHub/GitLab CI, because agents need fast, explainable validation. ([GitHub][8])

**9. The cache/security story is headed in the right direction.** There is a SmartCache layer with Docker registry mirror/proxy concepts, cache GC, disk reporting, CAS mismatch errors, and manager cache tracking; there are trust tiers, cache promotion rules, witness generation for Rust/Docker cacheability, and taint propagation across dependent cache objects. ([GitHub][9])

**10. The TUI vision is good.** The TUI docs describe nine tabs: Mission, Release, Jobs, Agents, Tests, Pools, Cache, Evidence, and Secrets; snapshot workers; flow screens; keyboard navigation; deterministic capture mode; and release-watch concepts. The action registry also classifies actions by risk tier, surface, side-effect class, grant requirement, and dry-run status. ([GitHub][10])

## Biggest gaps and risks

**1. It is not yet a full GitLab fork.** It is currently a Rust orchestrator/control plane that boots and controls GitLab CE, runners, Vault, RedlineDB/RedlineDB, webhooks, caches, and agent operations. That may actually be the better architecture, but the language should be honest: it is “agent-first GitLab control plane,” not “GitLab fork.” The README itself calls it a single-binary Rust control plane for local GitLab orchestration. ([GitHub][1])

**2. The autonomous agent loop is mostly not implemented.** `agent.rs` documents a loop where an agent creates a branch, performs a task, commits/pushes, opens an MR, watches CI, analyzes failures, fixes, force-pushes, and auto-merges or flags review. But the implementation shown creates a project bot, issue, branch, and MR-like flow; `list_agents` is TODO/empty, race cleanup is noted as manual/TODO, and winner auto-merge is explicitly not implemented. ([GitHub][11])

**3. Merge evidence is not yet trustworthy enough.** In `capability.rs`, `RequestMerge` checks selector misses and cache taints, but the proof object uses `successful_jobs: 1`, `failed_jobs: 0`, `pending_jobs: 0`, and a policy with evidence requirements disabled. That is a high-severity issue if agents are meant to request or perform merges. The action registry describes request merge as “pipeline green, no selector misses, no taint,” but the implementation needs to actually prove those conditions from GitLab and state. ([GitHub][6])

**4. Admission enforcement defaults are too soft.** The API docs say agent refs without a matching capability grant are audit-only by default unless `JERYU_ADMISSION_ENFORCE=1` is set. For an agent-first system, the default should be deny-by-default for ungranted agent write paths, with explicit development-mode override. ([GitHub][12])

**5. TLS safety needs a production/dev split.** `GitlabClient::new` uses `danger_accept_invalid_certs(true)`. That can be acceptable for a local self-signed GitLab dev stack, but it must be gated behind a “local insecure dev” setting and loudly forbidden for production/remote GitLab. ([GitHub][13])

**6. The GitLab client needs pagination, rate-limit handling, and backoff.** The code uses many list-style GitLab endpoints. GitLab’s own docs note that list endpoints are paginated, default to 20 items, max `per_page` is 100, and keyset pagination is recommended for large collections where available. A world-class agent platform cannot silently miss jobs/MRs/projects after page 1. ([GitHub][13])

**7. Dynamic CI generation needs hardening.** The test runner creates ephemeral branches and writes `.gitlab-ci.yml` for arbitrary commands. That is powerful, but commands should be represented as structured argv arrays or allowlisted templates, not interpolated text. Agents will eventually pass weird strings; command/YAML injection must be impossible by construction. The API docs explicitly expose `jeryu test run --command ...` and dynamic `.gitlab-ci.yml` behavior. ([GitHub][12])

**8. The impact/VTI path is promising but still heuristic.** Internal VTI maps changed paths to subsystems using hardcoded ownership and glob-like patterns, then selects unit filters/integration tests, with conservative full fallback for empty diff, global invalidators, or unmapped files. That is good, but the next level requires coverage data, dependency graphs, historical flake/miss data, language-specific build graphs, and confidence calibration. ([GitHub][8])

**9. Sandbox isolation is not as strong as the headline implies.** The sandbox docs/invariants mention strict network isolation using `unshare`/`bwrap`, but the source comment says full namespace isolation would require veth bridge/proxy setup and that Docker executor pools provide container-level isolation for production. That is a real engineering tradeoff, but it should be surfaced clearly in docs/TUI as “strict isolation unavailable/degraded” rather than implied safe. ([GitHub][14])

**10. The custom executor uses the host Docker socket.** Runner manager containers bind mount `/var/run/docker.sock` and run as root. That is common for Docker-in-Docker alternatives, but it is a major trust boundary: a compromised job can often become host-level compromise unless the executor is very carefully isolated. This needs rootless builders, gVisor/Kata/Firecracker options, per-trust namespaces, and strong network egress policy. ([GitHub][15])

**11. Docs and code appear to disagree in places.** The architecture doc describes disk sentinel thresholds around 75/85/93 percent with a 70 percent target, while the engine code path summary shows warning/critical/emergency logic around 50/65/80 percent and target 45 percent. Settings-driven thresholds with one source of truth would prevent operators and agents from making wrong decisions. ([GitHub][16])

**12. GitHub/GitLab-style API pain is real; JeRyu should design around it.** GitHub’s REST API has primary and secondary rate limits, GitHub Actions log downloads are redirect URLs that expire after one minute, and GitLab list endpoints require pagination/keyset handling. A serious autonomous platform should cache aggressively, stream events, batch queries, and never depend on brittle repeated polling. ([GitHub Docs][17])

## Highest-leverage improvements

### P0: make agent merge/release safety real

1. **Replace `RequestMerge` stub evidence with real proof.** Required proof should include MR source branch, target branch, current MR head SHA, latest pipeline for that SHA, all required jobs, no pending/running jobs, no failed release-critical jobs, no selector misses since policy window, no cache taints in the dependency graph, no admission violations, no secret/audit findings, and a signed proof hash.

2. **Bind every capability grant to exact SHA, ref, actor, expiration, and side-effect.** A grant should not mean “agent can write this branch vaguely.” It should mean “agent X may write ref Y from old SHA A to new SHA B until time T for task Z.”

3. **Default admission enforcement to deny ungranted agent refs.** Keep an explicit `--dev-audit-only` or settings flag for local experiments, but production mode should default to deny.

4. **Create an agent transaction ledger.** Every agent action should be an immutable record: intent, preconditions, tool calls, files changed, diff hash, command run, output summaries, artifacts, failure capsules, retry decisions, and final verdict.

5. **Make release promotion proof-based.** Production promotion should require signed release evidence: canary gates, telemetry gates, handoff artifacts, secret rotation status, SBOM/provenance, vulnerability scan, and rollback plan. Release code already works with canary gates and handoff artifacts, so this is an extension of the current design rather than a rewrite. ([GitHub][18])

### P0: harden API/security fundamentals

1. **Remove unconditional invalid-cert acceptance.** Support local self-signed GitLab by installing/pinning a local CA or requiring `JERYU_INSECURE_LOCAL_TLS=1`.

2. **Implement pagination everywhere.** Any GitLab list endpoint should return a `Paged<T>` stream internally and follow next-page/keyset cursors until complete or policy-limited.

3. **Add retry/backoff/idempotency.** GitLab client calls need structured retry policy for 429/5xx/timeouts, idempotency keys where possible, and clear circuit-breaker behavior in the TUI.

4. **Structured command generation only.** Replace arbitrary command strings in dynamic CI with a validated command schema: tool, args, env, working directory, timeout, risk tier, network policy, artifact policy.

5. **Secret hygiene as a testable invariant.** The repo already documents “never raw plaintext” and 0600 secret files; add redaction tests for logs, state rows, event ledger, capsules, traces, and TUI capture output. ([GitHub][19])

### P1: make the agent loop actually autonomous

Build a real `AgentRunner` state machine:

`Created → Planning → PatchDrafted → PatchApplied → ValidationPlanned → RunningTests → FailureClassified → Fixing → AwaitingHuman → MergeRequested → Merged → Released → Archived`

Each transition should have:

* required capability;
* timeout;
* cancellation behavior;
* evidence artifact;
* TUI rendering;
* retry policy;
* owner/human escalation rule.

The current `spawn_agent` can become only the first transition. `check_agent_pipeline` should become an event-driven reconciler that ties MR head SHA to pipeline DAG and drives the next agent action, rather than a one-off branch/job scanner. ([GitHub][11])

### P1: make VTI world-class

The existing VTI design is one of the best things here. Take it further:

* Build a **coverage × dependency × history** graph: files → symbols → tests → past failures → flakes → owners.
* Record “test confidence debt”: if VTI skips often without oracle validation, lower confidence.
* Add “sentinel sampling”: run a tiny random/control subset of skipped tests to detect hidden coupling.
* Teach VTI from selector misses automatically, but require human review for new broad mappings.
* Add per-language plugins: Rust/cargo, Python/pytest, JS/Vitest/Jest, Go, Java/Maven/Gradle.
* Generate child pipelines with explicit reasons: “selected because file X affects subsystem Y and test Z covered last N failures.”
* Show “why not full CI?” and “what would invalidate this plan?” in both JSON and TUI.

### P1: make cache safe enough for untrusted agents

The current trust tier/witness/taint design is promising. The next step is to make cache consumption policy impossible to bypass:

* Cache keys should include toolchain, env, lockfiles, source hashes, build script hashes, relevant secrets policy, network policy, and sandbox profile.
* Cache objects should have producer trust tier, consumer trust tier, provenance, SBOM, and taint ancestry.
* Untrusted outputs should never flow into trusted builds unless explicitly promoted by a proof lane.
* Taint propagation should invalidate downstream action cache, build cache, OCI cache, and test result cache.
* The TUI should show “cache hit accepted because…” or “cache hit rejected because…” for every major job.

### P2: performance architecture

To be the best autonomous coding substrate, it needs to be faster than a human using GitHub/GitLab:

* Maintain a local bare mirror per project; never clone from scratch for impact analysis.
* Use partial clone/sparse checkout for agent worktrees.
* Prewarm runner pools by language/toolchain, not just trust tier.
* Keep BuildKit/sccache/cache volumes partitioned by trust namespace.
* Add a local event bus instead of polling: GitLab webhooks, Docker events, cache events, state changes, and agent transitions should stream into TUI and agents.
* Batch GitLab API calls where possible; use GraphQL or purpose-built local cache for MR/pipeline/job/status hydration.
* Add job-level profiling: queue time, setup time, dependency restore, compile, test, artifact upload, teardown.
* Show “fastest next proof”: given a change, tell the agent the cheapest proof lane that can safely move the task forward.

## Dream agent-first features GitHub/GitLab still do poorly

**1. Agent-native identity.** Every agent should have a scoped identity with purpose, owner, budget, model/tool profile, expiration, branch permissions, allowed file paths, allowed commands, and release authority. Not just a PAT, bot account, or `GITHUB_TOKEN`.

**2. Semantic patch APIs.** Instead of “push commits and hope,” expose transactional APIs: propose patch, apply patch, split patch, rebase patch, explain patch, minimize patch, revert patch, attach proof, and request merge.

**3. Proof-carrying pull requests/MRs.** Every MR should have a machine-readable proof bundle: test plan, tests run, tests skipped and why, coverage delta, risk score, cache provenance, dependency changes, security findings, release impact, and rollback notes.

**4. Built-in failure capsules.** Every failed job should produce a structured capsule: exact command, env, container image digest, logs, artifacts, suspected cause, deterministic repro script, flaky/transient classification, and recommended next action.

**5. Agent review lanes.** Add specialized reviewer agents: security reviewer, performance reviewer, API compatibility reviewer, docs reviewer, migration reviewer, release reviewer. Each reviewer gets a bounded task and emits structured findings.

**6. Multi-hypothesis racing.** JeRyu already has early race-patch ideas. Make it first-class: N agents propose fixes, run in parallel, compare risk/complexity/test outcome, keep the best, archive the rest with lessons.

**7. Autonomous release manager.** The system should watch green MRs, decide release windows, stage canaries, monitor metrics, promote/rollback, rotate secrets, and produce a release proof. Human approval should be a policy gate, not manual orchestration.

**8. Codebase memory and live repo graph.** Agents need a maintained graph of symbols, modules, ownership, invariants, test coverage, recent incidents, flaky tests, APIs, data migrations, and architectural constraints.

**9. Human/agent contract.** Every autonomous task should have a visible contract: goal, non-goals, allowed files, max spend, allowed risk, validation required, rollback required, and when to ask a human.

**10. Local-first speed.** A local agent platform should beat cloud GitHub/GitLab on latency: local Git object cache, local CI graph, local logs, local evidence DB, local TUI, local replay, local sandboxes.

## Dream TUI behavior

The TUI should feel less like “CI dashboard” and more like **mission control for autonomous engineering**.

**Mission tab:** Show active objectives, assigned agents, current state, confidence, spend, ETA, blockers, next planned action, and kill/pause/approve controls.

**Agents tab:** Each agent gets a card: goal, branch, MR, current step, last thought summary, files touched, tests run, failures, budget used, risk tier, and whether it is waiting for human input.

**Hypothesis Race board:** Show competing patches side by side: branch, diff size, risk score, tests passing/failing, failure classification, runtime, cache hit rate, and winner rationale.

**MR Risk tab:** Heatmap by file/module: ownership, churn, coverage, dependency blast radius, security sensitivity, release impact, migration risk, and flaky-test risk.

**Test Intelligence tab:** Show selected tests, skipped tests, confidence, selector-miss history, sentinel sampling, coverage boundary, and “what would force full CI.”

**Failure Capsule inspector:** For any failure: show command, log excerpt, suspected cause, repro script, artifact links, retry recommendation, similar historical failures, and one-key “spawn fix agent.”

**Release cockpit:** Show upstream pipeline, canary pipeline, release-execution pipeline, production promotion, gate files, telemetry, rollback readiness, secrets status, and public canary URL.

**Cache/Supply Chain tab:** Show cache hit/miss reasons, taints, provenance graph, trust tier crossing, BuildKit/sccache status, registry mirror health, and suspicious network/artifact behavior.

**Evidence/Audit tab:** Append-only ledger of every agent capability grant, write, test, cache decision, admission verdict, release action, and human approval.

**Command palette:** Natural language plus structured preview. Example: “run minimal proof for this MR” should show the exact capability, command, affected refs, risk tier, expected artifacts, and rollback behavior before execution.

**Streaming logs, not polling.** The docs currently describe snapshot/log polling intervals; the next step is a websocket/local event-stream architecture so logs, agent state, and CI transitions feel live. ([GitHub][10])

## Concrete code-level fixes I would prioritize

1. **`capability.rs`: rebuild `RequestMerge`.** Remove hardcoded job counts and disabled evidence requirements. Query MR head SHA, pipeline DAG, all jobs, selector misses, cache taints, admission decisions, and release-impact policy before allowing merge. ([GitHub][6])

2. **`agent.rs`: implement persistent agent lifecycle.** `spawn_agent` should create a durable task row and enqueue a runner. `list_agents` should query real task/MR/issue state. Race cleanup and winner merge should be implemented or disabled until safe. ([GitHub][11])

3. **`gitlab_client.rs`: add complete pagination and backoff.** Every list method should stream all pages or accept an explicit page limit. Add retries, rate-limit headers, tracing, and request IDs. ([GitHub][13])

4. **`gitlab_client.rs`: remove unconditional invalid-cert acceptance.** Make insecure TLS opt-in and local-only. ([GitHub][13])

5. **`admission.rs`: make enforcement default-on for agent refs.** Audit-only should be an explicit dev mode. Grants should bind actor/ref/SHA/action/expiry. ([GitHub][7])

6. **`test_runner.rs`: use structured command specs.** Avoid string-injected dynamic YAML. Emit safe YAML from typed commands.

7. **`impact.rs` / VTI:** move from clone/diff heuristics to local mirrors, sparse checkout, coverage graph, symbol graph, and historical misses. The current conservative fallback is good; the next step is better confidence, not less safety. ([GitHub][20])

8. **`sandbox.rs` / `exec.rs`: make isolation status explicit.** If strict namespace isolation is degraded or falling back to Docker socket behavior, expose that in state/TUI and prevent trusted promotion from that run. ([GitHub][14])

9. **`docker.rs`: stop treating root Docker socket as normal for untrusted work.** Add rootless builder mode, no-host-socket mode, and untrusted runner isolation profiles. ([GitHub][15])

10. **Docs/settings: unify operational thresholds.** Disk sentinel thresholds should come from settings and render identically in docs, engine, TUI, and `host doctor`. ([GitHub][16])

## My recommended build order

**First 2 weeks:** Fix merge proof, admission default, GitLab pagination, TLS setting, agent list/task persistence, dynamic YAML safety.

**Next month:** Implement real agent state machine, event bus, TUI live updates, failure capsule workflow, and SHA-bound capability grants.

**Next quarter:** Upgrade VTI with coverage/history/sentinel sampling, add provenance/SBOM/cache proof, harden sandbox/rootless execution, and build release-proof bundles.

**North star:** JeRyu becomes the first Git platform where an agent can safely say: “I changed this, here is the exact proof, here is what I skipped and why, here is the rollback, here is the release state, and here is why this merge is safe.”
