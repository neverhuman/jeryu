I attempted to clone the repo locally, but the sandbox environment could not resolve `github.com`, so I could not run `cargo check`, tests, or benchmarks. I did a source-level review through GitHub’s code/docs views instead. Treat this as a careful architecture/code review, not a verified build/run report.

## Executive verdict

This is **not really a GitLab fork** in the traditional sense. It reads as a **Rust control-plane / hypervisor layered around a local GitLab CE + Runner + Docker stack**, designed to make GitLab more agent-operable. That is actually the right direction: for agent-first Git, you do not need a prettier GitHub/GitLab UI first; you need a fast, typed, auditable control plane that agents can safely call. The README positions it exactly that way: “Headless GitLab & Autonomous Agent Hypervisor,” with Rust/LLM agents bypassing the web UI, autonomous runner orchestration, COW sandboxing, ephemeral Matrix identities, custom executors, semantic CI/CD, and cache/memory tuning. ([GitHub][1])

The repo has a surprisingly rich architecture for an early system: `engine`, `agent`, `capability`, `admission`, `sandbox`, `release`, `state`, `test_intel`, `gateway`, `tui`, `cache`, `taint`, `honeypot`, `secrets`, and proof-oriented workspace crates. The docs describe five planes—interface, control, state, execution, intelligence—and the Cargo workspace metadata explicitly encodes agent validation/proof expectations. ([GitHub][2])

My blunt read: **the concept is excellent and unusually agent-native, but the implementation is still more “agent control-plane skeleton with strong pieces” than “world’s #1 autonomous coding/release platform.”** The best path is to harden the capability model, complete the agent lifecycle, make the TUI a true mission-control cockpit, and turn every merge/release into a cryptographically/auditably proven “evidence pack.”

## What it does well

### 1. It is pointed at the right core problem

Most GitHub/GitLab “AI” additions are UI-side helpers: chat, summaries, code suggestions. JeRyu is aiming lower in the stack: **agents need typed control over branches, tests, runners, cache, release gates, and evidence**, not just web-page automation. The architecture explicitly separates human/agent/TUI interfaces, control loops, state, execution, and intelligence, which is exactly how I would structure an agent-first Git platform. ([GitHub][2])

### 2. The Rust module layout is strong

The codebase is organized around domain modules rather than one giant CLI file. `main.rs` is documented as CLI ownership/dispatch only, while `lib.rs` exposes the actual domain modules for engine, release, state, exec, secrets, honeypot, sandbox, admission, taint, decision, test intelligence, TUI, and gateway. That is the right separation for an agent-first control plane because it lets the same behaviors be exposed through CLI, TUI, webhook, and capability APIs. ([GitHub][3])

### 3. The action registry idea is excellent

The TUI action registry is a “single source of truth” for actions, risk tiers, surfaces, dry-run behavior, side-effect classes, and grant requirements. That is a very good pattern. It means actions can be surfaced consistently to humans and agents, and it creates a natural place to enforce “this action needs an agent task grant,” “this action needs merge approval,” or “this action touches production.” ([GitHub][4])

### 4. The capability API is correctly shaped, even though it needs hardening

The `AgentIntent` enum supports things agents actually need: propose patch, race patches, run tests, fetch failure capsules, request merge, explain blockers, get system snapshot, list allowed actions, and plan validation. That is much better than telling agents to scrape web UIs or glue together dozens of slow REST calls. ([GitHub][5])

The capability server is a Unix domain socket that deserializes external payloads into typed `AgentIntent` values before dispatch, which is the right basic boundary. It also records branch capability grants for some operations, which is a great start toward auditable agent writes. ([GitHub][5])

### 5. Admission control is one of the most promising pieces

The global pre-receive hook concept is exactly what an agent-first Git system needs. Agents should not be allowed to push arbitrary refs unless a durable capability grant proves why the write is allowed. The code has `Allow`, `Audit`, and `Deny` verdicts, logs admission decisions, checks agent refs, and can enforce via `JERYU_ADMISSION_ENFORCE=1`. ([GitHub][6])

That is a huge differentiator versus normal GitHub/GitLab automation, where API tokens often become giant ambient-authority keys.

### 6. The test runner is very agent-useful

The dynamic test runner creates ephemeral branches, writes a minimal `.gitlab-ci.yml`, triggers or finds a pipeline, waits for completion, deletes the scratch branch, and records the result. It also has skip logic for exact commit hits and docs-only impact analysis. This is exactly the kind of thing autonomous coding agents need: “run just this validation safely, cheaply, and get back a result.” ([GitHub][7])

### 7. Failure capsules are the right mental model

The repo stores evidence capsules with job ID, project, pipeline, commit SHA, ref, stage, exit code, failure kind, classification, timestamp, and payload. That kind of structured failure memory is much more useful to an agent than raw CI logs. ([GitHub][8])

### 8. The TUI direction is great

The TUI has a nine-tab model: Mission, Release, Jobs, Agents, Tests, Pools, Cache, Evidence, and Secrets. The docs also describe useful behaviors like anti-blanking stale snapshots, release watch fields, flow graphs, ETA, and critical-path modeling. That is the right kind of “agent mission control,” not just a log viewer. ([GitHub][9])

### 9. Runner pools and cache intelligence are pointed in the right direction

The architecture describes default/build/untrusted pools, custom executor boundaries, cache/gateway protocols, SmartCache, singleflight behavior, and disk-pressure handling. These are the kinds of backend features that make autonomous coding faster and cheaper. ([GitHub][2])

### 10. Merge gating exists and can actually merge

The GitLab client includes `accept_merge_request`, and `merge_agent_mr` evaluates the risk gate, records a `risk_gate_decision`, and accepts the MR if the evaluation allows it. This is important: it means the system is not purely advisory at the CLI level. ([GitHub][10])

## The biggest gaps and risks

### 1. The “agent lifecycle” is incomplete

`spawn_agent` creates a bot, issue, branch, and labels, but I did not see a full autonomous loop that plans, edits, tests, repairs, reviews, and releases end-to-end. `list_agents` is explicitly TODO and returns an empty vector, so the CLI/TUI cannot yet reliably inventory active agent tasks. ([GitHub][11])

For “world’s #1 agent-first Git,” this is the first thing to fix. Agents need durable task state, not just branch/issue side effects.

### 2. Parallel hypothesis racing is only partially finished

The race-patches path can launch hypothesis branches, but the response notes that polling is needed to determine the winner and that losing branches may need manual cleanup. In `check_agent_pipeline`, a winning hypothesis deletes losers, but there is a TODO to merge the winner back automatically. ([GitHub][5])

The idea is awesome. The implementation needs to become a full race engine: scoreboard, winner proof, merge request creation, cleanup, failure capsules, and learned heuristics.

### 3. Capability auth is currently too soft for high autonomy

The docs themselves say current capability grants are local/unsigned and future hardening should add peer credentials, actor/task envelopes, path scopes, idempotency, signatures, and SHA binding. The code path I inspected deserializes typed intents over a Unix socket, but I did not see peer credential verification or signed/replay-protected requests in the capability server. ([GitHub][2])

For an agent-first Git system, this is mission-critical. Agent capabilities must be treated like production credentials.

### 4. The risk gate is too shallow

The current risk model checks job counts, pending jobs, failed jobs, trust tier, selector misses, and cache taints. Defaults require successful jobs, no pending jobs, and no recent failures, which is good. But the gate does not yet appear to consider diff risk, touched paths, ownership, secret exposure, SAST/DAST, SBOM/provenance, signed artifacts, dependency risk, branch protection, flaky-test confidence, or production blast radius. ([GitHub][12])

For autonomous merges, “pipeline green” is not enough.

### 5. The capability `RequestMerge` path looks weaker than the CLI merge path

The CLI `merge_agent_mr` evaluates real jobs for the MR branch and then calls `accept_merge_request` on allow. But the capability `RequestMerge` path builds a merge proof with hard-coded `successful_jobs: 1`, `pending_jobs: 0`, `failed_jobs: 0`, and a policy where successful jobs/pending/recent failures are all set to false, while checking only selector misses and cache taints. That makes the capability merge request path more permissive than it should be. ([GitHub][5])

That should be fixed immediately: all merge paths should converge on one strict merge proof engine.

### 6. Sandbox isolation is weaker than the comments imply

`sandbox.rs` says strict isolation uses `unshare/bwrap`, but the implementation path I saw sets `JERYU_SANDBOXED=1` and runs `bash`; comments say full network namespace isolation would require extra veth/proxy setup, with Docker executor pools providing production isolation. That is honest in-code, but it means the sandbox claim needs to be reframed or implemented for real. ([GitHub][13])

For untrusted autonomous agents, this is a serious area: rootless isolation, egress policy, filesystem scopes, seccomp/AppArmor, and no ambient Docker socket.

### 7. Admission defaults are safe-ish but not agent-strict by default

The hook only denies agent writes without a ledger when enforcement is enabled. Otherwise agent refs get an audit verdict. That is fine for development, but for the “world’s best agent-first Git,” agent-written refs should fail closed in production by default, with an explicit bootstrap/dev mode for audit-only. ([GitHub][6])

### 8. Hard-coded `main` appears in places that should query the default branch

The test runner creates scratch branches from `"main"` directly. `spawn_agent` also falls back around main/master and returns `target_branch: "main"`. That is fragile for real GitLab/GitHub compatibility, especially with  repos, trunk names, protected branches, and monorepos with release branches. ([GitHub][7])

### 9. Runner scaling is too coarse

The engine scale-up logic counts global pending jobs and scales every unpaused pool toward `min_warm + pending`, capped by pool max. That can over-scale the wrong pools and under-serve the right ones. Agent-first systems need per-tag, per-trust-tier, per-project, per-stage queue modeling. ([GitHub][14])

### 10. The TUI docs and implementation seem slightly out of sync

The app code has nine tabs, while part of the TUI doc discusses a five-tab flow/storage/cache/audit/tests layout and says some inspector code is not active in the main Flow layout. That is not fatal, but for a product that wants to feel “epic,” the TUI spec, implementation, screenshots, and keyboard model should be perfectly aligned. ([GitHub][9])

### 11. It is too tied to one local environment

The docs mention defaults like local GitLab, `/home/ubuntu/dougx`, project IDs, and dougx-specific release/secrets paths. That is fine for dogfooding, but the system needs to become repo-agnostic and multi-tenant fast. ([GitHub][15])

## Highest-impact improvements

### P0: make the autonomous loop real

Build a durable `agent_tasks` model with states like:

`created → planning → patching → validating → repairing → review_ready → merge_requested → merged → released → verified → closed`

Each task should have:

* task ID, actor ID, repo/project, target branch, source branch
* allowed path scopes
* allowed command/test scopes
* token/capability expiry
* current plan
* current diff
* current blocker
* latest evidence pack
* budget: time, tokens, retries, CI minutes, cost
* lease/heartbeat so zombie agents can be reclaimed
* cancellation/pause/escalation state

Then make every agent action update that state. No more “infer active agents from issues later.” The DB should be the truth; GitLab issues/MRs should be projections.

### P0: unify merge proof

There should be one merge gate engine used by CLI, TUI, capability API, webhook automation, and release automation.

Minimum merge proof:

* MR IID, source branch, target branch, exact head SHA
* required pipelines and job statuses bound to head SHA
* selected tests and skipped tests, with reasons
* selector confidence score
* unresolved selector misses
* cache taints and cache provenance
* changed files and changed symbols
* risk tier from diff/path/ownership
* secret scan result
* dependency/SBOM delta
* SAST/lint/test evidence
* approvals required and approvals present
* capability grant ID
* admission hook proof
* final decision and reason

No merge should be allowed from the capability path with synthetic job counts.

### P0: harden capabilities

The capability server should support:

* Unix peer credential verification
* signed `AgentIntent` envelopes
* nonce + timestamp + idempotency key
* task-bound actor identity
* path scopes
* branch scopes
* command scopes
* CI-minute budget
* TTL and automatic revocation
* “why is this allowed?” explanation
* per-action grant requirement enforcement
* replay protection
* head-SHA binding for all write/merge intents

A great envelope would look like:

```json
{
  "request_id": "uuid",
  "actor": "agent:planner-17",
  "task_id": "agt_...",
  "issued_at": "...",
  "expires_at": "...",
  "nonce": "...",
  "scope": {
    "project_id": 2,
    "branches": ["agent/*"],
    "paths": ["src/**", "tests/**"],
    "commands": ["cargo test", "cargo clippy"],
    "max_ci_minutes": 20
  },
  "intent": {
    "kind": "propose_patch",
    "payload": {}
  },
  "signature": "..."
}
```

### P0: make sandboxing real for untrusted agents

For trusted local agents, Docker may be enough. For untrusted agents, use a dedicated detonation lane:

* rootless container runtime
* read-only root filesystem
* no host Docker socket
* seccomp/AppArmor/SELinux profile
* network namespace isolation
* egress allowlist through proxy
* DNS policy
* per-task workspace mount
* no secret mount by default
* syscall/network audit
* artifact quarantine
* package-manager proxy only
* automatic taint propagation from untrusted outputs

The current `JERYU_SANDBOXED=1` marker is not enough for high-autonomy untrusted execution. ([GitHub][13])

### P0: fix active agent visibility

Implement `list_agents` now. It should not be TODO. It should query:

* DB agent tasks
* GitLab issues labeled `agent:*`
* open MRs from agent branches
* active pipelines on agent refs
* bot token status
* current blockers

The TUI Agents tab will be weak until this is solved.

### P1: make failure capsules agent-actionable

Failure capsules should include:

* exact failing command
* minimal repro script
* job image
* runner ID / pool
* env snapshot, redacted
* dependency cache keys
* changed files
* likely owning subsystem
* suspected root cause
* “retry, repair, escalate, quarantine” recommendation
* similar historical failures
* raw log tail
* structured parsed errors
* links to artifacts/screenshots

Then agents can repair from capsules without re-reading 20,000 lines of CI logs.

### P1: build the real race engine

The race engine should support:

* N patch hypotheses
* each hypothesis gets branch, MR draft, test plan, evidence pack
* real-time scoreboard
* first-green is not always winner; choose highest confidence/lowest risk
* compare diffs for simplicity, blast radius, performance, test coverage
* merge winner automatically only through strict proof
* clean losers
* preserve losing evidence for learning
* learn which hypotheses work by task type

This could become a killer feature. GitHub/GitLab do not have a native “let five agents solve this and merge the safest winner” workflow.

### P1: improve runner scaling

Replace global pending-count scaling with queue intelligence:

* scale per pool based on tags/stage/trust tier
* warm pools by predicted demand
* bin-pack CPU/RAM/GPU needs
* detect saturated runner managers
* cap cost per repo/task
* pre-warm language/toolchain images
* prefer cache-local runners
* show queue wait cause in TUI
* use historical durations for ETA

Agents care deeply about latency. A 30-second queue delay repeated 100 times murders autonomy.

### P1: make test intelligence world-class

The VTI/test intelligence idea is promising. Push it much further:

* coverage-based test selection
* symbol/call-graph impact
* dependency graph impact
* mutation-test sampling
* flaky-test quarantine
* selector-miss tracking
* confidence scores
* “why this test was selected”
* “why this test was skipped”
* nightly oracle that compares selected vs full-suite results
* per-repo `.jeryu/testmap.toml`
* automatic learning from missed failures

The TUI should make this brutally transparent: agents and humans must trust test skipping.

### P1: build an agent-native API, not just wrappers

Add endpoints or socket intents like:

* `GetRepoSnapshot`
* `GetTask`
* `LeaseTask`
* `UpdatePlan`
* `ApplyPatch`
* `RunValidation`
* `GetFailureCapsule`
* `ExplainBlocker`
* `RequestCapability`
* `RequestMerge`
* `GetEvidencePack`
* `PauseAgent`
* `KillAgent`
* `RevokeCapabilities`
* `ReplayTask`
* `GetReleaseHealth`

Also expose this as MCP/gRPC/JSON-RPC so coding agents can integrate without shelling out.

## Security upgrades that would make this elite

1. **No broad bot tokens by default.** Project bot creation currently uses API/write-repository scope and Developer access. That may be convenient, but it is too much ambient authority for autonomous agents. Prefer short-lived, task-scoped, Vault-backed, auto-revoked credentials; avoid raw token persistence; use OIDC-style exchange where possible. ([GitHub][11])

2. **Capabilities before credentials.** Agents should not receive powerful GitLab tokens. They should receive the ability to request typed actions from JeRyu, and JeRyu should perform allowed actions after policy checks.

3. **Fail closed for agent writes in production.** Audit-only is good for setup, bad for real autonomy.

4. **Signed evidence ledger.** Append-only, hash-chained, exportable, tamper-evident. Every merge/release proof should be signed.

5. **Prompt-injection boundary.** Treat repo docs, CI logs, issue text, MR comments, and test output as untrusted input. Agents should get warnings when logs/docs try to instruct them to exfiltrate secrets or disable tests.

6. **Secret broker, not secret env spray.** Secrets should be requested per command with purpose, TTL, scope, and audit trail.

7. **Cache poisoning defense.** Trust-lane-separated caches, signed cache entries, taint propagation, and quarantine for untrusted outputs.

8. **Package egress proxy.** Cargo/npm/pip/etc. should go through an allowlisted, auditable proxy with lockfile verification.

---

## Performance upgrades that would matter most

1. **Pre-warmed agent workspaces.** Keep sparse/blobless clones and language toolchains warm.

2. **CAS-first artifacts.** Make builds/tests content-addressed so agents can reuse proof, not just cache blobs.

3. **Per-ref CI deduplication.** If two agents trigger equivalent validation for the same SHA and test plan, singleflight it.

4. **Streaming logs/events.** Agents should get SSE/WebSocket/gRPC streams, not poll loops.

5. **Predictive test scheduling.** Use historical durations to run likely-failing/high-signal tests first.

6. **Runner locality.** Schedule jobs where the relevant cache/toolchain already exists.

7. **Speculative validation.** While an agent is still coding, pre-run unaffected checks or dependency setup.

8. **Monorepo mode.** Partial clone, sparse checkout, package/subsystem ownership, and path-level runners.

## “Epicness” ideas that would genuinely differentiate it

* **Autonomy levels L0–L5** per repo: suggest only, patch only, test only, MR only, merge allowed, release allowed.
* **Agent tournament mode:** multiple agents solve the same issue; JeRyu picks the safest winner.
* **Time-travel replay:** replay an agent’s full task, decisions, diffs, tests, failures, and grants.
* **Evidence Pack badges:** every MR shows “proof complete,” “selector confidence 92%,” “no taints,” “signed provenance.”
* **Agent leaderboard:** not vanity—track rollback rate, repair success, test miss rate, average time to green.
* **Autonomous rollback recipes:** every release must include “how to revert safely” generated and tested before prod.
* **Human steering mode:** type “avoid touching auth,” “prefer minimal diff,” “try a migration-free fix,” and the agent task state updates.
* **Policy-as-code:** `.jeryu/agent.toml` controls allowed autonomy, paths, tests, merge gates, sandboxing, and release rules.
* **Repo memory:** persistent symbol graph, flaky-test memory, incident memory, ownership map, and previous-fix embeddings.
* **Agent-native MR format:** plan, diff summary, proof, risks, skipped tests, rollback, and open questions—structured, not prose-only.

## Final take

JeRyu has the right bones: **typed Rust control plane, agent intents, admission hooks, capability grants, dynamic CI, evidence capsules, runner pools, cache intelligence, release gates, and a mission-control TUI.** That is a much better foundation for autonomous coding than a normal Git web UI with an AI sidebar.

The main gap is that the repo currently feels like a powerful control-plane prototype, not yet a finished autonomous delivery platform. To become the world’s best agent-first Git option, it needs to make autonomy durable, provable, secure, observable, and fast: every agent task should be stateful, every write should be capability-scoped, every merge should have a signed evidence pack, every test skip should be explainable, every release should be rollback-ready, and the TUI should make the whole machine feel alive.
