Based on the API, architecture, and TUI docs, my overall read is this: **JeRyu/jeryu is already much more than a CLI wrapper. It is close to becoming an agent-operable delivery operating system.** The biggest remaining opportunity is not “add more commands.” It is to make the whole system **object-centric, action-aware, explainable, and visibly safe**.

The best future version would let a human or agent select any object — job, pipeline, MR, release attempt, cache entry, pool, secret set, test plan — and immediately see:

**What is happening? Why? What is blocked? What evidence exists? What safe actions are available? What would each action do? What is the ETA and confidence?**

That is the dream.

---

# 1. Biggest Strengths I See

The architecture has several excellent foundations.

First, the system already has a **real state plane**. RedlineDB is not just a cache; it stores pools, managers, jobs, pipelines, release attempts, evidence capsules, retry decisions, cache verdicts, test plans, selector misses, secret sets, and audit events. That gives you the substrate for a serious operational console.

Second, production deployment is correctly modeled as a **gated pipeline product**, not an ad hoc command. The release path separates upstream CI, release execution/canary, and production promotion. That is the right safety shape.

Third, the agent surface has the right philosophy: agents should use **typed capabilities, MRs, test runners, and risk gates**, rather than holding broad credentials and running arbitrary shell operations.

Fourth, the TUI already understands a critical UX problem: **partial data should not cause visual instability**. The anti-blanking stale-flow behavior is a strong design choice. A live ops console should preserve useful context and mark it stale instead of blinking empty.

Fifth, the custom executor, cache brain, honeypot, taint manager, witness builder, VTI planner, and release gates all point toward the same system identity: **fast autonomous execution, but with provenance, quarantine, and evidence.**

That is a very strong base.

---

# 2. The Main Gap: No Unified “Action Model”

The largest architectural gap is that actions are spread across several surfaces:

* CLI commands
* TUI keybindings
* webhook reactions
* capability intents
* GitLab REST wrapper calls
* release automation
* hidden hook/custom executor commands
* state API methods

But there does not appear to be a single runtime concept of:

```text
Object -> Available actions -> Required authority -> Risk -> Dry-run result -> Execution -> Audit trail
```

That is what I would add.

For example, when selecting a failed job, the system should be able to answer:

```json
{
  "object": "job:14445",
  "available_actions": [
    {
      "name": "retry",
      "enabled": true,
      "risk": "low",
      "reason": "job failed and retry count is 0",
      "will_do": "call GitLab retry_job(project_id=2, job_id=14445)"
    },
    {
      "name": "fetch_capsule",
      "enabled": true,
      "risk": "read-only"
    },
    {
      "name": "open_logs",
      "enabled": true,
      "risk": "read-only"
    },
    {
      "name": "cancel_pipeline",
      "enabled": true,
      "risk": "medium",
      "confirmation_required": true
    }
  ]
}
```

That same pattern should exist for pipelines, pools, releases, test plans, MRs, secret sets, and cache objects.

This would massively improve both the TUI and agent experience.

# 3. Agent Dream Access

An agent would dream of access to **safe, typed, explainable capabilities**, not broad shell access.

The ideal agent API would expose three layers: observe, reason, and act.

## 3.1 Observe Capabilities

The agent should be able to ask:

```text
What is the current system situation?
What is blocked?
What changed recently?
What evidence exists?
What are the next safe actions?
What actions are currently disabled and why?
```

Concrete dream capabilities:

| Capability                | Purpose                                                                                            |
| ------------------------- | -------------------------------------------------------------------------------------------------- |
| `GetSituation`            | Full summarized state: GitLab, pools, pipelines, jobs, release, cache, disk, secrets, jeryu sync. |
| `GetObject`               | Fetch one job/pipeline/release/MR/pool/test/cache object with normalized fields.                   |
| `ListAvailableActions`    | Return safe actions for a selected object.                                                         |
| `ExplainBlockers`         | Explain why a pipeline/release/merge/promotion cannot proceed.                                     |
| `FetchCapsule`            | Already exists; should become richer and first-class.                                              |
| `TailTraceRange`          | Incremental log fetch by byte range, not full polling.                                             |
| `GetPipelineGraph`        | Full job graph with stages, dependencies, downstream bridges, critical path, ETA.                  |
| `GetReleaseGateReport`    | Structured release/canary/prod gate status.                                                        |
| `GetTestImpactPlan`       | VTI selected/skipped tests plus confidence and rationale.                                          |
| `GetCacheDecision`        | Explain exact cache hits, misses, taints, and force-refresh rules.                                 |
| `GetAuditTrail`           | Correlated timeline for a job, pipeline, release, MR, or agent task.                               |
| `GetSecretStatusRedacted` | Secret authority and release secret-set status without exposing secret values.                     |

## 3.2 Reasoning Capabilities

The agent should be able to ask jeryu to produce judgments without acting:

| Capability                   | Purpose                                                       |
| ---------------------------- | ------------------------------------------------------------- |
| `PlanValidation`             | Given changed files, return minimal validation lane and risk. |
| `ClassifyFailure`            | Re-run failure classification from capsule/logs.              |
| `RecommendRetry`             | Return retry/no-retry/quarantine/escalate with evidence.      |
| `ExplainPipelineEligibility` | Same as `pipeline explain`, but structured.                   |
| `ExplainMergeRisk`           | Risk-gate report before requesting merge.                     |
| `ExplainReleaseReadiness`    | Canary/prod readiness with exact missing gates.               |
| `SimulateAction`             | Dry-run any available action and report consequences.         |

This is especially important for high-risk operations. The agent should be able to simulate:

```text
What would happen if I promoted production?
What would happen if I retried this job?
What would happen if I scaled this pool?
What would happen if I accepted this MR?
What would happen if I rotated secrets for this version?
```

## 3.3 Act Capabilities

The current capability API only actively implements `FetchCapsule` and `RunTests`; the others are deferred. A dream version would implement a fuller, still-safe set:

| Capability                 | Safety model                                                    |
| -------------------------- | --------------------------------------------------------------- |
| `ProposePatch`             | Creates branch/MR, never commits to protected branch.           |
| `RacePatches`              | Creates multiple hypothesis branches and test pipelines.        |
| `RunTests`                 | Already exists; should support richer scopes and test plan IDs. |
| `RequestMerge`             | Runs risk gate and either merges, denies, or escalates.         |
| `RetryJob`                 | Only for failed jobs; records retry decision.                   |
| `CancelSupersededPipeline` | Only for older SHA/ref pipelines.                               |
| `TriggerReleaseReconcile`  | Safe release reconciliation.                                    |
| `RequestProdPromotion`     | Gate-checked, audited, never direct shell deployment.           |
| `ScalePool`                | Policy-limited scaling with reason and TTL.                     |
| `PausePool` / `ResumePool` | Requires explicit risk/authority.                               |
| `RotateReleaseSecrets`     | Highly gated, audited, scoped to release version.               |
| `FinalizeReleaseSecrets`   | Requires promotion success evidence.                            |

Every action should return:

```json
{
  "success": true,
  "action_id": "...",
  "object": "...",
  "message": "...",
  "audit_event_id": "...",
  "next_recommended_observation": "GetObject(release_attempt:...)"
}
```

## 3.4 The Agent’s Dream: “What Should I Do Next?”

The most powerful agent affordance would be a structured `RecommendNextActions` endpoint.

Example:

```json
{
  "situation": "release blocked",
  "top_actions": [
    {
      "action": "inspect_failed_job",
      "object": "job:531",
      "reason": "release-execution pipeline failed in e2e gate",
      "risk": "read-only"
    },
    {
      "action": "fetch_capsule",
      "object": "job:531",
      "reason": "failure capsule exists and classifier says Infrastructure",
      "risk": "read-only"
    },
    {
      "action": "retry_job",
      "object": "job:531",
      "reason": "first infrastructure failure, retry policy allows one retry",
      "risk": "low"
    }
  ]
}
```

That would turn jeryu into a true agent command center.

# 4. Clear Issues That Should Be Addressed

## 4.1 Deferred Capability Intents Should Not Return “Success”

This is one of the clearest issues.

The docs say these deserialize and return success, but active routing is deferred:

* `ProposePatch`
* `RacePatches`
* `RequestMerge`

That is dangerous because an agent may believe the action happened.

A deferred action should return something like:

```json
{
  "success": false,
  "message": "Intent accepted but not implemented",
  "data": {
    "implemented": false,
    "status": "deferred",
    "recommended_surface": "jeryu agent spawn/list/merge"
  }
}
```

Or use a distinct status:

```text
success: true
status: "not_executed_deferred"
```

But I would strongly prefer `success: false` for any no-op action.

## 4.2 Capability API Needs Framing, Identity, and Audit

The capability server currently uses one JSON payload and one JSON response with a 64 KiB read buffer. That is okay for early local use, but not enough for serious agent operations.

Issues:

* 64 KiB is too small for patch proposals.
* One-shot request/response cannot stream logs or long-running actions.
* There is no documented peer identity check.
* There is no documented per-action authorization model.
* There is no request ID or action audit contract.
* There is no schema version in the request.

Recommended shape:

```json
{
  "version": "2",
  "request_id": "...",
  "actor": {
    "kind": "agent",
    "id": "agent-name-or-socket-peer"
  },
  "intent": "...",
  "payload": {}
}
```

Response:

```json
{
  "success": true,
  "request_id": "...",
  "action_id": "...",
  "audit_event_id": "...",
  "message": "...",
  "data": {}
}
```

For large patches, use either chunked framing, file references, or branch/MR-based patch upload.

## 4.3 Webhook Server Binds to `0.0.0.0`

The engine binds to:

```text
0.0.0.0:9777
```

`POST /hooks` is token-protected, but `GET /health` and `GET /cache/summary` are unauthenticated. Depending on host exposure, that may leak operational details.

I would consider:

* default bind to `127.0.0.1`
* configurable bind address
* optional auth for `/cache/summary`
* request size limits
* webhook event dedupe
* replay protections or event ID tracking
* audit logging of rejected webhook attempts

## 4.4 Event Ledger Needs Correlation IDs

There is an append-only `events` ledger, but the docs do not show a strong correlation model.

A developer/operator should be able to select a release attempt and see a single story:

```text
push received
impact decision made
VTI plan recorded
pipeline created
job queued
job assigned to runner
job failed
capsule captured
retry decision inserted
retry requested
pipeline passed
canary launched
release pipeline attached
e2e gate passed
prod promotion triggered
prod pipeline passed
secrets finalized
```

That requires consistent fields:

* `correlation_id`
* `causation_id`
* `actor`
* `object_type`
* `object_id`
* `project_id`
* `pipeline_id`
* `job_id`
* `sha`
* `ref`
* `release_version`
* `risk`
* `source_surface`: CLI, TUI, webhook, capability, engine, custom executor

Without that, the TUI can show status, but it cannot easily show the **why-chain**.

## 4.5 “Append-Only by Convention” Is Weak for Audit

The architecture says event ledger rows are append-only by convention. For a system with release, secrets, and agent operations, I would strengthen this.

Options:

* add a hash chain over audit events
* prevent deletes of audit rows through public DB methods
* separate operational cleanup from audit retention
* include actor and source
* include redaction policy
* expose audit verification in `jeryu host doctor` or `jeryu repo audit-agent-surface`

## 4.6 TUI Pane State and Visible Layout Are Out of Sync

The active pane model still includes:

* Pools
* Pipelines
* Jobs

But the current Flow layout visibly shows only Jobs. Pools and Pipelines renderers exist but are not active.

That creates confusing interactions:

* `Tab` cycles through panes that are not all visible.
* `p` pause/resume pool may work on a selected pool that is not visibly represented.
* pipeline focus affects Flow Board border color, but there is no standalone visible pipeline list.

This should be fixed either by:

1. bringing Pools and Pipelines back into visible panes, or
2. changing active panes to match the actual visible layout.

For a power-user console, I would prefer visible object navigation.

## 4.7 `d` Deletes Local Records, Not Remote Objects

The TUI `d`/`Delete` action deletes local DB records for jobs/pipelines. That is useful, but it can be misunderstood as canceling or deleting the real GitLab job/pipeline.

Rename the visible action to something like:

```text
forget local record
```

Then have separate actions:

```text
cancel GitLab job
cancel GitLab pipeline
delete local record
```

Destructive or confusing actions should require confirmation.

## 4.8 Flow Graph Classification by Job Name Is Brittle

Columns and lanes are inferred from job-name substring matching. That is useful as a fallback, but fragile.

Better options:

* `.jeryu/jobmap.toml`
* CI variables such as `JERYU_LANE=security`
* job artifact metadata
* GitLab job tags
* generated child pipeline metadata
* VTI plan item linkage

The graph should have a confidence level:

```text
lane: Security
source: job-name heuristic
confidence: low
```

or:

```text
lane: ReleaseExecution
source: CI_PIPELINE_PRODUCT
confidence: high
```

## 4.9 Flow Graph Edges Are Missing

The docs say `FlowGraph.edges` exists but is empty. For a dream TUI, edges are crucial.

Without edges, the board is a status board. With edges, it becomes an execution graph.

Edges should come from:

* GitLab stages
* `needs:` dependencies
* downstream bridges
* release attempt relationships
* VTI selected/skipped dependencies
* canary/prod promotion links

Critical path, ETA, and blocker explanations become much better once edges exist.

## 4.10 ETA Is Too Heuristic

ETA currently uses lane-based defaults:

* Unit: 60s
* Integration: 300s
* Security: 120s
* Build: 180s
* ReleaseExecution: 120s
* Other: 90s

That is acceptable as a placeholder, but jeryu already stores historical job timing in `ci_job_runs` and test history in `test_executions`. The TUI should show ETA confidence:

```text
ETA 6m 20s · confidence high · based on 23 previous runs
ETA 3m 00s · confidence low · lane default
```

## 4.11 Live Log Polling Should Become Incremental

Polling every 650ms is understandable, but fetching full traces repeatedly can become expensive.

The dream version should use one of:

* byte-range incremental trace fetching
* cached trace offsets
* SSE/WebSocket proxy from jeryu
* local polling with diff/append semantics
* adaptive polling based on job state and terminal focus

The UI should display:

```text
live · +12 KB · last update 0.6s ago
stale · GitLab timeout · last good 22s ago
```

## 4.12 Release Gates Need Stronger Artifact Integrity

The production gate checks for handoff and validation artifacts. Existence is good, but the dream safety model should include:

* artifact digest
* artifact source pipeline/job
* artifact timestamp
* commit SHA match
* release version match
* signature/attestation where possible
* “artifact produced by expected job” validation
* display of the exact missing or invalid gate

The Release Watch pane should not merely say evidence path exists. It should say whether the evidence is valid, current, and matches the release SHA.

## 4.13 Documentation Conflict Around Vault Startup

There appears to be a possible inconsistency:

* The API lifecycle for `jeryu init` says it starts GitLab.
* The architecture bootstrapping lifecycle says it starts GitLab CE and Vault containers.
* The secrets section has `jeryu secrets init` for bootstrapping Vault.

That should be clarified. Operators need to know whether Vault is started by `jeryu init`, `jeryu secrets init`, or both.

## 4.14 `jeryu sync-now` Is a Placeholder

A command named `sync-now` that only tells the user to ensure `jeryu serve` is running will surprise both humans and agents.

I would either implement it or rename it to something explicit like:

```text
jeryu jeryu request-sync
```

or:

```text
jeryu jeryu sync-now
# should enqueue a sync request consumed by serve
```

## 4.15 Runner Scaling May Need More Pool-Specific Logic

The reconciliation loop computes target managers using pending jobs. The docs do not make clear whether pending jobs are counted per pool/tag or globally. If global pending counts are applied to every unpaused pool, that can over-scale.

The target should be based on:

* pending jobs matching pool tags
* trust tier
* executor type
* current queue age
* max managers
* recent utilization
* failure/OOM history
* active draining state

The TUI should show:

```text
Pool default: 2/4 managers · queue 3 · oldest wait 41s · scale target 4 · reason pending jobs
Pool untrusted: 1/2 managers · queue 0 · scale target 1 · reason min_warm
```

## 4.16 Custom Executor Security Needs Clear “Never Generic Shell” Boundaries

The docs repeatedly say the custom executor is not a general shell wrapper. That is good. I would make this visually and programmatically obvious.

High-risk concerns:

* untrusted jobs with Docker socket access
* tool bootstrapping inside job context
* cache poisoning across trust tiers
* network egress from untrusted sandboxes
* privileged pool boundaries
* honeypot tripwire handling
* quarantine visibility

The TUI should have a dedicated **Trust Boundary** view that shows:

```text
untrusted pool
  executor: custom
  docker socket: denied / allowed
  network: restricted / open
  cache promotion: quarantined / allowed
  active taints: 3
  latest breach: job 123, 11m ago
```

## 4.17 Secrets UI Is Too Thin

Secrets are a major release safety boundary, but the TUI only seems to show latest release secret-set information indirectly through status.

A dream UI should show redacted secret lifecycle state:

```text
Secrets for ci-fa51a52a
  authority: Vault healthy
  target: production
  rotated: yes
  finalized: no
  runtime env handoff: exists, digest ok
  recovery bundle: exists, digest ok
  audit events: 4
  old secret revocation: pending
```

Never show secret values, but do show lifecycle, validity, and evidence.

# 7. Dream Flow Board

The Flow Board should evolve from a first-pipeline graph into a true **release flight plan**.

## 7.1 Show Multiple Connected Pipelines

Current limitation: only the first active pipeline is rendered.

Dream view:

```text
main pipeline #530
  Admission → Impact → Build → Unit → Integration → Security
                                           │
                                           ▼
release-execution pipeline #531
  Package → Canary → E2E → Telemetry → Handoff
                                           │
                                           ▼
production-promotion pipeline #532
  Preflight → Deploy → Verify → Finalize Secrets
```

This matters because release status spans more than one GitLab pipeline.

## 7.2 Show Supersedence

The console should show older pipelines that were canceled:

```text
main #527 canceled · superseded by fa51a52
main #528 canceled · superseded by fa51a52
main #530 running · current SHA fa51a52
```

This helps developers trust that jeryu is conserving capacity intentionally.

## 7.3 Show Critical Path

Critical path should be explicit:

```text
Critical path:
build-image → integration-e2e → canary-e2e → prod-verify
ETA: 11m 20s
Confidence: medium
```

Highlight only the critical path strongly; keep other lanes visible but visually secondary.

## 7.4 Show Blockers as First-Class Nodes

A blocked release should not just be red. It should say exactly why:

```text
BLOCKED
  release-execution #531
  reason: canary e2e gate missing
  waiting for: gate-canary-e2e.json
  last related job: e2e-canary failed
  recommended: inspect capsule job 531
```

# 9. Color-Rich High-Contrast Design

I would use color aggressively, but not as the only signal. Every status should have a label and icon too.

## 9.1 Semantic Color System

Use a dark high-contrast background with semantic colors:

| Meaning                     | Color direction    | Label examples                |
| --------------------------- | ------------------ | ----------------------------- |
| Success                     | bright green       | `OK`, `PASS`, `GREEN`         |
| Running                     | bright blue        | `RUN`, `LIVE`                 |
| Waiting                     | yellow             | `WAIT`, `PEND`                |
| Warning / capacity pressure | orange/amber       | `WARN`, `PRESSURE`            |
| Blocked / escalation        | magenta            | `BLOCK`, `ESCALATE`           |
| Failed / destructive        | red                | `FAIL`, `DENY`, `DANGER`      |
| Selected / interactive      | cyan               | `SELECTED`, borders           |
| Agent activity              | violet/purple      | `AGENT`, `AUTO`               |
| Stale / unknown             | gray               | `STALE`, `UNKNOWN`            |
| Production                  | gold/white-on-dark | `PROD`                        |
| Secret/security boundary    | purple/red accent  | `SECRET`, `TAINT`, `TRIPWIRE` |

## 9.2 Status Tokens

Every row should have a compact status token:

```text
[OK]
[RUN]
[WAIT]
[FAIL]
[BLOCK]
[STALE]
[RISK]
[PROD]
[SECRET]
[AGENT]
```

This helps colorblind users and makes the UI readable in screenshots, logs, and low-color terminals.

## 9.3 Border Language

Use borders consistently:

* cyan border: focused pane
* green border: healthy complete section
* blue border: live/running section
* yellow border: waiting or degraded
* magenta border: blocked/escalated
* red border: failure or destructive confirmation
* gray border: stale/inactive

## 9.4 Destructive Action Modals

For dangerous actions, use a full-width modal with unmistakable styling:

```text
┌────────────────────────────── DANGER ──────────────────────────────┐
│ Cancel pipeline #531                                                │
│                                                                     │
│ This will cancel the active release-execution pipeline.              │
│ Release ci-fa51a52a will become blocked or failed.                   │
│                                                                     │
│ Required confirmation: type CANCEL                                   │
└─────────────────────────────────────────────────────────────────────┘
```

Production promotion and secret rotation should have similarly serious modals, but with gate evidence shown.

# 12. One Potential Bug-Looking Detail

In the capability `RunTests` scope table, integration generates:

```text
cargo test --test '*'
```

That looks suspicious. In Cargo, `--test <name>` generally selects a specific integration test target. A quoted `'*'` may not behave as intended. The more likely command for all integration tests is:

```text
cargo test --tests
```

or simply:

```text
cargo test
```

depending on desired scope.

That is worth verifying in code/tests.

That is the right shape.

# 15. My Strongest Design Recommendation

The single most powerful redesign would be:

> Build a unified object/action/evidence model, then make the TUI and capability API both consume it.

That means creating something like:

```rust
struct ActionDescriptor {
    object_type: String,
    object_id: String,
    action: String,
    enabled: bool,
    reason: String,
    risk: RiskLevel,
    confirmation: ConfirmationPolicy,
    dry_run: Option<ActionPreview>,
}
```

Then every interface can use it:

* TUI action panel
* command palette
* agent capability API
* CLI `--dry-run`
* audit logging
* risk gate reporting
* future web dashboard

This would make jeryu feel less like a collection of commands and more like a coherent control plane.

# 16. Final Summary

The system is already architecturally impressive. The foundations are strong: typed state, runner pools, release gates, VTI, evidence capsules, SmartCache, Vault, risk gates, and a real TUI.

The biggest gaps are:

1. deferred capabilities pretending to succeed
2. no unified action model
3. limited agent introspection
4. weak object/action/audit correlation
5. TUI pane mismatch and incomplete command surface
6. release gate evidence needing stronger provenance
7. heuristic flow/ETA logic needing historical data
8. hidden critical state around secrets, trust, taints, and audit

The dream version is a **developer mission-control console**: a color-rich, object-centric, real-time TUI where every job, pipeline, release, test, pool, cache object, secret set, and agent task has visible state, evidence, blockers, ETA, and safe actions.

The most important information to surface is:

```text
current release
current blocker
critical path
ETA + confidence
failure evidence
next safe actions
disabled actions + reasons
pool capacity
test impact
security/trust state
audit timeline
```

If I were redesigning it, I would make the TUI less like a dashboard and more like an **interactive flight deck for CI, release, agents, runners, cache, secrets, and production gates**.
